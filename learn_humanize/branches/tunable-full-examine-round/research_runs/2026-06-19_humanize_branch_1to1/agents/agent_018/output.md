### TUNABLE_FULL_EXAMINE_ROUND-HZ-018 `file` `commands/cancel-pr-loop.md`
- cursor: `[_]`
- core_role: Defines the assistant-facing workflow for cancelling an active PR review loop. The file is a command wrapper, not the cancellation algorithm itself: it restricts allowed tool use to `scripts/cancel-pr-loop.sh` and `scripts/cancel-pr-loop.sh --force` in frontmatter, then instructs the assistant to run the script and interpret sentinel output. See `commands/cancel-pr-loop.md:1-5` and `commands/cancel-pr-loop.md:9-21`.

- algorithmic_behavior: The command performs a two-stage workflow. First, it invokes `"${CLAUDE_PLUGIN_ROOT}/scripts/cancel-pr-loop.sh"` as the sole cancellation authority (`commands/cancel-pr-loop.md:11-15`). Second, it branches on the first output line: `NO_LOOP` or `NO_ACTIVE_LOOP` maps to the user-facing message "No active PR loop found"; `CANCELLED` maps to reporting the script's cancellation message (`commands/cancel-pr-loop.md:17-20`). The command explicitly delegates active-loop detection and state mutation to the script, stating that a PR loop is active when `state.md` exists in the newest `.humanize/pr-loop/` directory (`commands/cancel-pr-loop.md:21`).

- inputs_outputs_state: Inputs are the command invocation, `CLAUDE_PLUGIN_ROOT`, and the current project context used by the script. The delegated script resolves `PROJECT_ROOT` from `CLAUDE_PROJECT_DIR` or `pwd`, then targets `$PROJECT_ROOT/.humanize/pr-loop` (`scripts/cancel-pr-loop.sh:72-73`). It selects the newest loop directory via sorted directory listing (`scripts/cancel-pr-loop.sh:75-77`). State transition on success is: create `.cancel-requested`, move `state.md` to `cancel-state.md`, print `CANCELLED`, report PR number and round, and preserve loop artifacts (`scripts/cancel-pr-loop.sh:88-97`, `scripts/cancel-pr-loop.sh:118-130`). Outputs are sentinel lines plus human-readable detail; exit code `0` means cancelled, `1` means no active loop, and `3` covers usage/other errors (`scripts/cancel-pr-loop.sh:11-15`, `scripts/cancel-pr-loop.sh:60-64`).

- gates_or_invariants: The command's key invariant is PR-only scope: it must not affect RLCR loops, and the file tells users to use `/humanize:cancel-rlcr-loop` for `.humanize/rlcr/` (`commands/cancel-pr-loop.md:23-25`). The delegated script enforces this by only reading `.humanize/pr-loop` (`scripts/cancel-pr-loop.sh:72-76`). Another invariant is that the loop directory is preserved for audit/reference instead of deleted (`commands/cancel-pr-loop.md:23`; `scripts/cancel-pr-loop.sh:121-130`). Active detection is intentionally narrow: the newest PR-loop directory must contain `state.md`; if the base directory is absent, output is `NO_LOOP`, and if the newest directory lacks `state.md`, output is `NO_ACTIVE_LOOP` (`scripts/cancel-pr-loop.sh:78-97`). The hook-side cancellation guard allows the protected `mv state.md cancel-state.md` only when `.cancel-requested` exists and the command shape passes strict validation (`hooks/loop-bash-validator.sh:133-153`; `hooks/lib/loop-common.sh:533-710`).

- dependencies_and_callers: Direct dependency is `scripts/cancel-pr-loop.sh`, referenced by both command instructions and allowed-tools (`commands/cancel-pr-loop.md:3`, `commands/cancel-pr-loop.md:14`). It is paired with `/humanize:start-pr-loop`, whose stopping conditions include user cancellation by `/humanize:cancel-pr-loop` (`commands/start-pr-loop.md:58-62`). PR-loop setup blocks starting a second PR loop and points users back to the cancel command when one is already active (`scripts/setup-pr-loop.sh:235-239`). Runtime stop-hook flows surface the cancel command as the manual escape path during bot timeout and waiting-for-bots states (`hooks/pr-loop-stop-hook.sh:1155-1161`, `hooks/pr-loop-stop-hook.sh:1376-1384`). The stop hook also observes `.cancel-requested` and exits its polling loop when cancellation is signaled (`hooks/pr-loop-stop-hook.sh:941-945`). Shared active-loop discovery mirrors the command's invariant: `find_active_pr_loop` returns only the newest PR-loop directory containing `state.md` (`hooks/lib/loop-common.sh:795-814`).

- edge_cases_or_failure_modes: If `.humanize/pr-loop` has no session directories, the delegated script returns `NO_LOOP` and exit code `1` (`scripts/cancel-pr-loop.sh:75-82`). If the newest session exists but has no `state.md`, it returns `NO_ACTIVE_LOOP`, also exit code `1`, so completed, cancelled, approved, or max-iteration sessions are not treated as cancellable active loops (`scripts/cancel-pr-loop.sh:88-98`). Metadata extraction is best effort: missing `current_round`, `max_iterations`, or `pr_number` becomes `?` rather than blocking cancellation (`scripts/cancel-pr-loop.sh:104-112`). `--force` is accepted but currently has no additional effect (`scripts/cancel-pr-loop.sh:24-31`, `scripts/cancel-pr-loop.sh:39-41`). Unknown options terminate with exit code `3` (`scripts/cancel-pr-loop.sh:60-64`). Because newest-directory selection is simple reverse sort, an older active directory can be ignored if a newer inactive PR-loop directory exists; this matches the "newest PR loop directory" rule in the command file (`commands/cancel-pr-loop.md:21`) and active finder behavior (`hooks/lib/loop-common.sh:806-813`).

- validation_or_tests: `tests/test-pr-loop-scripts.sh` covers the delegated script's help path, no-loop sentinel behavior, and active-loop cancellation. The active-loop test creates `.humanize/pr-loop/<timestamp>/state.md`, invokes the script, expects `CANCELLED`, verifies `cancel-state.md` exists, and verifies `state.md` is gone (`tests/test-pr-loop-scripts.sh:181-247`). `tests/robustness/test-concurrent-state-robustness.sh` validates `find_active_pr_loop` detection for a session containing `state.md` (`tests/robustness/test-concurrent-state-robustness.sh:398-413`).
