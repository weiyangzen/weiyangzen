- core_role: `hooks/loop-write-validator.sh` is the Claude `PreToolUse` gate for `Write` operations, registered by `hooks/hooks.json:14-21`. Its core role is to preserve RLCR and PR-loop state-machine ownership boundaries by allowing ordinary writes, but blocking writes to generated loop control surfaces such as prompt files, state files, wrong-round summaries, post-round-0 goal tracker files, `plan.md` loop backups, PR-loop generated files, and most todos files. It is core algorithm support because it prevents the agent from bypassing the loop protocol through direct file writes.
- algorithmic_behavior: The hook reads the full hook JSON from stdin (`hooks/loop-write-validator.sh:25`), validates JSON/tool shape through shared helpers (`hooks/loop-write-validator.sh:27-35`), exits early for non-`Write` tools (`hooks/loop-write-validator.sh:37-41`), then requires `tool_input.file_path` (`hooks/loop-write-validator.sh:43-49`). It first handles special hard blocks for `round-*-todos.md` unless allowlisted in the active RLCR loop (`hooks/loop-write-validator.sh:55-63`) and all `round-*-prompt.md` writes (`hooks/loop-write-validator.sh:65-68`). It then branches into PR-loop protection for `.humanize/pr-loop/`: block `state.md`, block generated PR prompt/comment/check/feedback files, and allow only current-round `round-N-pr-resolve.md` after `validate_pr_resolve_round` (`hooks/loop-write-validator.sh:74-93`). For RLCR paths, it classifies summary/finalize-summary/in-loop paths (`hooks/loop-write-validator.sh:100-117`), finds the active loop (`hooks/loop-write-validator.sh:123-130`), chooses `finalize-state.md` over `state.md` when finalize phase exists (`hooks/loop-write-validator.sh:132-138`), strict-parses state (`hooks/loop-write-validator.sh:140-145`), blocks state mutation (`hooks/loop-write-validator.sh:152-160`), allows exactly the active-loop `finalize-summary.md` (`hooks/loop-write-validator.sh:168-172`), blocks RLCR `plan.md` backups (`hooks/loop-write-validator.sh:179-185`), blocks `goal-tracker.md` after round 0 (`hooks/loop-write-validator.sh:192-195`), blocks summary files outside `.humanize/rlcr/` (`hooks/loop-write-validator.sh:202-209`), validates summary round number against `current_round` except explicit allowlist (`hooks/loop-write-validator.sh:228-245`), and finally ensures the target path is in the active loop directory (`hooks/loop-write-validator.sh:252-263`).
- inputs_outputs_state: Input is Claude hook JSON on stdin with at least `tool_name` and, for `Write`, `tool_input.file_path`; `tool_input.content` is ignored except as part of JSON parsing. Environment input is `CLAUDE_PROJECT_DIR`, falling back to `pwd`, used to locate `.humanize/rlcr` and `.humanize/pr-loop` (`hooks/loop-write-validator.sh:56-58`, `hooks/loop-write-validator.sh:123-126`). State input is the newest active RLCR loop directory with `state.md` or `finalize-state.md`, found by `find_active_loop` (`hooks/lib/loop-common.sh:156-174`), and parsed by `parse_state_file_strict` for `current_round`, `max_iterations`, `review_started`, and `base_branch` (`hooks/lib/loop-common.sh:255-329`). Output is exit code `0` to allow the write, exit code `1` for malformed hook input/deep nesting/malformed state safety failures, or exit code `2` for intentional policy blocks with a rendered or fallback message on stderr. The hook does not mutate loop state; it only reads state and enforces whether a proposed write is allowed.
