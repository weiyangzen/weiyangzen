### TUNABLE_FULL_EXAMINE_ROUND-HZ-025 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: PreToolUse Bash validator for active Humanize loops. It is registered for Bash commands in `hooks/hooks.json:42-48`, reads Claude hook JSON from stdin, and blocks Bash-based attempts to mutate RLCR/PR-loop control files that should instead be managed by loop machinery or safer Write/Edit validators. Within the RLCR state machine, its central role is preserving loop state integrity: `state.md`, `finalize-state.md`, `plan.md`, `goal-tracker.md`, round prompt/summary/todos files, and `.humanize` git hygiene are guarded while an active loop exists.
- algorithmic_behavior: The script starts with `set -euo pipefail`, sources `hooks/lib/loop-common.sh`, reads all hook input from stdin, validates JSON, rejects excessive nesting, and exits early unless `.tool_name` is `Bash` (`hooks/loop-bash-validator.sh:14-48`). It locates active RLCR and PR loops under `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize/rlcr` and `.humanize/pr-loop` (`hooks/loop-bash-validator.sh:54-66`). With an active RLCR loop, it chooses `finalize-state.md` over `state.md` during finalize phase and parses the selected state strictly; malformed state fails closed (`hooks/loop-bash-validator.sh:74-86`). It then blocks `git push` unless `push_every_round: true` in state (`hooks/loop-bash-validator.sh:88-105`), blocks `git add` patterns that would add `.humanize` (`hooks/loop-bash-validator.sh:109-118`), blocks state/finalize-state destination writes via `command_modifies_file`, and also separately scans segmented shell commands for `mv`/`cp` where `state.md` or `finalize-state.md` is the source (`hooks/loop-bash-validator.sh:139-328`). It blocks RLCR plan backup writes (`hooks/loop-bash-validator.sh:330-341`), goal-tracker writes with round-sensitive messaging (`hooks/loop-bash-validator.sh:343-358`), prompt writes (`hooks/loop-bash-validator.sh:360-368`), summary writes through Bash (`hooks/loop-bash-validator.sh:370-379`), and unauthorized round todos writes (`hooks/loop-bash-validator.sh:381-393`). With an active PR loop, it blocks PR-loop `state.md` and read-only round artifacts (`hooks/loop-bash-validator.sh:403-443`). Successful non-blocked commands exit 0.
- inputs_outputs_state: Input is a JSON hook payload on stdin with `tool_name` and `tool_input.command`; missing `command` for Bash is an error (`hooks/loop-bash-validator.sh:24-48`). It also reads ambient `CLAUDE_PROJECT_DIR`, the filesystem under `.humanize/rlcr` and `.humanize/pr-loop`, active state files, optional `.cancel-requested`, templates under `prompt-template/block`, and git ignore state for `.humanize` through shared helpers. Outputs are process exit codes and stderr block messages. Exit 0 allows the Bash command, exit 1 indicates invalid hook input or malformed loop state, and exit 2 is a policy block with a rendered/fallback reason. It does not mutate repository files itself; the only “state transition” it permits is the authorized cancel move from active `state.md` or `finalize-state.md` to `cancel-state.md` when `.cancel-requested` exists and the command is structurally exact. That authorization logic lives in `is_cancel_authorized` and requires exactly two paths, same active loop dir, no shell chaining/substitution, and no symlink source (`hooks/lib/loop-common.sh:551-710`).
- gates_or_invariants: The validator enforces “loop control files are system-owned.” `state.md`/`finalize-state.md` cannot be written, deleted, truncated, copied, or moved except for the cancel path (`hooks/loop-bash-validator.sh:127-328`). State parsing is strict enough to require YAML frontmatter plus fields such as `current_round`, `max_iterations`, `review_started`, and `base_branch`; nonnumeric rounds/iteration counts and invalid booleans fail (`hooks/lib/loop-common.sh:246-330`). `.humanize` must not be force-added or broad-added to git; helper coverage includes direct `.humanize`, `-f`, `--force`, `-A/--all`, and broad `.`/`*` targets depending on ignore state (`hooks/lib/loop-common.sh:908-1066`). RLCR round todos are only allowed for active-loop `round-1-todos.md` or `round-2-todos.md` with the full active loop path; same basename in `/tmp`, old loop dirs, or relative names are blocked (`hooks/loop-bash-validator.sh:381-393`). PR-loop state and generated prompt/comment/check/feedback files are read-only while active (`hooks/loop-bash-validator.sh:403-443`). The script is intentionally targeted: unrelated Bash commands are allowed through and left to the outer sandbox, as reflected in tests (`tests/robustness/test-hook-system-robustness.sh:369-382`).
- dependencies_and_callers: Direct caller is Claude’s hook framework via `hooks/hooks.json:42-48`. Shared dependencies are `jq`, `sed`, `grep`, `tr`, `wc`, `sort`, `head`, optional `iconv`, and Git for `.humanize` add checks. It depends heavily on `hooks/lib/loop-common.sh`: JSON validation (`hooks/lib/loop-common.sh:59-133`), active loop lookup (`hooks/lib/loop-common.sh:156-174`, `hooks/lib/loop-common.sh:798-814`), strict state parsing (`hooks/lib/loop-common.sh:246-330`), cancel authorization (`hooks/lib/loop-common.sh:551-710`), PR-loop block messages (`hooks/lib/loop-common.sh:889-906`), `.humanize` git-add detection (`hooks/lib/loop-common.sh:930-1026`), and generic modification detection (`hooks/lib/loop-common.sh:1071-1096`). Block text is loaded through `hooks/lib/template-loader.sh`, whose resolver maps the hook library location to `prompt-template` (`hooks/lib/template-loader.sh:24-31`) and falls back safely if a template is missing (`hooks/lib/template-loader.sh:167-203`). Related sibling hooks are Write/Edit/Read validators and Stop hooks in `hooks/`, but this assigned file specifically owns Bash command bypass prevention.
- edge_cases_or_failure_modes: It deliberately handles shell-command evasions around `state.md`/`finalize-state.md`: chained segments split on `;`, `&&`, `||`, pipes, background `&`, and `|&`; redirection operators are temporarily protected or stripped so leading redirections like `2>/tmp/x mv state.md /tmp/foo` remain detectable (`hooks/loop-bash-validator.sh:169-284`). It has special checks for `sh -c`/`bash -c` payloads containing `mv`/`cp` of protected state files (`hooks/loop-bash-validator.sh:306-328`). The cancel exception is narrow: missing/wrong `.cancel-requested`, wrong destination, extra args, command substitution, backticks, newlines, mixed quote styles, remaining `$`, shell operators, and symlink source all reject (`hooks/lib/loop-common.sh:557-706`). A notable limitation is regex/string parsing rather than a real shell AST; it is broad for common bypasses but can over-block benign files named `state.md` during an active PR loop because the PR-loop bare filename check intentionally catches relative-path use (`hooks/loop-bash-validator.sh:411-415`). Another edge is that commands are lowercased before matching (`hooks/loop-bash-validator.sh:47-48`), which is necessary for case-insensitive policy but can blur case-sensitive paths; helpers document this especially for Git flags (`hooks/lib/loop-common.sh:912-914`). If no active RLCR or PR loop is found, all Bash commands pass (`hooks/loop-bash-validator.sh:64-67`), so active-loop discovery is a critical precondition.
