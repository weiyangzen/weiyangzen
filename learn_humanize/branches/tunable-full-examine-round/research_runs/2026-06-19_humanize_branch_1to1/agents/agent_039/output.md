- core_role: Provides the repository's portable timeout abstraction for shell workflows that must run on macOS and Linux. It normalizes command execution behind `run_with_timeout <seconds> <command> [args...]`, choosing Homebrew GNU `gtimeout`, GNU `timeout`, Python subprocess timeout, or a no-timeout fallback. This file is core support for workflow safety because RLCR/PR setup scripts and hooks wrap potentially hanging `git`, `gh`, and `codex` calls with it.
- algorithmic_behavior: The script first detects a timeout backend in priority order: `gtimeout`, then GNU `timeout`, then `python3`, then `python`, otherwise `none` (`scripts/portable-timeout.sh:10-27`). It stores the detected backend in the global `TIMEOUT_IMPL` once at source time (`scripts/portable-timeout.sh:29`). `run_with_timeout` parses the first argument as `timeout_secs`, shifts it away, stores the remaining argv in a Bash array, and dispatches by `TIMEOUT_IMPL` (`scripts/portable-timeout.sh:33-38`). Native backends call `gtimeout "$timeout_secs" "${cmd[@]}"` or `timeout "$timeout_secs" "${cmd[@]}"` and return the exact backend exit code (`scripts/portable-timeout.sh:39-45`). Python backends run an inline script using `subprocess.run(sys.argv[1:], timeout=$timeout_secs)`, return the child return code on normal completion, return `124` on `subprocess.TimeoutExpired`, and return `1` for other Python-side exceptions after printing an error to stderr (`scripts/portable-timeout.sh:47-62`). If no backend exists, the wrapper emits a warning and runs the command without a timeout (`scripts/portable-timeout.sh:64-68`).
- inputs_outputs_state: Inputs are the timeout duration string and a command argv list. The command is passed as an array (`local cmd=("$@")` at `scripts/portable-timeout.sh:36`), so normal shell argument boundaries are preserved for direct commands and callers can still use `sh -c` for pipelines. Outputs are the wrapped command stdout/stderr plus the wrapper's process exit code. Timeout output behavior depends on backend: GNU tools may print their own diagnostics, while the Python path is quiet on timeout and returns `124`. State is limited to the exported `TIMEOUT_IMPL` environment variable (`scripts/portable-timeout.sh:73-76`); there are no files, locks, counters, or persistent state transitions. The key state transition is one-time backend selection at source time, then per-call transition from pending command to completed, timed out, backend exception, or unbounded execution in `none` mode.
- gates_or_invariants: The backend gate rejects non-GNU `timeout` by requiring `timeout --version` to succeed before selecting `timeout` (`scripts/portable-timeout.sh:13-19`), which avoids accidentally relying on incompatible platform behavior. The intended timeout invariant is that timed-out commands return `124`, matching GNU timeout semantics; this is explicit in the Python fallback (`scripts/portable-timeout.sh:56-57`) and assumed by callers such as `hooks/loop-plan-file-validator.sh`, which treats exit `124` as timeout and fails closed for plan tracking checks (`hooks/loop-plan-file-validator.sh:126-137`, `hooks/loop-plan-file-validator.sh:151-163`).
