### TUNABLE_FULL_EXAMINE_ROUND-HZ-009 `directory` `hooks/lib`
- cursor: `[_]`
- core_role: Shared shell algorithm library for the Humanize RLCR and PR-loop hook system. `hooks/lib` contains exactly two recursively inspected files: `hooks/lib/template-loader.sh` and `hooks/lib/loop-common.sh`. The directory centralizes prompt-template resolution/rendering, hook JSON validation, RLCR state parsing, active-loop discovery, path/file policy predicates, cancel authorization, code-review issue extraction, PR-loop bot/author mapping, `.humanize` git-add protection, bash write-detection patterns, loop termination, and PR goal-tracker updates.
- algorithmic_behavior: `template-loader.sh` resolves the plugin template root from a caller-provided script dir (`get_template_dir`, lines 26-31), loads template text or emits an empty string with warning (`load_template`, lines 36-48), renders `{{VAR}}` placeholders in a single awk pass using `TMPL_VAR_` environment variables (`render_template`, lines 58-132), composes load/render calls (`load_and_render`, lines 136-147), appends template content (`append_template`, lines 151-161), falls back to inline messages when templates are missing or render empty (`load_and_render_safe`, lines 170-203), and validates required template subdirectories `block`, `codex`, and `claude` (`validate_template_dir`, lines 208-222). `loop-common.sh` sources the template loader and initializes `TEMPLATE_DIR` (`loop-common.sh`, lines 139-148), then provides policy functions consumed by hooks. It validates hook JSON by rejecting null bytes, invalid UTF-8 where `iconv` exists, malformed JSON, and missing `tool_name` (`loop-common.sh`, lines 59-96); rejects deeply nested JSON with `jq '[paths | length] | max'` (`lines 119-133`); discovers only the newest active RLCR/PR loop directory to avoid reviving stale sessions (`find_active_loop`, lines 156-174; `find_active_pr_loop`, lines 798-814); parses state frontmatter in tolerant and strict modes (`parse_state_file`, lines 209-244; `parse_state_file_strict`, lines 255-330); scans Codex review logs for `[P0-9]` markers in the first ten characters of each line and extracts from the first issue to EOF (`detect_review_issues`, lines 349-392); classifies loop-controlled filenames and directories (`is_round_file_type`, `extract_round_number`, `is_*_path`, lines 399-530); authorizes the narrow cancel transition from `state.md` or `finalize-state.md` to `cancel-state.md` only with `.cancel-requested` and strict command/path checks (`is_cancel_authorized`, lines 551-710); maps PR bot names (`map_bot_to_author`, `map_author_to_bot`, lines 727-749); blocks dangerous `git add` operations that would stage `.humanize` (`git_adds_humanize`, lines 930-1026); detects bash commands that modify protected files (`command_modifies_file`, lines 1071-1096); terminates loops by renaming active state to `<reason>-state.md` (`end_loop`, lines 1119-1145); and updates PR goal trackers idempotently with totals, table rows, and issue-log entries (`update_pr_goal_tracker`, lines 1168-1315).
- inputs_outputs_state: Inputs are hook JSON from stdin, file paths, bash command strings, loop base dirs, state/frontmatter files, Codex review logs under `$CACHE_DIR`, prompt templates under `prompt-template`, PR bot result JSON, and current repository state such as `.humanize` existence and git-ignore status. Outputs are shell return codes, stderr block messages, rendered prompts, extracted review-result markdown files, global parser variables such as `VALIDATED_TOOL_NAME`, `VALIDATED_TOOL_INPUT`, and `STATE_*`, renamed state files, and updated PR goal-tracker markdown. State transitions are mostly file-based: newest `.humanize/rlcr/<timestamp>/state.md` or `finalize-state.md` marks the active RLCR session (`loop-common.sh`, lines 156-170); cancel requires `mv <loop>/state.md|finalize-state.md <loop>/cancel-state.md` plus `.cancel-requested` (`lines 551-710`); normal loop endings rename state to `complete-state.md`, `cancel-state.md`, `maxiter-state.md`, `stop-state.md`, or `unexpected-state.md` (`lines 1119-1138`); Codex stop handling uses `detect_review_issues` to either continue a review round or enter finalize phase (`hooks/loop-codex-stop-hook.sh`, lines 999-1027 and 1032-1114); PR goal tracking increments totals only when adding a new summary row, skips fully duplicate round/reviewer entries, and repairs partial summary/log updates (`loop-common.sh`, lines 1184-1265).
- gates_or_invariants: Template rendering is intentionally single-pass so substituted values containing `{{OTHER_VAR}}` are not rescanned (`template-loader.sh`, lines 50-57 and 71-129). Template lookup degrades to inline fallback messages instead of failing hard for block prompts (`template-loader.sh`, lines 170-203; `loop-common.sh`, lines 443-500, 889-905, 1028-1066, 1098-1110). Strict state parsing requires at least two `---` separators, `current_round`, `max_iterations`, `review_started`, and `base_branch`, plus numeric rounds/limits and boolean `review_started` (`loop-common.sh`, lines 263-323). Hook input validation requires syntactically valid JSON with `tool_name`, optional required tool-input fields per caller, UTF-8, and max depth 30 (`loop-common.sh`, lines 59-133; callers in `hooks/loop-read-validator.sh`, lines 25-45; `hooks/loop-write-validator.sh`, lines 25-45; `hooks/loop-bash-validator.sh`, lines 24-44). Protected file gates block direct or bash-mediated modification of todos, prompt files, state files, finalize-state files, plan backups, stale/wrong-round summaries, post-round-0 goal trackers, PR-loop state, and PR-loop generated review/prompt files; these gates are applied in sibling validators sourcing this library (`hooks/loop-read-validator.sh`, lines 55-168; `hooks/loop-write-validator.sh`, lines 55-263; `hooks/loop-edit-validator.sh`, lines 38-181; `hooks/loop-bash-validator.sh`, lines 96-379).
