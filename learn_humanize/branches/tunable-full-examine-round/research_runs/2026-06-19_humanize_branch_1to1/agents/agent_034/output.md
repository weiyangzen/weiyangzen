### TUNABLE_FULL_EXAMINE_ROUND-HZ-034 `file` `scripts/check-bot-reactions.sh`
- cursor: `[_]`
- core_role: Runtime PR-loop gate that converts GitHub bot reaction state into machine-readable success/failure. It supports two workflow signals: Codex approval via a `+1` reaction on the PR issue body, and Claude receipt confirmation via an `eyes` reaction on a trigger comment. The script is not a general monitor; it is a narrow reaction detector used by setup and stop-hook orchestration to decide whether to proceed, block, or remove a bot from active tracking.

- algorithmic_behavior: The command dispatcher reads the first argument as `codex-thumbsup` or `claude-eyes`, falls back to help for empty/help arguments, and exits with documented code categories: found, not found, or error (`scripts/check-bot-reactions.sh:39-86`). Both modes resolve the GitHub repository through `gh repo view`, then try to identify the PR base repository for fork support by calling `gh pr view ... --json baseRepository`; if that fails, they try the parent repository and finally fall back to the current repo (`scripts/check-bot-reactions.sh:119-147`, `scripts/check-bot-reactions.sh:229-257`).  
  For `codex-thumbsup`, the script fetches PR body reactions from `repos/$PR_BASE_REPO/issues/$PR_NUMBER/reactions`, paginates, merges page arrays with `jq -s 'add // []'`, then selects the first reaction where user is `chatgpt-codex-connector[bot]` and content is `+1` (`scripts/check-bot-reactions.sh:149-165`). If `--after` is provided, it compares the selected reaction timestamp to the supplied timestamp and rejects older reactions (`scripts/check-bot-reactions.sh:172-183`).  
  For `claude-eyes`, the script parses a comment id plus optional `--retry`, `--delay`, and undocumented-but-used `--pr`, then loops `seq 1 "$MAX_RETRIES"`; each attempt sleeps first, fetches `repos/$PR_BASE_REPO/issues/comments/$COMMENT_ID/reactions`, selects user `claude[bot]` with content `eyes`, and exits as soon as one is found (`scripts/check-bot-reactions.sh:187-295`).

- inputs_outputs_state: Inputs are CLI arguments, current GitHub repository context, optional PR number for fork-aware comment lookup, GitHub API credentials available to `gh`, and local tools `bash`, `gh`, `jq`, `seq`, and `sleep`. The script sources `scripts/portable-timeout.sh` and wraps `gh` calls with a 30 second timeout (`scripts/check-bot-reactions.sh:24-33`; `scripts/portable-timeout.sh:31-71`). Outputs are a compact JSON reaction object on stdout when found, progress/error text on stderr, and exit status as the control channel. It writes no persistent state itself. State transitions happen in callers: setup deletes the loop directory and fails startup if Claude eyes is absent (`scripts/setup-pr-loop.sh:484-520`); the stop hook removes Codex from `active_bots` or completes the loop when a valid thumbs-up is detected (`hooks/pr-loop-stop-hook.sh:442-475`, `hooks/pr-loop-stop-hook.sh:975-1005`); the stop hook also blocks on missing Claude eyes with a rendered timeout message (`hooks/pr-loop-stop-hook.sh:757-775`).

- gates_or_invariants: The script runs under `set -euo pipefail` (`scripts/check-bot-reactions.sh:18`). Missing required PR/comment arguments, unknown commands, unknown options, or duplicate positional identifiers are intended to produce error exit status (`scripts/check-bot-reactions.sh:92-117`, `scripts/check-bot-reactions.sh:194-227`, `scripts/check-bot-reactions.sh:298-302`). It uses the GitHub Issues reactions endpoints because PR bodies and PR comments are issue resources (`scripts/check-bot-reactions.sh:149-154`, `scripts/check-bot-reactions.sh:264-270`). Fork safety is an invariant in both modes: reactions are checked against the PR base repository rather than blindly against a fork (`scripts/check-bot-reactions.sh:119-121`, `scripts/check-bot-reactions.sh:229-231`). Matching is exact on bot login and reaction content, so any bot rename or reaction content change becomes a false negative. The `--after` gate assumes normalized ISO-8601 timestamps whose lexicographic order matches time order (`scripts/check-bot-reactions.sh:174-180`).

- dependencies_and_callers: Direct dependency is `scripts/portable-timeout.sh`, which selects `gtimeout`, GNU `timeout`, Python subprocess timeout, or no timeout in that priority order (`scripts/portable-timeout.sh:9-29`, `scripts/portable-timeout.sh:38-70`). Runtime dependencies are GitHub CLI and `jq`; failures in those paths affect the gate result. Direct callers are `scripts/setup-pr-loop.sh` for mandatory Claude eyes verification during loop setup (`scripts/setup-pr-loop.sh:505-520`) and `hooks/pr-loop-stop-hook.sh` for first-round Codex thumbs-up approval, recurring Codex thumbs-up polling, and Claude trigger receipt verification (`hooks/pr-loop-stop-hook.sh:459`, `hooks/pr-loop-stop-hook.sh:766`, `hooks/pr-loop-stop-hook.sh:985`). Test support routes reaction API calls through `tests/mocks/gh`, with specific endpoint cases for comment reactions and PR/issue reactions (`tests/mocks/gh:131-148`).

- edge_cases_or_failure_modes: Option flags that require values read `$2` directly; with `set -u`, a missing value can terminate abruptly rather than cleanly returning the documented argument error path (`scripts/check-bot-reactions.sh:94-97`, `scripts/check-bot-reactions.sh:196-206`). `claude-eyes` sleeps before the first API check, so `--retry 3 --delay 5` waits before every attempt and can take at least 15 seconds even when the reaction already exists (`scripts/check-bot-reactions.sh:259-262`). In `codex-thumbsup`, API failure is treated as an error; in `claude-eyes`, API fetch failure inside an attempt is swallowed with `continue`, so repeated API failures can collapse into the same final not-found result as a real missing reaction (`scripts/check-bot-reactions.sh:154-159`, `scripts/check-bot-reactions.sh:268-273`, `scripts/check-bot-reactions.sh:293-295`). If repository resolution fails completely, the fallback can leave an empty repo slug and produce an invalid GitHub API path. Retry values are not validated as numeric before being passed to `seq` or `sleep` (`scripts/check-bot-reactions.sh:191-202`, `scripts/check-bot-reactions.sh:260-262`). The first matching Codex reaction drives the `--after` decision, so behavior depends on API ordering if multiple matching reaction records are ever returned (`scripts/check-bot-reactions.sh:161-180`).

- validation_or_tests: Existing focused tests cover Codex thumbs-up success, Codex `--after` rejection, and Claude eyes success with `--retry 1 --delay 0` (`tests/test-pr-loop-system.sh:199-229`). The mock `gh` fixture layer returns `tests/fixtures/reactions.json` for PR/issue reactions and `tests/fixtures/comment-reactions.json` for comment reactions; the default fixtures contain the expected Codex `+1` and Claude `eyes` records (`tests/mocks/gh:131-148`, `tests/fixtures/reactions.json:1`, `tests/fixtures/comment-reactions.json:1-10`).
