### TUNABLE_FULL_EXAMINE_ROUND-HZ-105 `file` `prompt-template/block/wrong-round-number.md`
- cursor: `[_]`
- core_role: This is a blocking-message template used by the RLCR loop write/edit validators when an agent targets a `round-N-summary.md` file whose embedded round number does not match the active loop's current round. The template is user-facing guard output, not the round-validation algorithm itself. Its concrete behavior is defined by `prompt-template/block/wrong-round-number.md:1-7`, with placeholders for action, attempted round, file type, current round, and correct path.
- algorithmic_behavior: The template communicates a failed transition attempt from the active round state to an incorrect summary file path. It says the attempted path is `round-{{CLAUDE_ROUND}}-{{FILE_TYPE}}.md`, states that the current round is `{{CURRENT_ROUND}}`, provides `{{CORRECT_PATH}}`, and explicitly instructs the agent not to increment the round manually (`prompt-template/block/wrong-round-number.md:3-7`). In the write validator, this block is rendered only when `CLAUDE_ROUND` is non-empty, differs from `CURRENT_ROUND`, and the file is not allowlisted (`hooks/loop-write-validator.sh:228-244`). The edit validator applies the same gate for summary edits (`hooks/loop-edit-validator.sh:155-179`).
- inputs_outputs_state: Inputs are render variables supplied by callers: `ACTION`, `CLAUDE_ROUND`, `FILE_TYPE`, `CURRENT_ROUND`, and `CORRECT_PATH` (`hooks/loop-write-validator.sh:238-243`, `hooks/loop-edit-validator.sh:172-177`). The output is Markdown text sent to stderr via `load_and_render_safe`, followed by validator exit code `2` to block the tool operation (`hooks/loop-write-validator.sh:238-244`, `hooks/loop-edit-validator.sh:172-178`). The template does not mutate state; state is read by callers from the active loop's `state.md` or `finalize-state.md`, then stored in `CURRENT_ROUND` before rendering (`hooks/loop-edit-validator.sh:99-110`). In the write path, active loop discovery and current-round parsing happen before summary round validation (`hooks/loop-write-validator.sh:120-130`, with the render gate at `hooks/loop-write-validator.sh:228-244`).
- gates_or_invariants: The invariant enforced around this template is that agents must write or edit only the summary file for the current active round. Round-file identification uses `round-[0-9]+-summary.md` pattern matching (`hooks/lib/loop-common.sh:399-406`) and round extraction from filenames (`hooks/lib/loop-common.sh:408-418`). The gate is bypassed when there is no active loop, no extractable round number, the attempted round equals the current round, or the path is allowlisted (`hooks/loop-write-validator.sh:128-130`, `hooks/loop-write-validator.sh:231`, `hooks/loop-edit-validator.sh:95-97`, `hooks/loop-edit-validator.sh:165`). The template itself encodes a second behavioral invariant: do not self-increment the round number (`prompt-template/block/wrong-round-number.md:7`).
- dependencies_and_callers: Direct dependencies are the repository's template renderer through `load_and_render_safe`, the active-loop/current-round helpers, filename pattern helpers, and allowlist logic. The direct callers are `hooks/loop-write-validator.sh` for `Write` tool attempts targeting wrong-round RLCR summaries (`hooks/loop-write-validator.sh:1-13`, `hooks/loop-write-validator.sh:228-244`) and `hooks/loop-edit-validator.sh` for `Edit` attempts (`hooks/loop-edit-validator.sh:1-12`, `hooks/loop-edit-validator.sh:155-179`). Related but separate behavior exists for PR-loop `round-N-pr-resolve.md`: that path validates wrong rounds in `validate_pr_resolve_round`, but emits its message inline instead of using this template (`hooks/lib/loop-common.sh:837-883`).
- edge_cases_or_failure_modes: If the template file is missing or cannot be rendered, both validators provide an inline fallback message with the same core variables (`hooks/loop-write-validator.sh:233-238`, `hooks/loop-edit-validator.sh:167-172`). If the filename does not match a supported round summary pattern, no wrong-round block is produced because `extract_round_number` returns empty (`hooks/lib/loop-common.sh:411-418`). If state parsing fails in the edit validator, the hook fails closed with a malformed-state error before reaching this template (`hooks/loop-edit-validator.sh:105-109`). The write validator validates hook JSON and blocks malformed or deeply nested input before path checks (`hooks/loop-write-validator.sh:25-35`). PR-loop wrong-round messages do not currently depend on this template, so changes here will not affect PR-loop resolve output (`hooks/lib/loop-common.sh:878-883`).

Broader template integration coverage exists in `tests/test-templates-comprehensive.sh:498-510`, including correct-path verification. Hook-level wrong-round behavior is indirectly covered for PR-loop resolve files in `tests/test-pr-loop-hooks.sh:1438-1459` and `tests/test-pr-loop-hooks.sh:1499-1527`, but those tests validate the shared wrong-round policy rather than this template file because PR-loop output is inline.
