### FIX_PR_LOOP-HZ-033 `file` `scripts/cancel-rlcr-loop.sh`
- cursor: `[_]`
- core_role: Runtime cancellation script for active RLCR loops. It is the authoritative action behind `/humanize:cancel-rlcr-loop`, including the special Finalize Phase guard.
- algorithmic_behavior: The script parses only `--force` and help flags (`scripts/cancel-rlcr-loop.sh:24-63`), locates the newest loop directory under `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize/rlcr` (`scripts/cancel-rlcr-loop.sh:69-78`), detects whether the active state is `state.md` or `finalize-state.md` (`scripts/cancel-rlcr-loop.sh:85-99`), extracts round metadata for reporting (`scripts/cancel-rlcr-loop.sh:105-111`), optionally blocks finalize cancellation without force (`scripts/cancel-rlcr-loop.sh:117-128`), then creates `.cancel-requested` and renames the active state file to `cancel-state.md` (`scripts/cancel-rlcr-loop.sh:134-138`).
- inputs_outputs_state: Inputs are CLI args, `CLAUDE_PROJECT_DIR`, filesystem state under `.humanize/rlcr/<timestamp>/`, and YAML-like fields `current_round` and `max_iterations` in the active state file. Outputs are status tokens on stdout: `NO_LOOP`, `NO_ACTIVE_LOOP`, `FINALIZE_NEEDS_CONFIRM`, `CANCELLED`, or `CANCELLED_FINALIZE` (`scripts/cancel-rlcr-loop.sh:75-78`, `scripts/cancel-rlcr-loop.sh:96-98`, `scripts/cancel-rlcr-loop.sh:117-127`, `scripts/cancel-rlcr-loop.sh:144-151`). State transition is `state.md -> cancel-state.md` for normal loops, or `finalize-state.md -> cancel-state.md` for forced finalize cancellation, plus creation of `.cancel-requested`.
- gates_or_invariants: Exit code contract is documented at `scripts/cancel-rlcr-loop.sh:11-15`. Finalize Phase is protected unless `--force` is passed (`scripts/cancel-rlcr-loop.sh:117-128`). The command spec adds a human confirmation gate before `--force` (`commands/cancel-rlcr-loop.md:23-34`). The newest loop directory is chosen lexicographically descending via `ls -1d ... | sort -r | head -1` (`scripts/cancel-rlcr-loop.sh:72-73`). Active-loop detection requires either `state.md` or `finalize-state.md`; `cancel-state.md` alone is not active.
- dependencies_and_callers: Called by `commands/cancel-rlcr-loop.md:11-15` and, with `--force`, by `commands/cancel-rlcr-loop.md:30-32`. It coordinates with hook-side active-loop detection that treats both `state.md` and `finalize-state.md` as active (`hooks/lib/loop-common.sh:154-169` from inspected reference output). It also coordinates with bash-validator cancel exceptions, where `.cancel-requested` allows state-file rename to `cancel-state.md` (`hooks/lib/loop-common.sh:547-555`, `hooks/lib/loop-common.sh:684-701` from reference output).
- edge_cases_or_failure_modes: If no loop directory exists, it returns exit 1 with `NO_LOOP` (`scripts/cancel-rlcr-loop.sh:75-78`). If the newest loop directory lacks both active state files, it returns exit 1 with `NO_ACTIVE_LOOP` (`scripts/cancel-rlcr-loop.sh:95-99`). Missing `current_round` or `max_iterations` becomes `?` in output (`scripts/cancel-rlcr-loop.sh:109-111`). Unknown options exit 3 (`scripts/cancel-rlcr-loop.sh:57-60`). If `touch` or `mv` fails, `set -euo pipefail` makes the script abort rather than emitting a success token.
- validation_or_tests: Finalize-state behavior is covered indirectly by `tests/test-finalize-phase.sh`, which verifies `finalize-state.md` is detected as active and validators block direct modification. Cancel-signal safety is covered by `tests/test-cancel-signal-file.sh`, including allowed `mv state.md cancel-state.md` only when `.cancel-requested` exists, and robustness coverage includes finalize-state cancellation and symlink/path attack cases in `tests/robustness/test-cancel-security-robustness.sh`. The command-level finalize confirmation contract is in `commands/cancel-rlcr-loop.md:17-34`. I inspected these references; I did not run them.
