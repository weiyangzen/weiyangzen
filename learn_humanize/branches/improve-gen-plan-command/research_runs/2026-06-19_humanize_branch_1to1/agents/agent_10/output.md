### IMPROVE_GEN_PLAN_COMMAND-HZ-010 `directory` `prompt-template/block`
- cursor: `[_]`
- core_role: Shared blocking-message template catalog for RLCR and PR-loop safety gates. It converts validator and stop-hook decisions into human-readable corrective instructions while preserving a central template/fallback contract.
- algorithmic_behavior: Contains 32 markdown templates, 407 total lines, with no nested subdirectories. The templates are selected by hooks through `load_and_render_safe "$TEMPLATE_DIR" "block/<name>.md" ...`, then rendered with `{{VAR}}` substitutions by `hooks/lib/template-loader.sh` using a single-pass awk renderer. The renderer intentionally leaves missing variables as placeholders and avoids recursively expanding placeholders inside variable values, preventing prompt/content injection (`hooks/lib/template-loader.sh:50-132`).
- inputs_outputs_state: Inputs are template names plus variables from validators and stop hooks, such as `FILE_PATH`, `CORRECT_PATH`, `CURRENT_ROUND`, `PLAN_FILE`, `AHEAD_COUNT`, `CURRENT_BRANCH`, `GIT_ISSUES`, `SUMMARY_FILE`, `BOT_MENTION_STRING`, and `CODEX_EXIT_CODE`. Outputs are markdown error/recovery messages written mostly to stderr and paired with hook exit code `2` for blocked tool use. The templates themselves hold no mutable state; they describe transitions the user/agent must take, while state remains in `.humanize/rlcr/<timestamp>/state.md`, `finalize-state.md`, PR-loop `state.md`, summaries, and goal trackers.
- gates_or_invariants: The directory encodes gate feedback for path correctness, current-round correctness, protected loop files, git cleanliness, push policy, goal-tracker ownership, template/schema validity, PR trigger state, force-push resets, and Codex review failures. Examples: `git-add-humanize.md` blocks staging local loop state and gives allowed alternatives (`prompt-template/block/git-add-humanize.md:1-34`); `prompt-file-write.md` marks round prompt files as read-only instructions (`prompt-template/block/prompt-file-write.md:1-12`); `state-file-modification.md` treats `state.md` as loop-managed state (`prompt-template/block/state-file-modification.md:1-10`); `incomplete-todos.md` stops review until native todos are complete (`prompt-template/block/incomplete-todos.md:1-12`).
- dependencies_and_callers: Called by `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, `hooks/loop-bash-validator.sh`, `hooks/loop-codex-stop-hook.sh`, `hooks/pr-loop-stop-hook.sh`, and helper functions in `hooks/lib/loop-common.sh`. Template reference integrity is itself tested by `tests/test-template-references.sh`, which scans hook references, checks existence, verifies common block templates, and enforces `load_and_render_safe` in critical validators (`tests/test-template-references.sh:51-200`).
- edge_cases_or_failure_modes: Missing template files degrade to inline fallback only where callers use `load_and_render_safe`; `tests/test-template-references.sh` treats missing referenced templates as critical because otherwise blocked actions could produce empty messages (`tests/test-template-references.sh:8-10`, `tests/test-template-references.sh:213-229`). Directory-level validation also depends on `validate_template_dir`, which requires `block`, `codex`, and `claude` template subdirectories under `prompt-template` (`hooks/lib/template-loader.sh:205-221`).
- validation_or_tests: `tests/test-template-references.sh` validates existence and safe rendering usage; `tests/test-templates-comprehensive.sh` renders real block templates like `block/wrong-round-number.md` and `block/unpushed-commits.md` with variables (`tests/test-templates-comprehensive.sh:493-528`); `tests/robustness/test-template-stress-robustness.sh` stress-tests the shared rendering engine used by these blocks.

- core_role: Corrective prompt block for active-loop path canonicalization failures. It tells the agent that a requested read/write target is in the wrong directory and supplies the exact active-loop path it should use.
- algorithmic_behavior: The template is six lines: title, attempted action/path, correct path, and a short command hint (`prompt-template/block/wrong-directory-path.md:1-6`). It has three variables: `{{ACTION}}`, `{{FILE_PATH}}`, and `{{CORRECT_PATH}}` (`prompt-template/block/wrong-directory-path.md:3-6`). It does not decide anything itself; validators decide the gate, then render this block through `load_and_render_safe`.
- inputs_outputs_state: Inputs are the attempted operation verb, attempted path, and computed canonical path. Output is markdown to stderr paired with validator exit code `2`. No persistent state is read or written by the template; state comes from active loop discovery and strict state parsing before the caller computes `CURRENT_ROUND` and active loop dir.
- gates_or_invariants: Enforces the invariant that round prompt/summary paths must address the newest active loop directory exactly, not another session directory or alternate relative spelling. The read validator computes `CORRECT_PATH="$ACTIVE_LOOP_DIR/$CLAUDE_FILENAME"` and blocks when `FILE_PATH != CORRECT_PATH` (`hooks/loop-read-validator.sh:153-168`). The write validator uses the same comparison for relevant RLCR summary/finalize/managed files (`hooks/loop-write-validator.sh:248-263`).
- dependencies_and_callers: Directly referenced by `hooks/loop-read-validator.sh:163` and `hooks/loop-write-validator.sh:258`. Both callers depend on `hooks/lib/loop-common.sh` for `find_active_loop`, strict state parsing, round-file matching, and allowlist behavior. The template renderer comes from `hooks/lib/template-loader.sh`.
- edge_cases_or_failure_modes: The final hint says `If you need this file, use: cat {{FILE_PATH}}` (`prompt-template/block/wrong-directory-path.md:6`), which repeats the attempted path rather than `{{CORRECT_PATH}}`; in the wrong-directory case this may be confusing or counterproductive because the preceding line identifies the correct path. The callers’ inline fallbacks correctly say the attempted path is wrong and show `{{CORRECT_PATH}}` (`hooks/loop-read-validator.sh:160-166`, `hooks/loop-write-validator.sh:255-261`), so the fallback behavior is clearer than the template’s command hint.
