### TEMPLIFY_PROMPT-HZ-014 `file` `hooks/check-todos-from-transcript.py`
- cursor: `[_]`
- core_role: Early stop-hook validator for the RLCR loop. It reads Claude Code hook input, locates the current transcript JSONL, extracts the latest native `TodoWrite` state, and blocks loop exit when any todo remains incomplete. This keeps the loop from spending a Codex review round before Claude has completed its own declared task list.
- algorithmic_behavior: The main algorithm is split between `find_latest_todos()` and `main()`. `find_latest_todos(transcript_path)` scans the transcript line by line as JSONL, ignores blank lines and malformed JSON, and keeps replacing `latest_todos` whenever it finds a non-empty `TodoWrite` payload, so the final result is the most recent todo list seen in file order. It recognizes three transcript shapes: Claude Code assistant messages with `message.content[]` tool-use blocks at lines 48-60, alternative `type: message` entries with top-level `content[]` at lines 62-73, and direct `type: tool_use` entries at lines 75-82. `main()` reads hook JSON from stdin, resolves `transcript_path`, retrieves the latest todos, then exits successfully if no todos exist or all statuses are `completed`; otherwise it prints an `INCOMPLETE_TODOS` marker plus one formatted line per unfinished todo and exits with code 1 at lines 117-122.
- inputs_outputs_state: Input is a JSON object on stdin containing `transcript_path`, shown in the usage comment at lines 9-10 and consumed at lines 89-100. The transcript itself is UTF-8 JSONL opened at lines 27-28. Output is process status plus stdout: exit code 0 means allow the stop hook to continue, exit code 1 means incomplete todos were found, with stdout beginning `INCOMPLETE_TODOS` at line 120. There is no persistent file write and no internal durable state; the effective state is reconstructed from the latest `TodoWrite` call in the transcript. Missing transcript, missing stdin JSON, absent path, absent todos, empty `todos` arrays, and malformed transcript lines all collapse to allow/exit 0 at lines 22-23, 33-36, 89-97, and 105-107.
- gates_or_invariants: The key gate is strict status equality: only `status == "completed"` passes; any other value, including `pending`, `in_progress`, empty, unknown, or missing, is treated as incomplete at lines 111-115. The scanner intentionally uses the latest non-empty todo list, not a union of all historical todos, which matches native TodoWrite state semantics but means an empty later TodoWrite payload is ignored rather than clearing prior todos. The script intentionally prints both marker and details to stdout, with a comment at lines 117-119 saying mixed stdout/stderr caused buffering-order problems. It is fail-open for invalid hook input and missing transcript, favoring not blocking a stop attempt when evidence cannot be parsed.
- dependencies_and_callers: Depends only on Python standard library modules `json`, `sys`, and `pathlib` at lines 12-14. The direct caller is `hooks/loop-codex-stop-hook.sh`: it sets `TODO_CHECKER` to this script at line 67, pipes the original hook input into `python3 "$TODO_CHECKER"` at line 71, captures nonzero exit as `TODO_EXIT`, and if the checker returns 1 it renders `prompt-template/block/incomplete-todos.md` and returns a Claude hook JSON decision of `"block"` at lines 74-94. This check runs before large-file, git cleanliness, unpushed-commit, summary, and Codex review gates, so it is a cheap pre-review blocker. The README lists this script as part of the hook lifecycle structure at README lines 214-230.
- edge_cases_or_failure_modes: Because `TODO_EXIT` is only set in the shell caller on failure, the stop hook’s surrounding shell logic is safe for this script’s expected 0/1 behavior, but unexpected script errors are treated the same as incomplete todos if they produce exit 1. The script assumes each todo is a dict; a malformed `todos` list containing non-dicts would raise on `todo.get()` at lines 111-113 rather than fail open. It treats any status spelling other than exact lowercase `completed` as blocking. It does not validate todo IDs, priorities, duplicate todos, or consistency between TodoWrite calls; it only gates on the latest parsed non-empty list. It silently ignores transcript JSON objects where tool-use blocks are nested differently from the three supported patterns.
- validation_or_tests: No dedicated unit test for this Python checker was found in the searched test references. It is indirectly integrated through `hooks/loop-codex-stop-hook.sh` lines 67-95 and through the incomplete-todos prompt template at `prompt-template/block/incomplete-todos.md` lines 1-12. The hook/template reference tests cover template existence and safe rendering behavior generally, but the Python parsing paths and malformed transcript cases appear untested.
