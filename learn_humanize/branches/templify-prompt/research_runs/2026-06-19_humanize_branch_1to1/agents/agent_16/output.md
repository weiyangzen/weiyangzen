### TEMPLIFY_PROMPT-HZ-016 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: PreToolUse validator for the RLCR loop’s `Bash` tool path. It prevents shell commands from bypassing the loop’s managed Write/Edit/Read hooks and protects loop-owned files such as `state.md`, `goal-tracker.md`, `round-N-prompt.md`, `round-N-summary.md`, and `round-N-todos.md`. It is registered for Bash in `hooks/hooks.json:32-38`.
- algorithmic_behavior: Reads hook JSON from stdin, extracts `.tool_name` and `.tool_input.command` with `jq`, and exits allow-state if the tool is not Bash (`hooks/loop-bash-validator.sh:22-30`). It resolves the active loop under `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize-loop.local`, using `find_active_loop`, and allows all commands when no active loop exists (`hooks/loop-bash-validator.sh:36-43`). With an active loop, it derives `CURRENT_ROUND` from `state.md`, then applies ordered gates: block `git push` unless `push_every_round: true`; block Bash writes to `state.md`; block Bash writes to `goal-tracker.md` with round-sensitive guidance; block prompt, summary, and todos file modifications (`hooks/loop-bash-validator.sh:45-124`).
- inputs_outputs_state: Input is the PreToolUse JSON payload on stdin plus environment/context from `CLAUDE_PROJECT_DIR`, current working directory, `.humanize-loop.local/*/state.md`, and `push_every_round:` in the state file. Output is either exit `0` to allow the Bash command or exit `2` with a rendered Markdown block on stderr to deny it. The script does not mutate loop state; its state transition is hook-level only: command allowed, command blocked, or no active loop. The active loop state is read from the newest timestamp-like loop directory selected by `find_active_loop` in `hooks/lib/loop-common.sh:23-44`; current round defaults to `0` if parsing fails or field is absent (`hooks/lib/loop-common.sh:46-58`).
- gates_or_invariants: The first invariant is that loop state is system-owned: any detected Bash modification of `state.md` is blocked (`hooks/loop-bash-validator.sh:67-75`). The second is that `goal-tracker.md` has phase-dependent ownership: in round `0`, Bash is blocked and Write/Edit is suggested; after round `0`, direct modification is blocked and the user must request changes through the current round summary (`hooks/loop-bash-validator.sh:77-92`). The third is that prompt files are read-only instructions from Codex to Claude (`hooks/loop-bash-validator.sh:94-102`). The fourth is that summary files must be written through Write/Edit so path and round validators can run (`hooks/loop-bash-validator.sh:104-113`). The fifth is that `round-*-todos.md` access is disallowed in favor of TodoWrite (`hooks/loop-bash-validator.sh:115-122`). The file-modification detector is centralized in `command_modifies_file`, which recognizes redirection, `tee`, in-place `sed`/`awk`/`perl`, `mv`/`cp` to a matching destination, and `dd of=` (`hooks/lib/loop-common.sh:155-178`).
- dependencies_and_callers: Called by Claude hook configuration as a `PreToolUse` Bash matcher (`hooks/hooks.json:32-38`). Sources `hooks/lib/loop-common.sh` (`hooks/loop-bash-validator.sh:14-16`), which in turn sources `hooks/lib/template-loader.sh` and initializes `TEMPLATE_DIR` (`hooks/lib/loop-common.sh:11-20`). Depends on shell tools and conventions: `bash`, `jq`, `grep`, `sed`, `tr`, `ls`, `sort`, `head`, and the prompt-template tree. Directly renders `block/git-push.md` for push blocking (`hooks/loop-bash-validator.sh:58-63`) and indirectly renders shared block templates through helper functions: `state-file-modification.md`, `goal-tracker-bash-write.md`, `goal-tracker-modification.md`, `prompt-file-write.md`, `summary-bash-write.md`, and `todos-file-access.md` (`hooks/lib/loop-common.sh:86-123`, `hooks/lib/loop-common.sh:126-134`, `hooks/lib/loop-common.sh:180-191`).
- edge_cases_or_failure_modes: If no active loop directory exists, all Bash commands are allowed, so this gate only applies while `.humanize-loop.local` has a newest directory with `state.md` (`hooks/loop-bash-validator.sh:40-43`). Because `find_active_loop` intentionally checks only the newest loop directory, older directories with `state.md` are ignored, preventing stale loop revival but also making newest-directory ordering critical (`hooks/lib/loop-common.sh:23-44`). The `git push` gate only matches commands that begin with optional whitespace and `git push`, so wrapped forms such as `cd repo && git push` or `env git push` are not caught by that specific gate (`hooks/loop-bash-validator.sh:55-64`). The Bash write detector is pattern-based and can miss nonstandard mutation paths such as `python -c 'open(...)'`, `truncate`, `install`, editor commands, or custom functions, while it may also block matching filenames outside the active loop because it matches command text against filename patterns rather than resolving paths (`hooks/lib/loop-common.sh:162-170`). Malformed hook JSON or missing `jq` would likely fail the hook under `set -euo pipefail` before reaching graceful allow/block logic (`hooks/loop-bash-validator.sh:12`, `hooks/loop-bash-validator.sh:22-30`).
- validation_or_tests: Static test coverage includes template-reference validation: `tests/test-template-references.sh` includes `hooks/loop-bash-validator.sh` in its script list (`tests/test-template-references.sh:57-64`) and scans `load_template`, `load_and_render`, and `load_and_render_safe` calls to verify referenced templates exist (`tests/test-template-references.sh:83-110`). Template-loader behavior and safe fallback rendering are also tested in `tests/test-template-loader.sh` and `tests/test-templates-comprehensive.sh` per repository search results. I did not find a dedicated behavioral test that executes this Bash validator against sample hook payloads and asserts each block/allow branch.

### TEMPLIFY_PROMPT-HZ-046 `file` `prompt-template/block/wrong-directory-path.md`
- cursor: `[_]`
- core_role: Markdown block template for the RLCR validator path-location gate. It is not executable logic, but it defines the user-facing failure contract when a Read or Write attempt targets the right round-style filename in the wrong directory. The template content is `# Wrong Directory Path`, attempted action/path, correct path, and a short `cat` fallback hint (`prompt-template/block/wrong-directory-path.md:1-6`).
- algorithmic_behavior: Rendered by validators through `load_and_render_safe` when `FILE_PATH` does not exactly equal `CORRECT_PATH`. In the read validator, `CORRECT_PATH` is computed as `$ACTIVE_LOOP_DIR/$CLAUDE_FILENAME`; mismatch renders this template with `ACTION=read` and exits `2` (`hooks/loop-read-validator.sh:122-132`). In the write validator, the same gate renders this template with `ACTION=write to` and exits `2` (`hooks/loop-write-validator.sh:154-164`). The template uses placeholders `{{ACTION}}`, `{{FILE_PATH}}`, and `{{CORRECT_PATH}}`, rendered by the shared template loader’s single-pass `{{VAR}}` substitution (`hooks/lib/template-loader.sh:35-43`, `hooks/lib/template-loader.sh:56-116`).
- inputs_outputs_state: Inputs are render variables supplied by caller validators: attempted action, attempted path, and computed active-loop path. Output is a Markdown denial message sent to stderr by the calling validator, followed by validator exit `2`. The template itself does not read files, compute loop state, or mutate state; it encodes the explanatory surface for a state-machine guard whose state dependency is the active loop directory and current round filename.
- gates_or_invariants: Supports the invariant that loop round artifacts must be accessed through the active loop directory, not through stale loop directories, copied filenames, or arbitrary working-directory-relative paths. It is reached after the read validator has already validated round number compatibility (`hooks/loop-read-validator.sh:100-116`) and after the write validator has handled summary round mismatch (`hooks/loop-write-validator.sh:130-148`). Its contract distinguishes attempted path from correct path so the agent can recover by targeting the active loop artifact.
- dependencies_and_callers: Called by `hooks/loop-read-validator.sh:128-131` and `hooks/loop-write-validator.sh:160-163`. Rendered through `hooks/lib/template-loader.sh`, where missing templates fall back to inline caller messages (`hooks/lib/template-loader.sh:152-187`) and unresolved placeholders are preserved rather than recursively substituted (`hooks/lib/template-loader.sh:100-107`). Template directory discovery is rooted at the plugin root’s `prompt-template` directory (`hooks/lib/template-loader.sh:9-16`), and shared hook initialization validates the expected template subdirectories while degrading to fallbacks on failure (`hooks/lib/loop-common.sh:15-21`).
- edge_cases_or_failure_modes: The last line says `If you need this file, use: cat {{FILE_PATH}}` (`prompt-template/block/wrong-directory-path.md:6`). For read blocks, that may intentionally provide a Bash escape hatch for inspecting the originally requested non-active-loop file; for write blocks, it is read-oriented wording and can be semantically awkward because the validator’s correction is `{{CORRECT_PATH}}`, not `cat` on the attempted path. If any render variable is omitted, the loader leaves the placeholder text visible rather than failing hard (`hooks/lib/template-loader.sh:100-107`). Because callers compare string paths directly, semantically equivalent paths with symlinks, `..`, or non-normalized forms may still be treated as wrong directory by the validator.
- validation_or_tests: `tests/test-template-references.sh` scans validator scripts for template references and fails if referenced templates are missing (`tests/test-template-references.sh:83-110`), covering the existence of this template because both read and write validators reference it. Search results show no dedicated content-specific test for the exact wording or `ACTION` variants of this template. The rendering and safe fallback behavior are covered at the loader level by template-loader tests, including `load_and_render_safe` missing/existing template cases referenced in the test inventory.
