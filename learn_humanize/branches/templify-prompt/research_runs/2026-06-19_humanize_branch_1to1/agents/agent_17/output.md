### TEMPLIFY_PROMPT-HZ-017 `file` `hooks/loop-codex-stop-hook.sh`
- cursor: `[_]`
- core_role: This is the Claude Code `Stop` hook for the RLCR loop. It prevents the loop from ending until local guard checks pass and an independent Codex review either approves completion with a terminal `COMPLETE` line or stops the loop with `STOP`. It is registered in `hooks/hooks.json:42-49` with a 7200 second hook timeout.
- algorithmic_behavior: The hook reads hook JSON from stdin, discovers the newest active loop under `.humanize-loop.local` via `find_active_loop`, then runs a staged state machine. If no active loop or no `state.md` exists, it allows exit (`hooks/loop-codex-stop-hook.sh:52-57`, `281-283`). Before invoking Codex, it blocks on incomplete Claude todos from the transcript (`67-95`), oversized changed code/doc files above 2000 lines (`104-177`), uncommitted git changes (`186-240`), and optionally unpushed commits when `push_every_round: true` (`246-274`). It parses YAML frontmatter from `state.md` for `current_round`, `max_iterations`, `codex_model`, `codex_effort`, `codex_timeout`, and `plan_file` (`285-301`), validates numeric state (`303-313`), requires the current round summary file (`319-340`), requires Round 0 goal tracker placeholders to be filled (`346-401`), stops gracefully at max iterations (`408-414`), builds a Codex review prompt (`428-506`), executes `codex exec` with timeout (`550-597`), validates review output (`635-707`), and then branches on the review result. Last non-empty line `COMPLETE` removes `state.md` and allows exit (`712-727`). Last non-empty line `STOP` also removes `state.md` and allows exit as a circuit breaker (`729-756`). Any other review content increments `current_round`, writes the next Claude prompt, and returns JSON `decision: block` (`763-829`).
- inputs_outputs_state: Inputs are hook stdin JSON, `CLAUDE_PROJECT_DIR` or `pwd` as project root (`43`), `.humanize-loop.local/<timestamp>/state.md`, `round-N-summary.md`, `goal-tracker.md`, git working tree status, latest TodoWrite data found through `transcript_path`, prompt templates, `codex` CLI, and optional `CODEX_TIMEOUT`. Outputs include Claude hook JSON with `decision: block` for guard failures or review feedback, stderr diagnostics for Codex and circuit-breaker events, generated `round-N-review-prompt.md`, `round-N-review-result.md`, next `round-(N+1)-prompt.md`, debug files in `$HOME/.cache/humanize/<sanitized-project-path>/<timestamp>/` (`536-549`), and mutation of `state.md` by either deleting it on completion/stop/max-iteration/corruption or updating `current_round` for the next iteration. Setup creates the expected state fields in `scripts/setup-rlcr-loop.sh:223-233`, and README documents the loop files at `README.md:241-257`.
- gates_or_invariants: The hook intentionally ignores `stop_hook_active` because each blocked stop should still trigger another review iteration (`31-38`). The newest timestamped loop is authoritative because `find_active_loop` only returns the newest directory with `state.md` (`hooks/lib/loop-common.sh:23-44`). Todos must be completed before review, using `check-todos-from-transcript.py` exit code 1 as a hard block (`67-95`; checker statuses at `hooks/check-todos-from-transcript.py:109-122`). Changed code/doc files above `MAX_LINES=2000` block continuation (`104-176`). Git must be clean before exit, and when `push_every_round` is true local ahead commits block exit (`186-274`). Round 0 must initialize the goal tracker before review (`346-401`). `COMPLETE` and `STOP` only count when they are the last non-empty line exactly after trimming, preventing false positives such as “CANNOT COMPLETE” (`712-716`). Review file existence and non-empty content are mandatory (`654-707`).

Shared dependencies are `hooks/lib/loop-common.sh` for loop discovery, lowercase/path helpers, template directory setup, and shared block messages (`hooks/loop-codex-stop-hook.sh:46-50`; helpers in `hooks/lib/loop-common.sh:11-21`, `27-63`), plus `hooks/lib/template-loader.sh` for safe fallback rendering (`template-loader.sh:152-188`). ### TEMPLIFY_PROMPT-HZ-047 `file` `prompt-template/block/wrong-file-location.md`
- cursor: `[_]`
- core_role: This is a small block-message template used by the RLCR read validator when Claude tries to read a round prompt or summary file outside the active `.humanize-loop.local/<timestamp>/` directory. It is not an algorithm by itself, but it is part of the guard contract because its rendered text is the user-facing reason for a blocked `Read` operation.
- algorithmic_behavior: The template renders `FILE_PATH`, `ACTIVE_LOOP_DIR`, and `CURRENT_ROUND` into a corrective message. It tells the model that loop files live in the active loop directory, names the current round prompt and summary paths, and ends with a direct `cat {{FILE_PATH}}` suggestion (`prompt-template/block/wrong-file-location.md:1-9`). Its only known direct caller is `hooks/loop-read-validator.sh:87-96`, where a read of `round-N-summary.md` or `round-N-prompt.md` outside `.humanize-loop.local/` triggers template rendering to stderr and exits 2, which blocks the tool use.
- inputs_outputs_state: Inputs are substitution variables from the read validator: attempted `FILE_PATH`, detected `ACTIVE_LOOP_DIR`, and `CURRENT_ROUND` (`hooks/loop-read-validator.sh:87-95`). Rendering is performed by `load_and_render_safe`, which loads the template, substitutes placeholders once, and falls back to an inline message if the file is missing or renders empty (`hooks/lib/template-loader.sh:152-188`). Output is markdown on stderr; no repository or loop state is mutated by the template. The validator’s hook input comes from Claude Code JSON and only applies to `tool_name == "Read"` (`hooks/loop-read-validator.sh:22-30`).
- gates_or_invariants: This template participates in the “right active loop, right round, right file path” invariant. The read validator first ignores non-Read tools (`hooks/loop-read-validator.sh:25-27`), blocks todos files separately (`36-39`), ignores non-round prompt/summary reads (`45-47`), requires an active loop (`56-62`), extracts current round from state (`64`), and only uses this template when the requested round file is not inside a `.humanize-loop.local/` path (`87-97`). Other path errors use separate templates: wrong round number (`103-115`) and wrong exact directory path (`122-132`). The shared helper defines the location check as a substring match for `.humanize-loop.local/` (`hooks/lib/loop-common.sh:149-153`).
- dependencies_and_callers: Direct caller is `hooks/loop-read-validator.sh:92`. Template loading comes from `hooks/lib/template-loader.sh`, whose single-pass substitution prevents placeholder injection from variable values (`35-43`, `56-116`). The template reference is discoverable through `rg`, and `tests/test-template-references.sh:56-64` includes `loop-read-validator.sh` in the scripts scanned for template references. Hook registration for read validation is in `hooks/hooks.json:23-30`.
- edge_cases_or_failure_modes: The final sentence says `If you need this file, use: cat {{FILE_PATH}}`, which points back to the attempted path rather than the computed current-round prompt/summary path. In this specific wrong-location gate, that can be confusing because the validator is blocking precisely because `FILE_PATH` is not in the active loop directory. The fallback in the caller is clearer: “Read from the active loop” (`hooks/loop-read-validator.sh:89-95`). Also, `CORRECT_PATH` is computed in the caller (`88`) but not passed to the template, so the template cannot show the exact target path except through the generic current prompt/summary list. The location predicate is substring-based, so a path containing `.humanize-loop.local/` but not equal to the active loop directory will bypass this template and be caught later by the exact directory path gate if the round number matches (`hooks/loop-read-validator.sh:122-132`).
- validation_or_tests: Existence of this template is covered indirectly by `tests/test-template-references.sh`, which scans shell scripts for `load_template`, `load_and_render`, and `load_and_render_safe` references and fails missing templates (`tests/test-template-references.sh:83-110`, `213-229`). Broader template-rendering tests cover real block templates and the template engine (`tests/test-template-loader.sh:147-164`; `tests/test-templates-comprehensive.sh:493-535`), but I did not find a dedicated assertion for `block/wrong-file-location.md` content or for the read-validator wrong-location branch.
