### TEMPLIFY_PROMPT-HZ-004 `directory` `prompt-template`
- cursor: `[_]`
- core_role: `prompt-template/` is the repository’s data-driven prompt and block-message corpus for the RLCR loop. It is not executable by itself; its algorithmic role is to externalize the text contracts used by hook scripts, Codex review prompts, Claude continuation prompts, and blocking validator messages. The loader contract is defined in `hooks/lib/template-loader.sh:3-7`, which states templates use `{{VARIABLE_NAME}}` placeholders, and `get_template_dir` resolves this directory from the plugin root at `hooks/lib/template-loader.sh:9-16`.
- algorithmic_behavior: The directory is organized into three coordinated subtrees. `prompt-template/block/` contains validator and stop-hook block messages for invalid or incomplete state transitions, such as wrong round files, summary-file misuse, dirty git state, incomplete todos, large files, unpushed commits, and blocked pushes. `prompt-template/codex/` contains the review prompts Codex receives on stop attempts: regular review, full alignment review every fifth round, and the goal-tracker update section. `prompt-template/claude/` contains the next-round continuation prompt fragments Claude receives when Codex does not return `COMPLETE`. Runtime behavior is single-pass placeholder rendering via `render_template`, which scans `{{VAR}}` patterns and replaces only values present in the initial environment table; injected placeholder-like text is intentionally not rescanned (`hooks/lib/template-loader.sh:35-43`, `hooks/lib/template-loader.sh:56-116`).
- inputs_outputs_state: Inputs are template paths plus runtime variables supplied by hooks: current round, plan path, summary content, review content, goal-tracker path, docs path, loop timestamp, previous rounds, summary file path, git branch/ahead counts, missing goal-tracker items, and validation-specific corrected paths. Outputs are rendered markdown strings written to hook JSON `reason` fields, review prompt files, or next-round prompt files. Examples: `loop-codex-stop-hook.sh` renders `codex/full-alignment-review.md` or `codex/regular-review.md` into `round-N-review-prompt.md` at `hooks/loop-codex-stop-hook.sh:475-506`; it later renders `claude/next-round-prompt.md` into `round-(N+1)-prompt.md` at `hooks/loop-codex-stop-hook.sh:768-784`. State transitions governed by these prompts are: active loop found, stop attempted, quick gates checked, Codex review generated, Codex returns `COMPLETE` or `STOP` to remove `state.md`, otherwise current round increments and a next prompt is created (`hooks/loop-codex-stop-hook.sh:712-766`).
- gates_or_invariants: The template directory must contain `block`, `codex`, and `claude` subdirectories; `validate_template_dir` enforces those required subdirs (`hooks/lib/template-loader.sh:190-207`). Missing or empty templates should not create empty user-facing blocks because critical hook call sites use `load_and_render_safe` with inline fallback text (`hooks/lib/template-loader.sh:152-188`). The prompt semantics impose major loop invariants: Codex review prompts require the original plan to be read first (`prompt-template/codex/regular-review.md:3-11`, `prompt-template/codex/full-alignment-review.md:5-10`); regular reviews only end with `COMPLETE` when all original plan tasks and ACs are complete with no deferred or pending work (`prompt-template/codex/regular-review.md:49-57`); full alignment checks add stagnation detection and allow `STOP` as the last line (`prompt-template/codex/full-alignment-review.md:59-85`). Claude continuation templates prohibit directly modifying `goal-tracker.md` after Round 0 and redirect updates to a summary request section (`prompt-template/claude/next-round-prompt.md:22-31`, `prompt-template/claude/goal-tracker-update-request.md:2-16`).
- dependencies_and_callers: Direct dependencies are `hooks/lib/template-loader.sh`, `hooks/lib/loop-common.sh`, the PreToolUse validators, and `hooks/loop-codex-stop-hook.sh`. `loop-common.sh` centralizes common block message rendering for todo, prompt, state, summary-bash, goal-tracker-bash, and goal-tracker-modification blocks (`hooks/lib/loop-common.sh:86-134`, `hooks/lib/loop-common.sh:180-192`). `loop-read-validator.sh`, `loop-write-validator.sh`, and `loop-edit-validator.sh` render wrong-location, wrong-round, prompt-write, state-file, and goal-tracker blocks (`hooks/loop-read-validator.sh:36-132`, `hooks/loop-write-validator.sh:37-164`, `hooks/loop-edit-validator.sh:36-114`). `loop-bash-validator.sh` renders `block/git-push.md` and other bash-block messages (`hooks/loop-bash-validator.sh:49-124`). `hooks/hooks.json:4-53` wires these validators into Claude Code PreToolUse and Stop lifecycle events. Setup creates the initial Round 0 prompt inline, then later rounds use `prompt-template/claude/*` fragments (`scripts/setup-rlcr-loop.sh:330-395`, `hooks/loop-codex-stop-hook.sh:768-814`).
- edge_cases_or_failure_modes: A missing template emits a warning and empty string from `load_template`, but safe call sites fall back (`hooks/lib/template-loader.sh:18-33`, `hooks/lib/template-loader.sh:152-188`). Unreplaced variables intentionally remain in output if not supplied, which can be useful for diagnostics but can leak raw `{{VAR}}` text into prompts if a caller forgets an argument (`hooks/lib/template-loader.sh:100-107`). Single-pass rendering prevents prompt corruption when injected review content itself contains `{{GOAL_TRACKER_FILE}}` or similar placeholders (`hooks/lib/template-loader.sh:39-42` and tested at `tests/test-template-loader.sh:488-543`). Templates are content contracts, so behavioral drift can happen if template instructions and hook gates diverge; for example, Round 0 prompt creation in `scripts/setup-rlcr-loop.sh` is inline rather than templated, while later rounds are templated. The directory also includes templates that are only conditionally appended, such as post-alignment instructions after rounds where `CURRENT_ROUND % 5 == 4` (`hooks/loop-codex-stop-hook.sh:786-792`) and push notes only when `push_every_round` is true (`hooks/loop-codex-stop-hook.sh:800-807`).
- validation_or_tests: `tests/test-template-loader.sh` validates directory resolution, loading `block/git-push.md`, missing-template behavior, placeholder substitution, safe fallback behavior, and shell metacharacter literal rendering (`tests/test-template-loader.sh:41-78`, `tests/test-template-loader.sh:81-222`, `tests/test-template-loader.sh:247-543`). `tests/test-template-references.sh` scans shell scripts for `load_template`, `load_and_render`, and `load_and_render_safe` calls and verifies referenced template files exist (`tests/test-template-references.sh:51-116`); it also verifies common block templates and requires critical validators to use safe rendering (`tests/test-template-references.sh:145-201`). `tests/test-templates-comprehensive.sh` validates the `block`, `codex`, and `claude` subdirs, loads every markdown template recursively, checks placeholder syntax, exercises fallback behavior with `block/git-push.md`, and stress-renders all templates with dummy placeholder values (`tests/test-templates-comprehensive.sh:61-107`, `tests/test-templates-comprehensive.sh:440-606`). I inspected these tests but did not run them because this branch export is read-only and the task requested research notes only.

### TEMPLIFY_PROMPT-HZ-034 `file` `prompt-template/block/git-push.md`
- cursor: `[_]`
- core_role: `prompt-template/block/git-push.md` is the block message shown when an RLCR loop is configured to keep commits local and Claude attempts `git push`. Its algorithmic role is a policy explanation paired with the executable bash validator gate in `hooks/loop-bash-validator.sh`.
- algorithmic_behavior: The file defines a short markdown block headed “Git Push Blocked” (`prompt-template/block/git-push.md:1`). It tells the agent that current commits should remain local and that the loop handles commits locally until completion (`prompt-template/block/git-push.md:3-4`). It then gives the opt-in transition for changing policy: start the loop with `--push-every-round` (`prompt-template/block/git-push.md:6-9`). The file itself has no placeholders, so render output is static and does not depend on runtime substitution.
- inputs_outputs_state: Runtime input is the hook input JSON for a Bash command. `loop-bash-validator.sh` extracts `.tool_input.command`, lowercases it, finds the active loop, reads `push_every_round` from `state.md`, and only applies this gate when the flag is not `true` (`hooks/loop-bash-validator.sh:22-30`, `hooks/loop-bash-validator.sh:36-45`, `hooks/loop-bash-validator.sh:53-55`). If the command matches a leading `git push`, the validator renders this template and writes it to stderr, then exits with code `2` to block the tool call (`hooks/loop-bash-validator.sh:55-64`). The state transition is therefore “Bash tool request for push” to “blocked pre-tool-use action” unless `push_every_round` has been enabled in loop state.
- gates_or_invariants: The core invariant is local-only commits by default. The gate pattern is strict but simple: `COMMAND_LOWER` must match `^[[:space:]]*git[[:space:]]+push` (`hooks/loop-bash-validator.sh:56-57`). Setup initializes `push_every_round` to `false` by default and only sets it true when the start command receives `--push-every-round` (`scripts/setup-rlcr-loop.sh:27-33`, `scripts/setup-rlcr-loop.sh:137-139`, `scripts/setup-rlcr-loop.sh:223-231`). The companion positive policy is that when `--push-every-round` is true, the stop hook requires unpushed commits to be pushed before exit via `block/unpushed-commits.md` (`hooks/loop-codex-stop-hook.sh:242-274`), and continuation prompts append `claude/push-every-round-note.md` (`hooks/loop-codex-stop-hook.sh:800-807`).
- dependencies_and_callers: The direct caller is `hooks/loop-bash-validator.sh:58-63`, through `load_and_render_safe "$TEMPLATE_DIR" "block/git-push.md" "$FALLBACK"`. The template directory is supplied by `loop-common.sh`, which sources the loader and computes `TEMPLATE_DIR` (`hooks/lib/loop-common.sh:11-20`). The loader path resolution and static rendering are from `hooks/lib/template-loader.sh:9-16` and `hooks/lib/template-loader.sh:152-188`. User-facing command documentation advertises `--push-every-round` as “Require git push after each round” in `scripts/setup-rlcr-loop.sh:45-52` and README usage (`README.md:190-199`).
- edge_cases_or_failure_modes: Because the command match only detects a simple command beginning with `git push`, indirect pushes hidden behind shell wrappers, aliases, functions, subshell scripts, or `command git push` may not be caught by this specific regex. Conversely, commands with leading whitespace are covered. If `state.md` is missing or no active loop exists, `loop-bash-validator.sh` exits early and this template is not used (`hooks/loop-bash-validator.sh:36-43`). If the template is missing, the safe fallback still blocks with a shorter inline message (`hooks/loop-bash-validator.sh:58-63`). The template’s example command uses `/humanize:start-rlcr-loop plan.md --push-every-round`; it is guidance only and does not alter the current loop’s state.
- validation_or_tests: `tests/test-template-loader.sh` explicitly loads `block/git-push.md` and expects content containing “Git Push Blocked” (`tests/test-template-loader.sh:55-65`), then verifies `load_and_render_safe` prefers this real template over fallback text (`tests/test-template-loader.sh:210-222`). `tests/test-templates-comprehensive.sh` repeats the fallback-preference check for `block/git-push.md` (`tests/test-templates-comprehensive.sh:455-463`) and also loads every template recursively (`tests/test-templates-comprehensive.sh:83-107`). `tests/test-template-references.sh` scans hook scripts for template references and validates referenced files exist (`tests/test-template-references.sh:51-116`), which covers the `loop-bash-validator.sh` reference found at `hooks/loop-bash-validator.sh:62`.
