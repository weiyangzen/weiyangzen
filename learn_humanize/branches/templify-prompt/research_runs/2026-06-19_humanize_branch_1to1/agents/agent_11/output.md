### TEMPLIFY_PROMPT-HZ-011 `directory` `prompt-template/codex`
- cursor: `[_]`
- core_role: This directory defines the Codex-side review prompt protocol for the RLCR loop. It is not executable code itself, but it is core algorithm surface because the stop hook renders these templates into `round-N-review-prompt.md`, runs Codex, then interprets the resulting `round-N-review-result.md` to either stop, circuit-break, or advance the loop. The three recursive children are:
  - `prompt-template/codex/regular-review.md`: normal per-round critical review template.
  - `prompt-template/codex/full-alignment-review.md`: mandatory deeper audit template used every fifth round.
  - `prompt-template/codex/goal-tracker-update-section.md`: embedded authority-transfer section telling Codex how to process goal-tracker update requests.
- algorithmic_behavior: `regular-review.md` instructs Codex to read the original plan first (`regular-review.md:3-9`), compare Claude’s claimed work and prompt against actual implementation (`regular-review.md:11-23`), reject deferred or pending work as incomplete (`regular-review.md:24-31`), run a mandatory goal-alignment check against the goal tracker (`regular-review.md:33-45`), inject the goal-tracker update subsection (`regular-review.md:47`), and only emit terminal `COMPLETE` when all original-plan tasks and ACs are fully done with no deferrals or pending work (`regular-review.md:49-57`). `full-alignment-review.md` adds a round checkpoint contract: it is mandatory every 5 rounds (`full-alignment-review.md:1-3`), requires reading the plan first (`full-alignment-review.md:5-10`), audits every acceptance criterion with evidence/blocker/deferred columns (`full-alignment-review.md:19-29`), checks forgotten plan items (`full-alignment-review.md:30-34`), audits deferred items (`full-alignment-review.md:36-40`), conducts implementation review (`full-alignment-review.md:50-55`), and performs a stagnation/circuit-breaker check over historical round files (`full-alignment-review.md:59-84`). `goal-tracker-update-section.md` makes Codex responsible for evaluating Claude’s requested goal-tracker changes after Round 0, then either editing the mutable goal-tracker sections or rejecting the request (`goal-tracker-update-section.md:1-17`).
- inputs_outputs_state: Inputs are placeholder values rendered by `hooks/loop-codex-stop-hook.sh`: `CURRENT_ROUND`, `PLAN_FILE`, `PROMPT_FILE`, `SUMMARY_CONTENT`, `GOAL_TRACKER_FILE`, `DOCS_PATH`, `GOAL_TRACKER_UPDATE_SECTION`, `COMPLETED_ITERATIONS`, `LOOP_TIMESTAMP`, `PREV_ROUND`, `PREV_PREV_ROUND`, and `REVIEW_RESULT_FILE` (`hooks/loop-codex-stop-hook.sh:428-438`, `hooks/loop-codex-stop-hook.sh:477-505`). The stop hook derives those from `.humanize-loop.local/<timestamp>/state.md` frontmatter, summary files, the current prompt path, and loop metadata (`hooks/loop-codex-stop-hook.sh:285-301`, `hooks/loop-codex-stop-hook.sh:319-332`, `hooks/loop-codex-stop-hook.sh:432-450`). Outputs are a rendered review prompt at `round-N-review-prompt.md`, a Codex-authored review result at `round-N-review-result.md`, optional goal-tracker edits by Codex, and a terminal marker on the review result’s last non-empty line. The state transition is implemented outside the templates: `COMPLETE` removes `state.md` and ends the loop (`hooks/loop-codex-stop-hook.sh:718-727`); `STOP` removes `state.md` and triggers circuit-breaker termination (`hooks/loop-codex-stop-hook.sh:729-756`); any other non-empty result increments `current_round` in `state.md`, creates the next prompt, and blocks Claude exit with the new instructions (`hooks/loop-codex-stop-hook.sh:763-828`).
- gates_or_invariants: The central completion invariant is strict: deferred items are incomplete and `COMPLETE` is valid only when all original-plan tasks and all acceptance criteria are fully met (`regular-review.md:53-56`, `full-alignment-review.md:88-93`). The goal tracker’s immutable section must never be modified (`goal-tracker-update-section.md:10`). Full alignment rounds are selected when `CURRENT_ROUND % 5 == 4` (`hooks/loop-codex-stop-hook.sh:440-444`), matching the README’s documented rounds 4, 9, 14, etc. (`README.md:130-151`). Codex result validation is strict: nonzero Codex exit blocks the loop, missing review result blocks, empty result blocks, and stdout is copied to the expected review file only as a recovery path (`hooks/loop-codex-stop-hook.sh:635-707`). Terminal markers are matched only on the last non-empty line to avoid false positives such as “cannot complete” (`hooks/loop-codex-stop-hook.sh:712-717`).
- dependencies_and_callers: The directory is consumed by `hooks/loop-codex-stop-hook.sh`, which renders `goal-tracker-update-section.md` first and injects it into either `regular-review.md` or `full-alignment-review.md` (`hooks/loop-codex-stop-hook.sh:434-438`, `hooks/loop-codex-stop-hook.sh:475-505`). Rendering is provided by `hooks/lib/template-loader.sh`: `get_template_dir` resolves `prompt-template` (`template-loader.sh:11-15`), `load_template` reads a template or warns if absent (`template-loader.sh:21-33`), `render_template` performs single-pass `{{VAR}}` substitution (`template-loader.sh:35-117`), and `load_and_render_safe` provides fallback text if a template is missing or renders empty (`template-loader.sh:155-188`). The broader RLCR caller contract is documented in `README.md:7-27` and `README.md:31-39`: Claude implements and summarizes, Codex reviews and either feeds back or ends the loop. Sibling templates under `prompt-template/claude` are used after Codex finds issues to build the next Claude prompt (`hooks/loop-codex-stop-hook.sh:772-814`).
- edge_cases_or_failure_modes: If a Codex template is missing, the loop continues with a fallback prompt, but the fallback is less detailed than the checked-in templates and may weaken review rigor (`hooks/loop-codex-stop-hook.sh:453-473`, `template-loader.sh:155-188`). Placeholder injection is intentionally reduced because `render_template` is single-pass; injected content containing `{{OTHER_VAR}}` is not re-scanned (`template-loader.sh:39-42`). If the summary file is missing, prompt generation never starts and the hook blocks Claude with a summary-required message (`hooks/loop-codex-stop-hook.sh:319-340`). If Round 0’s goal tracker still contains placeholders, review execution is blocked until initialized (`hooks/loop-codex-stop-hook.sh:342-401`). If `current_round` is non-numeric, the hook treats state as corrupted and removes `state.md` (`hooks/loop-codex-stop-hook.sh:303-308`). `STOP` is designed for full alignment rounds, but the hook still honors it during regular rounds while labeling it unexpected (`hooks/loop-codex-stop-hook.sh:744-752`). A notable validation gap is that tests directly exercise loading/rendering and the goal-tracker subsection, but the full regular/full-alignment prompt semantics are mostly enforced by review text rather than executable assertions.
- validation_or_tests: `tests/test-template-references.sh` scans hook scripts for `load_template`, `load_and_render`, and `load_and_render_safe` references and verifies referenced templates exist (`tests/test-template-references.sh:51-115`); it also scans all templates and warns about unreferenced ones (`tests/test-template-references.sh:117-140`). `tests/test-templates-comprehensive.sh` verifies the `prompt-template` directory and required `block`, `codex`, and `claude` subdirectories exist (`tests/test-templates-comprehensive.sh:61-80`), loads every markdown template (`tests/test-templates-comprehensive.sh:82-107`), validates syntax after that section, and has a direct real-template rendering check for `codex/goal-tracker-update-section.md` (`tests/test-templates-comprehensive.sh:531-539`). `tests/test-template-loader.sh` covers loader/render integration with real templates and variable replacement behavior (`tests/test-template-loader.sh:147-164`). ### TEMPLIFY_PROMPT-HZ-041 `file` `prompt-template/block/state-file-modification.md`
- cursor: `[_]`
- core_role: This is a guard-message template for the loop’s state-file integrity invariant. It is small, but algorithmically relevant because the write/edit/bash validators render it when Claude attempts to modify `.humanize-loop.local/<timestamp>/state.md`, which is the loop controller’s state cursor.
- algorithmic_behavior: The template is static and has no placeholders. It tells the actor that `state.md` cannot be modified because it is managed by the loop system (`state-file-modification.md:1-4`), and identifies the protected state fields as current round number, max iterations, and Codex configuration (`state-file-modification.md:5-9`). It does not perform validation itself; it is the explanatory output of validator decisions.
- inputs_outputs_state: Direct template input is none. Runtime inputs are the attempted tool action and path/command parsed by validators. `hooks/lib/loop-common.sh` exposes `state_file_blocked_message`, which loads `block/state-file-modification.md` with a short fallback (`loop-common.sh:106-113`). The output is a rendered block reason written to stderr by validators, followed by exit code `2` to block the tool action. State transition is prevention-only: the template helps preserve `state.md` so only loop code can advance `current_round`, terminate the loop by removing `state.md`, or update state after Codex review (`hooks/loop-codex-stop-hook.sh:718-727`, `hooks/loop-codex-stop-hook.sh:729-756`, `hooks/loop-codex-stop-hook.sh:763-766`).
- gates_or_invariants: The protected invariant is that `state.md` is loop-owned, not Claude-owned. State file paths are detected by `is_state_file_path`, which matches lowercase paths ending in `state.md` (`loop-common.sh:143-147`). The Write validator blocks writes to state files after it finds an active loop (`loop-write-validator.sh:71-87`). The Edit validator blocks edits to state files inside `.humanize-loop.local` after it finds the active loop (`loop-edit-validator.sh:58-74`). The Bash validator blocks shell commands that appear to modify `state.md` through redirection, `tee`, in-place editors, `mv`, `cp`, or `dd` patterns (`loop-common.sh:155-177`, `loop-bash-validator.sh:67-75`).
- dependencies_and_callers: The template depends on the shared template loader through `state_file_blocked_message` (`loop-common.sh:11-17`, `loop-common.sh:106-113`). Its direct callers are `loop-write-validator.sh` for `Write` tools (`loop-write-validator.sh:82-87`), `loop-edit-validator.sh` for `Edit` tools (`loop-edit-validator.sh:69-74`), and `loop-bash-validator.sh` for shell bypass attempts (`loop-bash-validator.sh:67-75`). It coordinates with sibling block templates that protect related loop artifacts such as prompt files, todos files, summary files, and goal-tracker edits.
- edge_cases_or_failure_modes: If no active loop directory is found, the Write/Edit/Bash validators allow the action, so stale or inactive loop directories are not protected by this guard (`loop-write-validator.sh:71-77`, `loop-edit-validator.sh:58-64`, `loop-bash-validator.sh:36-43`). The Bash detector is regex-based rather than shell-AST-based, so indirect writes, quoted edge cases, path aliases, or multi-step commands may evade it, while unrelated paths ending in `state.md` may be conservatively blocked. If the template is missing or empty, `load_and_render_safe` falls back to a shorter message that still blocks but omits the detailed list of state fields (`loop-common.sh:108-113`, `template-loader.sh:155-188`). Because this template has no variables, placeholder rendering failures are unlikely to corrupt the message content.
- validation_or_tests: `tests/test-template-references.sh` includes `block/state-file-modification.md` in the common-template existence set (`tests/test-template-references.sh:149-166`) and verifies critical validators use safe template rendering (`tests/test-template-references.sh:170-200`). `tests/test-templates-comprehensive.sh` loads all markdown templates under `prompt-template` (`tests/test-templates-comprehensive.sh:82-107`), which covers this file for loadability and non-empty content. The targeted search did not show a dedicated behavior test that invokes Write/Edit/Bash validators against `state.md`; coverage is mostly reference/load validation plus the shared validator logic.
