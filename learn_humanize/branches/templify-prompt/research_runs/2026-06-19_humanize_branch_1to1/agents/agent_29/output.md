### TEMPLIFY_PROMPT-HZ-029 `file` `hooks/lib/template-loader.sh`
- cursor: `[_]`
- core_role: Shared prompt-template loading and rendering library for RLCR loop hooks. It is not the RLCR state machine controller, but it is a core support boundary for state-machine enforcement because validators and stop hooks use it to produce block messages, review prompts, next-round prompts, and fallback messages that guide or constrain the loop. `hooks/lib/loop-common.sh` sources it at `hooks/lib/loop-common.sh:11-13`, initializes `TEMPLATE_DIR` through it at `hooks/lib/loop-common.sh:15-16`, and warns rather than aborts on template directory validation failure at `hooks/lib/loop-common.sh:18-21`.
- algorithmic_behavior: The file exposes six shell functions. `get_template_dir` maps a caller-supplied hook/lib directory to the plugin root by moving two levels up, then appends `prompt-template` (`hooks/lib/template-loader.sh:11-16`). `load_template` constructs `template_dir/template_name`, emits file contents if present, otherwise writes a warning to stderr and returns an empty stdout string (`hooks/lib/template-loader.sh:21-33`). `render_template` performs single-pass `{{VAR}}` replacement (`hooks/lib/template-loader.sh:43-117`): it converts `VAR=value` arguments into environment variables with a `TMPL_VAR_` prefix (`hooks/lib/template-loader.sh:47-54`), then uses AWK to scan each line for `{{` and matching `}}`, replace known placeholders, and preserve unknown placeholders (`hooks/lib/template-loader.sh:59-114`). The single-pass design is explicit: inserted values are appended to output and not rescanned, preventing injected placeholder text from being expanded later (`hooks/lib/template-loader.sh:35-42`, `hooks/lib/template-loader.sh:100-110`). `load_and_render` composes `load_template` and `render_template`, returning nothing if the template load is empty (`hooks/lib/template-loader.sh:121-132`). `append_template` prints base content followed by loaded template content (`hooks/lib/template-loader.sh:136-146`). `load_and_render_safe` suppresses loader warnings, falls back to an inline message if the template is missing/empty or rendering produces empty output, and renders fallback variables when provided (`hooks/lib/template-loader.sh:155-188`). `validate_template_dir` gates required directory shape: root directory plus `block`, `codex`, and `claude` subdirectories (`hooks/lib/template-loader.sh:193-207`).
- inputs_outputs_state: Inputs are a script/lib path for `get_template_dir`, a template root path, a repo-relative template name such as `block/wrong-round-number.md`, raw template content, fallback text, and zero or more `VAR=value` assignments. Outputs are rendered Markdown/plaintext to stdout, warnings/errors to stderr, and shell return codes for directory validation. The library does not write persistent state files and does not mutate RLCR round state. Its only transient state is local shell variables, AWK `vars[]`, and prefixed environment variables passed to the AWK process. Callers redirect output into validator feedback or generated prompt files: read/write/bash validators send block text to stderr before exiting with status 2 (`hooks/loop-read-validator.sh:87-96`, `hooks/loop-write-validator.sh:104-111`, `hooks/loop-bash-validator.sh:55-63`), while the Codex stop hook writes rendered review and next-round prompts to files (`hooks/loop-codex-stop-hook.sh:475-488`, `hooks/loop-codex-stop-hook.sh:781-798`).
- gates_or_invariants: The template directory invariant is structural, not content-deep: `validate_template_dir` only requires the root and three subdirectories (`hooks/lib/template-loader.sh:196-204`). `loop-common.sh` deliberately treats that gate as soft by warning and continuing with inline fallbacks (`hooks/lib/loop-common.sh:18-21`). Rendering invariants are: placeholders must be delimited as `{{...}}`; unknown variables are preserved exactly as placeholders (`hooks/lib/template-loader.sh:100-107`); unmatched opening braces are treated literally (`hooks/lib/template-loader.sh:87-93`); and replacement is single-pass, so values containing `{{OTHER_VAR}}` remain literal. The environment prefix `TMPL_VAR_` isolates substitution variables from ambient environment names (`hooks/lib/template-loader.sh:47-54`, `hooks/lib/template-loader.sh:61-66`). The repository’s template tree matches the validator’s required shape: `prompt-template/block`, `prompt-template/codex`, and `prompt-template/claude`; observed template files include block messages, Codex review prompts, and Claude next-round prompt fragments.
- dependencies_and_callers: Runtime dependencies are Bash, `cd`, `pwd`, `cat`, `env`, and `awk`; tests additionally use tools such as `grep`, `sed`, `find`, `sort`, and `mktemp`. The main transitive import path is `hooks/lib/loop-common.sh`, which is sourced by `loop-read-validator.sh`, `loop-write-validator.sh`, `loop-edit-validator.sh`, `loop-bash-validator.sh`, and `loop-codex-stop-hook.sh`. Common message helpers in `loop-common.sh` depend on `load_and_render_safe` for todos/prompt/state/summary/goal-tracker block text (`hooks/lib/loop-common.sh:86-134`, `hooks/lib/loop-common.sh:180-191`). Direct callers include read validation for wrong file location, round, and directory (`hooks/loop-read-validator.sh:87-132`); write validation for summary location, round, and directory (`hooks/loop-write-validator.sh:104-164`); bash validation for blocked pushes (`hooks/loop-bash-validator.sh:55-63`); and the stop hook for incomplete todos, large files, git cleanliness, unpushed commits, missing summaries, goal tracker initialization, Codex review prompts, Claude next-round prompts, append-only notes, and footer/request fragments (`hooks/loop-codex-stop-hook.sh:430-488`, `hooks/loop-codex-stop-hook.sh:781-814`). Template reference validation scans hook scripts for `load_template`, `load_and_render`, and `load_and_render_safe` calls and verifies referenced files exist (`tests/test-template-references.sh:66-110`).
- edge_cases_or_failure_modes: Missing template files produce an empty stdout string from `load_template`; normal loader mode emits a warning to stderr, while safe mode suppresses it and uses fallback text (`hooks/lib/template-loader.sh:26-32`, `hooks/lib/template-loader.sh:161-171`). Missing or incomplete template directories fail `validate_template_dir`, but the hook system continues with inline fallbacks via `loop-common.sh`, so user-facing enforcement can still work (`hooks/lib/template-loader.sh:193-207`, `hooks/lib/loop-common.sh:18-21`). Plain `load_and_render` on a missing template returns empty output, which can be risky when a caller expects feedback; `tests/test-error-scenarios.sh:48-58` records that behavior as safe-but-empty, and `tests/test-error-scenarios.sh:114-120` calls out the empty-feedback risk. Variable assignment parsing splits only on the first `=`, so values may contain additional equals signs, but names containing unusual characters are not validated before becoming environment variable suffixes (`hooks/lib/template-loader.sh:50-54`). Multiline values are stored in environment variables and emitted by AWK; covered by tests, but shell/environment size limits still apply for very large prompt bodies. Placeholder syntax permits whatever appears between braces during render, while comprehensive tests separately validate expected uppercase placeholder syntax in template files (`tests/test-templates-comprehensive.sh:118-135`). `append_template` has no safe fallback and always prints base content plus whatever `load_template` returns; missing appended templates can therefore add a blank segment and warning unless callers suppress stderr.
