### TEMPLIFY_PROMPT-HZ-022 `file` `scripts/portable-timeout.sh`
- cursor: `[_]`
- core_role: Portable command timeout abstraction for the RLCR Codex review path. The file is intended to be sourced by Bash callers and provides `run_with_timeout`, letting the loop enforce a per-review time budget across macOS and Linux where GNU `timeout` may not be installed.
- algorithmic_behavior: On source, it detects a timeout implementation once via `detect_timeout_impl` and stores the result in `TIMEOUT_IMPL` (`scripts/portable-timeout.sh:10-29`). Detection priority is `gtimeout`, GNU `timeout`, `python3`, `python`, then `none` (`scripts/portable-timeout.sh:11-26`). `run_with_timeout` accepts timeout seconds followed by a command argv, preserves command arguments in a Bash array, and dispatches to the selected implementation (`scripts/portable-timeout.sh:33-70`). Native implementations call `gtimeout "$timeout_secs" "${cmd[@]}"` or `timeout "$timeout_secs" "${cmd[@]}"` (`scripts/portable-timeout.sh:38-45`). Python fallback runs `subprocess.run(sys.argv[1:], timeout=<seconds>)` and maps `TimeoutExpired` to exit code `124`, matching GNU timeout convention (`scripts/portable-timeout.sh:47-61`). If no implementation is available, it emits a stderr warning and executes the command without a timeout (`scripts/portable-timeout.sh:64-68`).
- inputs_outputs_state: Inputs are `timeout_secs` and command argv supplied to `run_with_timeout` (`scripts/portable-timeout.sh:31-36`). Output is the child command’s exit status, except Python timeout returns `124` and Python wrapper exceptions return `1` (`scripts/portable-timeout.sh:53-60`). State is limited to the exported shell variable `TIMEOUT_IMPL`, computed at source time and available to the sourcing script (`scripts/portable-timeout.sh:29`, `scripts/portable-timeout.sh:73-76`). The main caller sources it from `hooks/loop-codex-stop-hook.sh` and then pipes the Codex prompt into `run_with_timeout "$CODEX_TIMEOUT" codex exec ...`, redirecting stdout/stderr to cache files (`hooks/loop-codex-stop-hook.sh:550-568`, `hooks/loop-codex-stop-hook.sh:593-597`). The timeout value originates from loop state `codex_timeout`, then env/default fallback (`hooks/loop-codex-stop-hook.sh:286-301`), and setup writes that state from `--codex-timeout` (`scripts/setup-rlcr-loop.sh:223-230`).
- gates_or_invariants: The wrapper maintains argv boundary safety by using a Bash array, so commands with spaces in arguments remain separate arguments (`scripts/portable-timeout.sh:36`, `scripts/portable-timeout.sh:40-44`). It validates GNU `timeout` by requiring `timeout --version` to succeed, avoiding non-GNU/BSD ambiguity (`scripts/portable-timeout.sh:13-19`). It does not validate `timeout_secs` itself; initial CLI setup validates `--codex-timeout` as numeric (`scripts/setup-rlcr-loop.sh:125-134`), but the stop hook’s state read only defaults `CODEX_TIMEOUT` and does not revalidate it (`hooks/loop-codex-stop-hook.sh:286-301`, `hooks/loop-codex-stop-hook.sh:310-313`). The caller gates Codex failure after execution: any nonzero exit, including timeout `124`, becomes a blocking hook response with diagnostic paths (`hooks/loop-codex-stop-hook.sh:635-652`).
- dependencies_and_callers: Depends on Bash features (`local`, arrays, `[[ ]]`, source semantics), `command -v`, optional `gtimeout`, optional GNU `timeout`, optional `python3`/`python`, and Python’s standard `subprocess` module. Direct repository caller is `hooks/loop-codex-stop-hook.sh`, which uses this file when present and otherwise defines a simpler inline fallback (`hooks/loop-codex-stop-hook.sh:550-568`). The user-facing timeout configuration is documented in `README.md` as `--codex-timeout <SECONDS>` with default `5400` (`README.md:190-198`).
- edge_cases_or_failure_modes: If `TIMEOUT_IMPL` is `none`, long-running Codex review can run indefinitely despite a configured timeout, with only a warning (`scripts/portable-timeout.sh:64-68`). If the state file is manually corrupted with a nonnumeric or shell-significant `codex_timeout`, this script interpolates the value directly into the Python `-c` source at `timeout=$timeout_secs`, which can cause Python syntax/runtime errors or worse shell parsing issues before Python runs (`scripts/portable-timeout.sh:49-61`); the setup path validates, but the runtime state path does not revalidate. Detection happens once at source time, so installing/removing `gtimeout` later in the same shell will not update `TIMEOUT_IMPL` unless re-sourced (`scripts/portable-timeout.sh:29`). Python fallback does not terminate a spawned process tree as GNU timeout may with additional options; it kills/manages the direct subprocess behavior provided by `subprocess.run(timeout=...)`, which may leave grandchildren depending on the invoked command. Native timeout paths pass only seconds and command, with no explicit signal/grace-period options (`scripts/portable-timeout.sh:39-45`).
- validation_or_tests: No dedicated test file was found for this script in the inspected references. Behavioral validation is indirect through `hooks/loop-codex-stop-hook.sh`: it logs the configured timeout, runs Codex through `run_with_timeout`, records the exit code, and blocks the loop on nonzero exit (`hooks/loop-codex-stop-hook.sh:578-599`, `hooks/loop-codex-stop-hook.sh:635-652`).
