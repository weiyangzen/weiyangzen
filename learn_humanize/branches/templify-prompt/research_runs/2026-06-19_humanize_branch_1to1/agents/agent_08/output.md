### TEMPLIFY_PROMPT-HZ-008 `directory` `hooks/lib`
- cursor: `[_]`
- core_role: `hooks/lib` is the shared hook-library layer for the RLCR loop. It centralizes template loading/rendering and common loop/path validation helpers used by the read, write, edit, bash, and stop hooks. The directory has two recursive descendants: `hooks/lib/loop-common.sh` and `hooks/lib/template-loader.sh`; there are no deeper child files.
- algorithmic_behavior: `loop-common.sh` sources `template-loader.sh` and initializes `TEMPLATE_DIR` from the plugin root unless already overridden (`hooks/lib/loop-common.sh:11-20`). It then exposes common operations: find the active loop directory by newest timestamped child with `state.md` (`hooks/lib/loop-common.sh:27-44`), parse `current_round` from YAML-like frontmatter (`hooks/lib/loop-common.sh:48-58`), normalize strings (`hooks/lib/loop-common.sh:60-63`), classify round artifacts by filename suffix (`hooks/lib/loop-common.sh:65-84`), detect protected files and loop-local paths (`hooks/lib/loop-common.sh:137-153`), detect shell commands that mutate protected files (`hooks/lib/loop-common.sh:155-178`), and render standardized block messages for todos, prompt files, state files, summary writes, and goal-tracker restrictions (`hooks/lib/loop-common.sh:86-135`, `hooks/lib/loop-common.sh:180-192`). `template-loader.sh` implements the template subsystem: compute `prompt-template` path from `hooks/lib` (`hooks/lib/template-loader.sh:11-16`), load a file or emit warning plus empty content when absent (`hooks/lib/template-loader.sh:21-33`), render `{{VAR}}` placeholders in a single pass with `awk` and environment variables (`hooks/lib/template-loader.sh:43-117`), compose load+render and append operations (`hooks/lib/template-loader.sh:121-146`), provide fallback-aware rendering (`hooks/lib/template-loader.sh:155-188`), and validate required template subdirectories (`hooks/lib/template-loader.sh:193-207`).
- inputs_outputs_state: Inputs are shell hook JSON-derived paths/commands supplied by caller scripts, loop state files under `.humanize-loop.local`, template names under `prompt-template`, and variable assignments such as `CORRECT_PATH=...`. Outputs are stdout/stderr strings, shell return codes, and rendered prompt/block text consumed by hook decisions. The library itself does not persist state; it reads filesystem state, especially newest loop directory and `state.md`, and computes transient classifications. `render_template` maps `VAR=value` pairs to `TMPL_VAR_VAR` environment entries, scans each input line for `{{...}}`, replaces known names, preserves unknown placeholders, and prints the rendered content (`hooks/lib/template-loader.sh:47-116`).
- gates_or_invariants: The library enforces several invariants indirectly through callers. Round file names must match `round-[0-9]+-(summary|prompt|todos).md` (`hooks/lib/loop-common.sh:67-84`). Only the newest loop directory can be active, preventing older loop directories from being revived after abnormal exits (`hooks/lib/loop-common.sh:23-44`). Template directory validity requires `block`, `codex`, and `claude` subdirectories (`hooks/lib/template-loader.sh:201-204`), but failure is a warning at source time rather than a hard hook failure (`hooks/lib/loop-common.sh:18-20`). Fallback-aware rendering guarantees block hooks can still return useful text when a template is missing or renders empty (`hooks/lib/template-loader.sh:155-188`). The template renderer’s important invariant is single-pass replacement: values containing `{{OTHER_VAR}}` are emitted literally and not rescanned (`hooks/lib/template-loader.sh:35-42`, `hooks/lib/template-loader.sh:56-114`).
- dependencies_and_callers: `loop-common.sh` depends on `template-loader.sh`, `sed`, `grep`, `sort`, `head`, `tr`, `cat`, `awk`, and the `prompt-template` tree. It is sourced by `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, `hooks/loop-bash-validator.sh`, and `hooks/loop-codex-stop-hook.sh` (`hooks/loop-read-validator.sh:14-17`, `hooks/loop-write-validator.sh:15-17`, `hooks/loop-edit-validator.sh:14-16`, `hooks/loop-bash-validator.sh:14-16`, `hooks/loop-codex-stop-hook.sh:47-50`). Those callers use the shared helpers to block todos-file access (`hooks/loop-read-validator.sh:36-39`, `hooks/loop-write-validator.sh:37-40`, `hooks/loop-edit-validator.sh:36-39`, `hooks/loop-bash-validator.sh:119-122`), block prompt/state/goal-tracker changes, and validate active loop paths/rounds. `template-loader.sh` is also directly sourced by template tests (`tests/test-template-loader.sh:10-12`, `tests/test-templates-comprehensive.sh:15-17`, `tests/test-error-scenarios.sh:10-12`).
- edge_cases_or_failure_modes: `find_active_loop` only checks the newest child directory; if the newest timestamp lacks `state.md`, older valid loops are ignored by design (`hooks/lib/loop-common.sh:23-44`). `get_current_round` defaults to `0` when parsing fails or the frontmatter key is missing (`hooks/lib/loop-common.sh:51-58`), which can make validators behave as Round 0. `is_in_humanize_loop_dir` only matches paths containing `.humanize-loop.local/`, so alternate path spellings or symlinks are not normalized (`hooks/lib/loop-common.sh:149-153`). `command_modifies_file` is regex-based and covers common redirection/edit tools but can miss more complex shell constructs, aliases, quoted paths with spaces, or indirect writes (`hooks/lib/loop-common.sh:155-178`). `load_template` treats missing files as empty output with warning rather than failure (`hooks/lib/template-loader.sh:26-32`), which is why safe fallbacks matter. `render_template` preserves unknown placeholders (`hooks/lib/template-loader.sh:100-107`) and treats malformed `{{` without `}}` as literal text (`hooks/lib/template-loader.sh:87-94`). ### TEMPLIFY_PROMPT-HZ-038 `file` `prompt-template/block/incomplete-todos.md`
- cursor: `[_]`
- core_role: This block template defines the user-facing stop gate when the loop detects unfinished native TodoWrite items. It is part of the prompt/template contract that prevents the stop hook from escalating to Codex review while local task state is still incomplete.
- algorithmic_behavior: The template renders a Markdown block headed `Incomplete Todos Detected` and injects `{{INCOMPLETE_LIST}}` as the concrete todo list (`prompt-template/block/incomplete-todos.md:1-5`). It then directs the agent to complete all remaining todos, mark each todo completed with TodoWrite, and only then write the summary and stop (`prompt-template/block/incomplete-todos.md:7-10`). The final instruction prevents Codex review until todos are finished (`prompt-template/block/incomplete-todos.md:12`). At runtime, `hooks/loop-codex-stop-hook.sh` performs this quick check before expensive Codex review (`hooks/loop-codex-stop-hook.sh:61-65`), runs `hooks/check-todos-from-transcript.py`, and when that script exits `1`, renders this template through `load_and_render_safe` with `INCOMPLETE_LIST` (`hooks/loop-codex-stop-hook.sh:67-84`). The hook then returns JSON with `"decision": "block"` and the rendered template as `"reason"` (`hooks/loop-codex-stop-hook.sh:86-94`).
- inputs_outputs_state: The template’s sole placeholder input is `INCOMPLETE_LIST`, which is derived from `TODO_RESULT` after dropping the marker line (`hooks/loop-codex-stop-hook.sh:74-84`). `check-todos-from-transcript.py` reads hook JSON from stdin, uses `transcript_path`, scans the latest TodoWrite call in Claude Code transcript formats, and emits incomplete todos as lines like `- [status] content` (`hooks/check-todos-from-transcript.py:87-122`). Output is rendered Markdown included in the stop-hook JSON `reason`; the surrounding hook output is a block decision and a system message telling the agent incomplete todos were detected (`hooks/loop-codex-stop-hook.sh:86-94`). The template does not mutate state directly; it causes a stop attempt to be blocked so the agent updates TodoWrite state and retries.
- gates_or_invariants: The gate fires only when the todo checker exists and exits `1` (`hooks/loop-codex-stop-hook.sh:67-75`). No active loop means the stop hook exits before this check, so the template is only relevant inside `.humanize-loop.local` loop execution (`hooks/loop-codex-stop-hook.sh:55-59`). The todo checker treats missing transcript, invalid hook JSON, missing `transcript_path`, no TodoWrite calls, or all todos completed as pass/exit `0` (`hooks/check-todos-from-transcript.py:87-107`, `hooks/check-todos-from-transcript.py:124-125`). The invariant expressed by the template is strict: all remaining todos must be completed before summary and stop, and Codex review must not proceed while incomplete todos remain (`prompt-template/block/incomplete-todos.md:7-12`).
- dependencies_and_callers: Direct caller is `hooks/loop-codex-stop-hook.sh`, via `load_and_render_safe "$TEMPLATE_DIR" "block/incomplete-todos.md"` (`hooks/loop-codex-stop-hook.sh:79-84`). The data producer is `hooks/check-todos-from-transcript.py`, which recognizes TodoWrite entries in three transcript shapes: assistant message content blocks, alternative message entries, and direct `tool_use` entries (`hooks/check-todos-from-transcript.py:48-83`). Rendering depends on `hooks/lib/template-loader.sh` and the `TEMPLATE_DIR` initialized by `hooks/lib/loop-common.sh` (`hooks/loop-codex-stop-hook.sh:47-50`). The broader policy is consistent with validator templates that block file-based todos and tell the agent to use native TodoWrite instead (`hooks/loop-read-validator.sh:36-39`, `hooks/loop-write-validator.sh:37-40`, `hooks/loop-edit-validator.sh:36-39`, `hooks/loop-bash-validator.sh:119-122`, `prompt-template/block/todos-file-access.md:5`).
- edge_cases_or_failure_modes: If `check-todos-from-transcript.py` is absent, the stop hook skips this gate entirely (`hooks/loop-codex-stop-hook.sh:67-69`). If hook input is invalid JSON or lacks `transcript_path`, the checker exits `0`, allowing the stop process to continue (`hooks/check-todos-from-transcript.py:87-97`). If transcript parsing encounters bad JSONL lines, those lines are ignored (`hooks/check-todos-from-transcript.py:27-36`). The checker only uses the latest non-empty TodoWrite todo list; if the latest list omits older incomplete items, the gate follows the latest list (`hooks/check-todos-from-transcript.py:25-84`). Any status other than literal `completed` is incomplete, including `pending`, `in_progress`, empty, or unexpected status values (`hooks/check-todos-from-transcript.py:109-116`). If this template is missing or empty, `load_and_render_safe` falls back to an inline shorter incomplete-todos message (`hooks/loop-codex-stop-hook.sh:79-84`; fallback text is in `hooks/loop-codex-stop-hook.sh:79-82`).
