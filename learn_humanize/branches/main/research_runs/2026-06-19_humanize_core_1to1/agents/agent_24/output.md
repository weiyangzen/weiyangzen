### HZ-054 `file` `scripts/cancel-rlcr-loop.sh`
- cursor: `[_]`
- core_role: Runtime workflow script that terminates an active RLCR loop by creating a cancel signal and renaming the active loop state file to `cancel-state.md`. It is part of the loop state machine, not a passive utility.
- algorithmic_behavior: Parses only `--force` and help; unknown options exit 3 (`scripts/cancel-rlcr-loop.sh:24`-`63`). It resolves the project root through shared loop code, sets `LOOP_BASE_DIR` to `.humanize/rlcr`, and finds the newest active loop via `find_active_loop` (`scripts/cancel-rlcr-loop.sh:69`-`91`). It selects the active state file in priority order `state.md`, `methodology-analysis-state.md`, then `finalize-state.md` (`scripts/cancel-rlcr-loop.sh:103`-`121`). It extracts `current_round` and `max_iterations` from the chosen state (`scripts/cancel-rlcr-loop.sh:127`-`133`), blocks unforced Finalize cancellation (`scripts/cancel-rlcr-loop.sh:139`-`150`), then touches `.cancel-requested`, removes transient markers, and moves the active state to `cancel-state.md` (`scripts/cancel-rlcr-loop.sh:156`-`166`).
- inputs_outputs_state: Inputs are CLI args and the current repo’s `.humanize/rlcr` loop directories. Outputs are status tokens/messages: `NO_LOOP`, `NO_ACTIVE_LOOP`, `FINALIZE_NEEDS_CONFIRM`, `CANCELLED`, `CANCELLED_METHODOLOGY_ANALYSIS`, or `CANCELLED_FINALIZE` (`scripts/cancel-rlcr-loop.sh:93`-`120`, `139`-`149`, `172`-`184`). State transitions: active state file becomes `cancel-state.md`; `.cancel-requested` authorizes the corresponding state move; `.humanize/.pending-session-id` and `.methodology-exit-reason` are removed (`scripts/cancel-rlcr-loop.sh:156`-`166`).
- gates_or_invariants: Uses `set -euo pipefail` (`scripts/cancel-rlcr-loop.sh:18`). Finalize phase requires `--force` to cancel because completion is otherwise imminent (`scripts/cancel-rlcr-loop.sh:139`-`149`). Cancel intentionally operates globally, without `session_id` filtering, because the slash command has no session ID and should cancel the project’s active loop (`scripts/cancel-rlcr-loop.sh:80`-`91`). The script relies on `find_active_loop` zombie-loop protection: without a session filter, only the newest loop directory is considered active (`hooks/lib/loop-common.sh:308`-`357`).
- dependencies_and_callers: Sources `hooks/lib/loop-common.sh`, which provides `resolve_project_root` and `find_active_loop` (`scripts/cancel-rlcr-loop.sh:69`-`77`). `resolve_project_root` uses `CLAUDE_PROJECT_DIR` first, then `git rev-parse --show-toplevel`, and deliberately avoids `pwd` fallback (`hooks/lib/project-root.sh:5`-`16`, `41`-`53`). The command wrapper exposes only the normal and forced invocations in `commands/cancel-rlcr-loop.md:3`-`14`. `skills/humanize/SKILL.md` and `skills/humanize-rlcr/SKILL.md` also reference this cancellation entry point.
- edge_cases_or_failure_modes: No project root exits 3 (`scripts/cancel-rlcr-loop.sh:73`-`77`). No active loop exits 1 (`scripts/cancel-rlcr-loop.sh:93`-`97`, `117`-`120`). Missing `current_round` or `max_iterations` becomes `?` for reporting (`scripts/cancel-rlcr-loop.sh:131`-`133`). An unforced Finalize cancel exits 2 and asks for `--force` (`scripts/cancel-rlcr-loop.sh:139`-`149`). Because cancel uses unfiltered `find_active_loop`, a newer terminal loop blocks stale older active-loop revival; that behavior is documented in `hooks/lib/loop-common.sh:310`-`313`.
- validation_or_tests: Session-independent cancel behavior is tested in `tests/test-session-id.sh:368`-`383`, including state rename to `cancel-state.md`. Zombie-loop protection is tested in `tests/test-session-id.sh:386`-`429`. Cancel authorization is also constrained by hook-side signal validation: `is_cancel_authorized` requires `.cancel-requested`, rejects command substitution/chaining/extra args/symlink aliases, and only authorizes moving active state files to `cancel-state.md` (`hooks/lib/loop-common.sh:1033`-`1250`). `tests/test-cancel-signal-file.sh:1`-`10` states the positive/negative signal-file contract and then covers allowed move with signal, blocked move without signal, wrong destination, injection, symlink aliases, and related cases.
