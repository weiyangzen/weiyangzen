- cursor: `[_]`
- core_role: Event-to-validator routing table for the RLCR hook state machine. It does not implement validation logic itself; it declares which command hook runs for each Claude Code event/tool surface. The file is still core because it is the dispatch contract that makes the validators and stop loop reachable.
- algorithmic_behavior: `UserPromptSubmit` routes every submitted prompt through `hooks/loop-plan-file-validator.sh` (`hooks/hooks.json:4-12`), enforcing loop state/schema, branch consistency, and plan-file immutability before normal interaction continues. `PreToolUse` routes tool-specific requests to validators for `Write`, `Edit`, `Read`, and `Bash` (`hooks/hooks.json:14-50`). `Stop` routes exit attempts through `hooks/loop-codex-stop-hook.sh` with a 7200 second timeout (`hooks/hooks.json:52-60`), which is the round-review transition driver.
- inputs_outputs_state: Input is Claude hook runtime metadata: event names plus tool matcher names. Output is not data emitted by this JSON; output is the selected command invocation using `${CLAUDE_PLUGIN_ROOT}` expansion. State is external in `.humanize/rlcr/<timestamp>/state.md` or `finalize-state.md`, read by the configured scripts rather than by the JSON. The dispatcher creates these state transitions indirectly: prompt submission checks loop invariants, tool use may allow/block operations, and stop attempts may advance review/finalize/end-loop behavior.
- gates_or_invariants: The invariant is one validator per protected tool category: `Write` cannot bypass `loop-write-validator.sh`, `Edit` cannot bypass `loop-edit-validator.sh`, `Read` cannot bypass `loop-read-validator.sh`, and `Bash` cannot bypass shell-command protections (`hooks/hooks.json:14-50`). Stop review is explicitly long-running via `timeout: 7200`, matching expensive Codex review behavior (`hooks/hooks.json:56-58`). The file assumes `CLAUDE_PLUGIN_ROOT` is correctly set; if it is unset or wrong, the hook table points to nonexistent commands.
- dependencies_and_callers: Loaded by the Claude plugin hook system. It depends on executable scripts under `hooks/`: `loop-plan-file-validator.sh`, `loop-write-validator.sh`, `loop-edit-validator.sh`, `loop-read-validator.sh`, `loop-bash-validator.sh`, and `loop-codex-stop-hook.sh`. Those scripts share `hooks/lib/loop-common.sh`, which provides JSON input validation, active-loop discovery, strict state parsing, round filename parsing, allowlists, path classifiers, and template-backed block messages. The stop hook also uses `scripts/portable-timeout.sh` for bounded git/Codex operations.
- edge_cases_or_failure_modes: This JSON is declarative and has no fallback behavior if a target script is missing, non-executable, or the environment variable resolves incorrectly. A malformed JSON file would disable or break hook registration; I validated it with `jq empty hooks/hooks.json` successfully. A matcher typo would silently leave a protected tool unguarded. The stop hook timeout is high enough for review, but if Codex review legitimately exceeds 7200 seconds the hook would be interrupted by the host runtime.
- validation_or_tests: Direct syntax validation passed with `jq empty hooks/hooks.json`. The scripts wired here are covered by broader test suites listed in `tests/run-all-tests.sh`, including template, validator pattern, plan-file, allowlist, finalize-phase, cancel, monitor, and robustness suites (`tests/run-all-tests.sh:32-63`). This assigned file itself is config, so useful validation is JSON parsing plus integration tests of the referenced hooks.

### ROBUST_EDGE_TEST_FIND_AND_RESOLVE-HZ-079 `file` `prompt-template/block/wrong-round-number.md`
- cursor: `[_]`
- core_role: Block-message template for the RLCR summary-file round gate. It tells the agent that it is attempting to write or edit a summary for a stale/future round and points to the correct active-round path.
- algorithmic_behavior: The template renders a short markdown error: title, attempted action and stale `round-{{CLAUDE_ROUND}}-{{FILE_TYPE}}.md`, current round, correct path, and the invariant “Do NOT increment the round number yourself” (`prompt-template/block/wrong-round-number.md:1-7`). It is passive content, but its placeholders are populated at the exact point the Write/Edit validators reject an operation.
- inputs_outputs_state: Inputs are five required render variables: `ACTION`, `CLAUDE_ROUND`, `FILE_TYPE`, `CURRENT_ROUND`, and `CORRECT_PATH` (`prompt-template/block/wrong-round-number.md:3-5`). Output is markdown written to stderr by pre-tool validators through `load_and_render_safe`. It does not modify state; it represents a failed transition where a proposed write/edit is blocked and the RLCR state remains at `CURRENT_ROUND`.
- gates_or_invariants: The enforced invariant is that agents must target the current active summary file, not manually advance or backfill round filenames. In `hooks/loop-write-validator.sh`, the gate extracts `CLAUDE_ROUND` from the requested summary filename and blocks when it differs from strict-parsed `STATE_CURRENT_ROUND`, unless the file is explicitly allowlisted (`hooks/loop-write-validator.sh:200-217`). In `hooks/loop-edit-validator.sh`, the same mismatch check blocks edits to wrong-round summaries (`hooks/loop-edit-validator.sh:127-151`). The shared filename parser only recognizes nonnegative numeric `round-N-(summary|prompt|todos).md` forms (`hooks/lib/loop-common.sh:312-322`), and the allowlist permits a small set of historical bootstrap files (`hooks/lib/loop-common.sh:324-345`).
- dependencies_and_callers: Rendered by `load_and_render_safe` in `hooks/loop-write-validator.sh` (`hooks/loop-write-validator.sh:210-215`) and `hooks/loop-edit-validator.sh` (`hooks/loop-edit-validator.sh:144-149`). Covered by `tests/test-templates-comprehensive.sh` integration rendering (`tests/test-templates-comprehensive.sh:497-514`) and by the smaller template-loader integration test in `tests/test-template-loader.sh` (`tests/test-template-loader.sh:151-164`). It depends on `hooks/lib/template-loader.sh` single-pass substitution semantics, where missing variables remain as placeholders and variable values are not rescanned.
- edge_cases_or_failure_modes: If any render variable is omitted, the template loader leaves the `{{VAR}}` token visible, which is useful for diagnosis but degrades the block message. The round mismatch gate only fires after validators identify the target as a summary file and can extract a numeric round; malformed names that do not match `round-[0-9]+-summary.md` may flow to directory/path validation instead. Write/Edit allowlisted paths bypass the wrong-round block for specific bootstrap summaries. Negative `STATE_CURRENT_ROUND` is accepted by strict state parsing but cannot match the nonnegative filename regex, so a negative current round would make normal round filenames mismatch in downstream gate logic.
- validation_or_tests: Direct file inspection confirms all placeholders use the uppercase `{{VAR}}` syntax expected by the comprehensive template test. `tests/test-templates-comprehensive.sh` asserts the rendered real template contains the title, stale `round-3-summary` reference, current round `5`, and `/tmp/round-5-summary.md` (`tests/test-templates-comprehensive.sh:506-510`). `tests/test-template-loader.sh` similarly validates real loading/rendering of this template and checks `round-3-summary.md` plus current round emphasis (`tests/test-template-loader.sh:151-164`).
