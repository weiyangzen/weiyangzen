- core_role: Prompt/block template for the RLCR stop gate when a worker/Claude attempts to exit without writing the required work summary. It is a small but core review-contract artifact because the stop hook cannot progress to Codex review without a summary file.
- algorithmic_behavior: The template renders a blocking instruction with one variable, `{{SUMMARY_FILE}}`, and enumerates the required summary contents: implemented work, files created/modified, tests added/passed, and remaining items (`prompt-template/block/work-summary-missing.md:1-16`). It does not perform logic itself; it is loaded by the stop hook when summary-file existence check fails.
- inputs_outputs_state: Input is `SUMMARY_FILE`, which the stop hook computes as either `$LOOP_DIR/finalize-summary.md` in Finalize Phase or `$LOOP_DIR/round-${CURRENT_ROUND}-summary.md` in normal rounds (`hooks/loop-codex-stop-hook.sh:525-530`). Output is a rendered markdown reason in the JSON block response. The state transition is intentionally “blocked but unchanged”: the hook returns a block decision and keeps the active loop state so the user can write the summary and retry exit.
- gates_or_invariants: The invariant is that every stop attempt must have a summary file before review proceeds. `hooks/loop-codex-stop-hook.sh` checks `[[ ! -f "$SUMMARY_FILE" ]]`, renders this template through `load_and_render_safe`, and sets the system message to either “Finalize Phase - summary file missing” or “Summary file missing for round N” (`hooks/loop-codex-stop-hook.sh:532-546`). The required summary fields in the template mirror setup’s final completion instructions at `scripts/setup-rlcr-loop.sh:745-754`.
- dependencies_and_callers: Called by `hooks/loop-codex-stop-hook.sh` via `load_and_render_safe "$TEMPLATE_DIR" "block/work-summary-missing.md"` (`hooks/loop-codex-stop-hook.sh:536-540`). Rendering depends on `hooks/lib/template-loader.sh`, where `load_and_render_safe` falls back if a template is missing or renders empty (`hooks/lib/template-loader.sh:167-198`). Template directory initialization and validation happen through `hooks/lib/loop-common.sh`, which sources the template loader and validates the `prompt-template` directory.
- edge_cases_or_failure_modes: If the template is missing or empty, the stop hook uses a fallback message that still includes `{{SUMMARY_FILE}}` (`hooks/loop-codex-stop-hook.sh:536-540`), so the block remains operational. The template only checks existence through its caller; it does not validate summary quality, content completeness, or test evidence. Finalize Phase swaps the expected file from round summary to `finalize-summary.md`, avoiding false blocking on normal round filenames (`hooks/loop-codex-stop-hook.sh:525-530`).
- validation_or_tests: `tests/test-finalize-phase.sh` explicitly checks that Finalize Phase blocks exit when `finalize-summary.md` is missing and the output mentions summary (`tests/test-finalize-phase.sh:370-385`). Template loading and missing-template fallback behavior are covered by `tests/test-templates-comprehensive.sh` and `tests/test-error-scenarios.sh`; `tests/test-template-references.sh` validates many common templates but its hard-coded common-template list does not include this specific file (`tests/test-template-references.sh:149-167`), so this template’s strongest direct coverage is through the stop-hook/finalize tests and general all-template loading tests.
