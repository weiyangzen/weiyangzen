### IMPROVE_PR_LOOP-HZ-025 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: PreToolUse Bash validator for RLCR and PR-loop control surfaces. It prevents shell-command bypasses of Write/Edit validators for loop-owned files and protects local `.humanize` state from accidental staging while an RLCR or PR loop is active. It is registered for Bash tool calls in `hooks/hooks.json:42-48`.
- algorithmic_behavior: The script reads hook JSON from stdin, validates JSON/tool shape, rejects deeply nested input, and exits early for non-Bash tools (`hooks/loop-bash-validator.sh:24-47`). It resolves `PROJECT_ROOT` from `CLAUDE_PROJECT_DIR` or `pwd`, discovers the newest active RLCR loop and newest active PR loop, and allows all commands if neither exists (`hooks/loop-bash-validator.sh:54-67`). With an RLCR loop, it chooses `finalize-state.md` over `state.md`, parses state strictly, reads `current_round` and `push_every_round`, blocks `git push` when push-every-round is false, blocks `.humanize` git-add attempts, then applies file-write gates for RLCR state, plan backup, goal tracker, prompt, summary, and todo files (`hooks/loop-bash-validator.sh:74-395`). With an active PR loop, it blocks state and read-only PR-loop round files (`hooks/loop-bash-validator.sh:403-443`).
- inputs_outputs_state: Inputs are hook JSON fields `tool_name` and `tool_input.command`, environment `CLAUDE_PROJECT_DIR`, active loop directories under `.humanize/rlcr` and `.humanize/pr-loop`, loop state frontmatter, and templates via `loop-common.sh`. Outputs are process exit codes: `0` allow, `1` malformed input/state safety failure, `2` command blocked with a rendered explanation to stderr. The script itself does not mutate loop state; it only reads state and enforces gates.
- gates_or_invariants: The input gate rejects null bytes, invalid UTF-8, invalid JSON, missing `tool_name`, missing Bash `command`, and JSON depth over 30 through shared helpers (`hooks/lib/loop-common.sh:59-133`). RLCR state parsing is fail-closed: malformed or missing required fields block the operation (`hooks/loop-bash-validator.sh:81-85`, `hooks/lib/loop-common.sh:255-310`). State/finalize-state files are loop-system-owned and protected both as write destinations and as `mv`/`cp` sources, with a narrow cancel exception (`hooks/loop-bash-validator.sh:139-328`). `goal-tracker.md` has a round-sensitive gate: round 0 points the agent to Write/Edit, later rounds require a summary-based update request (`hooks/loop-bash-validator.sh:349-357`). Todo files are only allowed for `round-1-todos.md` or `round-2-todos.md` when the full path is inside the active loop directory; same basename in another root is blocked (`hooks/loop-bash-validator.sh:385-393`). PR-loop state and generated comment/prompt/review/feedback files are read-only (`hooks/loop-bash-validator.sh:407-442`).
- dependencies_and_callers: Depends on `hooks/lib/loop-common.sh` for active-loop discovery, strict state parsing, path/write detection, cancel authorization, `.humanize` git-add detection, lowercasing, and rendered block messages; `hooks/lib/template-loader.sh` supplies fallback-safe rendering (`hooks/lib/template-loader.sh:170-203`). It uses `jq`, `grep`, `sed`, `find/ls/sort/head`, `git check-ignore`, and standard shell tools. Related helper behavior includes active RLCR loop discovery by newest timestamp directory with `state.md` or `finalize-state.md` (`hooks/lib/loop-common.sh:156-174`), PR-loop discovery by newest timestamp with `state.md` (`hooks/lib/loop-common.sh:798-814`), `.humanize` add blocking (`hooks/lib/loop-common.sh:930-1026`), and command modification patterns (`hooks/lib/loop-common.sh:1071-1096`).
- edge_cases_or_failure_modes: If no active RLCR or PR loop exists, the hook allows all Bash commands, including commands that would otherwise be considered risky (`hooks/loop-bash-validator.sh:64-67`). PR-loop protection includes a bare `state.md` check, so an unrelated `state.md` write can be blocked while a PR loop is active; this is intentional to catch relative-path bypasses from inside the loop directory (`hooks/loop-bash-validator.sh:411-415`). The source-file `mv`/`cp` detector strips leading redirections and splits shell operators with placeholder handling for forms such as `&>`, `&>>`, and `2>&1` (`hooks/loop-bash-validator.sh:174-284`). Shell wrappers `sh -c` and `bash -c` get a separate bypass check (`hooks/loop-bash-validator.sh:306-328`). Cancel authorization is deliberately narrow: it requires `.cancel-requested`, rejects command substitution/backticks/newlines/operators, enforces the state-to-cancel-state move shape, and rejects symlink sources (`hooks/lib/loop-common.sh:533-575`). `command_modifies_file` has a documented limitation for multi-source `cp file1 file2 goal-tracker.md` style cases in `tests/test-bash-validator-patterns.sh:165-168`.
- validation_or_tests: `tests/test-bash-validator-patterns.sh:69-194` exercises redirection, `tee`, in-place editors, `mv/cp/rm`, `dd`, `truncate`, `exec`, false positives, and state/summary patterns. `tests/test-allowlist-validators.sh:290-380` verifies the todo allowlist and same-basename/different-root blocking. `tests/test-pr-loop-hooks.sh:193-205` verifies PR-loop Bash state protection works even without an RLCR loop. `tests/robustness/test-hook-system-robustness.sh:336-379` verifies state and goal-tracker blocking while unrelated shell commands pass through to the outer sandbox. `tests/test-cancel-signal-file.sh:1259-1343` covers direct cancel authorization cases.
