### IMPROVE_PR_LOOP-HZ-038 `file` `scripts/poll-pr-reviews.sh`
- cursor: `[_]`
- core_role: Runtime PR-loop poller. It converts GitHub API issue comments, review comments, and review records into one normalized JSON response filtered by watched bots and timestamp.
- algorithmic_behavior: Parses `<pr_number> --after <timestamp> --bots <bot1,bot2>`, validates required fields and numeric PR number, checks `gh` and `jq`, resolves the correct repository for PR operations, maps bot aliases to GitHub authors, fetches three comment streams with retries, normalizes them, filters by timestamp and author regex, and emits JSON with `comments`, `bots_responded`, `has_new_comments`, and `comment_count` (scripts/poll-pr-reviews.sh:23-102, 108-187, 193-326).
- inputs_outputs_state: Inputs are PR number, ISO-like timestamp string, and comma-separated bot names (scripts/poll-pr-reviews.sh:7-8, 45-53). Output is a JSON object; `has_new_comments` is emitted as a boolean derived from filtered comment count, not a string (scripts/poll-pr-reviews.sh:317-326). Temporary state lives in a `mktemp -d` directory and is removed by trap on exit (scripts/poll-pr-reviews.sh:193-199).
- gates_or_invariants: Hard gates reject missing arguments, unknown options, nonnumeric PR numbers, missing `gh`, or missing `jq` (scripts/poll-pr-reviews.sh:23-116). Fork PR handling is an invariant: first try current repo, then parent repo, and fail if the PR cannot be found in either (scripts/poll-pr-reviews.sh:122-150). API calls are retried three times, but fetch failures intentionally degrade to empty arrays so polling can continue (scripts/poll-pr-reviews.sh:200-235).
- dependencies_and_callers: Called by `hooks/pr-loop-stop-hook.sh` as `POLL_SCRIPT="$PLUGIN_ROOT/scripts/poll-pr-reviews.sh"` during stop-hook polling (hooks/pr-loop-stop-hook.sh:801-812, 960-965). It depends on GitHub CLI API endpoints for issue comments, PR review comments, and PR reviews (scripts/poll-pr-reviews.sh:240-294). The stop hook consumes `.bots_responded[]` and maps authors back to bot names (hooks/pr-loop-stop-hook.sh:967-982).
- edge_cases_or_failure_modes: Approval-only PR reviews with empty body are preserved by substituting a review-state placeholder so approvals are not silently dropped (scripts/poll-pr-reviews.sh:276-289). Timestamps are compared lexically in jq using `.created_at >= $after`, so correctness assumes GitHub ISO 8601 UTC-compatible strings (scripts/poll-pr-reviews.sh:299-305). Bot regex construction trims spaces and escapes brackets, but arbitrary unescaped regex metacharacters in bot names other than brackets could affect matching (scripts/poll-pr-reviews.sh:175-187). Warnings from retry failures go to stderr, which callers often suppress.
- validation_or_tests: Fixture-backed tests assert approval-only reviews appear and timestamp filtering excludes earlier comments (tests/test-pr-loop-hooks.sh:1322-1411). Robustness tests assert help, missing-argument rejection, valid JSON shape, slow API tolerance, and API failure output with boolean `has_new_comments:false` (tests/robustness/test-pr-loop-api-robustness.sh:619-843).
