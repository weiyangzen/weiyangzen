### FEATURE__CODEX_BYPASS_SANDBOX_ENV-HZ-035 `file` `scripts/check-pr-reviewer-status.sh`
- cursor: `[_]`
- core_role: Runtime classifier for PR-loop startup state. It inspects the whole PR, not only comments after the latest commit, and emits JSON that tells setup/stop-hook logic whether bots have reviewed and whether those reviews are stale relative to the latest commit.
- algorithmic_behavior: The script parses `<pr_number> --bots <bot1,bot2>` with hard errors for missing PR number, missing `--bots`, unknown options, or multiple PR numbers (`scripts/check-pr-reviewer-status.sh:49-85`). It maps configured bot names to GitHub authors, with a special Codex mapping to `chatgpt-codex-connector[bot]` and default `<bot>[bot]` for others (`scripts/check-pr-reviewer-status.sh:91-100`). It resolves the PR base repository for fork support by trying the current repo, then the parent repo, before falling back to current repo with a warning (`scripts/check-pr-reviewer-status.sh:113-140`). It fetches latest commit SHA/date via `gh pr view --json headRefOid,commits` and sorts commits by `committedDate` before selecting the latest (`scripts/check-pr-reviewer-status.sh:142-151`). It fetches issue comments, inline review comments, and PR review submissions with pagination, normalizes each to `{author, created_at, body[, state]}`, and combines them with `jq -s 'add // []'` (`scripts/check-pr-reviewer-status.sh:153-168`).
- inputs_outputs_state: Inputs are GitHub CLI data, `jq`, the configured bot list, and the sourced portable timeout wrapper (`scripts/check-pr-reviewer-status.sh:35-40`). The script does not persist state. Output is JSON containing `case`, `reviewers_commented`, `reviewers_missing`, `latest_commit_sha`, `latest_commit_at`, `newest_review_at`, and `has_commits_after_reviews` (`scripts/check-pr-reviewer-status.sh:259-275`). Internal state arrays track commented, missing, and stale reviewers (`scripts/check-pr-reviewer-status.sh:174-178`).
- gates_or_invariants: Freshness is per-bot, not global. For each configured bot with comments/reviews, it finds that bot’s newest timestamp and marks the bot stale if `LATEST_COMMIT_AT > BOT_NEWEST` (`scripts/check-pr-reviewer-status.sh:179-205`). Classification is: case 1 when no bot commented; case 2 when some but not all commented and none of the commented bots is stale; case 5 when some commented and at least one commented bot is stale; case 3 when all commented and none stale; case 4 when all commented and at least one bot is stale (`scripts/check-pr-reviewer-status.sh:211-245`). `has_commits_after_reviews` is set true only for cases 4 and 5.
- dependencies_and_callers: `scripts/setup-pr-loop.sh` calls this script during initialization, parses the JSON, writes `startup_case`, `latest_commit_sha`, and `latest_commit_at` into `state.md`, and uses cases 4/5 to post or reuse a trigger comment after the latest commit (`scripts/setup-pr-loop.sh:437-463`, `scripts/setup-pr-loop.sh:615-634`). `hooks/pr-loop-stop-hook.sh` re-evaluates startup case dynamically after polling so state can change as bot comments arrive (`hooks/pr-loop-stop-hook.sh:1553-1588`). The stop hook also maps startup cases into trigger requirements: round 0 cases 1/2/3 do not require a fresh trigger, cases 4/5 do, and any round greater than 0 requires one (`hooks/pr-loop-stop-hook.sh:612-643`).
- edge_cases_or_failure_modes: If PR base repo resolution fails, the script falls back to `CURRENT_REPO`; if that is empty, later `gh` operations can fail. Failure to fetch commit info is fatal (`scripts/check-pr-reviewer-status.sh:142-148`), while failures fetching comments/review comments/reviews degrade to empty arrays (`scripts/check-pr-reviewer-status.sh:156-165`). Timestamp comparison is lexical string comparison, which is valid for normalized UTC ISO-8601 timestamps but fragile for non-normalized dates. Bot list parsing does not trim whitespace, so `--bots "claude, codex"` would create a bot named `" codex"`. The comment in the header describes case 5 as all commented like case 4 (`scripts/check-pr-reviewer-status.sh:27`), but implementation uses case 5 for partial commented plus stale review (`scripts/check-pr-reviewer-status.sh:223-230`).
- validation_or_tests: `tests/test-pr-loop-system.sh` directly validates case 1 no comments (`tests/test-pr-loop-system.sh:152-171`), case 2 partial comments (`tests/test-pr-loop-system.sh:173-193`), PR review submissions counting as reviewer activity (`tests/test-pr-loop-system.sh:236-250`), case 3 all commented (`tests/test-pr-loop-system.sh:493-507`), case 4 all commented plus newer commit (`tests/test-pr-loop-system.sh:1315-1341`), and case 5 partial commented plus newer commit (`tests/test-pr-loop-system.sh:1347-1378`). The mock `gh` supports `MOCK_GH_LATEST_COMMIT_AT`, `MOCK_GH_HEAD_SHA`, fixture-backed issue/review comments, and PR reviews (`tests/mocks/gh:17-22`, `tests/mocks/gh:66-69`, `tests/mocks/gh:154-188`).

- inputs_outputs_state: Inputs are `PR_NUMBER`, `BOT_MENTION_STRING`, and `RESOLVE_PATH`. - gates_or_invariants: The important rules are explicit: do not modify `.humanize/pr-loop/` state files, always push fixes so remote bots can review them, use the correct comment format to trigger bots, and address all valid reviewer concerns (`prompt-template/pr-loop/round-0-task-has-comments.md:30-35`).
