- algorithmic_behavior: The hook reads stdin into `INPUT` but does not inspect it (`hooks/loop-plan-file-validator.sh:26-27`). It derives `PROJECT_ROOT` from `CLAUDE_PROJECT_DIR` or `pwd` (`hooks/loop-plan-file-validator.sh:13-14`), finds the newest active loop under `.humanize/rlcr` with `find_active_loop` (`hooks/loop-plan-file-validator.sh:29-35`), and exits silently when no active loop exists. State parsing is strict via `parse_state_file_strict`; malformed state fails closed with stderr and exit 1 (`hooks/loop-plan-file-validator.sh:44-48`). After parsing, it maps `STATE_PLAN_TRACKED`, `STATE_PLAN_FILE`, and `STATE_START_BRANCH` into local variables (`hooks/loop-plan-file-validator.sh:50-53`). It then applies three main checks in order: required v1.1.2 fields, current git branch, and plan-file tracking cleanliness.

Outputs are either no output with exit 0 for allow, JSON `{ "decision": "block", "reason": ... }` with exit 0 for policy blocks, or plain stderr with exit 1 for malformed state. It does not mutate files or advance loop state. It observes state transitions indirectly by selecting `finalize-state.md` when the loop is in Finalize Phase.

When `plan_tracked=true`, the plan must still be tracked and have empty `git status --porcelain` output (`hooks/loop-plan-file-validator.sh:123-192`). When `plan_tracked` is anything else, the plan must not be tracked (`hooks/loop-plan-file-validator.sh:193-228`). Timeout exit 124 is treated as a fail-closed block in both tracking paths (`hooks/loop-plan-file-validator.sh:129-137`, `hooks/loop-plan-file-validator.sh:154-162`, `hooks/loop-plan-file-validator.sh:199-207`).

### FIX_TOO_STRICT_RULE_AND_ENHANCE_PLAN_GEN-HZ-053 `file` `hooks/lib/template-loader.sh`
- cursor: `[_]`
- core_role: Shared prompt-template loading and rendering library for RLCR hooks. It centralizes template directory discovery, Markdown template loading, single-pass placeholder substitution, fallback rendering, and template directory validation.
- algorithmic_behavior: `get_template_dir` converts a caller-supplied `hooks/lib` path into the plugin `prompt-template` directory (`hooks/lib/template-loader.sh:24-31`). `load_template` cats an existing template or emits a warning plus empty stdout for missing files (`hooks/lib/template-loader.sh:33-48`). `render_template` accepts content plus `VAR=value` arguments, exports them as `TMPL_VAR_*` environment entries, and uses an awk scanner to replace `{{VAR}}` spans in one pass (`hooks/lib/template-loader.sh:50-132`). The scanner appends literal text until `{{`, finds the next `}}`, replaces only if the exact name exists in its environment-derived lookup table, and preserves unknown placeholders (`hooks/lib/template-loader.sh:84-128`). `load_and_render` combines load and render only when content is non-empty (`hooks/lib/template-loader.sh:134-147`). `append_template` concatenates a base string with another template (`hooks/lib/template-loader.sh:149-161`). `load_and_render_safe` suppresses missing-template warnings and renders a fallback if loading or rendering yields empty output (`hooks/lib/template-loader.sh:167-203`). `validate_template_dir` requires the root plus `block`, `codex`, and `claude` subdirectories (`hooks/lib/template-loader.sh:205-222`).
- inputs_outputs_state: Inputs are filesystem template paths, raw template content, and `VAR=value` substitution arguments. Outputs are rendered Markdown on stdout, warnings/errors on stderr, and function return codes. The library has no durable state; it uses local shell arrays and a temporary `TMPL_VAR_` environment namespace for each render call.
- gates_or_invariants: Documented placeholder syntax is `{{VARIABLE_NAME}}` with uppercase letters, numbers, and underscores (`hooks/lib/template-loader.sh:7-13`), but runtime replacement is exact-match and does not enforce the uppercase syntax itself. The main rendering invariant is single-pass substitution: inserted variable values are not scanned again, preventing placeholder injection from values such as review content (`hooks/lib/template-loader.sh:54-57`, `hooks/lib/template-loader.sh:116-121`). Missing variables remain as their original placeholders (`hooks/lib/template-loader.sh:119-122`). Missing templates are non-fatal by default, and safe rendering falls back rather than aborting (`hooks/lib/template-loader.sh:176-203`). Directory validation is advisory for `loop-common.sh`, which warns and continues with inline fallbacks.
- dependencies_and_callers: Sourced by `hooks/lib/loop-common.sh:135-140`, making it available to the loop validators and stop hook. `loop-common.sh:142-145` validates the directory but degrades to fallbacks. `hooks/loop-codex-stop-hook.sh:1080-1083` renders the next-round prompt and `hooks/loop-codex-stop-hook.sh:1093-1097` renders the assigned footer template through this library. `hooks/loop-plan-file-validator.sh:64-65` uses it for schema-outdated blocks. Tests source it directly in `tests/test-template-loader.sh`, `tests/test-error-scenarios.sh`, `tests/test-templates-comprehensive.sh`, and `tests/robustness/test-template-stress-robustness.sh`.
- edge_cases_or_failure_modes: Missing files intentionally produce empty content and a warning unless callers redirect stderr or use `load_and_render_safe`. Command substitution in `render_template` can trim trailing newlines from awk output, and final output uses `echo "$content"` (`hooks/lib/template-loader.sh:74-131`), so exact trailing-newline preservation is not guaranteed. Variable assignments without `=` are not rejected. Runtime rendering will process any text between `{{` and `}}` if a matching `TMPL_VAR_...` exists, even though template validation tests enforce uppercase syntax separately. Very large values are bounded by shell/environment limits rather than explicit code limits. Unclosed `{{` is treated literally and scanning resumes after those two characters (`hooks/lib/template-loader.sh:102-109`).
- validation_or_tests: `tests/test-template-loader.sh:41-244` verifies directory discovery, existing/missing template loading, single and multiple substitutions, multiline content, fallback rendering, and directory validation. `tests/test-template-loader.sh:247-360` exercises shell metacharacters literally. `tests/test-error-scenarios.sh:42-180` verifies graceful behavior for missing files/directories, strict-mode survivability, fallback substitution, and validation errors. `tests/robustness/test-template-stress-robustness.sh:82-151` covers 10KB/100KB values/templates and many variables; `tests/robustness/test-template-stress-robustness.sh:218-228` specifically verifies placeholder injection prevention. `tests/test-templates-comprehensive.sh:61-194` loads all templates and validates placeholder syntax across `prompt-template`.

- cursor: `[_]`
- core_role: Footer prompt block appended to the next RLCR round prompt after the review gate decides more work is required. It encodes the worker-facing contract for not bypassing the loop and for producing commit plus summary evidence.
- algorithmic_behavior: The template is static Markdown with one placeholder, `{{NEXT_SUMMARY_FILE}}` (`prompt-template/claude/next-round-footer.md:6-9`). `hooks/loop-codex-stop-hook.sh` appends it after generating the next-round prompt and any post-alignment action-item block (`hooks/loop-codex-stop-hook.sh:1093-1097`). Rendering is done by `load_and_render_safe`, with fallback text that only says to commit and write a summary (`hooks/loop-codex-stop-hook.sh:1093-1097`). In the normal path, the rendered footer tells the next worker not to exit the RLCR loop by lying, editing loop state, or executing `cancel-rlcr-loop` (`prompt-template/claude/next-round-footer.md:4`). It then instructs the worker to optionally use the `code-simplifier` plugin, commit changes, and write the work summary to the rendered summary file path (`prompt-template/claude/next-round-footer.md:6-9`).
- inputs_outputs_state: Input is `NEXT_SUMMARY_FILE`, computed as `$LOOP_DIR/round-${NEXT_ROUND}-summary.md` immediately before rendering (`hooks/loop-codex-stop-hook.sh:1067-1070`). Output is Markdown appended to `$NEXT_PROMPT_FILE`; the template itself does not read or mutate loop state. It participates in the state machine by shaping the next blocked prompt after `current_round` is advanced in the state file (`hooks/loop-codex-stop-hook.sh:1062-1065`) and before the stop hook returns a block response with the new prompt (`hooks/loop-codex-stop-hook.sh:1118-1120`).
- gates_or_invariants: The key invariant is behavioral: the next-round agent must not forge loop completion or alter loop state to escape the RLCR cycle (`prompt-template/claude/next-round-footer.md:4`). The evidence invariant is that each round must leave a commit and a summary at `@{{NEXT_SUMMARY_FILE}}` (`prompt-template/claude/next-round-footer.md:8-9`). Push behavior is intentionally not in this footer; `loop-codex-stop-hook.sh` appends a separate push note only when `push_every_round` is true (`hooks/loop-codex-stop-hook.sh:1099-1106`).
- dependencies_and_callers: Called from `hooks/loop-codex-stop-hook.sh:1093-1097`. Depends on `hooks/lib/template-loader.sh` for single-pass rendering and missing-placeholder preservation. The template directory inventory includes this file under `prompt-template/claude`, which is required by `validate_template_dir` through the `claude` subdirectory gate. Related search results show it is one of the prompt templates loaded by comprehensive template tests.
- edge_cases_or_failure_modes: If this template is missing or empty, the fallback footer omits the anti-bypass warning and `code-simplifier` instruction, reducing the strength of the next-round contract. If `NEXT_SUMMARY_FILE` is not passed, the placeholder remains literal because the renderer preserves unknown variables. Since rendering is single-pass, a malicious or unusual summary-file value containing `{{...}}` would not trigger secondary substitution. The template says `cancel-rlcr-loop` without the leading slash in the anti-exit note, while other user-facing cancellation text often uses `/humanize:cancel-rlcr-loop`; this still names the forbidden operation but is less command-specific.
- validation_or_tests: `tests/test-templates-comprehensive.sh:90-107` loads every Markdown template under `prompt-template`, which includes this footer. `tests/test-templates-comprehensive.sh:118-194` validates placeholder syntax and would catch malformed `{{NEXT_SUMMARY_FILE}}` syntax. There is no narrow test that asserts this exact footer text appears in the next-round prompt; coverage is currently structural/template-level plus indirect rendering through `loop-codex-stop-hook.sh`.
