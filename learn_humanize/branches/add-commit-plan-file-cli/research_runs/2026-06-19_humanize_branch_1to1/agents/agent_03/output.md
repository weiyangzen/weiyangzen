### ADD_COMMIT_PLAN_FILE_CLI-HZ-033 `file` `hooks/lib/template-loader.sh`
- cursor: `[_]`
- core_role: `hooks/lib/template-loader.sh` is the prompt/block-message rendering engine for RLCR hook decisions. It resolves the plugin’s `prompt-template` directory, loads markdown templates, renders `{{VARIABLE_NAME}}` placeholders, and provides safe fallbacks so validators can still return actionable block messages when a template is missing.
- algorithmic_behavior: `get_template_dir` computes the plugin template root by going two levels up from `hooks/lib` and appending `prompt-template` (`hooks/lib/template-loader.sh:11-16`). `load_template` concatenates a template file if present, otherwise emits a warning to stderr and returns an empty string (`hooks/lib/template-loader.sh:21-33`). `render_template` accepts a template string plus `VAR=value` assignments, stores them as environment variables with a `TMPL_VAR_` prefix, then runs a single `awk` pass over the content to replace `{{VAR}}` tokens (`hooks/lib/template-loader.sh:43-117`). The awk algorithm builds a lookup from `ENVIRON`, scans each line for `{{`, finds the next `}}`, extracts the variable name, replaces known placeholders, preserves unknown placeholders, and appends replacement values directly to output without re-scanning them (`hooks/lib/template-loader.sh:59-114`). `load_and_render` composes load then render (`hooks/lib/template-loader.sh:121-132`). `append_template` concatenates base and additional template content (`hooks/lib/template-loader.sh:136-146`). `load_and_render_safe` suppresses missing-template warnings, renders fallback text if the file is absent/empty, and also falls back if render output is empty (`hooks/lib/template-loader.sh:155-188`). `validate_template_dir` requires the root plus `block`, `codex`, and `claude` subdirectories (`hooks/lib/template-loader.sh:193-207`).
- inputs_outputs_state: Inputs are a template directory, a repository-relative template name, a raw template content string, and zero or more `VAR=value` assignments. Outputs are rendered markdown on stdout or warnings/errors on stderr. The file itself has no persistent state; it is sourced by other hook scripts and exports shell functions into their process. Its key state-like invariant is the placeholder namespace: assignments are converted to environment variables named `TMPL_VAR_<VAR>`, keeping substitutions isolated from ambient shell variables (`hooks/lib/template-loader.sh:47-54`, `hooks/lib/template-loader.sh:60-67`).
- gates_or_invariants: The critical invariant is single-pass rendering. Lines `39-42` explicitly describe the injection guard: if a variable value contains another `{{OTHER_VAR}}`, it is not replaced. The awk implementation supports that by never feeding substituted values back into the scanner (`hooks/lib/template-loader.sh:100-110`). Unknown placeholders are preserved rather than removed (`hooks/lib/template-loader.sh:104-107`), allowing templates to remain diagnosable if a caller omitted an assignment. `load_and_render_safe` guarantees non-empty fallback behavior for blocking UX when template files are missing or render empty (`hooks/lib/template-loader.sh:161-187`). `validate_template_dir` is intentionally strict about expected subdirectories but callers such as `loop-common.sh` only warn on failure, preserving graceful degradation (`hooks/lib/loop-common.sh:18-21`).
- dependencies_and_callers: This file depends on POSIX-ish shell tools available in Bash scripts: `cd`, `pwd`, `cat`, `env`, and `awk`. It is sourced by `hooks/lib/loop-common.sh:11-16`, which in turn is sourced by all hook validators and the Stop hook. Direct call sites include block messages in `loop-common.sh`, validator templates in `loop-read-validator.sh`, `loop-write-validator.sh`, `loop-edit-validator.sh`, `loop-bash-validator.sh`, and Codex/Claude prompt generation in `loop-codex-stop-hook.sh`, including the assigned full-alignment review template at `hooks/loop-codex-stop-hook.sh:596-607`.
- edge_cases_or_failure_modes: Missing template files produce empty content from `load_template`; unsafe callers using `load_template` or `load_and_render` directly must handle that, while `load_and_render_safe` provides fallback rendering. A malformed placeholder with `{{` but no closing `}}` treats only the opening braces literally and continues scanning after them (`hooks/lib/template-loader.sh:87-94`). Assignments split at the first `=`, so values may contain `=`, but variable names cannot. Because substitutions pass through environment variables into awk, very large template values are constrained by process environment limits. Placeholder names are not syntactically validated by this function; any text between `{{` and `}}` can be a lookup key, though tests and template validation enforce uppercase placeholder conventions elsewhere.
- validation_or_tests: `tests/test-template-loader.sh` directly sources this file (`tests/test-template-loader.sh:10-13`) and covers template directory resolution, existing/missing load behavior, single and multiple substitutions, multiline rendering, special characters, unknown placeholders, `load_and_render`, `load_and_render_safe`, and `validate_template_dir` (`tests/test-template-loader.sh:41-244`). The same test file specifically verifies placeholder injection prevention: a replacement value containing `{{VAR_B}}` or `{{VAR_A}}` remains literal, including a realistic `REVIEW_CONTENT` scenario (`tests/test-template-loader.sh:488-543`). `tests/test-template-references.sh` checks hook scripts for missing referenced templates and requires critical validators to use the safe rendering path (`tests/test-template-references.sh:51-115`, `tests/test-template-references.sh:172-200`). `tests/test-templates-comprehensive.sh` also validates template syntax and safe rendering behavior; targeted references show checks for malformed placeholders and missing-template fallback (`tests/test-templates-comprehensive.sh:122-174`, `tests/test-templates-comprehensive.sh:446-458`).
