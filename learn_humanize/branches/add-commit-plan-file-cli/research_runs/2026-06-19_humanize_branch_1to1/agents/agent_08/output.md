### ADD_COMMIT_PLAN_FILE_CLI-HZ-038 `file` `prompt-template/block/git-push.md`
- cursor: `[_]`
- core_role: Static block-message template used by the Bash pre-tool guard to deny `git push` during an active RLCR loop unless the loop was started with `--push-every-round`. It is not executable logic by itself, but it is part of the guard contract: when the algorithm blocks a push, this file supplies the user-facing transition instruction.
- algorithmic_behavior: The template has no placeholders and no branching. It emits a heading, states that commits should remain local, and tells the user to start the loop with `--push-every-round` if pushing each round is required (`prompt-template/block/git-push.md:1-9`). The actual gate is in `hooks/loop-bash-validator.sh`: after parsing hook input, it finds the active loop, reads `push_every_round` from `state.md`, lowercases the Bash command, and if `push_every_round` is not exactly `true`, blocks commands whose lowercased text starts with `git push` (`hooks/loop-bash-validator.sh:22-31`, `hooks/loop-bash-validator.sh:36-64`). On block, it calls `load_and_render_safe "$TEMPLATE_DIR" "block/git-push.md" "$FALLBACK"` and exits `2`, the PreToolUse blocking signal (`hooks/loop-bash-validator.sh:55-64`).
- inputs_outputs_state: Inputs are indirect: active loop state at `.humanize-loop.local/<timestamp>/state.md`, specifically `push_every_round`, and the Bash hook JSON command string. The template itself takes no variables. Output is the rendered Markdown block to stderr, followed by validator exit code `2`; no repository or loop state is modified. The state field is created by setup: `--push-every-round` sets `PUSH_EVERY_ROUND=true` during argument parsing, and the generated `state.md` stores `push_every_round: $PUSH_EVERY_ROUND` (`scripts/setup-rlcr-loop.sh:141-143`, `scripts/setup-rlcr-loop.sh:406-418`).
- gates_or_invariants: Default invariant is local-only commits during the loop; README documents `--push-every-round` as opt-in and says the default is local commits (`README.md:199`). With `push_every_round` false or missing, direct `git push` commands are blocked before execution (`hooks/loop-bash-validator.sh:53-64`). With `push_every_round` true, Bash push commands are not blocked by this guard; the stop hook then enforces the opposite completion gate by checking for ahead-of-remote commits and blocking exit until they are pushed (`hooks/loop-codex-stop-hook.sh:402-430`). The stop hook also appends a push reminder to next-round prompts only in the opt-in mode (`hooks/loop-codex-stop-hook.sh:919-925`).
- dependencies_and_callers: Primary caller is `hooks/loop-bash-validator.sh` via `load_and_render_safe` (`hooks/loop-bash-validator.sh:58-63`). Template loading depends on `hooks/lib/loop-common.sh` setting `TEMPLATE_DIR`, which in turn depends on `hooks/lib/template-loader.sh` resolving `prompt-template` from the plugin root (`hooks/lib/loop-common.sh:11-19`, `hooks/lib/template-loader.sh:9-16`). Related sibling template `prompt-template/block/unpushed-commits.md` handles the opt-in mode’s stop-hook exit block, while `prompt-template/claude/push-every-round-note.md` handles next-round prompt reminders (`hooks/loop-codex-stop-hook.sh:417-419`, `hooks/loop-codex-stop-hook.sh:921-925`).
- edge_cases_or_failure_modes: The command regex only matches commands beginning with optional whitespace followed by `git push`; it catches case variants because the command is lowercased, but it can miss wrappers or compound commands such as `command git push`, `env git push`, `cd repo && git push`, aliases, functions, or other tools that push (`hooks/loop-bash-validator.sh:29-31`, `hooks/loop-bash-validator.sh:56-57`). The state comparison requires the parsed value to be exactly `true`; setup writes that exact value, but manually edited or malformed state values fall back to blocking behavior (`hooks/loop-bash-validator.sh:53-55`). If there is no active loop, the validator exits before this gate and does not block pushes (`hooks/loop-bash-validator.sh:36-43`). If the template is missing, `load_and_render_safe` emits the inline fallback instead, so guard behavior remains intact but wording changes (`hooks/loop-bash-validator.sh:58-63`, `hooks/lib/template-loader.sh:155-188`).
- validation_or_tests: `tests/test-template-loader.sh` directly verifies that `load_template` can load `block/git-push.md` and that `load_and_render_safe` prefers the real template over fallback text (`tests/test-template-loader.sh:55-65`, `tests/test-template-loader.sh:211-221`). `tests/test-templates-comprehensive.sh` also checks safe loading of the same template and includes an all-template load/render sweep (`tests/test-templates-comprehensive.sh:455-463`, `tests/test-templates-comprehensive.sh:563-592`). I found template-loading coverage for this file, but not a dedicated behavioral test that feeds Bash hook JSON containing `git push` through `loop-bash-validator.sh`.
