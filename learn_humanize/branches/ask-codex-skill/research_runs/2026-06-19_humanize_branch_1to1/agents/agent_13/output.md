### ASK_CODEX_SKILL-HZ-043 `file` `scripts/portable-timeout.sh`
- cursor: `[_]`
- core_role: Shared runtime timeout abstraction used by setup scripts, PR/RLCR hooks, GitHub API polling, git commands, and Codex invocations to keep the workflow bounded across macOS and Linux.
- algorithmic_behavior: `detect_timeout_impl` selects the first available implementation in priority order: Homebrew `gtimeout`, GNU `timeout`, `python3`, `python`, then `none` (`scripts/portable-timeout.sh:10`, `:11`, `:13`, `:20`, `:22`, `:25`). `TIMEOUT_IMPL` is computed at source time (`:29`). `run_with_timeout` takes seconds plus command/args, dispatches to the selected implementation, preserves command arguments as an array, returns the wrapped command exit code, and normalizes Python timeout expiry to GNU-style exit `124` (`:33`, `:36`, `:38`, `:49`, `:56`). If no implementation exists, it warns and runs the command unbounded (`:64`).
- inputs_outputs_state: Inputs are `timeout_secs` and command argv. Outputs are the command’s stdout/stderr and exit status, with `124` signaling timeout when an implementation supports enforcement. State is a sourced-shell global `TIMEOUT_IMPL`, exported for child visibility (`scripts/portable-timeout.sh:73`, `:76`).
- gates_or_invariants: GNU `timeout` is accepted only if `timeout --version` succeeds, avoiding non-GNU/BSD incompatibility (`scripts/portable-timeout.sh:14`). Python fallback uses `subprocess.run(sys.argv[1:], timeout=N)`, so shell syntax requires explicit `sh -c` from callers. Exit code preservation is part of the contract for non-timeout commands.
- dependencies_and_callers: Direct callers include `scripts/ask-codex.sh` for `codex exec` (`scripts/ask-codex.sh:28`, `:296`), `scripts/setup-pr-loop.sh` for git/gh setup and PR lookup (`scripts/setup-pr-loop.sh:33`, `:244`, `:296`), PR stop hooks for Codex validation (`hooks/pr-loop-stop-hook.sh:56`, `:1342`), RLCR stop hooks for `codex review` and `codex exec` (`hooks/loop-codex-stop-hook.sh:1012`), plan validators, bot reaction checks, reviewer-status checks, and multiple robustness suites.
- edge_cases_or_failure_modes: `none` mode deliberately degrades to no timeout, which can hang on stuck commands (`scripts/portable-timeout.sh:64`). Python fallback returns `1` for wrapper exceptions and may not mimic all signal semantics of GNU timeout. Zero-second timeouts are implementation-dependent. Empty command invocation is treated as an error-like edge case by tests rather than a strict semantic guarantee.
- validation_or_tests: `tests/robustness/test-timeout-robustness.sh` is the direct executable spec. It validates implementation detection, allowed implementation values, quick command success, timeout exit `124`, argument preservation, exit-code preservation, pipeline use via `sh -c`, short timeout elapsed time, zero timeout tolerance, large output, rapid cycles, nonexistent command nonzero exit, signal behavior, fallback-chain validity, subshell use, and exported `TIMEOUT_IMPL` (`tests/robustness/test-timeout-robustness.sh:31`, `:62`, `:79`, `:89`, `:120`, `:139`, `:157`, `:170`, `:188`, `:221`, `:235`, `:259`).
