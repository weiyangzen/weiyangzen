- algorithmic_behavior: The prompt defines a three-step decision procedure: quickly inspect repository docs/structure/technologies, compare the draft against repository concepts/files/features, then emit a strict verdict prefix of either `RELEVANT:` or `NOT_RELEVANT:` at `agents/draft-relevance-checker.md:14-29`. - gates_or_invariants: The key invariant is verdict prefix stability: callers can branch on `RELEVANT:` versus `NOT_RELEVANT:`. The decision threshold is deliberately lenient, so only drafts with no reasonable repository connection should be rejected. The agent is also bounded to repository-inspection tools, avoiding write-side effects.
- dependencies_and_callers: `commands/gen-plan.md` Phase 2 explicitly invokes `humanize:draft-relevance-checker` with the draft content, asks it to inspect README/CLAUDE/main files, and continue only if the verdict is relevant at `commands/gen-plan.md:50-73`. - edge_cases_or_failure_modes: Ambiguous drafts are intentionally accepted, which reduces false negatives but may let weakly related drafts proceed. A malformed verdict prefix would break the caller’s Phase 2 branching. If repo docs are missing, the prompt still tells the agent to inspect structure and main files, so relevance can be inferred from code paths and features.
- validation_or_tests: Covered structurally by `tests/test-gen-plan.sh`; there is no deterministic content-level fixture proving semantic classifications. The command contract is further documented in `commands/gen-plan.md:56-65`, which is the caller-side acceptance contract for the verdict format.
