- cursor: `[_]`
- core_role: Runtime cancellation command for active PR loops. It converts a `.humanize/pr-loop/<timestamp>/state.md` active loop into a terminal `cancel-state.md` and creates `.cancel-requested` so hooks can observe the cancellation intent.
- algorithmic_behavior: Parses only `--force`, `-h`, and `--help`; unknown options exit 3 (`scripts/cancel-pr-loop.sh:24-66`). Determines `PROJECT_ROOT` from `CLAUDE_PROJECT_DIR` or `pwd`, then uses `.humanize/pr-loop` as the loop base (`scripts/cancel-pr-loop.sh:72-76`). It selects the newest loop directory by lexicographic descending sort (`ls -1d ... | sort -r | head -1`) and requires `state.md` to be present (`scripts/cancel-pr-loop.sh:76-98`). It extracts `current_round`, `max_iterations`, and `pr_number` with grep/sed/tr, defaulting missing values to `?` (`scripts/cancel-pr-loop.sh:104-112`). Cancellation is a two-step state transition: `touch .cancel-requested`, then `mv state.md cancel-state.md` (`scripts/cancel-pr-loop.sh:118-122`).
- inputs_outputs_state: Inputs are command-line flags, `CLAUDE_PROJECT_DIR`, current directory, and the newest `.humanize/pr-loop/*/state.md`. Outputs are status tokens and human-readable messages: `NO_LOOP`, `NO_ACTIVE_LOOP`, or `CANCELLED` (`scripts/cancel-pr-loop.sh:78-81`, `95-97`, `128-130`). State outputs are `.cancel-requested` and `cancel-state.md`; active `state.md` is removed by rename.
- gates_or_invariants: A PR loop is cancellable only if the newest PR loop directory exists and contains `state.md`. This script intentionally does not touch `.humanize/rlcr`, as documented in help (`scripts/cancel-pr-loop.sh:54-56`). `--force` is parsed but has no behavioral effect (`scripts/cancel-pr-loop.sh:28-30`, help at `39-40`). It does not run the stricter `is_cancel_authorized` hook gate directly; that gate is for validating tool commands in hook context (`hooks/lib/loop-common.sh:964-1144`).
- dependencies_and_callers: Exposed by `commands/cancel-pr-loop.md`, which allows only this script and this script with `--force`, and maps `NO_LOOP`/`NO_ACTIVE_LOOP`/`CANCELLED` to user-facing responses (`commands/cancel-pr-loop.md:3-19`). Documented in the humanize skill command reference (`skills/humanize/SKILL.md:127-131`) and usage docs. The PR stop hook also checks `.cancel-requested` (`hooks/pr-loop-stop-hook.sh:953` from search evidence).
- edge_cases_or_failure_modes: If no loop directories exist, exits 1 with `NO_LOOP`; if the newest directory has no active `state.md`, exits 1 with `NO_ACTIVE_LOOP` and does not scan older active directories. This matches zombie-loop protection but can surprise users if a newer terminal/stale directory masks an older active one. Metadata extraction is tolerant; missing fields become `?`. `mv` can fail under permissions/races because `set -euo pipefail` is active.
- validation_or_tests: PR loop cancellation is covered by `tests/test-pr-loop-scripts.sh` around help/no-loop/cancel behavior (`tests/test-pr-loop-scripts.sh:169-243` from search evidence). Cancel-command security surrounding `.cancel-requested` and `cancel-state.md` is broadly specified in `tests/robustness/test-cancel-security-robustness.sh` tests 1-26 (`tests/robustness/test-cancel-security-robustness.sh:39-392`), though those primarily exercise `is_cancel_authorized`.
