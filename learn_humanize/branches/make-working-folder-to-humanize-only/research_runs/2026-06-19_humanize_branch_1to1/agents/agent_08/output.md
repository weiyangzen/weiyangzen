- core_role: `hooks/lib` is the shared shell-library layer for RLCR hook algorithms. It contains `hooks/lib/loop-common.sh` and `hooks/lib/template-loader.sh`; all loop validators and the Codex stop hook source it rather than duplicating active-loop detection, round parsing, path classification, command mutation detection, state termination, and prompt-template rendering.
- algorithmic_behavior: `loop-common.sh` initializes its own directory from `BASH_SOURCE`, sources `template-loader.sh`, derives `TEMPLATE_DIR`, and warns while continuing if template validation fails (`hooks/lib/loop-common.sh:11-21`). Its main algorithms are: newest active loop lookup that only accepts a newest child directory containing `state.md` (`hooks/lib/loop-common.sh:27-44`); YAML-frontmatter `current_round` extraction with default `0` (`hooks/lib/loop-common.sh:48-58`); lowercase/path/round-file helpers (`hooks/lib/loop-common.sh:60-84`, `137-153`); standard block-message construction through `load_and_render_safe` (`hooks/lib/loop-common.sh:86-135`, `185-197`); shell-command write detection for protected files using a fixed regex family for redirection, `tee`, in-place editors, `mv`, `cp`, `rm`, `dd`, `truncate`, `printf`, and `exec` (`hooks/lib/loop-common.sh:155-183`); and loop termination by renaming `state.md` to `<reason>-state.md` after reason validation (`hooks/lib/loop-common.sh:199-232`). `template-loader.sh` computes `prompt-template` from the plugin root (`hooks/lib/template-loader.sh:11-16`), loads templates or returns empty with a warning (`hooks/lib/template-loader.sh:21-33`), performs single-pass `{{VAR}}` substitution via environment-backed `awk` scanning (`hooks/lib/template-loader.sh:35-117`), composes load/render helpers (`hooks/lib/template-loader.sh:119-188`), and validates that `prompt-template/block`, `prompt-template/codex`, and `prompt-template/claude` exist (`hooks/lib/template-loader.sh:190-207`).
- inputs_outputs_state: Inputs are hook JSON decoded by callers, `.humanize/rlcr` loop directories, `state.md` frontmatter, file paths, lowercased shell commands, template names, fallback strings, and `VAR=value` substitutions. Outputs are stdout values for active loop path, current round, rendered messages, and predicates by exit status; warnings/errors are emitted on stderr for missing templates, invalid template dirs, and invalid loop-end attempts. State mutation is intentionally narrow: only `end_loop` mutates loop state by moving `state.md` to one of `complete-state.md`, `cancel-state.md`, `maxiter-state.md`, `stop-state.md`, or `unexpected-state.md` (`hooks/lib/loop-common.sh:211-231`).
- gates_or_invariants: Active loops are recognized only under `.humanize/rlcr` and only when the newest directory has `state.md`, which intentionally prevents older “zombie” state files from being revived (`hooks/lib/loop-common.sh:23-44`). `end_loop` rejects any reason outside `complete|cancel|maxiter|stop|unexpected` (`hooks/lib/loop-common.sh:211-219`). Template rendering is single-pass so placeholder-like text injected through a value is not recursively substituted (`hooks/lib/template-loader.sh:39-42`, `56-116`). Template validation is non-fatal in `loop-common.sh`, preserving hook behavior through inline fallbacks (`hooks/lib/loop-common.sh:18-21`).

### MAKE_WORKING_FOLDER_TO_HUMANIZE_ONLY-HZ-038 `file` `prompt-template/block/git-push.md`
- cursor: `[_]`
- core_role: This is the user-facing block template for the RLCR loop’s default “local commits only” gate. It defines the denial message shown when an agent tries to run `git push` without the loop having been started with `--push-every-round`.
- algorithmic_behavior: The template itself is static Markdown with no placeholders. It declares “Git Push Blocked,” states that current commits should stay local, and gives the enabling command form `/humanize:start-rlcr-loop plan.md --push-every-round` (`prompt-template/block/git-push.md:1-9`). The executable gate is in `hooks/loop-bash-validator.sh`: after finding an active loop and reading `push_every_round` from `state.md`, it blocks commands matching leading `git push` when the state value is not exactly `true`, renders this template through `load_and_render_safe`, writes the reason to stderr, and exits with code `2` (`hooks/loop-bash-validator.sh:49-64`).
- inputs_outputs_state: Inputs are the active loop state field `push_every_round`, the Bash tool command string lowercased by `to_lower`, and the template file loaded as `block/git-push.md`. Output is the block message on stderr and a validator exit code of `2` when blocked; otherwise no output and exit `0`. This file has no state transition by itself, but it participates in the state-driven transition where `push_every_round: false` or missing means “push blocked,” and `push_every_round: true` means the bash validator does not apply this block.
- gates_or_invariants: The invariant is local-only by default. The public command option documents `--push-every-round` as requiring push after each round, while the default is “commits stay local” (`README.md:188-200`). Setup parses `--push-every-round` and sets `PUSH_EVERY_ROUND=true` (`scripts/setup-rlcr-loop.sh:141-143`); initial and next-round prompts add push instructions only when this flag is true (`scripts/setup-rlcr-loop.sh:574-579`, `hooks/loop-codex-stop-hook.sh:919-925`). The bash gate checks only commands beginning with optional whitespace plus `git push`, so non-push git commands are outside this block (`hooks/loop-bash-validator.sh:55-64`).
- dependencies_and_callers: Loaded through `load_and_render_safe "$TEMPLATE_DIR" "block/git-push.md"` in `hooks/loop-bash-validator.sh:58-63`; `TEMPLATE_DIR` comes from `hooks/lib/loop-common.sh:15-21` and `hooks/lib/template-loader.sh:11-16`. Related positive push instruction is a separate template, `prompt-template/claude/push-every-round-note.md`, used by the stop hook when pushing is enabled (`hooks/loop-codex-stop-hook.sh:919-925`). The setup script has an inline equivalent note for round 0 (`scripts/setup-rlcr-loop.sh:574-579`).
- edge_cases_or_failure_modes: If `prompt-template/block/git-push.md` is missing or empty, the bash validator falls back to an inline “Git Push Blocked” message (`hooks/loop-bash-validator.sh:58-62`; fallback mechanism in `hooks/lib/template-loader.sh:155-188`). The state comparison is literal: only `push_every_round: true` disables the block, while missing, uppercase, extra values, or other spellings block pushes (`hooks/loop-bash-validator.sh:53-55`). The command detector matches a simple leading `git push`; aliases, wrapper scripts, `command git push`, subshells, or `git -C path push` are not covered by this specific regex.
