### GENERAL_REFACTOR_AND_REVIEW-HZ-016 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: PreToolUse Bash gate for the RLCR loop. It prevents shell commands from bypassing the stricter Read/Write/Edit hook model by blocking mutations to loop-owned state surfaces and selected round artifacts. It is registered as the Bash matcher in `hooks/hooks.json:42-48`, so it executes before Bash tool calls.
- algorithmic_behavior: The hook reads JSON from stdin, extracts `.tool_name`, and exits immediately unless the tool is Bash (`hooks/loop-bash-validator.sh:22-30`). It locates the newest active loop under `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize/rlcr` via `find_active_loop` (`hooks/loop-bash-validator.sh:36-42`; helper at `hooks/lib/loop-common.sh:58-79`). If no active loop exists, Bash is allowed. With an active loop, it parses `state.md` frontmatter through `parse_state_file` to obtain `current_round` and `push_every_round` (`hooks/loop-bash-validator.sh:45-58`; helper defaults at `hooks/lib/loop-common.sh:110-136`). It lowercases the raw command and tests it against command mutation regexes from `command_modifies_file` (`hooks/lib/loop-common.sh:257-285`).
- inputs_outputs_state: Input is hook JSON on stdin with `tool_name` and `tool_input.command`. Inputs from repo/session state include `CLAUDE_PROJECT_DIR`, active `.humanize/rlcr/<loop>/state.md`, and prompt templates under `prompt-template/`. Output is no stdout on allow, or a rendered block/fallback on stderr and exit code `2` on rejection. It does not mutate state. It consumes state fields `current_round` and `push_every_round`, defaulting `current_round=0`, `max_iterations=10`, and `push_every_round=false` when absent (`hooks/lib/loop-common.sh:131-135`).
- gates_or_invariants: Non-Bash tools are ignored (`hooks/loop-bash-validator.sh:25-27`). Git push is blocked unless `push_every_round` is true (`hooks/loop-bash-validator.sh:59-68`). Any command detected as modifying `state.md` is blocked (`hooks/loop-bash-validator.sh:76-79`). Plan backups under `.humanize/rlcr/.../plan.md` are blocked (`hooks/loop-bash-validator.sh:87-91`). `goal-tracker.md` mutations are blocked in all rounds when attempted via Bash: round 0 redirects the agent to Write/Edit, later rounds direct the agent to a summary-file update request (`hooks/loop-bash-validator.sh:95-108`). Prompt files matching `round-N-prompt.md` are blocked (`hooks/loop-bash-validator.sh:116-119`). Summary file Bash writes are blocked and redirected to the active round summary path (`hooks/loop-bash-validator.sh:126-129`). Todos files are mostly blocked, but full-path writes to active-loop `round-1-todos.md` and `round-2-todos.md` are allowed by the explicit active-loop path check (`hooks/loop-bash-validator.sh:136-143`).
- dependencies_and_callers: Depends on `jq`, `find/ls/sort/head`, `grep`, `sed`, `tr`, Bash regex behavior, and shared functions from `hooks/lib/loop-common.sh` (`hooks/loop-bash-validator.sh:14-16`). It shares message templates through `load_and_render_safe` in `loop-common.sh`, including `block/goal-tracker-modification.md` via `goal_tracker_blocked_message` (`hooks/lib/loop-common.sh:287-299`). It coordinates with `loop-write-validator.sh` and `loop-edit-validator.sh`, which also block post-round-0 goal tracker writes through the same message helper (`hooks/loop-write-validator.sh:113-120`; `hooks/loop-edit-validator.sh:98-105`). Setup creates round 0 as the only direct initialization phase for the goal tracker (`scripts/setup-rlcr-loop.sh:605-615`), which explains why Bash is disallowed for goal tracker writes even in round 0.
- edge_cases_or_failure_modes: The active-loop finder only checks the newest timestamp-named loop directory and ignores older loops even if they still contain `state.md` (`hooks/lib/loop-common.sh:58-60`, `70-78`). If the newest loop has invalid or missing state content but still has `state.md`, the parser silently defaults some fields, which can make `push_every_round` false and `current_round` zero. Mutation detection is regex-based over the command string rather than shell AST parsing; covered cases include redirection, `tee`, in-place `sed/awk/perl`, `mv/cp/rm`, `dd`, `truncate`, and fd redirection (`hooks/lib/loop-common.sh:264-277`). Known test-documented limitation: multi-source `cp file1 file2 goal-tracker.md` is not detected because the pattern expects one source and one destination (`tests/test-bash-validator-patterns.sh:165-168`). Variable indirection such as `echo x > $FILE` is intentionally not detected by literal filename matching (`tests/test-bash-validator-patterns.sh:170-172`). Todos allowlist enforcement for Bash requires the active loop full path and only allows round 1 or round 2 todos (`hooks/loop-bash-validator.sh:136-143`), so generic same-basename writes are blocked.
- validation_or_tests: Pattern-level tests are in `tests/test-bash-validator-patterns.sh`, covering redirection (`75-80`), `tee` (`89-93`), in-place editors (`102-107`), file ops (`116-121`), other mutators (`130-133`), false positives for read-only commands (`142-152`), and state/summary patterns (`181-194`).
