- cursor: `[_]`
- core_role: Shared RLCR hook library. It centralizes state-field constants, active-loop discovery, state frontmatter parsing, round-file recognition, protected-path classification, block-message rendering, Bash write-detection patterns, and terminal state renaming.
- algorithmic_behavior: On source, it loads `template-loader.sh`, computes `TEMPLATE_DIR`, and warns rather than failing if template validation fails so hook messages can fall back inline (`hooks/lib/loop-common.sh:46-56`). `find_active_loop` checks only the newest timestamp directory under `.humanize/rlcr/` and returns it only when `state.md` exists, intentionally avoiding older zombie loops (`hooks/lib/loop-common.sh:58-79`). `parse_state_file` extracts YAML frontmatter and populates `STATE_*` variables for plan tracking, branch, plan path, current round, maximum iterations, push setting, Codex model/effort/timeout, with defaults for round, max iterations, and push behavior (`hooks/lib/loop-common.sh:96-137`). Round-file helpers classify `round-N-summary.md`, `round-N-prompt.md`, and `round-N-todos.md`, and extract `N` portably via `sed` (`hooks/lib/loop-common.sh:144-163`). `end_loop` validates an exit reason and renames `state.md` to `<reason>-state.md` (`hooks/lib/loop-common.sh:301-334`).
- inputs_outputs_state: Inputs include loop base paths, state file paths, candidate file paths, lowercased commands, and target file regex patterns. Outputs are stdout strings for active loop/current round/rendered block messages, shell return codes for predicates, global `STATE_*` variables from parsing, and filesystem state mutation only in `end_loop` where `state.md` is renamed. Constants define the accepted state schema fields (`plan_tracked`, `start_branch`, `plan_file`, `current_round`, `max_iterations`, `push_every_round`, `codex_model`, `codex_effort`, `codex_timeout`) and Codex terminal markers `COMPLETE`/`STOP` (`hooks/lib/loop-common.sh:15-40`).
- gates_or_invariants: Active loop detection is newest-directory-only and requires literal `state.md` (`hooks/lib/loop-common.sh:58-79`). Protected loop files include `goal-tracker.md`, `state.md`, and paths under `.humanize/rlcr/` (`hooks/lib/loop-common.sh:239-255`). `is_allowlisted_file` permits only four bootstrap-era files in the active loop directory: `round-1-todos.md`, `round-2-todos.md`, `round-0-summary.md`, and `round-1-summary.md` (`hooks/lib/loop-common.sh:165-186`). `command_modifies_file` treats redirection, append, `tee`, in-place `sed`/`awk`/`perl`, `mv`, `cp`, `rm`, `dd`, `truncate`, `printf >`, and file-descriptor redirection as modification attempts (`hooks/lib/loop-common.sh:257-285`). `end_loop` only accepts `complete`, `cancel`, `maxiter`, `stop`, or `unexpected` (`hooks/lib/loop-common.sh:313-321`).
- dependencies_and_callers: Direct dependency is `hooks/lib/template-loader.sh`, especially `get_template_dir`, `validate_template_dir`, and `load_and_render_safe` (`hooks/lib/loop-common.sh:46-56`, `hooks/lib/template-loader.sh:23-49`, `hooks/lib/template-loader.sh:157-186`). Callers include `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, `hooks/loop-bash-validator.sh`, `hooks/loop-plan-file-validator.sh`, and `hooks/loop-codex-stop-hook.sh`. The read/write/edit validators use active loop, round parsing, allowlists, and protected-path messages to block stale/wrong round files and protected artifacts; the Bash validator uses `command_modifies_file` to prevent bypassing tool validators; the stop hook uses `parse_state_file`, terminal marker constants, and `end_loop` to advance or end the RLCR state machine (`hooks/loop-codex-stop-hook.sh:46-88`, `hooks/loop-codex-stop-hook.sh:916-970`).
- edge_cases_or_failure_modes: `find_active_loop` ignores older directories even if they contain `state.md`, so a newer closed/corrupt directory can suppress an older still-active loop by design (`hooks/lib/loop-common.sh:58-79`). `parse_state_file` is line-oriented YAML parsing, not a full YAML parser; it strips double quotes only on `start_branch` and `plan_file`, while booleans/numerics are whitespace-stripped (`hooks/lib/loop-common.sh:117-129`). - cursor: `[_]`
- core_role: Small Claude prompt appendix inserted immediately after a full goal-alignment review fails to terminate the loop. It translates Codex’s alignment-audit findings into priorities for the next Claude implementation round.
- algorithmic_behavior: The template adds a “Post-Alignment Check Action Items” section instructing Claude to focus on forgotten items, acceptance criteria marked not met, and unjustified deferrals (`prompt-template/claude/post-alignment-action-items.md:2-7`). It contains no placeholders and performs no rendering logic itself. The algorithmic transition is in the stop hook: after a review with issues, the hook increments `current_round`, builds the next prompt, and if the just-reviewed round was a full alignment round (`CURRENT_ROUND % 5 == 4`), appends this template to `round-(N+1)-prompt.md` (`hooks/loop-codex-stop-hook.sh:967-995`).
- inputs_outputs_state: Input is implicit loop state: `CURRENT_ROUND`, Codex review content, and the full-alignment cadence. The file output is appended prose in the next Claude prompt, not a separate state artifact. Its expected downstream behavioral output is Claude prioritizing Codex-flagged forgotten tasks, unmet ACs, and invalid deferrals in the following round.
- gates_or_invariants: It is gated by the full-alignment cadence; the stop hook sets full alignment when `CURRENT_ROUND % 5 == 4` and appends this file only for the next round after such checks (`hooks/loop-codex-stop-hook.sh:644-648`, `hooks/loop-codex-stop-hook.sh:990-995`). If the template is missing or empty, the hook silently skips the appendix because it uses `load_template` and only appends non-empty content (`hooks/loop-codex-stop-hook.sh:992-994`). The template itself reinforces existing goal-tracker invariants: forgotten items must be addressed, unmet ACs are priority work, and unjustified deferrals should be reversed (`prompt-template/claude/post-alignment-action-items.md:4-7`).
- dependencies_and_callers: Sole identified caller is `hooks/loop-codex-stop-hook.sh`, after it has already built the base `claude/next-round-prompt.md` and before adding the footer, push note, and goal-tracker update request (`hooks/loop-codex-stop-hook.sh:976-1018`). It coordinates with `prompt-template/codex/full-alignment-review.md`, which defines full-alignment audit duties and allows `STOP` for stagnation, and with `prompt-template/claude/goal-tracker-update-request.md`, which follows in the generated next prompt.
- edge_cases_or_failure_modes: Since the file is generic and has no variable interpolation, it cannot name the specific forgotten items or unmet ACs; it relies on the Codex review text already embedded in the next-round prompt. Missing template produces no explicit fallback, unlike many block templates. The README says full alignment checks occur at rounds 4, 9, 14 (`README.md:143`), while the hook comment describes “every 5 rounds” and implements modulo-5 round numbers (`hooks/loop-codex-stop-hook.sh:644-648`); these are consistent for zero-based `current_round` but can be misread by users.
- validation_or_tests: Template existence is indirectly covered by template reference scanning: `tests/test-template-references.sh` scans hook template references and verifies referenced template files exist (`tests/test-template-references.sh:56-111`). The broader template suite validates loader/render behavior and real templates, though this specific no-placeholder appendix has no dedicated semantic test. Runtime behavior is tied to `hooks/loop-codex-stop-hook.sh:990-995`.
