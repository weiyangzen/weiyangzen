### IMPL_DCCB_LOOP-HZ-019 `file` `scripts/humanize.sh`
- cursor: `[_]`
- core_role: User-facing shell utility dispatcher for live monitoring of Humanize loop executions. It does not implement the loop scheduler itself; it implements the terminal monitor/runtime dashboard that reads loop state, tracker files, git state, and cached Codex logs for RLCR and DCCB loops. The public entrypoint is `humanize()` at `scripts/humanize.sh:790`, currently supporting `humanize monitor rlcr-loop` and `humanize monitor dccb-loop` at `scripts/humanize.sh:794-821`.

- algorithmic_behavior: The file has two near-parallel monitoring algorithms. `_humanize_monitor_codex()` monitors `.humanize-rlcr.local` sessions (`scripts/humanize.sh:9-518`), while `_humanize_monitor_dccb()` monitors `.humanize-dccb.local` sessions (`scripts/humanize.sh:522-787`). Each monitor first verifies the local loop directory exists (`scripts/humanize.sh:16-21`, `scripts/humanize.sh:529-533`), finds the latest timestamp-named session directory by lexicographic timestamp order (`scripts/humanize.sh:23-41`, `scripts/humanize.sh:535-548`), then maps the current project path to a sanitized cache path and selects the latest `round-*-codex-run.log` by newest session and highest round number (`scripts/humanize.sh:45-96`, `scripts/humanize.sh:550-587`). Once a log is found, it clears the terminal, reserves a 10-line status bar via ANSI scroll region setup (`scripts/humanize.sh:383-391`, `scripts/humanize.sh:696-700`), displays the last 50 log lines (`scripts/humanize.sh:469-471`, `scripts/humanize.sh:753-756`), then polls every 0.5 seconds for appended bytes and newer log files (`scripts/humanize.sh:473-512`, `scripts/humanize.sh:758-782`). When a newer log appears, it clears the scroll region, announces the switch, resets read position, and continues following the new file (`scripts/humanize.sh:499-511`, `scripts/humanize.sh:774-781`).

- inputs_outputs_state: Inputs are the current working directory, a local loop state tree, a project-specific cache tree, terminal capabilities, and git metadata. RLCR expects `.humanize-rlcr.local/<timestamp>/rlcr-state.md` and `rlcr-tracker.md`; DCCB expects `.humanize-dccb.local/<timestamp>/dccb-state.md` and `dccb-tracker.md` (`scripts/humanize.sh:269-270`, `scripts/humanize.sh:622-623`). The cache lookup uses `$HOME/.cache/humanize/<sanitized-project-path>/<timestamp>/round-N-codex-run.log` for RLCR and `$HOME/.cache/humanize-dccb/<sanitized-project-path>/<timestamp>/round-N-codex-run.log` for DCCB (`scripts/humanize.sh:49-55`, `scripts/humanize.sh:554-557`). Outputs are purely terminal UI: status bar text, log tail content, usage text, error messages, and terminal control sequences. Internal state is process-local only: `current_file`, `current_session_dir`, `last_size`, `file_size`, `monitor_running`, and `cleanup_done` (`scripts/humanize.sh:10-14`, `scripts/humanize.sh:401-405`, `scripts/humanize.sh:523-527`, `scripts/humanize.sh:707-709`). It does not write repository state.

- gates_or_invariants: The monitor refuses to start if the expected loop directory is missing, returning nonzero with a contextual error (`scripts/humanize.sh:17-20`, `scripts/humanize.sh:529-532`). Session directories must match `YYYY-MM-DD_HH-MM-SS`; other children are ignored (`scripts/humanize.sh:34-38`, `scripts/humanize.sh:540-545`). Cached log files must match `round-*-codex-run.log`, and round ordering is numeric within the newest session (`scripts/humanize.sh:75-90`, `scripts/humanize.sh:570-582`). Missing state/tracker files degrade to `N/A` or default display values rather than aborting (`scripts/humanize.sh:101-114`, `scripts/humanize.sh:591-602`, `scripts/humanize.sh:606-615`). Terminal cleanup is guarded by `cleanup_done` so repeated `INT`/`TERM` does not repeatedly reset state (`scripts/humanize.sh:406-425`, `scripts/humanize.sh:711-723`). The public dispatcher rejects unknown monitor targets and unknown commands with usage text (`scripts/humanize.sh:804-819`, `scripts/humanize.sh:823-829`).

- dependencies_and_callers: This file is intended to be sourced by user shell setup; README references sourcing `$HUMANIZE_PLUGIN_ROOT/scripts/humanize.sh` and running `humanize monitor rlcr-loop` (`README.md:130-138`). Its producers are the setup and stop-hook scripts: `scripts/setup-rlcr-loop.sh:207-240` creates `.humanize-rlcr.local/<timestamp>/rlcr-state.md` and `rlcr-tracker.md`; `scripts/setup-dccb-loop.sh:188-223` creates `.humanize-dccb.local/<timestamp>/dccb-state.md` and `dccb-tracker.md`; `hooks/loop-codex-stop-hook.sh:690-702` writes RLCR cached `round-N-codex-run.log`; `hooks/loop-dccb-stop-hook.sh:669-677` writes DCCB cached `round-N-codex-run.log`. Runtime command dependencies are standard shell tools and terminal APIs: `grep`, `sed`, `sort`, `wc`, `head`, `cut`, `git`, `stat`, `tail`, `tput`, `clear`, `sleep`, `basename`, `seq`, `kill`, and `wait`.

- edge_cases_or_failure_modes: If no cached log exists but the loop directory exists, the monitor waits indefinitely for the first log and polls every 2 seconds (`scripts/humanize.sh:434-442`, `scripts/humanize.sh:730-738`). If a log is truncated or rotated in place, the code only reads new bytes when `file_size > last_size`; smaller sizes do not reset unless a different newer file is detected (`scripts/humanize.sh:483-490`, `scripts/humanize.sh:763-768`). `tail_pid` is declared and cleaned up, but the implementation does not actually background a `tail` process, so that cleanup branch is inert (`scripts/humanize.sh:401-420`, `scripts/humanize.sh:707-719`). The pipe-delimited parser can mis-split displayed values containing `|`, especially tracker status/goal text, because `_split_to_array` uses `|` as an unescaped field separator (`scripts/humanize.sh:248-259`, `scripts/humanize.sh:626-640`). Several numeric comparisons assume parsed values are numeric; missing files are handled, but malformed tracker counts could produce shell arithmetic/comparison warnings in the status renderer (`scripts/humanize.sh:347-350`, `scripts/humanize.sh:610-612`, `scripts/humanize.sh:674`). The status bar uses `{1..10}`, array handling, and negative substring slicing, with explicit zsh compatibility handling only for array indexing (`scripts/humanize.sh:253-258`, `scripts/humanize.sh:316-323`, `scripts/humanize.sh:337`, `scripts/humanize.sh:664`).

- validation_or_tests: Direct validation performed for this research: `bash -n scripts/humanize.sh` completed successfully with no output, so the script parses under Bash. No repository test specifically for `humanize monitor` was found in the inspected references; behavior is mainly exercised through manual shell sourcing and live loop state/log production documented in `README.md`. The monitor’s validation behavior at runtime is defensive parsing and display fallback, not formal acceptance gating: missing state/tracker files become defaults, missing loop directories return failure, and missing log files enter a wait loop.
