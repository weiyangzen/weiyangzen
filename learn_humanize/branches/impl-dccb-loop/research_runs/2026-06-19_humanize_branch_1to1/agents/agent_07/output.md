### IMPL_DCCB_LOOP-HZ-007 `file` `commands/cancel-dccb-loop.md`
- cursor: `[_]`
- core_role: Defines the hidden slash-command workflow for explicit user cancellation of an active DCCB loop. It is part of the loop-control algorithm because DCCB loop liveness is represented by the presence of `.humanize-dccb.local/*/dccb-state.md`; this command terminates the loop by deleting that sentinel state file while preserving the session directory and artifacts. The command metadata describes it as "Cancel active DCCB loop" and restricts available operations to listing, reading, and removing DCCB state files plus `Read` (`commands/cancel-dccb-loop.md:1-5`).
- algorithmic_behavior: The workflow is a three-step decision procedure. First, probe for active state files with `ls .humanize-dccb.local/*/dccb-state.md 2>/dev/null || echo "NO_LOOP"` (`commands/cancel-dccb-loop.md:11-15`). Second, if the sentinel probe emits `NO_LOOP`, report that no active loop exists (`commands/cancel-dccb-loop.md:17`). Third, if any state file exists, read it to recover the current round, delete all matching DCCB state files with `rm .humanize-dccb.local/*/dccb-state.md`, and report cancellation using the stored `N of M` progress values (`commands/cancel-dccb-loop.md:19-22`). The preservation rule is explicit: only state files are removed; "documentation drafts and review results" remain available in the loop directory (`commands/cancel-dccb-loop.md:24`).
- inputs_outputs_state: Inputs are the project-relative glob `.humanize-dccb.local/*/dccb-state.md`, the contents of matching `dccb-state.md` files, and the implicit user action invoking `/humanize:cancel-dccb-loop`. The setup script creates the state file under `.humanize-dccb.local/<timestamp>/dccb-state.md` with YAML-frontmatter fields including `current_round`, `max_iterations`, `codex_model`, `codex_effort`, `codex_timeout`, `output_dir`, and `started_at` (`scripts/setup-dccb-loop.sh:188-215`). The stop hook also treats `dccb-state.md` as the active-loop marker by scanning newest session directories for that file (`hooks/loop-dccb-stop-hook.sh:43-58`). Output is either the user-visible no-loop message or a cancellation message containing round progress. The state transition is active-to-inactive by deleting the state sentinel; after that, the DCCB stop hook sees no `LOOP_DIR` and exits without blocking (`hooks/loop-dccb-stop-hook.sh:55-58`). The command intentionally does not delete `dccb-tracker.md`, round prompts, summaries, review results, or generated documentation.
- gates_or_invariants: The main invariant is that active DCCB loop state is represented by at least one `dccb-state.md` beneath `.humanize-dccb.local/*/`. The command’s allowed-tool gate limits shell access to only `ls`, `rm`, and `cat` against that state-file glob, plus `Read` (`commands/cancel-dccb-loop.md:3`). The hidden-command flag prevents normal slash-command exposure while still allowing internal/user-directed invocation (`commands/cancel-dccb-loop.md:4`). The command depends on the state file being readable enough to obtain `current_round` and `max_iterations`; those keys are produced by setup (`scripts/setup-dccb-loop.sh:205-215`) and parsed similarly by monitoring code (`scripts/humanize.sh:589-601`). Cancellation is a direct bypass of the normal validation/review gates: the ordinary stop hook blocks on incomplete todos, large docs, missing docs, tracker initialization, Codex review gaps, and round advancement while the state file exists; deleting the sentinel makes the hook no-op on later stops (`hooks/loop-dccb-stop-hook.sh:55-58`). Normal non-cancel completion also removes the same sentinel on max iterations (`hooks/loop-dccb-stop-hook.sh:355-358`), Codex `COMPLETE` (`hooks/loop-dccb-stop-hook.sh:786-794`), or Codex `STOP` circuit breaker (`hooks/loop-dccb-stop-hook.sh:797-818`).
