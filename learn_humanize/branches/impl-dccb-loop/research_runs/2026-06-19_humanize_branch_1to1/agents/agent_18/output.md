### IMPL_DCCB_LOOP-HZ-018 `file` `hooks/loop-write-validator.sh`
- cursor: `[_]`
- core_role: `hooks/loop-write-validator.sh` is a Claude Code `PreToolUse` gate for the `Write` tool. It protects RLCR loop control files and enforces that Claude writes only the current round summary in the active `.humanize-rlcr.local/<timestamp>/` session. It is registered in `hooks/hooks.json:4-11` for `PreToolUse` matcher `Write`. Although this branch also has DCCB hooks, this file is explicitly RLCR-scoped: its header says “Validate Write paths for RLCR loop” at `hooks/loop-write-validator.sh:3`, and it uses `.humanize-rlcr.local` and `rlcr-state.md`.

- algorithmic_behavior: The script is a deterministic allow/block validator driven by hook JSON from stdin. It reads `tool_name` and exits successfully for anything except `Write` (`hooks/loop-write-validator.sh:23-28`). For `Write`, it extracts `.tool_input.file_path`, lowercases it, and classifies it using shared helpers (`hooks/loop-write-validator.sh:30-31`). It immediately blocks writes to `round-*-todos.md`, forcing native TodoWrite instead (`hooks/loop-write-validator.sh:37-40`), and blocks writes to `round-*-prompt.md`, treating prompt files as Codex-authored instructions (`hooks/loop-write-validator.sh:42-45`). It then determines whether the target is a summary file and whether it is inside `.humanize-rlcr.local/` (`hooks/loop-write-validator.sh:51-59`). Non-summary files outside the loop directory are allowed (`hooks/loop-write-validator.sh:61-64`). Non-summary files inside the loop directory are allowed unless they are recognized as state or goal-tracker paths (`hooks/loop-write-validator.sh:66-72`). For protected loop files, it finds the newest active loop directory, reads the current round, blocks state writes, blocks goal tracker writes after round 0, blocks summary files outside the loop directory, validates the summary round number, and finally validates the exact active-loop path (`hooks/loop-write-validator.sh:78-173`).

- inputs_outputs_state: Input is Claude hook JSON on stdin, requiring `tool_name` and `tool_input.file_path` (`hooks/loop-write-validator.sh:23-31`). It also reads environment `CLAUDE_PROJECT_DIR`, falling back to `pwd`, to locate `$PROJECT_ROOT/.humanize-rlcr.local` (`hooks/loop-write-validator.sh:78-80`). The active state source is `$ACTIVE_LOOP_DIR/rlcr-state.md`, specifically YAML frontmatter field `current_round`, read through `get_current_round` (`hooks/loop-write-validator.sh:80-86`; helper implementation at `hooks/lib/loop-common.sh:35-47`). Outputs are process exit status and optional stderr guidance. Exit `0` means allow the write; exit `2` means block the tool call with a human-readable error. This validator does not mutate state. The broader RLCR state transition is handled elsewhere: `scripts/setup-rlcr-loop.sh:223-234` creates `rlcr-state.md` with `current_round: 0`, and `hooks/loop-codex-stop-hook.sh:864-871` increments `current_round` and creates next round paths after Codex review finds issues. This write validator consumes that state to enforce “Claude writes only the current round summary.”

- gates_or_invariants: The key invariants are: only `Write` tool calls are evaluated (`hooks/loop-write-validator.sh:26-28`); todos files are never written directly (`hooks/loop-write-validator.sh:37-40`); prompt files are read-only from Claude’s perspective (`hooks/loop-write-validator.sh:42-45`); `rlcr-state.md` is managed by the loop system and cannot be written by Claude (`hooks/loop-write-validator.sh:92-95`); goal tracker writes are allowed only in round 0, then redirected into a summary-file update request (`hooks/loop-write-validator.sh:101-105`); summary writes must be inside the active loop directory (`hooks/loop-write-validator.sh:111-120`); summary filenames must match the current round (`hooks/loop-write-validator.sh:139-154`); and final path equality must match `$ACTIVE_LOOP_DIR/$CLAUDE_FILENAME`, preventing writes into stale session directories or wrong timestamp folders (`hooks/loop-write-validator.sh:161-170`). Active-loop selection itself is newest-directory-only to avoid reviving older “zombie” loops (`hooks/lib/loop-common.sh:11-33`).

- dependencies_and_callers: The direct caller is Claude Code’s hook system via `hooks/hooks.json:4-11`. The script depends on `bash`, `jq`, `sed`, `grep`, `tr`, `find_active_loop`, `get_current_round`, `to_lower`, `is_round_file_type`, `is_in_humanize_loop_dir`, `is_state_file_path`, `is_goal_tracker_path`, `extract_round_number`, and shared block-message helpers from `hooks/lib/loop-common.sh` (`hooks/loop-write-validator.sh:13-17`). It coordinates with sibling validators: `hooks/loop-edit-validator.sh` applies similar protections to `Edit`, but only for paths already inside `.humanize-rlcr.local` after prompt/todos checks (`hooks/loop-edit-validator.sh:22-31`, `hooks/loop-edit-validator.sh:50-67`, `hooks/loop-edit-validator.sh:91-117`); `hooks/loop-read-validator.sh` blocks stale/wrong-location reads of round prompt/summary files (`hooks/loop-read-validator.sh:48-55`, `hooks/loop-read-validator.sh:98-149`); `hooks/loop-bash-validator.sh` blocks shell bypasses such as redirects, `tee`, and `sed -i` against protected files (`hooks/loop-bash-validator.sh:69-124`). The state and artifact contract is documented in `README.md:411-421`, which lists `rlcr-state.md`, `rlcr-tracker.md`, `round-N-prompt.md`, and `round-N-summary.md`.

- edge_cases_or_failure_modes: If there is no active loop directory, writes are allowed (`hooks/loop-write-validator.sh:82-84`), so protection depends on the active `rlcr-state.md` existing. Active-loop discovery considers only the newest timestamp directory and ignores older directories even if they contain state (`hooks/lib/loop-common.sh:11-33`), which intentionally avoids stale-loop revival but means a malformed newest directory can disable validation for older active-looking sessions. If `current_round` is missing or unreadable, `get_current_round` defaults to `0` (`hooks/lib/loop-common.sh:35-47`), potentially making round-0 summary paths the allowed target. The filename extraction logic is regex-based and assumes a `.humanize-rlcr.local/<session>/<filename>` shape, falling back to `.humanize-rlcr.local/<filename>` (`hooks/loop-write-validator.sh:127-132`); unusual nested files may get treated as a relative filename and then fail exact path equality. A significant apparent bug is the goal-tracker predicate: comments and docs refer to `rlcr-tracker.md` (`hooks/loop-write-validator.sh:66`, `README.md:416-417`, `scripts/setup-rlcr-loop.sh:240`), but `is_goal_tracker_path` actually matches `goal-tracker.md$` (`hooks/lib/loop-common.sh:160-164`). Because non-summary loop files are allowed unless `is_state_file_path` or `is_goal_tracker_path` matches (`hooks/loop-write-validator.sh:68-72`), a direct `Write` to the real `rlcr-tracker.md` after round 0 appears likely to bypass the intended gate. The same naming mismatch appears in the Bash validator’s command matching for `goal-tracker\.md` while its messages cite `rlcr-tracker.md` (`hooks/loop-bash-validator.sh:85-93`). Another edge: `set -euo pipefail` plus missing `jq` or invalid hook JSON can make the hook fail before producing a tailored block message (`hooks/loop-write-validator.sh:13`, `hooks/loop-write-validator.sh:23-31`).
