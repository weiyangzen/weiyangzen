The fourth invariant is round ownership: in round `0`, goal tracker setup is allowed only through `Write`/`Edit`; after round `0`, requested tracker changes must be placed in the current round summary for Codex review ([hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/loop-bash-validator.sh:85)). - edge_cases_or_failure_modes: The most important edge case is a likely filename mismatch in the goal-tracker gate. The RLCR setup creates `rlcr-tracker.md` ([scripts/setup-rlcr-loop.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/scripts/setup-rlcr-loop.sh:240)), and the block message explicitly discusses `rlcr-tracker.md` ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/lib/loop-common.sh:151)), but both `is_goal_tracker_path` and the Bash validator’s `command_modifies_file` call match `goal-tracker.md`, not `rlcr-tracker.md` ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/lib/loop-common.sh:160), [hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/loop-bash-validator.sh:85)). As written, Bash commands like `cat > .humanize-rlcr.local/<ts>/rlcr-tracker.md` appear not to match that goal-tracker gate, even though the comments say this file should be protected. Another mismatch: the comments at the top say `rlcr-tracker.md`, but the actual pattern says `goal-tracker.md` ([hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/loop-bash-validator.sh:5)). The state-file pattern is broad, `state\.md`, so it can block any Bash write target ending in state-like names, not only `rlcr-state.md` ([hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/loop-bash-validator.sh:74)). The Bash parser is heuristic string matching, not shell AST parsing; quoted paths with spaces, multi-command constructs, process substitution, here-doc variants that do not match `>[[:space:]]*`, or generated filenames may bypass or false-trigger depending on text shape. Because `set -euo pipefail` is active and `jq` is required, malformed JSON or missing `jq` can fail the hook rather than cleanly allow/deny. `get_current_round` defaults missing round to `0` but does not validate numeric content here, so a corrupted non-numeric `current_round` could break arithmetic comparison in `[[ "$CURRENT_ROUND" -eq 0 ]]` ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/lib/loop-common.sh:37), [hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/impl-dccb-loop/hooks/loop-bash-validator.sh:86)). Finally, despite this branch’s DCCB naming and hook registration description covering both loop types, this Bash validator only looks under `.humanize-rlcr.local`; it does not enforce equivalent Bash protections for `.humanize-dccb.local` sessions.
