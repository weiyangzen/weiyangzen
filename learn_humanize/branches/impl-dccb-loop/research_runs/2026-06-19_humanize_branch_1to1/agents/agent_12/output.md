### IMPL_DCCB_LOOP-HZ-012 `file` `hooks/hooks.json`

- core_role: `hooks/hooks.json` is the Claude plugin hook dispatch table for the loop state machines. It does not implement the review algorithms itself; it binds Claude Code lifecycle events to executable validators and stop hooks. It registers `PreToolUse` gates for `Write`, `Edit`, `Read`, and `Bash`, then registers two `Stop` hooks, first the RLCR/Codex implementation loop hook and then the DCCB documentation distillation hook. See `hooks/hooks.json:2-3`, `hooks/hooks.json:4-41`, and `hooks/hooks.json:42-57`.
- algorithmic_behavior: On every matching tool event, Claude Code supplies hook JSON to the configured shell command through stdin. The dispatch table maps:
  - `Write` to `${CLAUDE_PLUGIN_ROOT}/hooks/loop-write-validator.sh` at `hooks/hooks.json:6-10`.
  - `Edit` to `${CLAUDE_PLUGIN_ROOT}/hooks/loop-edit-validator.sh` at `hooks/hooks.json:15-19`.
  - `Read` to `${CLAUDE_PLUGIN_ROOT}/hooks/loop-read-validator.sh` at `hooks/hooks.json:24-28`.
  - `Bash` to `${CLAUDE_PLUGIN_ROOT}/hooks/loop-bash-validator.sh` at `hooks/hooks.json:33-37`.
  - `Stop` to both `${CLAUDE_PLUGIN_ROOT}/hooks/loop-codex-stop-hook.sh` and `${CLAUDE_PLUGIN_ROOT}/hooks/loop-dccb-stop-hook.sh`, each with a 7200-second timeout, at `hooks/hooks.json:42-54`.
  The DCCB algorithmic path is primarily activated by the second stop hook. `loop-dccb-stop-hook.sh` discovers the newest `.humanize-dccb.local/<timestamp>/dccb-state.md`, blocks exit if work is incomplete, creates a Codex review prompt, runs `codex exec`, interprets the last non-empty review line as `COMPLETE`, `STOP`, or "continue", and either removes the state file or advances `current_round` and returns a blocking JSON payload with next-round instructions. Relevant lines: `hooks/loop-dccb-stop-hook.sh:36-60`, `hooks/loop-dccb-stop-hook.sh:66-98`, `hooks/loop-dccb-stop-hook.sh:198-227`, `hooks/loop-dccb-stop-hook.sh:353-359`, `hooks/loop-dccb-stop-hook.sh:365-445`, `hooks/loop-dccb-stop-hook.sh:561-659`, `hooks/loop-dccb-stop-hook.sh:721-777`, `hooks/loop-dccb-stop-hook.sh:782-818`, and `hooks/loop-dccb-stop-hook.sh:825-925`.

- edge_cases_or_failure_modes: Because `hooks.json` wires both stop hooks for every Stop event, RLCR checks run before DCCB checks. If both `.humanize-rlcr.local` and `.humanize-dccb.local` active state files exist, the RLCR stop hook can block or terminate before the DCCB hook gets meaningful control; this is an ordering interaction from `hooks/hooks.json:46-53`. The pre-tool validators are RLCR-specific despite the file description mentioning both RLCR and DCCB; they protect `.humanize-rlcr.local` and `rlcr-*` paths, not `.humanize-dccb.local` or `dccb-*` paths. DCCB’s main guard is therefore the Stop hook, not the PreToolUse validators.
  DCCB has several graceful bypass or termination paths: no active DCCB loop exits 0 (`loop-dccb-stop-hook.sh:55-58`); missing `dccb-state.md` exits 0 (`loop-dccb-stop-hook.sh:161-163`); corrupted nonnumeric `current_round` removes the state file and exits 0 (`loop-dccb-stop-hook.sh:183-188`); exceeding max iterations removes the state file and exits 0 without a successful reconstruction-ready review (`loop-dccb-stop-hook.sh:353-359`). If Codex does not create the expected review result file, the hook copies stdout if present, otherwise blocks with debug paths and stderr tail (`loop-dccb-stop-hook.sh:729-777`). If no system `timeout` or `gtimeout` exists and portable timeout is unavailable, `run_with_timeout` runs the command directly, so the configured timeout becomes advisory rather than enforced (`loop-dccb-stop-hook.sh:679-695`). The todo checker treats malformed hook JSON, missing transcript paths, missing transcript files, or no TodoWrite calls as "allow" rather than fail-closed at `hooks/check-todos-from-transcript.py:87-107`.
