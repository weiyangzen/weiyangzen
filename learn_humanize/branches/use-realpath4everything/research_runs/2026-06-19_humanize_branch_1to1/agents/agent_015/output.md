### USE_REALPATH4EVERYTHING-HZ-015 `directory` `prompt-template/claude`
- cursor: `[_]`
- core_role: This directory is the Claude-facing prompt policy layer for the Humanize RLCR loop. It is not executable code, but it is core algorithm input: the stop hook, setup script, and methodology-analysis helper load these Markdown fragments to decide what Claude must do in each loop phase, what state artifacts it must update, which gates it must respect, and how review feedback is converted into the next round contract. The directory is flat, with 14 Markdown templates totaling 22,925 bytes.

- algorithmic_behavior: Template rendering is handled by `hooks/lib/template-loader.sh`: `get_template_dir` resolves the parent `prompt-template` directory from the hook library path (`hooks/lib/template-loader.sh:24-31`), `load_template` reads a named file or emits a warning (`hooks/lib/template-loader.sh:33-46`), and `render_template` performs single-pass `{{VAR}}` substitution (`hooks/lib/template-loader.sh:48-135`). Single-pass rendering is an important invariant because inserted values such as review content are not recursively scanned for placeholder injection (`hooks/lib/template-loader.sh:52-55`, `hooks/lib/template-loader.sh:115-124`). `load_and_render_safe` provides fallback text when required phase templates are missing or render empty (`hooks/lib/template-loader.sh:185-210`).

  Contained child roles:
  - `agent-teams-instructions.md`: initial Agent Teams mode header for round 0. It tells the team leader to read the plan and goal tracker and write a summary after teammates finish (`prompt-template/claude/agent-teams-instructions.md:1-8`). It is appended during loop setup when `AGENT_TEAMS=true` (`scripts/setup-rlcr-loop.sh:1370-1379`).
  - `agent-teams-core.md`: shared delegation algorithm for Agent Teams. It makes the leader coordination-only, requires task splitting, file ownership boundaries, progress monitoring, review of teammate output, teammate commits, and BitLesson discipline (`prompt-template/claude/agent-teams-core.md:1-31`). It also encodes a key failure mode: parallel teammates editing the same file can silently overwrite each other, so overlapping files must be sequenced (`prompt-template/claude/agent-teams-core.md:16-23`).
  - `agent-teams-continue.md`: continuation wrapper for later implementation rounds. It resets team state by stating previous teammates no longer exist, requires reviewing Codex feedback first, avoids redoing prior correct work, and tells the leader to inspect current file state before assigning ownership (`prompt-template/claude/agent-teams-continue.md:1-12`). The stop hook appends it with `agent-teams-core.md` only during implementation-phase continuations, not review phase (`hooks/loop-codex-stop-hook.sh:2145-2164`).
  - `next-round-prompt.md`: normal next-round prompt after Codex feedback. It requires re-reading the original plan, goal tracker, and recent summaries, then writing a round contract before implementation (`prompt-template/claude/next-round-prompt.md:3-28`). It classifies work into `mainline`, `blocking`, and `queued` lanes, requires BitLesson selection before tasks, embeds `{{REVIEW_CONTENT}}`, and keeps the goal tracker mutable section current (`prompt-template/claude/next-round-prompt.md:29-75`). The stop hook selects this template when drift recovery is not required (`hooks/loop-codex-stop-hook.sh:2028-2036`).
  - `drift-replan-prompt.md`: recovery prompt selected when mainline progress has stalled or regressed enough to require replanning. It takes `{{STALL_COUNT}}` and `{{LAST_MAINLINE_VERDICT}}`, demands a recovered mainline objective, root-cause statement, blocking/queued classification, and success criteria that would restore an `ADVANCED` verdict (`prompt-template/claude/drift-replan-prompt.md:3-68`). It is selected under `DRIFT_REPLAN_REQUIRED=true` (`hooks/loop-codex-stop-hook.sh:2017-2026`).
  - `review-phase-prompt.md`: review-fix prompt when Codex review reports issues. It requires re-reading the plan and goal tracker, refreshing the round contract, preserving a single mainline objective, classifying findings as blocking or queued, fixing only blocking issues, committing, and writing a structured summary (`prompt-template/claude/review-phase-prompt.md:1-58`). It is rendered by `continue_review_loop_with_issues` (`hooks/loop-codex-stop-hook.sh:1456-1520`).
  - `finalize-phase-prompt.md`: final simplification prompt after review passes. It instructs Claude to run the code-simplifier agent, allows only functionality-equivalent cleanup, requires tests to remain passing, focuses on changes between `{{BASE_BRANCH}}` and `{{START_BRANCH}}`, and requires a finalize summary (`prompt-template/claude/finalize-phase-prompt.md:1-52`). It is rendered when entering finalize normally (`hooks/loop-codex-stop-hook.sh:1328-1353`).
  - `finalize-phase-skipped-prompt.md`: alternate finalize prompt when code review was skipped. It surfaces `{{REVIEW_SKIP_REASON}}`, requires manual verification and tests if possible, makes simplification optional, and still enforces no functionality change (`prompt-template/claude/finalize-phase-skipped-prompt.md:1-51`). It is used by the same finalize prompt function when the review skip path is active (`hooks/loop-codex-stop-hook.sh:1320-1326`).
  - `methodology-analysis-prompt.md`: exit-phase methodology prompt. It instructs Claude to spawn an Opus analysis agent, analyze only loop summaries/review results, write `methodology-analysis-report.md`, derive all user-facing content solely from that sanitized report, optionally ask the user about filing a GitHub issue, and write `methodology-analysis-done.md` before exit (`prompt-template/claude/methodology-analysis-prompt.md:1-73`). `hooks/lib/methodology-analysis.sh` renames active state, records exit reason, creates a placeholder done file, renders this template, and blocks with it (`hooks/lib/methodology-analysis.sh:38-99`).
  - `next-round-footer.md`: mandatory footer appended to follow-up prompts. It forbids false exits, state-file tampering, or running `cancel-rlcr-loop`, then requires optional code-simplifier use, commit, and summary write to `{{NEXT_SUMMARY_FILE}}` (`prompt-template/claude/next-round-footer.md:1-9`). It is always appended to next-round prompts (`hooks/loop-codex-stop-hook.sh:2122-2127`).
  - `goal-tracker-update-request.md`: optional summary section template for cases where Claude cannot safely edit the goal tracker. It gives a structured `Goal Tracker Update Request` with requested changes and justification (`prompt-template/claude/goal-tracker-update-request.md:1-17`). The stop hook appends it to next-round prompts or falls back to a one-line instruction if missing (`hooks/loop-codex-stop-hook.sh:2138-2143`).
  - `open-question-notice.md`: one-line high-priority notice telling Claude to use `AskUserQuestion` before other findings when Codex reports open questions (`prompt-template/claude/open-question-notice.md:1`). The stop hook injects it only when `ASK_CODEX_QUESTION=true` and a short review-result line contains `Open Question` (`hooks/loop-codex-stop-hook.sh:2078-2110`).
  - `post-alignment-action-items.md`: append-only guidance after a Full Goal Alignment Check. It prioritizes forgotten items, unmet ACs, unjustified deferrals, and keeps queued issues from taking over unless they now block mainline progress (`prompt-template/claude/post-alignment-action-items.md:1-8`). It is appended when `FULL_ALIGNMENT_CHECK=true` (`hooks/loop-codex-stop-hook.sh:2114-2119`).
  - `push-every-round-note.md`: append-only note requiring remote push after each round when `--push-every-round` is enabled (`prompt-template/claude/push-every-round-note.md:1-2`). The stop hook appends it when `PUSH_EVERY_ROUND=true` (`hooks/loop-codex-stop-hook.sh:2129-2135`).

Required phase prompts use safe loading with fallback, while optional addenda are appended only when present or use small inline fallback text (`hooks/loop-codex-stop-hook.sh:2091-2094`, `hooks/loop-codex-stop-hook.sh:2131-2141`).
