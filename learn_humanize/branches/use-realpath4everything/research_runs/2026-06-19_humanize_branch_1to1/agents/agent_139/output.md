### USE_REALPATH4EVERYTHING-HZ-139 `file` `prompt-template/block/state-file-modification.md`
- cursor: `[_]`
- core_role: Static block-message template for the normal loop state file protection gate. The file contains the user-facing reason emitted when an agent tries to modify `state.md`: title at `prompt-template/block/state-file-modification.md:1` and the prohibition/rationale at `prompt-template/block/state-file-modification.md:3`. It does not implement matching logic itself; its core role is to provide the canonical prompt text used by validators after they detect a forbidden `state.md` modification.
- algorithmic_behavior: The template encodes the policy that `state.md` is loop-system-owned, and that manual modification would corrupt loop state. It also gives the recovery behavior: if the work is done, the agent should stop and let the automated review round trigger. The actual algorithm is split between this template and hook code: `state_file_blocked_message()` loads `block/state-file-modification.md` with a fallback in `hooks/lib/loop-common.sh:859-865`; the loader returns fallback text if the template is missing or renders empty in `hooks/lib/template-loader.sh:185-210`. Write and edit validators call this message after path matching and exit with block status, as shown in `hooks/loop-write-validator.sh:188-204` and `hooks/loop-edit-validator.sh:151-168`. Bash command validation also calls it when a command writes, redirects, moves, or copies normal `state.md`, including source-side `mv/cp` bypass attempts and shell-wrapper payloads in `hooks/loop-bash-validator.sh:271-278`, `hooks/loop-bash-validator.sh:430-436`, and `hooks/loop-bash-validator.sh:461-468`.
- inputs_outputs_state: The template itself has no variables/placeholders; its only input is successful template lookup via `$TEMPLATE_DIR` and `load_and_render_safe`. Upstream inputs are the requested file path or bash command, lowered to match `state.md` patterns, plus active-loop discovery and strict state parsing before validators proceed. Outputs are plain Markdown text written to stderr by validators through `state_file_blocked_message`, followed by validator exit code `2` for a blocked operation in the write/edit/bash paths. State transition is intentionally negative: no mutation of `state.md` is allowed. The intended transition is external and automatic: the user/agent stops, then the stop hook/review loop performs the next state transition rather than a manual edit.
- gates_or_invariants: The invariant is that normal `state.md` cannot be modified directly by agent file-write, edit, or bash tools. `is_state_file_path()` matches lowercase paths ending in `state.md` in `hooks/lib/loop-common.sh:994-998`, which is deliberately broad enough to also match names such as `finalize-state.md`; therefore validators check more-specific state files first to avoid issuing the wrong message. That ordering is documented and enforced in write validation at `hooks/loop-write-validator.sh:190-203`, edit validation at `hooks/loop-edit-validator.sh:154-167`, and bash validation at `hooks/loop-bash-validator.sh:253-278`. Bash validation has one narrow exception path: authorized cancellation can allow a controlled move to `cancel-state.md` when the cancel signal passes `is_cancel_authorized`; the authorization constraints are documented in `hooks/lib/loop-common.sh:1033-1051`. Even with a cancel signal, arbitrary writes like `echo > state.md` or `sed -i state.md` remain blocked.
- dependencies_and_callers: Direct dependency is `load_and_render_safe` in `hooks/lib/template-loader.sh:185-210`, called by `state_file_blocked_message()` in `hooks/lib/loop-common.sh:859-865`. Primary callers are `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, and `hooks/loop-bash-validator.sh`.
