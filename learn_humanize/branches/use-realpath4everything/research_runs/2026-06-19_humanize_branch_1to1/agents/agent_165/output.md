- core_role: This file is a Codex review-phase audit template, not an executable algorithm by itself. Its core role is to document the review invocation and encode the expected phase contract for RLCR: run `codex review --base {{BASE_BRANCH}}`, interpret priority markers, send issues back for remediation, or transition to finalization when no issues are present. The template states that `codex review` does not accept prompt input and operates from the git diff, so the generated prompt file is an audit artifact rather than stdin for Codex review (`prompt-template/codex/code-review-phase.md:3-4`).
- algorithmic_behavior: The template defines a four-step review transition: invoke Codex review against a base branch, scan output for `[P0-9]` markers, return a fix prompt to Claude if any are found, and move to Finalize Phase if none are found (`prompt-template/codex/code-review-phase.md:12-18`). The executable realization is in `hooks/loop-codex-stop-hook.sh`: `run_codex_code_review()` renders this template into `round-N-review-prompt.md`, records a command file, then runs `codex review --base "$review_base"` from `PROJECT_ROOT` with combined stdout/stderr redirected to a cache log (`hooks/loop-codex-stop-hook.sh:1185-1229`). `run_and_handle_code_review()` then gates on the review command exit code and parser result: command failure blocks, parser hard error blocks, detected issues continue the review loop, and no detected issues enters finalize (`hooks/loop-codex-stop-hook.sh:1250-1278`).
- inputs_outputs_state: Template inputs are placeholders: `{{REVIEW_ROUND}}`, `{{BASE_BRANCH}}`, and `{{TIMESTAMP}}` in the checked-in template (`prompt-template/codex/code-review-phase.md:1`, `prompt-template/codex/code-review-phase.md:8-10`). The caller also passes `BASE_COMMIT`, `REVIEW_BASE`, and `REVIEW_BASE_TYPE`, but this specific template does not currently expose those placeholders; the fallback audit prompt does include them (`hooks/loop-codex-stop-hook.sh:1190-1208`). - gates_or_invariants: The primary gate is the severity-marker contract: `[P0]` through `[P9]` indicate review issues. The actual invariant is stricter in `detect_review_issues()`: it scans only the last 50 lines of the combined log, and only matches `[P<digit>]` within the first 10 characters of a line (`hooks/lib/loop-common.sh:728-739`, `hooks/lib/loop-common.sh:755-766`). A missing or empty review log returns hard error code `2`, which blocks finalization (`hooks/lib/loop-common.sh:745-749`, `hooks/loop-codex-stop-hook.sh:1268-1270`). A separate anti-tamper gate requires `.review-phase-started` whenever state says `review_started=true`, preventing manual state-file toggles from bypassing proper review entry (`hooks/loop-codex-stop-hook.sh:1884-1899`). The base selection invariant prefers `BASE_COMMIT` over `BASE_BRANCH` to avoid comparing a branch against itself after the branch advances (`hooks/loop-codex-stop-hook.sh:1176-1183`), but the template’s visible `Base Branch` label does not fully represent that runtime behavior.
