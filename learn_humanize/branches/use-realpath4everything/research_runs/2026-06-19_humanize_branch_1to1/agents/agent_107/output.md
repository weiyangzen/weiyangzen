### USE_REALPATH4EVERYTHING-HZ-107 `file` `hooks/lib/project-root.sh`
- cursor: `[_]`
- core_role: Shared deterministic project-root and path-canonicalization library for RLCR hooks/scripts. It is core because the active RLCR loop path is derived from the resolved project root, and validators use its canonicalization helpers to decide whether reads/writes/moves target the active loop or escape it. The file explicitly avoids `pwd` fallback because session `cd` drift can make `.humanize/rlcr` state lookups target the wrong directory (`hooks/lib/project-root.sh:5-12`).
- algorithmic_behavior: The file is guarded against repeated sourcing with `_HUMANIZE_PROJECT_ROOT_SOURCED`, changing shell-library load state from unset to loaded and then returning early on future sources (`hooks/lib/project-root.sh:23-26`). `resolve_project_root` selects root in priority order: `CLAUDE_PROJECT_DIR`, then `git rev-parse --show-toplevel`, then failure return `1` if neither exists (`hooks/lib/project-root.sh:41-48`). On success it runs the chosen root through `canonicalize_path` and prints the canonical result, falling back to the original root only if canonicalization prints empty (`hooks/lib/project-root.sh:50-52`). `canonicalize_path_prefix` canonicalizes only the parent directory and reattaches the original basename, deliberately preserving leaf symlink identity for authorization comparisons (`hooks/lib/project-root.sh:55-96`). `canonicalize_path` canonicalizes the whole path when possible, otherwise canonicalizes the parent for not-yet-existing files, otherwise uses a Python `os.path.realpath` fallback, otherwise prints the input (`hooks/lib/project-root.sh:98-144`).
- inputs_outputs_state: Inputs are environment variable `CLAUDE_PROJECT_DIR`, current repository context visible to `git rev-parse --show-toplevel`, and function path arguments. Outputs are stdout path strings and return status: `resolve_project_root` returns `0` with a path or `1` with no path; both canonicalization helpers return `0`, print nothing for empty input, and print either canonical or original paths (`hooks/lib/project-root.sh:30-31`, `hooks/lib/project-root.sh:70-75`, `hooks/lib/project-root.sh:111-116`). State is process-local shell state only: `_HUMANIZE_PROJECT_ROOT_SOURCED=1` after sourcing. It does not mutate RLCR files directly. Its resolved `PROJECT_ROOT` feeds RLCR state lookup roots such as `$PROJECT_ROOT/.humanize/rlcr`, for example in `loop-common.sh` config loading (`hooks/lib/loop-common.sh:176-189`), setup mutual exclusion (`scripts/setup-rlcr-loop.sh:326-370`), and validators (`hooks/loop-read-validator.sh:76-83`, `hooks/loop-write-validator.sh:79-86`).
- gates_or_invariants: Primary invariant: never use `pwd` as project root fallback, because `pwd` can drift inside long Claude/Codex sessions (`hooks/lib/project-root.sh:10-12`). Second invariant: project-root comparisons must canonicalize both sides so symlinked prefixes like `/var` versus `/private/var` do not diverge (`hooks/lib/project-root.sh:14-20`). Third invariant: do not use full leaf dereferencing to authorize user-supplied target filenames; `canonicalize_path` has a security note warning callers to use `canonicalize_path_prefix` for expected-filename checks (`hooks/lib/project-root.sh:106-109`). This invariant is applied in write/read directory validation, where the validator canonicalizes only parent prefixes before comparing the requested path to the active-loop path (`hooks/loop-write-validator.sh:334-343`, `hooks/loop-read-validator.sh:307-313`). The same parent-only invariant appears in cancel authorization for `mv state.md cancel-state.md`, preserving leaf symlink identity so a symlink alias cannot authorize state movement outside the loop (`hooks/lib/loop-common.sh:1191-1229`).
- dependencies_and_callers: Dependencies are Bash, `git`, `realpath`, `dirname`, `basename`, and optional `python3`. `resolve_project_root` is pulled into the shared RLCR library before config loading (`hooks/lib/loop-common.sh:171-189`). Direct or transitive callers include `loop-read-validator.sh`, `loop-write-validator.sh`, `loop-edit-validator.sh`, `loop-bash-validator.sh`, `loop-plan-file-validator.sh`, `loop-post-bash-hook.sh`, `loop-codex-stop-hook.sh`, `setup-rlcr-loop.sh`, `cancel-rlcr-loop.sh`, `rlcr-stop-gate.sh`, `ask-codex.sh`, `ask-gemini.sh`, and `bitlesson-select.sh`. Examples: `loop-post-bash-hook.sh` explicitly says no root means no active loop and exits cleanly (`hooks/loop-post-bash-hook.sh:29-35`); `loop-bash-validator.sh` resolves root before finding the session-filtered active loop (`hooks/loop-bash-validator.sh:48-59`); setup treats missing root as a hard error with guidance to set `CLAUDE_PROJECT_DIR` or run inside a git repo (`scripts/setup-rlcr-loop.sh:326-330`). `canonicalize_path_prefix` is consumed by read/write validators and `is_cancel_authorized` (`hooks/loop-read-validator.sh:311-313`, `hooks/loop-write-validator.sh:341-343`, `hooks/lib/loop-common.sh:1207-1225`).
- edge_cases_or_failure_modes: If both `CLAUDE_PROJECT_DIR` and git toplevel are unavailable, hooks generally exit `0` to avoid false blocking, while setup-style scripts fail loudly (`hooks/lib/project-root.sh:33-39`, `scripts/setup-rlcr-loop.sh:326-330`). Empty canonicalization input prints nothing and returns success, which means callers must treat empty results intentionally (`hooks/lib/project-root.sh:70-75`, `hooks/lib/project-root.sh:111-116`). If `realpath` cannot resolve a nonexistent leaf, `canonicalize_path` canonicalizes the parent and reattaches the basename (`hooks/lib/project-root.sh:126-132`). If `realpath` fails and Python is unavailable or fails, helpers print the original path, which keeps execution portable but means comparison callers usually fail to match canonical expected paths rather than accidentally approving them (`hooks/lib/project-root.sh:66-68`, `hooks/lib/project-root.sh:135-143`). A security-sensitive edge case is a leaf symlink: full `realpath` may be correct for resolving the project root, but wrong for authorizing a write/read/move target because it can follow a symlink out of or into the loop.
