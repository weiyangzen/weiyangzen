- core_role: This file is the shipped QA ledger template for the `refine-plan` workflow. It defines the required evidence surface produced after an annotated `gen-plan` document is refined: summary, comment ledger, answers, research findings, applied changes, remaining decisions, and metadata. The caller contract is explicit in `commands/refine-plan.md`: Phase 6 reads `${CLAUDE_PLUGIN_ROOT}/prompt-template/plan/refine-plan-qa-template.md` and must populate it completely; the QA document is not optional (`commands/refine-plan.md:457-471`). The flow skill also identifies this template as the QA state after plan validation passes (`skills/humanize-refine-plan/SKILL.md:35-46`, `skills/humanize-refine-plan/SKILL.md:85-86`).
- algorithmic_behavior: The template does not execute code, but it encodes the output schema and review contract for the refine-plan algorithm. Its sections map directly to algorithm phases: `## Summary` captures aggregate refinement outcome (`prompt-template/plan/refine-plan-qa-template.md:3-5`); `## Comment Ledger` requires one row per extracted comment with classification, location, excerpt, and disposition (`prompt-template/plan/refine-plan-qa-template.md:7-17`); `## Answers` stores question handling and plan clarifications (`prompt-template/plan/refine-plan-qa-template.md:19-35`); `## Research Findings` records scoped repository investigation and impact (`prompt-template/plan/refine-plan-qa-template.md:38-57`); `## Plan Changes Applied` records mutations and cross-reference reconciliation (`prompt-template/plan/refine-plan-qa-template.md:60-83`); `## Remaining Decisions` carries unresolved decision state (`prompt-template/plan/refine-plan-qa-template.md:86-105`); and `## Refinement Metadata` records paths, counts, modified sections, convergence, and date (`prompt-template/plan/refine-plan-qa-template.md:108-119`). This mirrors the command’s ordered algorithm: setup, validation, extraction, classification, processing, plan refinement, QA generation, atomic write (`commands/refine-plan.md:30-42`).
- inputs_outputs_state: Inputs are the extracted raw comment records from the annotated plan, their classifications, location labels, full original text, dispositions, research results, applied plan edits, unresolved decisions, resolved input/output/QA paths, classification counts, modified plan sections, convergence status, and refinement date. The upstream command defines extraction state including `id`, original and normalized text, line/column positions, nearest heading, `location_label`, form, and context excerpt (`commands/refine-plan.md:184-265`). It then classifies each raw block as exactly one dominant `question`, `change_request`, or `research_request`, with possible deterministic sub-items for mixed intent (`commands/refine-plan.md:280-328`). Output is a populated QA Markdown document, normally written under `QA_DIR`, with `QA_FILE` derived from the input basename rather than output basename (`commands/refine-plan.md:65-76`). State transitions are: extracted comments become ledger rows; question comments become `Answers`; research comments become `Research Findings`; change requests become `Plan Changes Applied`; unresolved or assumption-heavy items become `Remaining Decisions`; final unresolved state determines `converged` versus `partially_converged` metadata (`commands/refine-plan.md:378-391`, `commands/refine-plan.md:500-504`).
- gates_or_invariants: The main invariant is complete population: all template sections are mandatory (`commands/refine-plan.md:459-471`). The `Comment Ledger` must contain exactly one row per raw `CMT-N` in document order, even when a raw block is split into sub-items (`commands/refine-plan.md:473-485`). The ledger disposition vocabulary is constrained to `answered`, `applied`, `researched`, `deferred`, or `resolved`, matching the template hint (`prompt-template/plan/refine-plan-qa-template.md:17`) and command resolution rules (`commands/refine-plan.md:378-386`). Original comment text must be preserved verbatim in fenced blocks (`commands/refine-plan.md:494-498`), which the template structurally supports under each detailed section (`prompt-template/plan/refine-plan-qa-template.md:25-28`, `prompt-template/plan/refine-plan-qa-template.md:44-47`, `prompt-template/plan/refine-plan-qa-template.md:66-69`). The QA document must use the same main language as the refined plan while keeping identifiers unchanged (`commands/refine-plan.md:429-440`, `commands/refine-plan.md:494-498`). The template’s metadata requires path and convergence evidence (`prompt-template/plan/refine-plan-qa-template.md:108-119`). Upstream gates must pass before QA generation: required plan sections present, no comment markers remain, AC/task references consistent, routing tags valid, and decisions/convergence aligned (`commands/refine-plan.md:442-453`).
