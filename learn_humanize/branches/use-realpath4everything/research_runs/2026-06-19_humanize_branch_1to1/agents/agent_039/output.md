### USE_REALPATH4EVERYTHING-HZ-039 `file` `hooks/hooks.json`

- core_role: `hooks/hooks.json` is the Claude Code plugin hook manifest for the RLCR loop. It is not algorithmic code itself; its core role is event-to-script routing. It binds Claude lifecycle events to the hook scripts that enforce the RLCR state machine, path restrictions, session scoping, and stop/review gates. The file declares this purpose in its description at `hooks/hooks.json:2`.
- algorithmic_behavior: The manifest registers four hook phases. `UserPromptSubmit` runs `loop-plan-file-validator.sh` for every prompt submit, giving the loop a chance to fail closed before the model continues if plan/state invariants are broken (`hooks/hooks.json:4-13`). `PreToolUse` dispatches by tool matcher: `Write` to `loop-write-validator.sh`, `Edit` to `loop-edit-validator.sh`, `Read` to `loop-read-validator.sh`, and `Bash` to `loop-bash-validator.sh` (`hooks/hooks.json:14-51`). `PostToolUse` only watches `Bash` and runs `loop-post-bash-hook.sh` (`hooks/hooks.json:52-62`). `Stop` runs `loop-codex-stop-hook.sh` with a 7200 second timeout, making loop exit conditional on the stop hook’s validation/review flow (`hooks/hooks.json:63-70`).
- inputs_outputs_state: The manifest input is Claude Code’s hook event stream plus the plugin root environment variable. Every command uses `${CLAUDE_PLUGIN_ROOT}/hooks/...`, so resolution depends on `CLAUDE_PLUGIN_ROOT` being set to the installed Humanize plugin root (`hooks/hooks.json:9`, `hooks/hooks.json:20`, `hooks/hooks.json:29`, `hooks/hooks.json:38`, `hooks/hooks.json:47`, `hooks/hooks.json:58`, `hooks/hooks.json:68`). The manifest output is declarative configuration consumed by Claude Code; the actual hook outputs are produced by the scripts as exit codes, stderr guidance, or JSON decisions such as `{"decision":"block"}`. State mutations are delegated: the post-Bash hook patches `session_id` into `state.md` after setup (`hooks/loop-post-bash-hook.sh:120-138`), while the stop hook renames active state files into phase or terminal files such as `finalize-state.md` and `complete-state.md` (`hooks/loop-codex-stop-hook.sh:1281-1288`, `hooks/loop-codex-stop-hook.sh:943-952`).
- gates_or_invariants: The manifest enforces coverage by ensuring the critical tool classes have validators before use. The Read/Write/Edit validators protect RLCR loop files from stale-round access, wrong directory access, state-file mutation, prompt/todos modification, and methodology-analysis leakage. The realpath-related invariant is explicit in path comparisons: project root resolution canonicalizes symlink prefixes (`hooks/lib/project-root.sh:14-20`, `hooks/lib/project-root.sh:41-53`), while directory checks use `canonicalize_path_prefix` to canonicalize parent directories without dereferencing the leaf (`hooks/lib/project-root.sh:55-96`). `loop-write-validator.sh` applies that helper before comparing requested and expected loop paths, explicitly avoiding approval of writes through a planted leaf symlink (`hooks/loop-write-validator.sh:332-352`). `loop-read-validator.sh` mirrors the same prefix-canonical comparison for reads (`hooks/loop-read-validator.sh:305-322`). Methodology-analysis read/write/edit gates also use `realpath`, parent realpath fallback for not-yet-existing paths, raw absolute fallback, and traversal rejection when canonicalization fails (`hooks/loop-write-validator.sh:88-132`, `hooks/loop-edit-validator.sh:71-113`, `hooks/loop-read-validator.sh:86-168`). The Bash validator blocks shell bypasses for state files, goal tracker, prompt, summary, contract, todos, direct stop-hook execution, and methodology-analysis writes (`hooks/loop-bash-validator.sh:81-153`, `hooks/loop-bash-validator.sh:160-184`, `hooks/loop-bash-validator.sh:239-553`).

- edge_cases_or_failure_modes: If `${CLAUDE_PLUGIN_ROOT}` is missing or wrong, all command paths in the manifest resolve incorrectly and the hook system cannot reach validators.
