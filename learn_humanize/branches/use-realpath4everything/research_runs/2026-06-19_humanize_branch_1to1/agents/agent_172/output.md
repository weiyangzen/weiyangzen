### USE_REALPATH4EVERYTHING-HZ-172 `file` `scripts/lib/config-loader.sh`
- cursor: `[_]`
- core_role: Shared Bash configuration resolver for Humanize runtime scripts. It centralizes JSON config discovery, validation, merge precedence, and scalar value extraction so callers such as `scripts/bitlesson-select.sh` and `hooks/lib/loop-common.sh` do not read `.humanize/config.json` directly. Its role is core runtime routing/config behavior, especially model/provider defaults and project-specific overrides, but it is not itself a realpath/path-canonicalization implementation.
- algorithmic_behavior: The script is source-guarded with `_CONFIG_LOADER_LOADED` to avoid double sourcing (`scripts/lib/config-loader.sh:2-4`) and enables strict Bash mode (`scripts/lib/config-loader.sh:6`). `load_merged_config <plugin_root> <project_root>` validates required arguments, requires `jq`, computes three concrete config paths, normalizes each layer into a temporary JSON object file, then deep-merges the layers with `jq` object multiplication (`scripts/lib/config-loader.sh:63-136`). The merge order is empty object, required plugin default config, optional user config, optional project config; later layers override earlier layers (`scripts/lib/config-loader.sh:113-132`). Before merging, `strip_nulls` recursively removes null-valued object entries and null array entries, so a higher-precedence null cannot erase a lower-precedence value (`scripts/lib/config-loader.sh:119-126`). `_config_loader_prepare_layer` is the per-layer gate: missing optional layers become `{}`, missing/malformed required layers are fatal, and malformed optional layers warn and become `{}` (`scripts/lib/config-loader.sh:24-61`). `get_config_value <merged_config_json> <key>` reads a top-level key using `jq -r`; strings are emitted as-is, null/missing are empty, and other JSON scalars/structures are stringified (`scripts/lib/config-loader.sh:139-159`).
- inputs_outputs_state: Inputs are `plugin_root`, `project_root`, environment variables `XDG_CONFIG_HOME`, `HOME`, and `HUMANIZE_CONFIG`, plus JSON files at `$plugin_root/config/default_config.json`, `${XDG_CONFIG_HOME:-$HOME/.config}/humanize/config.json`, and `${HUMANIZE_CONFIG:-$project_root/.humanize/config.json}` (`scripts/lib/config-loader.sh:77-88`). Output of `load_merged_config` is merged JSON on stdout; warnings/errors go to stderr. Output of `get_config_value` is a newline-terminated string value or empty output. The only persistent shell state is the source guard variable `_CONFIG_LOADER_LOADED`; merge scratch state is isolated in a `mktemp -d` directory and removed by an EXIT trap inside the subshell (`scripts/lib/config-loader.sh:90-101`). No repo files are modified by normal operation.
- gates_or_invariants: `jq` must exist or loading fails (`scripts/lib/config-loader.sh:17-21`, `scripts/lib/config-loader.sh:75`). `load_merged_config` requires both root arguments (`scripts/lib/config-loader.sh:70-73`). The default config is required and must be a JSON object (`scripts/lib/config-loader.sh:109`; fatal behavior implemented at `scripts/lib/config-loader.sh:40-56`). User and project config files are optional, but if present they must be JSON objects to participate in the merge; malformed optional files are ignored with a warning (`scripts/lib/config-loader.sh:52-59`). The loader intentionally uses `exit 1` inside `_config_loader_prepare_layer` for fatal errors because the helper is called inside the `load_merged_config` subshell and the comment notes that `set -e` does not reliably propagate through nested if-body calls (`scripts/lib/config-loader.sh:41-46`). Merge semantics are top-level/deep object merge via jq `*`; arrays are replaced rather than concatenated by jq’s merge semantics after null stripping.
- dependencies_and_callers: Runtime dependencies are Bash, `jq`, `mktemp`, `printf`, and `rm`. Direct runtime caller `scripts/bitlesson-select.sh` sources this file, loads merged config, reads `bitlesson_model`, `codex_model`, and `provider_mode`, then uses those values to choose Codex vs Claude selector routing and Codex-only fallback behavior (`scripts/bitlesson-select.sh:9-26`, `scripts/bitlesson-select.sh:115-130`). `hooks/lib/loop-common.sh` also sources the loader while preserving/restoring the caller’s `errexit`, `nounset`, and `pipefail` settings, then best-effort loads config and derives `DEFAULT_BITLESSON_MODEL`, `DEFAULT_CODEX_MODEL`, `DEFAULT_CODEX_EFFORT`, and `DEFAULT_AGENT_TEAMS` (`hooks/lib/loop-common.sh:180-238`). `commands/gen-plan.md` documents this loader as the authoritative config mechanism and explicitly says to reuse it rather than reading `.humanize/config.json` directly (`commands/gen-plan.md:62-74`). `docs/usage.md` describes the public 4-layer hierarchy as plugin defaults, user config, project config, then CLI flags where available (`docs/usage.md:256-274`). `config/default_config.json` supplies the required default layer, currently including `codex_model`, `codex_effort`, `bitlesson_model`, `agent_teams`, `alternative_plan_language`, and `gen_plan_mode` (`config/default_config.json:1-8`).
- edge_cases_or_failure_modes: If `jq` is absent, all consumers that call `load_merged_config` fail unless they intentionally swallow the error, as `loop-common.sh` does with `|| true` (`hooks/lib/loop-common.sh:189-200`). If `plugin_root` points somewhere without `config/default_config.json`, `load_merged_config` exits nonzero; tests assert that missing default config is fatal (`tests/test-config-error-handling.sh:36-53`). If optional user/project config is malformed, the loader emits a warning and falls back to valid lower layers (`tests/test-config-error-handling.sh:55-110`). Empty optional config `{}` is valid and preserves defaults (`tests/test-config-error-handling.sh:112-128`). Missing optional project/user config is nonfatal (`tests/test-config-error-handling.sh:130-172`). `HUMANIZE_CONFIG` replaces the default project config path entirely, so `$project_root/.humanize/config.json` is ignored when the environment override is set (`scripts/lib/config-loader.sh:84-88`; tested at `tests/test-config-merge.sh:152-173`). Nulls in a higher layer are stripped before merge and therefore cannot clear defaults (`tests/test-config-merge.sh:134-150`); this is useful for safe fallback, but it also means users cannot intentionally unset a default with JSON null.
- validation_or_tests: `tests/test-config-merge.sh` directly sources `scripts/lib/config-loader.sh` and validates default-only behavior, project-over-default precedence, project-over-user precedence, additive merge, recursive null stripping, `HUMANIZE_CONFIG` override path behavior, and all-layer contribution (`tests/test-config-merge.sh:1-200`). `tests/test-config-error-handling.sh` validates fatal missing default config, warning/fallback for malformed optional project/user configs, empty object handling, and missing optional config handling (`tests/test-config-error-handling.sh:1-174`). `tests/test-unified-codex-config.sh` also exercises config-backed Codex defaults through the broader runtime surface, including fallback and warnings for invalid model/effort values in `loop-common.sh` (`hooks/lib/loop-common.sh:211-230`; references found in `tests/test-unified-codex-config.sh`). Documentation-level validation is present in `commands/gen-plan.md`, which requires consumers to preserve loader semantics and treat fatal loader failures as configuration errors while allowing malformed optional configs to warn and continue (`commands/gen-plan.md:68-126`).
