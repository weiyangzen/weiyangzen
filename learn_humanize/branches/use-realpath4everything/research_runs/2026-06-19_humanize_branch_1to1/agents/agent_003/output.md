### USE_REALPATH4EVERYTHING-HZ-003 `directory` `commands`

- core_role: `commands` is a flat user-facing command-spec directory for the Humanize RLCR workflow. It contains four Claude slash-command Markdown specs: `commands/gen-plan.md`, `commands/refine-plan.md`, `commands/start-rlcr-loop.md`, and `commands/cancel-rlcr-loop.md`. These files are not shell implementations themselves; their core role is to define allowed tools, preflight gates, sequencing rules, user-interaction gates, and handoff contracts into sibling executable logic under `scripts/`, `hooks/`, `agents/`, and `prompt-template/`. This is core algorithm surface because it defines how planning, refinement, loop startup, Codex review routing, and cancellation are orchestrated.

- algorithmic_behavior: `commands/start-rlcr-loop.md` defines the RLCR start pipeline. It first performs a plan compliance pre-check unless skipped for `--skip-impl` or help, extracts the plan path from `$ARGUMENTS`, applies a relative-path/safe-character prefilter, reads the plan, and invokes `humanize:plan-compliance-checker`; malformed or failing checker output blocks fail-closed (`commands/start-rlcr-loop.md:13-57`). It then optionally runs a non-blocking plan-understanding quiz via `humanize:plan-understanding-quiz`, asks two multiple-choice questions, and lets the user proceed or stop after wrong answers (`commands/start-rlcr-loop.md:60-106`). If gates pass, it runs `scripts/setup-rlcr-loop.sh` with the original arguments (`commands/start-rlcr-loop.md:109-115`). The underlying setup script parses loop flags, resolves project root, checks dependencies, enforces one active loop, validates plan path/content/git cleanliness, creates `.humanize/rlcr/<timestamp>/state.md`, prompt, summary scaffold, goal tracker, and pending session marker (`scripts/setup-rlcr-loop.sh:188-427`, `scripts/setup-rlcr-loop.sh:430-620`, `scripts/setup-rlcr-loop.sh:620-918`, `scripts/setup-rlcr-loop.sh:1195-1541`).

- algorithmic_behavior: `commands/cancel-rlcr-loop.md` is a thin cancellation controller. It runs `scripts/cancel-rlcr-loop.sh`, dispatches on the first output token, and only asks the user when the active loop is in Finalize Phase (`commands/cancel-rlcr-loop.md:11-35`). The command’s key invariant is delegated cancellation: active loop status is determined by `state.md`, `methodology-analysis-state.md`, or `finalize-state.md` in the newest loop directory (`commands/cancel-rlcr-loop.md:37`). The script resolves project root, finds the newest active loop, selects the active state file, returns `FINALIZE_NEEDS_CONFIRM` without `--force`, otherwise creates `.cancel-requested`, removes pending/session methodology markers, and renames the active state file to `cancel-state.md` (`scripts/cancel-rlcr-loop.sh:69-120`, `scripts/cancel-rlcr-loop.sh:139-186`).

- algorithmic_behavior: `commands/gen-plan.md` defines a sequential planning algorithm from draft to implementation plan. It explicitly forbids coding during plan generation and limits writes to the output plan plus optional language variant before any optional auto-start (`commands/gen-plan.md:20-29`). The phases are ordered and non-parallel: mode setup, merged config load, IO validation, relevance check, first Codex analysis, Claude candidate plan, Claude/Codex convergence rounds, issue resolution, final plan generation, write/report (`commands/gen-plan.md:32-45`). It uses `scripts/validate-gen-plan-io.sh` for input/output validation and template discovery (`commands/gen-plan.md:132-150`; `scripts/validate-gen-plan-io.sh:98-179`), uses `scripts/ask-codex.sh` before Claude synthesis and again during convergence (`commands/gen-plan.md:184-213`, `commands/gen-plan.md:255-299`), consolidates unresolved questions into `## Pending User Decisions` (`commands/gen-plan.md:314-329`), and emits a plan schema matching `prompt-template/plan/gen-plan-template.md` (`commands/gen-plan.md:372-530`; `prompt-template/plan/gen-plan-template.md:1-120`).

- algorithmic_behavior: `commands/refine-plan.md` defines an annotated-plan refinement algorithm. It is planning-only, permits writes only to refined plan, QA document, and optional translated variants, and requires reuse of the existing `gen-plan` schema (`commands/refine-plan.md:19-28`). Its phases parse arguments, load config, validate IO, statefully extract comments, classify each comment, process comments, produce a comment-free refined plan, generate QA, and perform atomic writes (`commands/refine-plan.md:30-43`). The command’s scanner contract supports `CMT:/ENDCMT`, `<cmt></cmt>`, and `<comment></comment>`, ignores markers inside fenced code/HTML comments, tracks nearest heading and line/column metadata, rejects nested/stray/missing markers, and removes empty blocks without allocating IDs (`commands/refine-plan.md:184-274`). The validator implements the scanner with awk and enforces required `gen-plan` sections after comment masking (`scripts/validate-refine-plan-io.sh:16-315`, `scripts/validate-refine-plan-io.sh:318-477`, `scripts/validate-refine-plan-io.sh:608-637`).
