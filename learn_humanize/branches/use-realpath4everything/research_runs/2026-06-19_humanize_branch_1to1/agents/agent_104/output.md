### USE_REALPATH4EVERYTHING-HZ-104 `file` `hooks/lib/loop-bg-tasks.sh`
- cursor: `[_]`
- core_role: Background-task state gate for the RLCR Stop hook. The file is sourced through `hooks/lib/loop-common.sh:1568-1578`, then called by `hooks/loop-codex-stop-hook.sh:61-83` immediately after active-loop discovery and before normal phase/state parsing, git gates, summaries, BitLesson gates, max-iteration checks, and Codex review. Its job is to detect Claude Code async Agent/Bash work that is still running and park the RLCR loop without falsely advancing or terminating it.

- algorithmic_behavior: The module exposes small helpers plus one Stop-hook entrypoint. `expand_leading_tilde` expands only bare `~` and `~/...` without `eval` (`hooks/lib/loop-bg-tasks.sh:23-36`). `extract_transcript_path` reads `.transcript_path` from hook JSON via `jq` and normalizes leading tilde (`hooks/lib/loop-bg-tasks.sh:38-46`). `derive_loop_start_iso_ts` converts a loop directory basename like `YYYY-MM-DD_HH-MM-SS` from local wall time to UTC ISO text so transcript launch events can be filtered lexically against the loop start (`hooks/lib/loop-bg-tasks.sh:48-95`). `derive_tasks_dir_from_transcript` maps Claude transcript path shape `<claude-home>/projects/<slug>/<session-id>.jsonl` to `/tmp/claude-<uid>/<slug>/<session-id>/tasks` (`hooks/lib/loop-bg-tasks.sh:97-118`). `is_bg_task_alive` treats a task as alive if its output file is missing or `lsof` shows an open holder, and as dead only when the output file exists and `lsof` reports no holder (`hooks/lib/loop-bg-tasks.sh:120-141`). `prune_dead_bg_task_ids` filters pending IDs through that liveness probe (`hooks/lib/loop-bg-tasks.sh:143-154`).

  The main transcript algorithm is `list_pending_background_task_ids` (`hooks/lib/loop-bg-tasks.sh:156-265`). It requires a readable transcript file and `jq`; otherwise it returns non-zero so callers treat the state as unknown. It collects launches from transcript records with `.toolUseResult.isAsync == true` and `.toolUseResult.agentId`, or `.toolUseResult.backgroundTaskId` for background shell tasks (`hooks/lib/loop-bg-tasks.sh:209-222`). If a `since_ts` is present, launch records with `.timestamp < since_ts` are ignored, while timestamp-less records remain included for older transcripts and fixtures (`hooks/lib/loop-bg-tasks.sh:179-185`, `210-216`). Completion IDs are collected from both current SDK task notifications, `type == "system"` and `subtype == "task_notification"`, and legacy queue-operation XML snippets containing `<task-id>` (`hooks/lib/loop-bg-tasks.sh:224-246`). Pending is `launched - completed` using sorted unique sets and `comm -23` (`hooks/lib/loop-bg-tasks.sh:248-252`), then orphaned IDs can be pruned with the derived task-output directory (`hooks/lib/loop-bg-tasks.sh:254-264`). `has_pending_background_tasks` and `count_pending_background_tasks` are thin wrappers over this list function (`hooks/lib/loop-bg-tasks.sh:267-297`).

  The Stop-hook entrypoint, `handle_bg_task_short_circuit`, runs four ordered guards (`hooks/lib/loop-bg-tasks.sh:299-435`). It first derives the loop-start timestamp from the loop directory and extracts the transcript path from hook input (`hooks/lib/loop-bg-tasks.sh:314-320`). Guard 1 exits silently when `bg-pending.marker` exists but the hook invocation has no `session_id`, because the caller may be the parked owner or a different session and the hook cannot prove authority (`hooks/lib/loop-bg-tasks.sh:322-336`). Guard 2 detects a parked loop owned by a different stored `session_id`, emits a JSON `systemMessage` saying the loop is parked by another Claude session, and exits without touching disk (`hooks/lib/loop-bg-tasks.sh:338-367`). Guard 3 scans for pending task IDs; if any remain, it writes `bg-pending.marker`, emits a JSON `systemMessage` with the pending count, exits 0, and leaves the RLCR loop state untouched (`hooks/lib/loop-bg-tasks.sh:369-405`). Guard 4 removes a stale same-session marker only after a fresh successful transcript parse returns an empty pending list (`hooks/lib/loop-bg-tasks.sh:407-434`).

- inputs_outputs_state: Inputs are the active RLCR loop directory, raw Stop-hook JSON, hook session ID, the Claude transcript JSONL path, current `HOME`, local timezone, current uid from `id -u`, optional `LSOF_BIN`, `jq`, `date`, `sort`, `comm`, `sed`, `grep`, `wc`, and `lsof`. Outputs are newline-delimited pending task IDs for helper calls, numeric pending counts, boolean-style shell exit codes, or Stop-hook JSON on stdout. The Stop-hook JSON intentionally has no `.decision` field for allow semantics; `scripts/rlcr-stop-gate.sh:162-173` maps that to `ALLOW:` and surfaces the `systemMessage`.

  Persistent state transition is narrow: pending background work creates or refreshes `$loop_dir/bg-pending.marker` (`hooks/lib/loop-bg-tasks.sh:397-403`); same-session verified completion removes it (`hooks/lib/loop-bg-tasks.sh:428-433`). The module does not rewrite `state.md`, `finalize-state.md`, or `methodology-analysis-state.md`. It reads `session_id` from the active state file using `resolve_active_state_file` and `FIELD_SESSION_ID` from `loop-common.sh` (`hooks/lib/loop-bg-tasks.sh:307-308`, `355-359`). Cross-session and ambiguous-call paths deliberately preserve marker and state artifacts.

- gates_or_invariants: The entrypoint must run before every normal Stop-hook gate; `hooks/loop-codex-stop-hook.sh:71-83` places it before active state resolution and phase parsing. Transcript failures are fail-closed for parking cleanup: missing transcript, non-file path, absent `jq`, parse error, or helper non-zero does not prove "no pending" and must not delete `bg-pending.marker` (`hooks/lib/loop-bg-tasks.sh:187-192`, `416-431`). Liveness is fail-open for running tasks: missing output file, missing `lsof`, or unexpected `lsof` failure keeps a task alive (`hooks/lib/loop-bg-tasks.sh:120-128`, `135-140`). Loop-start filtering must compare UTC timestamps derived from local loop-dir names because setup creates loop dirs in local wall-clock time (`hooks/lib/loop-bg-tasks.sh:51-59`, `75-94`). Session authority is an invariant: a foreign session with a parked marker may be informed and allowed to stop, but it cannot inspect, advance, clean, or rewrite the parked loop (`hooks/lib/loop-bg-tasks.sh:338-365`). Empty stored session IDs retain backward compatibility via `find_active_loop`, but the cross-session guard requires both stored and hook session IDs to be non-empty before treating them as a mismatch (`hooks/lib/loop-bg-tasks.sh:349-361`).

- dependencies_and_callers: Direct dependency on `loop-common.sh` symbols is documented at `hooks/lib/loop-bg-tasks.sh:15-16` and `307-308`: `FIELD_SESSION_ID` and `resolve_active_state_file`. The source chain is `hooks/loop-codex-stop-hook.sh` -> `hooks/lib/loop-common.sh` -> `hooks/lib/loop-bg-tasks.sh`; the hook extracts `HOOK_SESSION_ID`, calls `find_active_loop "$LOOP_BASE_DIR" "$HOOK_SESSION_ID" true`, then delegates to this module (`hooks/loop-codex-stop-hook.sh:42-83`). `find_active_loop` in `hooks/lib/loop-common.sh:325-329` has an opt-in `allow_bg_marker_fallback` used only by the Stop hook. - edge_cases_or_failure_modes: A literal `~/...` transcript path is normalized, but `~user/...`, relative paths, empty strings, and non-tilde paths are left as passed (`hooks/lib/loop-bg-tasks.sh:23-36`). Invalid loop directory names or unparsable local dates produce an empty `since_ts`, which disables launch-time filtering rather than failing the hook (`hooks/lib/loop-bg-tasks.sh:65-95`, `314-320`). Timestamp-less transcript launches are included even when a boundary exists, which preserves older Claude transcript compatibility but can include pre-loop async work from old formats (`hooks/lib/loop-bg-tasks.sh:179-185`, `210-216`). Completion status values are all treated as terminal once a matching task notification appears; the code does not distinguish completed, failed, stopped, or other statuses (`hooks/lib/loop-bg-tasks.sh:165-171`). The legacy completion parser uses regex over XML-like snippets and intentionally guards `grep -oE` with `|| true` so no legacy match does not poison the pipeline under `pipefail` (`hooks/lib/loop-bg-tasks.sh:224-245`). If the derived `/tmp/claude-.../tasks` directory cannot be derived, orphan pruning is skipped and pending transcript IDs remain pending (`hooks/lib/loop-bg-tasks.sh:254-261`). If `lsof` is unavailable, tasks with output files are still treated as alive (`hooks/lib/loop-bg-tasks.sh:137-140`). A wrapper invocation without `session_id` and an existing marker exits silently with no state mutation, which avoids ambiguous ownership but means non-hook automation cannot clear that marker without an authoritative session context (`hooks/lib/loop-bg-tasks.sh:322-336`).
