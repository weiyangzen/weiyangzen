### USE_REALPATH4EVERYTHING-HZ-124 `file` `prompt-template/block/git-tracked-humanize.md`
- cursor: `[_]`
- core_role: This file is a static stop/block prompt shown when Humanize runtime state under `.humanize/` has been accidentally staged or tracked by git. It is part of the RLCR stop gate’s version-control hygiene contract: local loop state must not enter project history. The template’s title and primary diagnosis are at `prompt-template/block/git-tracked-humanize.md:1-5`, and its remediation steps are at `prompt-template/block/git-tracked-humanize.md:7-14`.
- algorithmic_behavior: The template itself contains no executable logic and no interpolation placeholders. Its algorithmic role is as the user-facing `reason` payload for a guard implemented in `hooks/lib/loop-common.sh`. The guard function `git_has_tracked_humanize_state` checks whether git lists files under `.humanize` with `git -C "$project_root" ls-files -- .humanize`; if any entry exists, it returns success to mean blocked state detected (`hooks/lib/loop-common.sh:1418-1439`). `git_tracked_humanize_blocked_message` then loads this template via `load_and_render_safe "$TEMPLATE_DIR" "block/git-tracked-humanize.md" "$fallback"` (`hooks/lib/loop-common.sh:1442-1465`). The stop hook consumes that message when the tracked-state predicate is true and returns JSON with `"decision": "block"` plus a system message telling the caller to remove tracked Humanize state first (`hooks/loop-codex-stop-hook.sh:634-650`).
- inputs_outputs_state: Inputs are indirect: the project root, git repository status, and the template directory selected by `TEMPLATE_DIR`, initialized from `get_template_dir` after sourcing the template loader (`hooks/lib/loop-common.sh:240-248`). The template has no dynamic variables, so rendering is equivalent to reading the markdown file when present; otherwise the identical inline fallback in `git_tracked_humanize_blocked_message` is used (`hooks/lib/loop-common.sh:1445-1465`). Output is a markdown block with a required fix: run `git rm --cached -r .humanize`, keep only real project files staged, and retry the stop action (`prompt-template/block/git-tracked-humanize.md:7-14`). State transition is external: tracked or staged `.humanize/` entries cause the stop action to move into a blocked response; after the user removes `.humanize` from the index, `git ls-files -- .humanize` no longer returns entries and this specific guard no longer blocks.
- gates_or_invariants: The core invariant is that `.humanize/` contains local Humanize loop state and must remain outside version control (`prompt-template/block/git-tracked-humanize.md:3-5`). The guard is intentionally scoped to the `.humanize/` directory, not similarly named paths, to stay consistent with git-add protection and avoid blocking unrelated `.humanize-backup` or `.humanizeconfig` paths (`hooks/lib/loop-common.sh:1422-1427`). It also treats `git rm --cached` as the unstick path because removed-from-index entries are omitted by `git ls-files` (`hooks/lib/loop-common.sh:1425-1427`). The template reinforces two negative constraints: do not force-add Humanize state, and do not commit RLCR trackers, round summaries, contracts, or cancel/finalize markers (`prompt-template/block/git-tracked-humanize.md:16-19`).
