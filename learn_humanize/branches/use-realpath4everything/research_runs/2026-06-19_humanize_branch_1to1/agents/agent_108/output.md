- algorithmic_behavior: The file exposes six active helpers plus one private fallback helper. `get_template_dir` computes the plugin template root by walking from the caller-supplied `hooks/lib` path up two directories and appending `prompt-template` at `hooks/lib/template-loader.sh:26-31`. `load_template` concatenates `template_dir` and `template_name`, emits file contents when present, and warns to stderr when missing at `hooks/lib/template-loader.sh:36-45`. `render_template` is the core renderer at `hooks/lib/template-loader.sh:56-136`: it accepts template content plus `VAR=value` arguments, maps them into `TMPL_VAR_*` environment entries at `hooks/lib/template-loader.sh:60-67`, then runs one `awk` pass over each line at `hooks/lib/template-loader.sh:73-128`. The `awk` loop searches for `{{`, finds the next `}}`, extracts the raw name between them, replaces it only if that exact name exists in the environment-derived map, and leaves unknown placeholders intact at `hooks/lib/template-loader.sh:87-124`. Replacement values are appended directly to the output string and are not rescanned, which is the explicit prompt-injection guard described at `hooks/lib/template-loader.sh:52-55`. `load_and_render` composes `load_template` and `render_template` but returns empty output for missing templates at `hooks/lib/template-loader.sh:140-151`. `append_template` appends a second template’s content to base content only when the second file loads non-empty at `hooks/lib/template-loader.sh:156-168`; `rg` found no current external caller, so it is presently a library utility rather than active hook flow. `_emit_fallback` renders fallback text when substitutions are supplied at `hooks/lib/template-loader.sh:175-183`. `load_and_render_safe` is the operationally important safe path: it suppresses missing-template load errors, falls back on missing or empty template content, renders the selected content, and falls back again if rendering produces empty output at `hooks/lib/template-loader.sh:188-211`. `validate_template_dir` checks the existence of the template root and required subdirectories `block`, `codex`, `claude`, and `plan` at `hooks/lib/template-loader.sh:216-238`.

- gates_or_invariants: The strongest renderer invariant is single-pass substitution: values containing `{{OTHER_VAR}}` remain literal and are not expanded, preventing review text or summaries from corrupting prompt structure. - edge_cases_or_failure_modes: `get_template_dir` trusts the caller-supplied `script_dir`; it performs no separate validation and uses `pwd` rather than an explicit physical `realpath`, so unusual symlink spellings or a bad caller path can produce an invalid template path until `validate_template_dir` catches it. `load_template` performs simple path concatenation and has no traversal defense; this is acceptable for current constant template names but would matter if a caller passed user-controlled template names. If a file exists but `cat` fails, the safe wrapper swallows the load failure and falls back, while bare `load_and_render` can propagate empty or failed output. `render_template` is line-oriented through `awk` and shell command substitution; trailing newlines are not a fidelity guarantee, binary/NUL content is out of scope, and very large environment payloads remain bounded by OS environment/argument limits despite tests covering up to 100KB values and templates. Malformed placeholders are not fatal: unclosed `{{` is treated as literal text by the branch at `hooks/lib/template-loader.sh:101-107`, unknown placeholders remain unchanged, empty placeholders can remain or be replaced only if an empty-name environment entry is supplied, and nested/extra braces are processed mechanically by the “next `}}`” scanner rather than a grammar. Duplicate variable assignments are resolved by the environment/awk map with the later effective environment value for a repeated `TMPL_VAR_*` name. `load_and_render_safe` treats an empty rendered result as failure and emits fallback at `hooks/lib/template-loader.sh:202-207`, which is desirable for block messages but means an intentionally empty template cannot be distinguished from a failed render through that API.
