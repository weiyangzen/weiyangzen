Its role is to seed a project-local BitLesson knowledge base and define the strict lesson-entry schema expected by the RLCR BitLesson workflow. The file is not executable code, but it is part of the algorithm’s control surface because downstream selector and stop-gate scripts rely on the presence or absence of concrete `## Lesson:` entries and `Lesson ID:` fields.
- algorithmic_behavior: `templates/bitlesson.md` defines the initial BitLesson KB shape. Lines 1-3 identify the file as a project-specific reusable knowledge base. Lines 5-19 define the exact entry template and field order: `## Lesson`, `Lesson ID`, `Scope`, `Problem Description`, `Root Cause`, `Solution`, `Constraints`, `Validation Evidence`, and `Source Rounds`. Lines 21-23 intentionally leave the `Entries` section empty with an HTML comment placeholder. This placeholder state is meaningful: `scripts/bitlesson-select.sh` treats a BitLesson file with no `## Lesson:` headings as an empty KB and returns `LESSON_IDS: NONE` without invoking a model (`scripts/bitlesson-select.sh:105-108`).
- inputs_outputs_state: The directory has no runtime inputs by itself. Its child template is an input to `scripts/bitlesson-init.sh`, which accepts `--project-root`, `--template`, and optional `--bitlesson-relpath` (`scripts/bitlesson-init.sh:7-14`). The output is a copied project KB at `<project-root>/.humanize/bitlesson.md` when missing (`scripts/bitlesson-init.sh:83-86`), and the script prints the resolved BitLesson path (`scripts/bitlesson-init.sh:88`). In normal RLCR setup, `scripts/setup-rlcr-loop.sh` sets `BITLESSON_FILE_REL=".humanize/bitlesson.md"`, points `PLUGIN_BITLESSON_TEMPLATE` at `../templates/bitlesson.md`, and invokes `bitlesson-init.sh` with those values (`scripts/setup-rlcr-loop.sh:860-866`). State transition is therefore: no project KB -> copy template -> empty-but-valid KB; existing KB -> no overwrite; concrete entries later added by users/agents -> selector and validator can reference IDs.
