### USE_REALPATH4EVERYTHING-HZ-036 `file` `docs/bitlesson.md`
- cursor: `[_]`
- core_role: `docs/bitlesson.md` is behavior-defining documentation for the RLCR BitLesson memory workflow. It specifies the project knowledge-base location, selector configuration/routing, initialization behavior, per-task lesson selection, and the stop-gate summary contract. The central runtime artifact is `.humanize/bitlesson.md`, documented at `docs/bitlesson.md:27`, initialized from `templates/bitlesson.md` as described at `docs/bitlesson.md:31`.
- algorithmic_behavior: The workflow is a small state machine around lesson memory and round summaries. At loop setup/start, the BitLesson file is created if absent from the strict template (`docs/bitlesson.md:29-32`; implementation in `scripts/bitlesson-init.sh:83-86`). During each task/sub-task, `scripts/bitlesson-select.sh` receives task text, related paths, and the BitLesson file, then either short-circuits to `LESSON_IDS: NONE` when no `## Lesson:` headings exist (`scripts/bitlesson-select.sh:105-109`) or sends a bounded prompt to Codex/Claude and normalizes the response to exactly `LESSON_IDS:` and `RATIONALE:` (`scripts/bitlesson-select.sh:148-179`, `scripts/bitlesson-select.sh:242-267`). At round stop, `scripts/bitlesson-validate-delta.sh` extracts the visible `## BitLesson Delta` block, validates the action/IDs/notes, and returns a JSON block decision when the summary violates the contract (`scripts/bitlesson-validate-delta.sh:78-90`, `scripts/bitlesson-validate-delta.sh:181-218`, `scripts/bitlesson-validate-delta.sh:242-385`).
- inputs_outputs_state: Inputs are merged config values, task description, path list, BitLesson knowledge-base content, round summary, current round, and the allow-empty-none flag. The documented config hierarchy is default config, user config, project config, then CLI flags where applicable (`docs/bitlesson.md:7-13`); the loader implements default/user/project merging through `config/default_config.json`, `${XDG_CONFIG_HOME:-$HOME/.config}/humanize/config.json`, and `.humanize/config.json` or `HUMANIZE_CONFIG` (`scripts/lib/config-loader.sh:77-88`, `scripts/lib/config-loader.sh:113-132`). Outputs are selected lesson IDs plus rationale during implementation (`docs/bitlesson.md:32-33`; `scripts/bitlesson-select.sh:266-267`), and either no output for a passing stop-gate or a JSON `decision: block` with a message when validation fails (`scripts/bitlesson-validate-delta.sh:78-90`, `tests/test-bitlesson-validate-delta.sh:86-94`). The persistent state transitions are knowledge base absent -> template file present, no applicable lesson -> `NONE`, selected lesson IDs -> implementation guidance, and round summary delta -> accepted or blocked.
- gates_or_invariants: The required summary block shape is fixed: `## BitLesson Delta`, `Action: none|add|update`, `Lesson ID(s): <IDs or NONE>`, and `Notes:` (`docs/bitlesson.md:36-45`). `Action: none` must have `Lesson ID(s): NONE` or an empty field (`docs/bitlesson.md:47-50`; enforced at `scripts/bitlesson-validate-delta.sh:242-252`). `Action: add` and `Action: update` require concrete IDs that match `BL-YYYYMMDD-short-name` and exist in `.humanize/bitlesson.md` (`docs/bitlesson.md:49-50`; enforced at `scripts/bitlesson-validate-delta.sh:269-339`, `scripts/bitlesson-validate-delta.sh:354-384`). Add/update also require non-empty, non-placeholder notes, an invariant present in implementation and tests though not fully spelled out in `docs/bitlesson.md` (`scripts/bitlesson-validate-delta.sh:281-297`; `tests/test-bitlesson-validate-delta.sh:101-114`). The delta heading must be visible normal markdown, not inside fences or HTML comments (`scripts/bitlesson-validate-delta.sh:92-178`; tested at `tests/test-bitlesson-validate-delta.sh:116-152`). BitLesson enforcement is enabled through loop state `bitlesson_required: true`, but skipped in finalize and disabled for skip-impl setup (`hooks/loop-codex-stop-hook.sh:823-838`; `scripts/setup-rlcr-loop.sh:875-901`).
- dependencies_and_callers: `docs/bitlesson.md` points to `templates/bitlesson.md`, `scripts/bitlesson-select.sh`, and the stop gate (`docs/bitlesson.md:31-34`). `templates/bitlesson.md` defines strict entry field order for new lessons, including `Lesson ID`, `Scope`, `Problem Description`, `Root Cause`, `Solution`, `Constraints`, `Validation Evidence`, and `Source Rounds` (`templates/bitlesson.md:5-19`). `scripts/setup-rlcr-loop.sh` initializes `.humanize/bitlesson.md`, writes BitLesson state fields, creates the summary scaffold, and appends selection instructions to round prompts (`scripts/setup-rlcr-loop.sh:857-867`, `scripts/setup-rlcr-loop.sh:880-901`, `scripts/setup-rlcr-loop.sh:1183-1188`, `scripts/setup-rlcr-loop.sh:1349-1368`). `hooks/loop-codex-stop-hook.sh` reads frontmatter fields, sanitizes the relative BitLesson path, invokes the validator on non-finalize rounds, and injects BitLesson instructions into next-round/review prompts (`hooks/loop-codex-stop-hook.sh:115-168`, `hooks/loop-codex-stop-hook.sh:820-838`, `hooks/loop-codex-stop-hook.sh:1521-1533`, `hooks/loop-codex-stop-hook.sh:2039-2051`).
