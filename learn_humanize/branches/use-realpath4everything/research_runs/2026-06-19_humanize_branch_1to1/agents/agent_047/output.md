### USE_REALPATH4EVERYTHING-HZ-047 `file` `scripts/ask-codex.sh`
- cursor: `[_]`
- core_role: `scripts/ask-codex.sh` is a one-shot Codex consultation runtime script. It is not the RLCR loop scheduler itself, but it is an active workflow/routing script used by the Humanize skill surface to invoke `codex exec`, persist an invocation record, and return Codex stdout cleanly to the caller. The script declares this role directly in its header and usage comments at `scripts/ask-codex.sh:3-17`, and the corresponding skill entry exposes it as the allowed Bash target at `skills/ask-codex/SKILL.md:2-5`.

- algorithmic_behavior: The script runs as a strict Bash program with `set -euo pipefail` at `scripts/ask-codex.sh:20`. Its algorithm is linear and gate-heavy:
  1. Resolve its own script directory and source dependencies: `scripts/portable-timeout.sh` for timeout execution and `hooks/lib/loop-common.sh` for shared Codex defaults and project-root resolution (`scripts/ask-codex.sh:26-33`).
  2. Initialize default model, effort, and timeout from shared config-backed defaults plus a local `3600` second fallback (`scripts/ask-codex.sh:39-43`).
  3. Parse CLI flags until the first positional argument or explicit `--`, preserving all remaining tokens as question text (`scripts/ask-codex.sh:86-147`). `--codex-model` accepts either `MODEL:EFFORT` or `MODEL`; `--codex-timeout` must be numeric.
  4. Validate prerequisites and safety constraints: `codex` must be on `PATH`, question must be non-empty, model and effort must match restricted character sets (`scripts/ask-codex.sh:153-186`).
  5. Resolve the canonical project root via `resolve_project_root` (`scripts/ask-codex.sh:192-196`). That resolver prioritizes `CLAUDE_PROJECT_DIR`, then `git rev-parse --show-toplevel`, and realpath-canonicalizes the result (`hooks/lib/project-root.sh:41-53`).
  6. Create per-run project-local storage under `.humanize/skill/<unique-id>` and cache storage under `${XDG_CACHE_HOME:-$HOME/.cache}/humanize/<sanitized-project-path>/skill-<unique-id>`, falling back to `.humanize/skill/<id>/cache` if home cache creation fails (`scripts/ask-codex.sh:202-218`).
  7. Write `input.md` with the question and resolved config (`scripts/ask-codex.sh:224-238`).
  8. Construct `codex exec` arguments: `-m <model>`, optional `-c model_reasoning_effort=<effort>`, sandbox flag, and `-C <project-root>` (`scripts/ask-codex.sh:244-256`).
  9. Save a debug command file with timestamp, working directory, timeout, command shape, and prompt content (`scripts/ask-codex.sh:262-276`).
  10. Pipe the question into `run_with_timeout "$CODEX_TIMEOUT" codex exec ... -`, redirecting stdout/stderr to cache files and recording the exit code without aborting the script (`scripts/ask-codex.sh:294-303`).
  11. Classify result state: timeout, non-zero Codex exit, empty stdout, or success. Each terminal path writes metadata; success also copies stdout to `.humanize/skill/<id>/output.md` and emits stdout to the caller (`scripts/ask-codex.sh:309-415`).

- inputs_outputs_state: Inputs are CLI arguments, environment variables, project context, and external command availability. CLI inputs are `--codex-model`, `--codex-timeout`, `--help`, `--`, and free-form question text (`scripts/ask-codex.sh:89-144`). Environment inputs include `CLAUDE_PROJECT_DIR` through `resolve_project_root`, `XDG_CACHE_HOME`, `HOME`, and `HUMANIZE_CODEX_BYPASS_SANDBOX` (`scripts/ask-codex.sh:211-218`, `scripts/ask-codex.sh:250-254`; resolver behavior at `hooks/lib/project-root.sh:41-53`). Shared model defaults come from `loop-common.sh`, which loads merged config and sets `DEFAULT_CODEX_MODEL`/`DEFAULT_CODEX_EFFORT` using caller preset, config value, or fallback `gpt-5.4/high` (`hooks/lib/loop-common.sh:208-230`). Outputs are:
  - stdout: Codex response only, on success (`scripts/ask-codex.sh:410-415`).
  - stderr: status, cache path, duration, errors, and last Codex stderr lines on relevant failures (`scripts/ask-codex.sh:282-303`, `scripts/ask-codex.sh:335-343`, `scripts/ask-codex.sh:361-370`).
  - project-local state: `.humanize/skill/<unique-id>/input.md`, `output.md` on success, and `metadata.md` on timeout/error/empty/success (`scripts/ask-codex.sh:205-238`, `scripts/ask-codex.sh:318-330`, `scripts/ask-codex.sh:345-357`, `scripts/ask-codex.sh:372-383`, `scripts/ask-codex.sh:391-406`).
  - cache state: `codex-run.cmd`, `codex-run.out`, `codex-run.log` (`scripts/ask-codex.sh:262-276`, `scripts/ask-codex.sh:297-298`).
  The per-run state transition is effectively `initialized -> input_saved -> command_saved -> codex_executed -> timeout|error|empty_response|success`, with the final state recorded in YAML-like metadata.

- gates_or_invariants: The script enforces strict option parsing before execution; unknown flags fail immediately (`scripts/ask-codex.sh:132-135`). `--codex-timeout` must match `^[0-9]+$`, though the error text says “positive integer” and the regex technically accepts `0` (`scripts/ask-codex.sh:120-130`). `CODEX_MODEL` is limited to alphanumeric, dot, underscore, hyphen; `CODEX_EFFORT` is limited to alphanumeric, underscore, hyphen (`scripts/ask-codex.sh:172-186`). Unlike `loop-common.sh` config validation, the script’s CLI effort validation does not restrict values to `xhigh|high|medium|low`; this is broader but injection-resistant. The `codex` executable gate runs before the empty-question gate (`scripts/ask-codex.sh:153-170`), so an empty invocation on a machine without Codex reports missing Codex first. Project root must resolve or the script fails (`scripts/ask-codex.sh:192-196`). Each invocation must have a unique storage id based on timestamp, PID, and 4 random bytes (`scripts/ask-codex.sh:202-203`). Sandbox mode defaults to `--full-auto`; only exact `HUMANIZE_CODEX_BYPASS_SANDBOX=true` or `1` switches to the dangerous bypass flag (`scripts/ask-codex.sh:250-254`). A non-empty stdout file is required for success (`scripts/ask-codex.sh:361-385`).
