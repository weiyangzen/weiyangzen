- algorithmic_behavior: Hook input validation rejects null bytes, invalid UTF-8 when `iconv` exists, invalid JSON, and missing `tool_name`, then exposes `VALIDATED_TOOL_NAME` and `VALIDATED_TOOL_INPUT` (`hooks/lib/loop-common.sh:59-96`). `require_tool_input_field` enforces required `tool_input` members (`hooks/lib/loop-common.sh:101-114`). `is_deeply_nested` uses `jq '[paths | length] | max'` with default max depth `30` to reject DoS-style nested payloads (`hooks/lib/loop-common.sh:119-133`). State parsing exists in tolerant and strict modes: tolerant defaults non-critical fields (`hooks/lib/loop-common.sh:209-244`), while strict mode requires YAML separators, `current_round`, `max_iterations`, `review_started`, `base_branch`, numeric rounds, numeric max iterations, and boolean review state (`hooks/lib/loop-common.sh:255-330`).
- inputs_outputs_state: Inputs are hook JSON payloads, loop directory paths, `state.md`/`finalize-state.md`, shell command strings, review logs, and PR tracker files. - gates_or_invariants: The library encodes several key validators’ invariants. Round files are identified by `round-N-summary.md`, `round-N-prompt.md`, and `round-N-todos.md` patterns (`hooks/lib/loop-common.sh:399-418`). The hardcoded allowlist permits only `round-1-todos.md`, `round-2-todos.md`, `round-0-summary.md`, and `round-1-summary.md` in the active loop directory (`hooks/lib/loop-common.sh:420-441`).
