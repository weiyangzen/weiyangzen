### CHANGE_TODOS_TO_TASKS-HZ-051 `file` `tests/test-codex-review-merge.sh`
- cursor: `[_]`
- core_role: Executable specification for the Codex review-log issue detection algorithm used by the RLCR review phase. It verifies `detect_review_issues()` from `hooks/lib/loop-common.sh`, which determines whether Codex review output contains actionable priority markers and whether the loop should continue or finalize.
- algorithmic_behavior: The test states the algorithm directly: scan the entire log from line 1, find the first line where `[P?]` with digit severity appears in the first 10 characters, extract from that line to EOF, return no issues if absent (`tests/test-codex-review-merge.sh:11-15`). The implementation under test constructs `round-${round}-codex-review.log` and `round-${round}-review-result.md`, rejects missing/empty logs with exit 2, uses awk against `substr($0, 1, 10) ~ /\[P[0-9]\]/`, writes extracted content to the result file, prints a `## Codex Review Issues` wrapper, returns 0 for issues, and returns 1 for no markers (`hooks/lib/loop-common.sh:332-392`).
- inputs_outputs_state: Inputs are `round_number`, plus required globals `LOOP_DIR` and `CACHE_DIR`; the test creates both in a temporary isolated environment (`tests/test-codex-review-merge.sh:47-54`). Input files are synthetic `round-N-codex-review.log` files in `$CACHE_DIR`. Outputs are stdout from `detect_review_issues`, exit code 0/1/2, and an audit file `$LOOP_DIR/round-N-review-result.md` when issues are found (`tests/test-codex-review-merge.sh:306-329`; implementation at `hooks/lib/loop-common.sh:349-386`). In runtime, `hooks/loop-codex-stop-hook.sh` consumes exit 0 as “continue review loop with issues,” exit 1 as “enter finalize phase,” and exit 2 as hard review failure (`hooks/loop-codex-stop-hook.sh:1012-1028`).
- gates_or_invariants: The marker must be within the first 10 characters of a line; later mentions of `[P1]` or `[P2]` are ignored (`tests/test-codex-review-merge.sh:83-107`). The scan covers the whole file, not a tail window, proven by markers at line 55 and line 5 in long logs (`tests/test-codex-review-merge.sh:171-227`). Extraction starts at the first matching marker and includes all following content, including later priority entries and final debug lines (`tests/test-codex-review-merge.sh:229-255`). Missing and empty logs are hard errors, not “no issues” (`tests/test-codex-review-merge.sh:133-169`).
- dependencies_and_callers: The test sources `hooks/lib/loop-common.sh` (`tests/test-codex-review-merge.sh:41-42`). Runtime caller is `run_and_handle_code_review()` in `hooks/loop-codex-stop-hook.sh`, which runs `codex review`, then calls `detect_review_issues "$round"` (`hooks/loop-codex-stop-hook.sh:1000-1028`). The review phase is entered after an implementation-phase `COMPLETE`, after `review_started` is set true and `.review-phase-started` is created (`hooks/loop-codex-stop-hook.sh:1395-1435`); during review phase it reruns Codex review on every exit attempt (`hooks/loop-codex-stop-hook.sh:1441-1469`).
- edge_cases_or_failure_modes: The test covers no marker at all (`tests/test-codex-review-merge.sh:109-131`), marker at byte/position 0 (`tests/test-codex-review-merge.sh:257-278`), common dash-prefixed marker (`tests/test-codex-review-merge.sh:280-303`), exactly 50-line logs (`tests/test-codex-review-merge.sh:332-354`), and logs longer than 50 lines to protect against previous tail-only behavior (`tests/test-codex-review-merge.sh:171-227`).
