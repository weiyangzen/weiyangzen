### FIX_HUMANIZE_ESCAPE-HZ-009 `directory` `prompt-template/block`
- cursor: `[_]`
- core_role: The `prompt-template/block` directory is the diagnostic/message payload layer for RLCR loop guard failures. It is not executable code by itself; each Markdown file is a rendered block reason used by hook scripts after those scripts detect an invalid operation. The directory coordinates with `hooks/lib/template-loader.sh`, whose `load_and_render_safe` function loads these files, substitutes `{{VAR}}` placeholders, and falls back to inline messages if a template is missing or empty (`hooks/lib/template-loader.sh:167-203`).
- algorithmic_behavior: The directory encodes guard outcomes as user-facing transition messages. Major template families are:
  - Round/path access routing: `wrong-file-location.md`, `wrong-round-file.md`, `wrong-round-number.md`, `wrong-directory-path.md`, `wrong-summary-location.md`. These explain transitions from an attempted stale/wrong path to the active loop path and current round.
  - Protected loop state: `state-file-modification.md`, `finalize-state-file-modification.md`, `plan-backup-protected.md`, `plan-file-modified.md`, `prompt-file-write.md`, `todos-file-access.md`. These preserve loop-owned files and redirect users to allowed tools or loop commands.
  - Goal tracker policy: `goal-tracker-bash-write.md`, `goal-tracker-modification.md`, `goal-tracker-not-initialized.md`. These distinguish Round 0 initialization from post-Round-0 Codex-only updates.
  - Git/session exit gates: `git-push.md`, `git-add-humanize.md`, `git-status-failed.md`, `git-not-clean.md`, `git-not-clean-humanize-local.md`, `git-not-clean-untracked.md`, `unpushed-commits.md`, `incomplete-todos.md`, `work-summary-missing.md`, `large-files.md`.
  - Review/schema failures: `codex-review-failed.md`, `schema-outdated.md`.
- inputs_outputs_state: Inputs are template file names plus variable assignments passed by hooks, for example `FILE_PATH`, `ACTIVE_LOOP_DIR`, `CURRENT_ROUND`, `CORRECT_PATH`, `GIT_ISSUES`, `SPECIAL_NOTES`, `AHEAD_COUNT`, `CURRENT_BRANCH`, `SUMMARY_FILE`, `MAX_LINES`, and `LARGE_FILES`. Outputs are rendered Markdown block reasons written to stderr by PreToolUse validators or embedded into JSON `decision: "block"` responses by stop hooks. State transitions happen in callers: validators exit with status `2` for blocked tool use, while stop hooks emit JSON block decisions and exit successfully from the hook process after denying stop.
- gates_or_invariants: The templates assume the loader preserves literal user/content values and does single-pass `{{VAR}}` substitution. The directory also depends on `validate_template_dir` requiring `prompt-template/block`, `prompt-template/codex`, and `prompt-template/claude` to exist (`hooks/lib/template-loader.sh:205-221`). Critical hook usage generally calls `load_and_render_safe` so missing templates do not fail open or produce empty block messages (`tests/test-template-references.sh:170-190`).
- dependencies_and_callers: Direct callers include:
  - `hooks/loop-read-validator.sh`, which uses `wrong-file-location.md`, `wrong-round-file.md`, and `wrong-directory-path.md` for invalid read paths (`hooks/loop-read-validator.sh:100-145`).
  - `hooks/loop-write-validator.sh`, which uses `plan-backup-protected.md`, `wrong-summary-location.md`, `wrong-round-number.md`, and `wrong-directory-path.md` (`hooks/loop-write-validator.sh:132-215`).
  - `hooks/loop-bash-validator.sh`, which blocks `git push` through `git-push.md`, `.humanize` staging through `git-add-humanize.md`, and plan backup writes through `plan-backup-protected.md` (`hooks/loop-bash-validator.sh:63-83`, `hooks/loop-bash-validator.sh:295-299`).
  - `hooks/lib/loop-common.sh`, which centralizes block-message helpers for todos, prompt writes, state-file writes, summary bash writes, goal-tracker bash writes, `.humanize` git-add protection, and goal-tracker modification policy (`hooks/lib/loop-common.sh:190-247`, `hooks/lib/loop-common.sh:568-612`).
  - `hooks/loop-codex-stop-hook.sh`, which uses stop-phase block templates for plan modification, incomplete todos, git status failures, large files, dirty git state, unpushed commits, and missing summaries (`hooks/loop-codex-stop-hook.sh:244-248`, `hooks/loop-codex-stop-hook.sh:289-305`, `hooks/loop-codex-stop-hook.sh:325-337`, `hooks/loop-codex-stop-hook.sh:399-419`, `hooks/loop-codex-stop-hook.sh:461-514`, `hooks/loop-codex-stop-hook.sh:529-537`).
- edge_cases_or_failure_modes: If a template is absent or renders empty, `load_and_render_safe` falls back to the caller-supplied inline message (`hooks/lib/template-loader.sh:176-203`). If a value contains shell metacharacters or nested placeholders, the intended behavior is literal output and no second-pass expansion; this is the branch’s core escape concern. Some note templates, such as `git-not-clean-humanize-local.md` and `git-not-clean-untracked.md`, are loaded with `load_template` and concatenated into `SPECIAL_NOTES` before `git-not-clean.md` is rendered, so their content can influence the final rendered block body (`hooks/loop-codex-stop-hook.sh:438-457`, `prompt-template/block/git-not-clean.md:1-15`).

### FIX_HUMANIZE_ESCAPE-HZ-039 `file` `tests/test-template-loader.sh`
- cursor: `[_]`
- core_role: This file is the executable specification for `hooks/lib/template-loader.sh`. It defines the expected contract for template directory discovery, template loading, rendering, safe fallback behavior, directory validation, escaping, and injection resistance. For the `fix-humanize-escape` branch, its most algorithmically relevant sections are the shell metacharacter and placeholder injection tests.
- algorithmic_behavior: The test script sources the loader (`tests/test-template-loader.sh:10-12`), maintains pass/fail counters (`tests/test-template-loader.sh:20-34`), then performs 41 named checks. The core flow is:
  - Resolve `PROJECT_ROOT`, source `hooks/lib/template-loader.sh`, and compute `TEMPLATE_DIR`.
  - Call loader functions directly and compare rendered output with exact expected strings.
  - Increment counters through local `pass`/`fail` helpers.
  - Exit `0` only when `TESTS_FAILED` is zero; otherwise exit `1` (`tests/test-template-loader.sh:641-659`).
- inputs_outputs_state: Inputs are the real repo template directory, selected real templates such as `block/git-push.md` and `block/wrong-round-number.md`, and inline synthetic templates containing placeholders. Outputs are colored PASS/FAIL lines plus a final pass/fail summary. Internal state is limited to `TESTS_PASSED` and `TESTS_FAILED`, updated by helper functions (`tests/test-template-loader.sh:20-34`).
- gates_or_invariants: The script establishes these invariants:
  - `get_template_dir "$PROJECT_ROOT/hooks/lib"` must resolve to `$PROJECT_ROOT/prompt-template` (`tests/test-template-loader.sh:41-52`).
  - Existing templates must load; missing templates must return empty for `load_template` (`tests/test-template-loader.sh:55-78`).
  - `render_template` must replace single and multiple placeholders, preserve multiline content, ignore unused variables, and leave unreplaced variables intact (`tests/test-template-loader.sh:81-194`).
  - `load_and_render_safe` must render fallback content for missing templates and prefer real template content when available (`tests/test-template-loader.sh:197-222`).
  - `validate_template_dir` must accept the real template tree and reject a missing path (`tests/test-template-loader.sh:225-244`).
  - Values containing shell metacharacters must be rendered literally: ampersand, backslash, dollar sign, backticks, pipe, semicolon, globs, parentheses, redirection characters, quotes, hash, tilde, combined shell fragments, Windows paths, regex patterns, JSON-like strings, and multiline metacharacter content (`tests/test-template-loader.sh:247-486`).
  - Values containing `{{VAR}}` patterns must not trigger placeholder injection or second-pass substitution (`tests/test-template-loader.sh:488-543`).
  - Empty substitutions, Unicode, underscore-prefixed variable names, and numeric variable names are valid (`tests/test-template-loader.sh:545-639`).
- dependencies_and_callers: It depends directly on `hooks/lib/template-loader.sh` and the repository’s `prompt-template` tree. It calls `get_template_dir`, `load_template`, `render_template`, `load_and_render`, `load_and_render_safe`, and `validate_template_dir`. It is included in `tests/run-all-tests.sh` as one of the aggregate test scripts (`tests/run-all-tests.sh:34` from reference scan).
- edge_cases_or_failure_modes: The test script is broad enough to catch the classic `awk gsub` replacement bug where `&` in a replacement value expands to matched text; Test 29 explicitly names this regression (`tests/test-template-loader.sh:425-435`). It also catches command-substitution hazards from backticks and `$`, shell pipeline/redirect characters, path backslashes, regex backslashes, JSON braces, multiline values, and nested placeholder strings. One implementation-level failure surfaced during execution in this read-only export: `render_template` uses an `awk ... <<< "$content"` here-string in the loader (`hooks/lib/template-loader.sh:74-129`), and Bash failed to create the here-document temp file under this sandbox. That made every render-dependent test fail despite load and directory tests passing.
- validation_or_tests: Direct execution result in this environment: `bash tests/test-template-loader.sh` exited `1`, with 5 passed and 38 failed. The passing tests were non-rendering or directory/file checks: template directory resolution, loading an existing file, missing-file empty output, valid directory, invalid directory. All rendering tests failed because the loader could not create a temp file for the here-string at `hooks/lib/template-loader.sh:129`; Test 12 additionally hit `env_vars[@]: unbound variable` when rendering a template with no variable assignments.

### FIX_HUMANIZE_ESCAPE-HZ-069 `file` `prompt-template/block/wrong-file-location.md`
- cursor: `[_]`
- core_role: This template is the specific block message used when a read request targets a round prompt or summary file outside the `.humanize/rlcr` loop area. It does not detect the violation; it renders the explanation and the correct active-loop alternatives once `hooks/loop-read-validator.sh` has classified the read as being in the wrong location.
- algorithmic_behavior: The template has three placeholders: `{{FILE_PATH}}`, `{{ACTIVE_LOOP_DIR}}`, and `{{CURRENT_ROUND}}` (`prompt-template/block/wrong-file-location.md:3-7`). It tells the user the attempted path, states that loop files are in the active loop directory, lists the current round prompt and summary file paths, and includes a final `cat {{FILE_PATH}}` line (`prompt-template/block/wrong-file-location.md:9`). Rendering is single-pass through `load_and_render_safe`.
- inputs_outputs_state: Inputs come from `hooks/loop-read-validator.sh`: the attempted `FILE_PATH`, the discovered `ACTIVE_LOOP_DIR`, and parsed `CURRENT_ROUND` (`hooks/loop-read-validator.sh:61-77`, `hooks/loop-read-validator.sh:100-108`). Output is Markdown written to stderr, after which the read validator exits with status `2` to block the tool call (`hooks/loop-read-validator.sh:105-109`). No persistent repository state is changed by this template.
- gates_or_invariants: The calling gate is: if the target is a prompt or summary round file, an active loop exists, a round number can be extracted, and `is_in_humanize_loop_dir "$FILE_PATH"` is false, block with this template (`hooks/loop-read-validator.sh:50-65`, `hooks/loop-read-validator.sh:83-100`). It assumes `ACTIVE_LOOP_DIR` is the canonical active loop returned by `find_active_loop`, and `CURRENT_ROUND` is parsed from `state.md` or `finalize-state.md` (`hooks/loop-read-validator.sh:69-77`).
- dependencies_and_callers: Its direct caller is `hooks/loop-read-validator.sh` through `load_and_render_safe "$TEMPLATE_DIR" "block/wrong-file-location.md"` (`hooks/loop-read-validator.sh:100-109`). It depends on shared functions and variables from `hooks/lib/loop-common.sh`, including active-loop discovery, state parsing, round-file type detection, and template loading via `TEMPLATE_DIR`.
- edge_cases_or_failure_modes: If the template is missing or empty, the read validator falls back to an inline message that still names the attempted path and active loop directory (`hooks/loop-read-validator.sh:102-105`). If placeholder rendering fails, the block output may be empty or fallback-dependent depending on `load_and_render_safe`; the read remains blocked because the caller exits `2`. One content-level edge case is the final line: “If you need this file, use: `cat {{FILE_PATH}}`” (`prompt-template/block/wrong-file-location.md:9`). That suggests reading the same wrong-location path via Bash, while the earlier lines direct users toward active-loop prompt/summary paths. It may be intentional for arbitrary file inspection, but it is slightly inconsistent with the guard’s purpose.
- validation_or_tests: `tests/test-template-loader.sh` does not directly render this exact file, but it verifies the loader mechanics used by this template, including real block template loading (`block/git-push.md`), integration rendering with another round/path template (`block/wrong-round-number.md`), safe fallback behavior, and literal escaping of shell-sensitive values (`tests/test-template-loader.sh:55-65`, `tests/test-template-loader.sh:147-164`, `tests/test-template-loader.sh:197-222`, `tests/test-template-loader.sh:247-543`). Runtime caller behavior is covered structurally by `hooks/loop-read-validator.sh`, which wires this template to the wrong-location read gate (`hooks/loop-read-validator.sh:97-110`).
