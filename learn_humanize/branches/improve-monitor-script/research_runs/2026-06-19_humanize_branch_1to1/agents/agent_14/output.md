### IMPROVE_MONITOR_SCRIPT-HZ-014 `file` `hooks/check-todos-from-transcript.py`
- cursor: `[_]`
- core_role: This is a stop-gate helper for the RLCR loop. It prevents the loop from proceeding to the expensive Codex review path while Claude still has native `TodoWrite` items that are not completed. It is called from [hooks/loop-codex-stop-hook.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/improve-monitor-script/hooks/loop-codex-stop-hook.sh:246), inside the “Quick Check: Are All Todos Completed?” phase before git cleanliness and Codex review checks.
- algorithmic_behavior: The script reads one JSON object from stdin, expects `transcript_path`, scans that JSONL transcript line by line, and keeps only the most recent non-empty `TodoWrite` payload. The scan is tolerant of malformed transcript lines: individual `json.JSONDecodeError` entries are ignored rather than failing the hook. It recognizes three transcript shapes: Claude Code assistant messages with `message.content[].type == tool_use`, an alternate top-level `type: message` format, and direct `type: tool_use` entries. For each recognized `TodoWrite`, it replaces `latest_todos` with the latest non-empty `todos` array, so later calls override earlier calls. After the scan, every todo whose `status` is not exactly `completed` is treated as incomplete.
- inputs_outputs_state: Input is hook JSON on stdin, normally including `transcript_path`; the referenced transcript is read as UTF-8 JSONL. Output is only process exit code unless incomplete todos exist. Exit `0` means no blocking evidence: all latest todos are complete, no todos were found, no transcript path was supplied, or the transcript path does not exist. Exit `1` prints `INCOMPLETE_TODOS` followed by one line per incomplete item in the form `  - [status] content`. Exit `2` prints `PARSE_ERROR: ...` to stderr when the hook input JSON itself cannot be parsed. The script has no persistent state; its derived state is the in-memory `latest_todos` list at [hooks/check-todos-from-transcript.py](/Users/wangweiyang/GitHub/humanize_branch_worktrees/improve-monitor-script/hooks/check-todos-from-transcript.py:28).
- gates_or_invariants: The main invariant is fail-closed for malformed hook input but fail-open for absent todo evidence. Invalid stdin JSON blocks via exit `2`, and the stop hook converts that into a block decision at [hooks/loop-codex-stop-hook.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/improve-monitor-script/hooks/loop-codex-stop-hook.sh:253). Missing `transcript_path`, missing transcript file, empty transcript, no `TodoWrite`, or only empty todo arrays all return `0`. Completion is exact-string based: only `status == "completed"` passes; `pending`, `in_progress`, missing status, empty status, or any other value blocks. This is a native Claude TodoWrite gate, not the execution-cron three-checkbox `[ ]` / `[_]` / `[x]` protocol; it enforces “finish your working todos before stop/review,” not master acceptance.
- dependencies_and_callers: The script depends only on Python standard library modules `json`, `sys`, and `pathlib.Path`. The caller is `hooks/loop-codex-stop-hook.sh`, which passes the full hook input JSON to Python and captures stdout/stderr. On exit `1`, the shell strips the first output line with `tail -n +2`, renders `prompt-template/block/incomplete-todos.md`, and returns a Claude hook block decision. Hook registration comes through the Stop hook in [hooks/hooks.json](/Users/wangweiyang/GitHub/humanize_branch_worktrees/improve-monitor-script/hooks/hooks.json:52), which invokes `loop-codex-stop-hook.sh`. The file is also listed as part of the hook directory structure in [README.md](/Users/wangweiyang/GitHub/humanize_branch_worktrees/improve-monitor-script/README.md:224).
- edge_cases_or_failure_modes: A transcript containing only malformed JSONL lines returns success because line-level JSON decode failures are ignored. If the latest `TodoWrite` has an empty `todos` array, it is ignored and an earlier non-empty todo list can remain authoritative; this matters because the update only happens under `if todos`. If todo content is empty, the item still blocks when status is incomplete. If status is missing, `todo.get("status", "")` makes it incomplete. The file-open path does not catch filesystem errors other than nonexistence, so permission or decode problems could surface as unhandled exceptions. Because completion requires exactly `"completed"`, any upstream status vocabulary change would block by default.
