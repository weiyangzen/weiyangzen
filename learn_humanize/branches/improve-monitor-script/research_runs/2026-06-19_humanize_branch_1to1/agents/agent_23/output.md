- core_role: Portable Bash timeout abstraction for RLCR setup and hook workflows. It gives macOS/Linux-compatible timeout behavior to git checks and Codex review execution without requiring GNU `timeout` to exist on every host. The file is intended to be sourced, not executed directly; usage is documented at `scripts/portable-timeout.sh:4`.
- algorithmic_behavior: On source, `detect_timeout_impl` selects one implementation in priority order: Homebrew `gtimeout`, GNU `timeout`, `python3`, `python`, or `none` (`scripts/portable-timeout.sh:10-27`). The selected value is stored once in global `TIMEOUT_IMPL` (`scripts/portable-timeout.sh:29`). `run_with_timeout` accepts `<seconds> <command> [args...]`, shifts off the seconds value, preserves the remaining command as a Bash array, and dispatches by `TIMEOUT_IMPL` (`scripts/portable-timeout.sh:33-38`). Native branches call `gtimeout "$timeout_secs" "${cmd[@]}"` or `timeout "$timeout_secs" "${cmd[@]}"` and return the child status (`scripts/portable-timeout.sh:39-45`). Python branches run `subprocess.run(sys.argv[1:], timeout=$timeout_secs)`, propagate the child return code, map `TimeoutExpired` to exit `124`, and map other exceptions to exit `1` after printing an error (`scripts/portable-timeout.sh:47-61`). If no timeout implementation is found, it emits a warning to stderr and runs the command without a timeout (`scripts/portable-timeout.sh:64-68`).
- inputs_outputs_state: Inputs are the timeout duration string and an argv-style command list passed to `run_with_timeout`. Outputs are the wrapped command stdout/stderr, plus the wrapper exit status: child status on normal completion, `124` on Python timeout, `1` on Python wrapper exception, or implementation-specific status for native timeout commands. Persistent state is limited to exported `TIMEOUT_IMPL` (`scripts/portable-timeout.sh:73-76`), computed at source time. The helper does not write files or mutate RLCR loop state directly.
- gates_or_invariants: The main invariant is that callers can treat `run_with_timeout` as a bounded execution gate for potentially blocking commands. GNU compatibility is explicitly checked with `timeout --version`; a non-GNU `timeout` is rejected (`scripts/portable-timeout.sh:13-19`). The Python fallback intentionally matches GNU timeout’s `124` timeout code (`scripts/portable-timeout.sh:56-57`). Command arguments are passed as an array, so spaces in command arguments are preserved after the initial timeout argument (`scripts/portable-timeout.sh:36`, `scripts/portable-timeout.sh:40`, `scripts/portable-timeout.sh:44`, `scripts/portable-timeout.sh:61`).
- dependencies_and_callers: Depends on Bash, `command -v`, optionally `gtimeout`, GNU `timeout`, `python3`, or `python`, and Python’s `subprocess` module. It is sourced by setup at `scripts/setup-rlcr-loop.sh:26-28`, where git operations use `GIT_TIMEOUT=30` (`scripts/setup-rlcr-loop.sh:23-24`) and later calls such as `git rev-parse`, `git status`, `git ls-files`, and branch capture are routed through `run_with_timeout` (`scripts/setup-rlcr-loop.sh:212`, `scripts/setup-rlcr-loop.sh:218`, `scripts/setup-rlcr-loop.sh:313`, `scripts/setup-rlcr-loop.sh:323`, `scripts/setup-rlcr-loop.sh:429`). The `none` branch weakens all timeout gates by running commands unbounded, which can hang setup, validation, or Codex review despite printing a warning (`scripts/portable-timeout.sh:64-68`). Python `subprocess.run(..., timeout=...)` times out the direct child process, but it is not a process-group watchdog.
