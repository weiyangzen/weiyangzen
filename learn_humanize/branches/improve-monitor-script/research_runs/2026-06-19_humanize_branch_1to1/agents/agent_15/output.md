- core_role: Declarative hook dispatch manifest for the RLCR loop. It wires Claude hook lifecycle events to the repository’s validator and stop-hook scripts: `UserPromptSubmit` to plan/state validation, `PreToolUse` to `Write`/`Edit`/`Read`/`Bash` validators, and `Stop` to the Codex review loop controller. See `hooks/hooks.json:4-12`, `hooks/hooks.json:14-51`, and `hooks/hooks.json:52-60`.
- algorithmic_behavior: The file itself does not implement validation logic; it is the routing table that determines which algorithm runs for each hook event. `UserPromptSubmit` invokes `hooks/loop-plan-file-validator.sh` to validate active loop schema, branch consistency, and plan tracking. `PreToolUse` dispatches tool-specific gates: `Write` protects todos, prompt files, summaries, state, plan backups, and post-round-zero goal tracker writes; `Edit` applies similar edit gates; `Read` prevents stale or wrong-location round-file reads; `Bash` blocks shell bypasses such as redirection, `tee`, in-place edits, destructive file ops, and disallowed `git push`. `Stop` invokes `hooks/loop-codex-stop-hook.sh`, which blocks exit until summary, todo, git, plan, goal tracker, and Codex review conditions pass.

Outputs are allow/block decisions: PreToolUse validators generally exit `0` to allow or `2` to block with a rendered message; UserPromptSubmit and Stop hooks emit JSON block decisions when blocking. State transitions are performed by called scripts, not the JSON: `loop-codex-stop-hook.sh` increments `current_round` when Codex finds follow-up work, or ends the loop by renaming `state.md` to an exit-reason state file via `end_loop` in `hooks/lib/loop-common.sh:301-334`.

Key invariants in referenced scripts include: newest active loop only, preventing older “zombie” loops from reviving (`hooks/lib/loop-common.sh:58-79`); required state fields and fixed branch during a loop (`hooks/loop-plan-file-validator.sh:48-108`); plan file tracking/integrity consistency (`hooks/loop-plan-file-validator.sh:110-224`, `hooks/loop-codex-stop-hook.sh:186-238`); no state or plan-backup modification through tools or Bash (`hooks/loop-write-validator.sh:91-110`, `hooks/loop-bash-validator.sh:71-92`); current-round summary location checks (`hooks/loop-write-validator.sh:123-188`, `hooks/loop-edit-validator.sh:108-139`); and stop-hook exit gates for incomplete todos, dirty git state, missing summary, uninitialized goal tracker, max iterations, and Codex completion markers (`hooks/loop-codex-stop-hook.sh:240-620`, `hooks/loop-codex-stop-hook.sh:916-969`). ### IMPROVE_MONITOR_SCRIPT-HZ-045 `file` `prompt-template/block/goal-tracker-bash-write.md`
- cursor: `[_]`
- core_role: Block-message template used by the Bash validator when a shell command attempts to modify `goal-tracker.md` during round zero. It is not an algorithm implementation by itself, but it is part of the enforcement contract because it tells the agent which tool path preserves validation. See `prompt-template/block/goal-tracker-bash-write.md:1-8`.
- algorithmic_behavior: The template renders a “Bash Write Blocked” response, rejects Bash modification of `goal-tracker.md`, substitutes `{{CORRECT_PATH}}`, and explains that `cat`, `echo`, `sed`, `awk`, and similar commands bypass validation hooks. It is loaded by `goal_tracker_bash_blocked_message` in `hooks/lib/loop-common.sh:228-237`, then emitted by `hooks/loop-bash-validator.sh:100-104` when `command_modifies_file` detects a goal-tracker write and `CURRENT_ROUND` is `0`. For later rounds, the Bash validator uses the separate goal-tracker modification message that instructs the agent to request updates through the round summary (`hooks/loop-bash-validator.sh:104-107`, `hooks/lib/loop-common.sh:287-299`).
- inputs_outputs_state: Inputs are the template text plus the `CORRECT_PATH` variable, normally set to `$ACTIVE_LOOP_DIR/goal-tracker.md` by `hooks/loop-bash-validator.sh:101-103`. Rendering is single-pass through `load_and_render_safe` and `render_template` in `hooks/lib/template-loader.sh:50-58` and `hooks/lib/template-loader.sh:170-203`. Output is a human-readable block reason written to stderr, after which the Bash validator exits with code `2`. The template performs no direct state transition and writes no files.
- gates_or_invariants: The invariant encoded by this template is that round-zero goal tracker initialization must use `Write` or `Edit`, not Bash, because Bash bypasses the path/round validators. This complements `loop-write-validator.sh`, which allows goal tracker writes during round zero but blocks them after round zero (`hooks/loop-write-validator.sh:113-121`). The block therefore channels initial goal tracker setup through tool hooks while preserving stronger restrictions after the initial round.
- dependencies_and_callers: Direct caller is `goal_tracker_bash_blocked_message` in `hooks/lib/loop-common.sh:230-237`. That helper is reached from `loop-bash-validator.sh`, which receives Bash tool hook input from the `PreToolUse` matcher in `hooks/hooks.json:42-49`. Template loading depends on `hooks/lib/template-loader.sh`; missing or empty template content falls back to an inline message with the same `CORRECT_PATH` placeholder flow.
- edge_cases_or_failure_modes: If `CORRECT_PATH` is not supplied, the loader intentionally leaves the placeholder unresolved rather than expanding it recursively or guessing a value (`hooks/lib/template-loader.sh:12-13`, `hooks/lib/template-loader.sh:115-122`). If the template is missing, `load_and_render_safe` falls back to the inline message in `hooks/lib/loop-common.sh:232-235`. The message is only as comprehensive as the Bash detector; regex limitations include variable destination paths and the documented multi-source `cp` edge case in `tests/test-bash-validator-patterns.sh:165-168`. The template is round-zero specific through its caller; after round zero, a different block message applies.
- validation_or_tests: `tests/test-template-references.sh:152-158` lists `block/goal-tracker-bash-write.md` as a required common template. `tests/test-bash-validator-patterns.sh:75-168` validates the command patterns that can trigger this message, including redirection, `tee`, in-place editors, `mv`, `cp`, `rm`, `dd`, `truncate`, and safe read-only false positives. Template-loader behavior is covered by `tests/test-template-loader.sh` and the comprehensive template reference tests. Tests were inspected only; no commands were executed beyond read-only file inspection.
