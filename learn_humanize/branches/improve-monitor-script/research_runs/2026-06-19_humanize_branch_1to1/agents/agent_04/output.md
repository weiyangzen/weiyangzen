- algorithmic_behavior: Templates are rendered through `hooks/lib/template-loader.sh`: `get_template_dir` resolves the root to `prompt-template` (`hooks/lib/template-loader.sh:24`), `load_template` returns file contents or an empty string on missing file (`hooks/lib/template-loader.sh:36`), `render_template` performs single-pass `{{VAR}}` replacement while preserving unknown placeholders (`hooks/lib/template-loader.sh:58`), and `load_and_render_safe` falls back when a template is missing or renders empty (`hooks/lib/template-loader.sh:170`). The `block/` templates are emitted when validators block unsafe tool use or stop-hook exits; `codex/` templates build review prompts and define `COMPLETE`/`STOP` criteria; `claude/` templates build the next round’s prompt after Codex finds more work.

### IMPROVE_MONITOR_SCRIPT-HZ-034 `file` `tests/test-template-references.sh`
- cursor: `[_]`
- core_role: Executable static specification for template-reference integrity. It prevents hook scripts from referencing missing templates and guards against empty validator messages caused by unsafe template loading.
- algorithmic_behavior: The script resolves `PROJECT_ROOT` and `TEMPLATE_DIR` from its own location (`tests/test-template-references.sh:14`). It scans a fixed script list for `load_template`, `load_and_render`, and `load_and_render_safe` calls using literal `"$TEMPLATE_DIR"` plus a quoted template path (`tests/test-template-references.sh:57`, `tests/test-template-references.sh:93`). Each extracted template path is checked for existence and recorded as pass or fail (`tests/test-template-references.sh:98`). It then enumerates every `*.md` template and warns if it is not directly referenced anywhere under `hooks/` (`tests/test-template-references.sh:121`). Finally, it checks required common templates and verifies critical validators do not use unsafe `load_and_render` calls (`tests/test-template-references.sh:152`, `tests/test-template-references.sh:176`).
- inputs_outputs_state: Inputs are the repository tree, the `prompt-template` directory, and selected hook scripts. Internal state is held in counters `TESTS_PASSED`, `TESTS_FAILED`, `WARNINGS`, plus arrays for `MISSING_TEMPLATES`, `TEMPLATE_FILES`, and `UNREFERENCED` (`tests/test-template-references.sh:25`, `tests/test-template-references.sh:71`, `tests/test-template-references.sh:122`). Outputs are colored PASS/FAIL/WARN lines, a final summary, a critical missing-template list if applicable, and exit code 0 or 1 (`tests/test-template-references.sh:206`).
- gates_or_invariants: Missing templates referenced by the fixed script list are hard failures (`tests/test-template-references.sh:102`). Missing common `loop-common.sh` templates are hard failures (`tests/test-template-references.sh:161`). Unsafe `load_and_render` usage in `loop-read-validator.sh`, `loop-write-validator.sh`, or `loop-edit-validator.sh` is a hard failure (`tests/test-template-references.sh:188`). Unreferenced templates are warnings only (`tests/test-template-references.sh:134`). Missing scripts in `SCRIPTS_TO_CHECK` warn and continue rather than failing (`tests/test-template-references.sh:74`). The script exits 0 only when `TESTS_FAILED` is zero (`tests/test-template-references.sh:221`).
- dependencies_and_callers: Depends on Bash, `find`, `grep`, `sed`, and the hook/template tree. The fixed scan list covers `loop-codex-stop-hook.sh`, read/write/edit/bash validators, and `hooks/lib/loop-common.sh` (`tests/test-template-references.sh:57`). It complements loader behavior in `hooks/lib/template-loader.sh`, especially safe fallback semantics.
- edge_cases_or_failure_modes: The hard missing-reference scan only covers the fixed `SCRIPTS_TO_CHECK` array; references in other hook files, such as `hooks/loop-plan-file-validator.sh`, are not hard-checked by Section 1. The extraction pattern is line-based and requires a literal `"$TEMPLATE_DIR"` followed by a quoted template path on the same line, so dynamic names, single quotes, unquoted paths, or multi-line call formatting can evade the check. Because unreferenced templates only warn, stale templates can remain. The script uses `set -uo pipefail` but not `set -e`, so failures are intentionally represented through counters rather than immediate shell aborts.

- core_role: Claude-facing contract for proposing goal-tracker mutations without directly editing `goal-tracker.md`. It is part of the round-transition prompt assembled after Codex rejects completion and the loop continues.
- algorithmic_behavior: The template tells Claude to include a `## Goal Tracker Update Request` section in its summary when goal-tracker changes are needed (`prompt-template/claude/goal-tracker-update-request.md:2`). It defines structured fields for requested changes and justification (`prompt-template/claude/goal-tracker-update-request.md:6`). Example request categories include marking tasks complete with evidence, adding open issues, recording plan evolution, and deferring tasks with acceptance-criteria impact (`prompt-template/claude/goal-tracker-update-request.md:7`). It ends by assigning approval/update authority to Codex (`prompt-template/claude/goal-tracker-update-request.md:16`).
- inputs_outputs_state: The file has no `{{VAR}}` placeholders, so it is appended literally. Its input is the loop condition where Codex found issues and a next prompt is being generated. Its output is Markdown appended to `round-N-prompt.md` by the stop hook (`hooks/loop-codex-stop-hook.sh:1013`). The state transition is indirect: Claude writes a summary request, then Codex evaluates and, if justified, updates `goal-tracker.md` through the Codex review/update contract rather than allowing Claude to mutate it directly.
- gates_or_invariants: Requires explicit requested changes plus justification. It reinforces the invariant that after round 0 Claude requests goal-tracker updates in the summary instead of editing the tracker. That invariant is also present in `claude/next-round-prompt.md` (`prompt-template/claude/next-round-prompt.md:30`), enforced by write/edit validators for direct edits after round 0 (`hooks/loop-write-validator.sh:117`, `hooks/loop-edit-validator.sh:102`), and enforced for Bash modifications through `hooks/loop-bash-validator.sh:100`.
- dependencies_and_callers: Loaded with `load_template` in `hooks/loop-codex-stop-hook.sh`; if missing, the hook falls back to “Include a Goal Tracker Update Request section in your summary if needed.” (`hooks/loop-codex-stop-hook.sh:1014`). The paired Codex-side evaluator is `prompt-template/codex/goal-tracker-update-section.md`, which instructs Codex to evaluate, approve, reject, or apply requested changes while never modifying the immutable section (`prompt-template/codex/goal-tracker-update-section.md:3`).
- edge_cases_or_failure_modes: Because the template is literal and optional in behavior, Claude can omit the section even when an update would help; then Codex may lack a structured request. The stop hook’s fallback keeps the prompt usable if the template is missing, but the fallback is much less specific than the full template. The template includes a deferral example; Codex templates still treat deferrals as incomplete unless strongly justified and accepted, so this block is a request protocol, not a permission to bypass completion.
