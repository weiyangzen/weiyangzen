### DO_NOT_WISH_CODING-HZ-007 `directory` `prompt-template`
- cursor: `[_]`
- core_role: Template source tree for RLCR, PR-loop, Codex review, Claude next-round, blocker, and plan-generation prompts. It is not executable by itself, but it defines the text contracts that production hooks render into state-transition messages, review prompts, retry prompts, and guard failures.
- algorithmic_behavior: The directory is organized into five coordination surfaces. `prompt-template/block/*` contains guard/block messages for invalid hook actions such as wrong round file access, protected state edits, missing trigger comments, unpushed commits, force-push retriggering, incomplete tasks, and goal-tracker modification. `prompt-template/claude/*` drives Claude-side continuation, review remediation, finalization, agent-teams coordination, push-every-round reminders, and goal-tracker update requests. `prompt-template/codex/*` drives Codex review obligations, goal alignment checks, severity-marker review phases, and goal-tracker update authority. `prompt-template/pr-loop/*` defines PR loop round-0 instructions, critical completion requirements, initial goal tracker shape, and Codex goal-tracker update rules. `prompt-template/plan/gen-plan-template.md` defines the generated plan schema: acceptance criteria, path boundaries, dependency sequence, task tags, deliberation, pending decisions, code-style requirements, and translated output convention.
- inputs_outputs_state: Inputs are template variables in `{{VAR}}` form, e.g. `{{PLAN_FILE}}`, `{{CURRENT_ROUND}}`, `{{GOAL_TRACKER_FILE}}`, `{{REVIEW_CONTENT}}`, `{{PR_NUMBER}}`, and `{{BOT_MENTION_STRING}}`. Rendering is handled by `hooks/lib/template-loader.sh`: `get_template_dir` resolves `prompt-template` from the plugin root, `load_template` reads files, and `render_template` performs single-pass replacement using `TMPL_VAR_` environment variables (`hooks/lib/template-loader.sh:24`, `hooks/lib/template-loader.sh:36`, `hooks/lib/template-loader.sh:50`). Output is Markdown text passed to hook stderr/stdout, round prompt files, review prompt files, and Codex/Claude task instructions.
- gates_or_invariants: The templates encode major invariants: state files are system-managed (`prompt-template/block/state-file-modification.md:1`), prompt files are read-only instructions (`prompt-template/block/prompt-file-write.md:1`), wrong round files are blocked (`prompt-template/block/wrong-round-number.md:1`), goal tracker becomes Codex-controlled after round 0 (`prompt-template/block/goal-tracker-modification.md:1`), incomplete tasks block review (`prompt-template/block/incomplete-todos.md:1`), PR-loop trigger requirements are explicit (`prompt-template/block/no-trigger-comment.md:1`), and force pushes require new bot triggers (`prompt-template/block/force-push-detected.md:1`). Template rendering itself preserves unprovided placeholders and avoids recursive placeholder injection (`hooks/lib/template-loader.sh:54`).
- dependencies_and_callers: Production callers include `hooks/loop-codex-stop-hook.sh` for RLCR review/finalize/next-round prompts and blocker messages, `hooks/pr-loop-stop-hook.sh` for PR-loop blocker and Codex goal-tracker update prompts, `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, and `hooks/loop-bash-validator.sh` for read/write/bash guard text. `tests/test-template-references.sh` scans shell references and checks template existence; `tests/test-templates-comprehensive.sh` validates loading/rendering behavior.
- edge_cases_or_failure_modes: Missing templates are not intended to hard-crash production hooks: `load_and_render_safe` returns a fallback when the template is missing or renders empty (`hooks/lib/template-loader.sh:167`). Unreferenced templates can be acceptable if dynamically used, but missing referenced templates are critical because they can remove actionable block messages. Variable values containing `{{OTHER_VAR}}` are intentionally not expanded in the same render pass, preventing injected review content from corrupting prompts (`hooks/lib/template-loader.sh:54`).
- validation_or_tests: Template coverage is validated by `tests/test-template-references.sh`, `tests/test-templates-comprehensive.sh`, `tests/test-error-scenarios.sh`, `tests/robustness/test-template-stress-robustness.sh`, and feature tests such as `tests/test-gen-plan.sh` and `tests/test-agent-teams.sh`. For new tasks, it reads `~/.claude/tasks/<session_id>/*.json` or a test override directory, treating statuses other than `completed` or `deleted` as incomplete (`hooks/check-todos-from-transcript.py:104`, `hooks/check-todos-from-transcript.py:117`, `hooks/check-todos-from-transcript.py:130`).
- inputs_outputs_state: Input is hook JSON on stdin with optional `session_id`, optional `tasks_base_dir`, and optional `transcript_path` (`hooks/check-todos-from-transcript.py:150`). Output is exit code `0` when no incomplete tasks exist, exit code `1` with `INCOMPLETE_TODOS` and bullet lines on stdout when incomplete items exist, and exit code `2` with `PARSE_ERROR` on stderr for invalid hook JSON (`hooks/check-todos-from-transcript.py:9`, `hooks/check-todos-from-transcript.py:158`, `hooks/check-todos-from-transcript.py:193`). It reads but does not mutate transcript/task state.
- gates_or_invariants: Empty stdin allows progression (`hooks/check-todos-from-transcript.py:153`). Missing transcript path or missing task directory is treated as no tasks, not a hard failure (`hooks/check-todos-from-transcript.py:64`, `hooks/check-todos-from-transcript.py:121`). Malformed transcript lines and malformed task JSON files are skipped, so only top-level hook JSON parse failure blocks with code `2` (`hooks/check-todos-from-transcript.py:76`, `hooks/check-todos-from-transcript.py:143`).
- dependencies_and_callers: `hooks/loop-codex-stop-hook.sh` invokes it during incomplete-task checks before Codex review; `prompt-template/block/incomplete-todos.md` renders its `INCOMPLETE_TODOS` output into the user-facing blocker. `tests/test-todo-checker.sh` is the direct executable spec and covers legacy TodoWrite, Task files, mixed sources, parse errors, missing dirs, latest TodoWrite precedence, and multiple transcript entry shapes.
- edge_cases_or_failure_modes: Because malformed task files are skipped, a corrupt task JSON could fail open. Because the legacy transcript logic only keeps the latest non-empty `TodoWrite`, older incomplete tasks are ignored if a later TodoWrite omits them or marks all completed. Status comparison is exact: anything not equal to `completed` is incomplete for legacy todos, while Task files additionally allow `deleted`.
- validation_or_tests: Direct test coverage appears in `tests/test-todo-checker.sh`; finalize integration coverage appears in `tests/test-finalize-phase.sh` where incomplete todos block Codex invocation. I did not execute tests in this read-only research pass.
