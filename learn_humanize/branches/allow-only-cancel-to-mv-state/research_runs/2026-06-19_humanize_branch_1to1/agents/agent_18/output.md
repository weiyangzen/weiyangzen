### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-018 `file` `hooks/loop-edit-validator.sh`
- cursor: `[_]`
- core_role: Claude `Edit` PreToolUse validator for RLCR loop-managed files. It is registered for the `Edit` matcher in `hooks/hooks.json:24-30`, reads hook JSON from stdin, and decides whether an edit is allowed or blocked. For the branch’s state-machine invariant, its relevant role is to block direct `Edit` modifications to `state.md`; cancellation must instead flow through the separate Bash-only `mv state.md -> cancel-state.md` path guarded by `.cancel-requested`.
- algorithmic_behavior: The script starts with strict shell settings and sources shared RLCR helpers from `hooks/lib/loop-common.sh` (`hooks/loop-edit-validator.sh:12-16`). It parses `.tool_name` and exits successfully for any tool other than `Edit` (`hooks/loop-edit-validator.sh:22-27`). For `Edit`, it lowercases `.tool_input.file_path` (`hooks/loop-edit-validator.sh:29-30`), then applies ordered gates. First, round todos files are blocked unless they are in the active-loop allowlist (`hooks/loop-edit-validator.sh:36-44`); prompt files are always blocked (`hooks/loop-edit-validator.sh:46-49`). If the path is not inside `.humanize/rlcr/`, the validator permits it (`hooks/loop-edit-validator.sh:55-57`). For loop paths, it resolves the active loop directory from `CLAUDE_PROJECT_DIR`/`PROJECT_ROOT`, parses `state.md`, and loads `current_round` (`hooks/loop-edit-validator.sh:63-73`). It then blocks `state.md`, `plan.md` backup edits, post-Round-0 `goal-tracker.md` edits, and wrong-round summary edits (`hooks/loop-edit-validator.sh:79-138`).
- inputs_outputs_state: Input is Claude hook JSON on stdin with `tool_name` and `tool_input.file_path` (`hooks/loop-edit-validator.sh:22-30`). Environment inputs include `CLAUDE_PROJECT_DIR`, optionally `PROJECT_ROOT`, `LOOP_BASE_DIR`, and `LOOP_DIR` (`hooks/loop-edit-validator.sh:37-39`, `hooks/loop-edit-validator.sh:63-65`). Filesystem inputs are the newest active `.humanize/rlcr/<timestamp>/state.md` located by `find_active_loop`, whose current round is parsed by `parse_state_file` (`hooks/lib/loop-common.sh:62-79`, `hooks/lib/loop-common.sh:110-136`). Output is exit `0` to allow or exit `2` with a rendered block message on stderr to deny. The validator does not mutate loop state; it enforces that state mutation is outside the `Edit` pathway.
- gates_or_invariants: The key invariant is “Claude cannot directly edit `state.md`” (`hooks/loop-edit-validator.sh:79-82`), backed by the shared state block template/message in `state_file_blocked_message` (`hooks/lib/loop-common.sh:208-215`). This complements the branch-specific cancel invariant implemented in Bash validation: only an authorized command exactly moving active-loop `state.md` to active-loop `cancel-state.md` with `.cancel-requested` may pass (`hooks/lib/loop-common.sh:251-389`, especially source/destination checks at `hooks/lib/loop-common.sh:373-385`). Other gates: todos/prompt files are protected (`hooks/loop-edit-validator.sh:36-49`), `plan.md` backups are read-only (`hooks/loop-edit-validator.sh:88-96`), the goal tracker is editable by Claude only in Round 0 (`hooks/loop-edit-validator.sh:102-106`), and summary file round numbers must match `current_round` unless allowlisted (`hooks/loop-edit-validator.sh:112-136`).
- dependencies_and_callers: Called by Claude hook configuration via `${CLAUDE_PLUGIN_ROOT}/hooks/loop-edit-validator.sh` (`hooks/hooks.json:24-30`). Depends on `jq`, `sed`, `basename`, shell builtins, and shared helpers from `hooks/lib/loop-common.sh`: `to_lower`, `is_round_file_type`, `find_active_loop`, `is_allowlisted_file`, `todos_blocked_message`, `prompt_write_blocked_message`, `is_in_humanize_loop_dir`, `parse_state_file`, `is_state_file_path`, `load_and_render_safe`, `is_goal_tracker_path`, `goal_tracker_blocked_message`, and `extract_round_number`. Its sibling validators enforce the same policy for other tool classes: `loop-write-validator.sh` blocks `state.md` writes (`hooks/loop-write-validator.sh:91-98`) and `loop-bash-validator.sh` blocks shell state modifications except authorized cancel moves (`hooks/loop-bash-validator.sh:71-89`, `hooks/loop-bash-validator.sh:91-225`). The user-facing cancel command documents the required sequence: touch `.cancel-requested`, then `mv "${LOOP_DIR}state.md" "${LOOP_DIR}cancel-state.md"` (`commands/cancel-rlcr-loop.md:28-32`).
- edge_cases_or_failure_modes: If the active loop cannot be found, loop-specific checks are skipped and the edit is allowed (`hooks/loop-edit-validator.sh:67-69`); this is intentional because active loop detection is based only on newest loop directory containing `state.md` (`hooks/lib/loop-common.sh:62-79`). The todos gate runs before the general `.humanize/rlcr` test, so even round todos paths outside an active loop can be blocked unless allowlisted (`hooks/loop-edit-validator.sh:36-44`). The state parser defaults missing `current_round` to `0` (`hooks/lib/loop-common.sh:131-133`), which means goal-tracker edits would be allowed if state parsing silently falls back to Round 0. Summary filename extraction handles both timestamped and direct `.humanize/rlcr/` forms (`hooks/loop-edit-validator.sh:114-117`). Allowlisted historical files bypass some round mismatch/todos restrictions via `is_allowlisted_file`, whose static list includes `round-1-todos.md`, `round-2-todos.md`, `round-0-summary.md`, and `round-1-summary.md` (`hooks/lib/loop-common.sh:165-186`).

### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-048 `file` `prompt-template/block/goal-tracker-not-initialized.md`
- cursor: `[_]`
- core_role: Rendered block template for the RLCR Stop hook when Round 0 cannot exit because `goal-tracker.md` still contains placeholder content. It defines the user-facing recovery contract for initializing the goal tracker’s immutable goal/acceptance-criteria section before the loop advances beyond Round 0.
- algorithmic_behavior: The template itself is declarative Markdown with placeholders `{{GOAL_TRACKER_FILE}}` and `{{MISSING_ITEMS}}` (`prompt-template/block/goal-tracker-not-initialized.md:5-6`). It instructs the agent to read the goal tracker, replace placeholders with actual Ultimate Goal, 3-7 testable Acceptance Criteria, and Active Tasks mapped to ACs, then write the updated `goal-tracker.md` (`prompt-template/block/goal-tracker-not-initialized.md:8-15`). The behavioral transition is enforced by `hooks/loop-codex-stop-hook.sh`, which in Round 0 extracts the Ultimate Goal, Acceptance Criteria, and Active Tasks sections from `goal-tracker.md`, checks each section for a generic placeholder pattern `\[To be [a-z]`, builds `MISSING_ITEMS`, renders this template, and returns a JSON block decision (`hooks/loop-codex-stop-hook.sh:537-605`).
- inputs_outputs_state: Template inputs are `GOAL_TRACKER_FILE`, the active loop’s `goal-tracker.md`, and `MISSING_ITEMS`, a newline list of missing sections produced by the Stop hook (`hooks/loop-codex-stop-hook.sh:572-594`). The template output is the block reason string embedded in JSON fields `decision: "block"`, `reason`, and `systemMessage: "Loop: Goal Tracker not initialized in Round 0"` (`hooks/loop-codex-stop-hook.sh:596-604`). It does not mutate state directly. Its intended state effect is to prevent the loop from advancing out of Round 0 until the goal tracker’s initial immutable content is populated.
- gates_or_invariants: Main invariant: the Goal Tracker immutable section can only be set in Round 0 and becomes read-only afterward (`prompt-template/block/goal-tracker-not-initialized.md:16`). The Stop hook enforces this as a Round-0-only exit gate: the check is guarded by `CURRENT_ROUND == 0` and file existence (`hooks/loop-codex-stop-hook.sh:539`). It detects placeholders section-specifically to avoid false positives from placeholder-like text elsewhere (`hooks/loop-codex-stop-hook.sh:540-543`). This pairs with edit/write/bash validators that block goal-tracker modification after Round 0 and redirect later changes into a summary-file “Goal Tracker Update Request” flow (`hooks/loop-edit-validator.sh:99-106`, `hooks/loop-write-validator.sh:113-121`, `hooks/loop-bash-validator.sh:257-270`).
- dependencies_and_callers: Called via `load_and_render_safe "$TEMPLATE_DIR" "block/goal-tracker-not-initialized.md"` in `hooks/loop-codex-stop-hook.sh:592-594`. Rendering is performed by `hooks/lib/template-loader.sh`, which uses single-pass `{{VAR}}` substitution and preserves missing placeholders rather than recursively expanding values (`hooks/lib/template-loader.sh:50-131`). The Stop hook is registered in `hooks/hooks.json:52-59`. The initial goal tracker file is created by setup logic in `scripts/setup-rlcr-loop.sh` around the “Create Goal Tracker File” section referenced by search results, and startup instructions require Round 0 initialization before implementation.
- edge_cases_or_failure_modes: If `goal-tracker.md` does not exist, this specific template is not used because the Stop-hook check requires `-f "$GOAL_TRACKER_FILE"` (`hooks/loop-codex-stop-hook.sh:539`); missing-file behavior must be handled elsewhere or may fall through depending on setup integrity. The placeholder detector only matches lowercase text after `[To be ` (`hooks/loop-codex-stop-hook.sh:553`, `hooks/loop-codex-stop-hook.sh:560`, `hooks/loop-codex-stop-hook.sh:568`), so uppercase variants or different placeholder wording may evade this specific gate. Section extraction depends on exact headings: `### Ultimate Goal`, `### Acceptance Criteria`, and `#### Active Tasks` (`hooks/loop-codex-stop-hook.sh:549-566`); heading drift could create false negatives. Missing template files fall back to an inline shorter message via `load_and_render_safe` (`hooks/loop-codex-stop-hook.sh:588-594`, `hooks/lib/template-loader.sh:167-180`).
- validation_or_tests: `tests/test-plan-file-hooks.sh` contains section-specific Stop-hook tests for this behavior. It verifies only Ultimate Goal is reported when that section still has a placeholder (`tests/test-plan-file-hooks.sh:737-807`), only Acceptance Criteria is reported when that section is missing (`tests/test-plan-file-hooks.sh:809-876`), only Active Tasks is reported when that section is missing (`tests/test-plan-file-hooks.sh:878-943`), and all three are reported when all placeholders remain (`tests/test-plan-file-hooks.sh:945-1010`). These tests assert the exact missing-item lines produced for the rendered template.
