- algorithmic_behavior: `start-rlcr-loop.md` delegates initialization to `"${CLAUDE_PLUGIN_ROOT}/scripts/setup-rlcr-loop.sh" $ARGUMENTS` and restricts allowed tools to that setup script only (`commands/start-rlcr-loop.md:3-14`). It also establishes the goal-tracker protocol: immutable goal/acceptance criteria, mutable tasks/evolution/deferred sections, and full alignment checks every fifth review cycle starting after round 4 (`commands/start-rlcr-loop.md:24-42`).
- inputs_outputs_state: Start input is `<path/to/plan.md>` plus options such as `--max`, `--codex-model`, `--codex-timeout`, and `--push-every-round` (`commands/start-rlcr-loop.md:3`). Its delegated setup script creates `.humanize/rlcr/<timestamp>/state.md`, a backup `plan.md`, `goal-tracker.md`, and `round-0-prompt.md`/`round-0-summary.md` paths as seen in `scripts/setup-rlcr-loop.sh:467-691`. Cancel input is the active loop discovered by newest timestamp directory under `.humanize/rlcr` (`commands/cancel-rlcr-loop.md:11-16`). Cancel output is a `.cancel-requested` signal followed by renaming `state.md` to `cancel-state.md` (`commands/cancel-rlcr-loop.md:28-32`).
- gates_or_invariants: The branch-relevant invariant is that cancellation is a signal-plus-rename transition, not arbitrary state mutation. `cancel-rlcr-loop.md` allows only `ls`, `touch .cancel-requested`, `mv state.md cancel-state.md`, `cat state.md`, and `Read` in its front matter (`commands/cancel-rlcr-loop.md:1-4`). It explicitly says active means the newest loop directory has `state.md`; if `state.md` is absent, no active loop exists (`commands/cancel-rlcr-loop.md:20-34`). Start also warns against exiting by editing state files or running cancel commands during normal work (`commands/start-rlcr-loop.md:43-49`).
- dependencies_and_callers: Start coordinates with `scripts/setup-rlcr-loop.sh`, which performs Git/plan validation, state creation, goal-tracker generation, prompt creation, and user-facing activation output (`scripts/setup-rlcr-loop.sh:186-423`, `scripts/setup-rlcr-loop.sh:467-720`). Cancel coordinates with `hooks/loop-bash-validator.sh` and `hooks/lib/loop-common.sh`: the Bash hook blocks `state.md` modification unless `is_cancel_authorized` approves the exact cancel rename (`hooks/loop-bash-validator.sh:81-89`, `hooks/loop-bash-validator.sh:91-241`), and the helper requires a `.cancel-requested` file plus exact two-argument `mv` from active `state.md` to active `cancel-state.md` (`hooks/lib/loop-common.sh:261-389`).
- edge_cases_or_failure_modes: No active loop if no loop directory exists, or if the newest loop directory lacks `state.md` (`commands/cancel-rlcr-loop.md:18-26`). Older loop directories are intentionally ignored by `find_active_loop`, preventing stale loops from being revived (`hooks/lib/loop-common.sh:58-79`). The cancel path must create the signal before the rename; a signal in a parent or wrong directory does not authorize cancellation (`tests/test-cancel-signal-file.sh:196-215`). Command injection around the allowed rename is rejected by helper checks for command substitution, newlines, chaining operators, extra args, remaining `$`, and wrong source/destination paths (`hooks/lib/loop-common.sh:272-388`).
- validation_or_tests: `tests/test-cancel-signal-file.sh` is the direct executable spec for the cancel transition. It covers allowed rename with signal (`tests/test-cancel-signal-file.sh:78-92`), allowed `./` path normalization (`tests/test-cancel-signal-file.sh:98-113`), block without signal (`tests/test-cancel-signal-file.sh:119-133`), block edits/removes/wrong destination even with signal (`tests/test-cancel-signal-file.sh:139-235`), and many injection/argument/path bypass cases. `tests/test-state-exit-naming.sh` validates that `cancel-state.md` is not treated as an active loop state (`tests/test-state-exit-naming.sh:93-107`).

### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-062 `file` `prompt-template/block/wrong-round-file.md`
- cursor: `[_]`
- core_role: This block template is the Read-hook denial message for attempts to read a round prompt or summary whose round number does not match the current active loop round. It is a small but core part of RLCR state isolation: it prevents stale prompt/summary context from being read as if it were current.
- algorithmic_behavior: The template renders `CLAUDE_ROUND`, `FILE_TYPE`, `CURRENT_ROUND`, `ACTIVE_LOOP_DIR`, and `FILE_PATH`. It tells the agent which file it tried to read and names the current prompt/summary files (`prompt-template/block/wrong-round-file.md:1-9`). The actual gate lives in `hooks/loop-read-validator.sh`: after identifying round prompt/summary paths, it parses active `state.md`, extracts the requested round, and blocks when requested round differs from `current_round` unless the file is allowlisted (`hooks/loop-read-validator.sh:69-82`, `hooks/loop-read-validator.sh:110-122`).
- inputs_outputs_state: Inputs are hook JSON for a `Read` tool call, active loop directory from `.humanize/rlcr`, `state.md` current round, and the target file path. Output is exit code `2` plus this rendered block message on denial; exit `0` allows the read. No state mutation occurs. The template’s final line includes `cat {{FILE_PATH}}` (`prompt-template/block/wrong-round-file.md:9`), which is advisory text for deliberate access, but the hook itself is the enforcing component.
- gates_or_invariants: Read access to `round-N-summary.md` and `round-N-prompt.md` must use the active loop directory and the current round. Wrong location is blocked earlier with `wrong-file-location.md` (`hooks/loop-read-validator.sh:90-104`), wrong round uses this template (`hooks/loop-read-validator.sh:106-123`), and wrong active directory path is blocked after round validation (`hooks/loop-read-validator.sh:126-140`). `round-*-todos.md` reads are also blocked unless allowlisted because native TodoWrite is preferred (`hooks/loop-read-validator.sh:32-44`).
- dependencies_and_callers: The caller is `hooks/loop-read-validator.sh`, which sources `hooks/lib/loop-common.sh` for `find_active_loop`, `parse_state_file`, `extract_round_number`, `is_round_file_type`, `is_allowlisted_file`, and `load_and_render_safe` (`hooks/loop-read-validator.sh:14-17`, `hooks/lib/loop-common.sh:58-79`, `hooks/lib/loop-common.sh:96-137`, `hooks/lib/loop-common.sh:153-186`). The template loader provides variable substitution and safe fallback behavior.
- edge_cases_or_failure_modes: If there is no active loop, read validation exits successfully (`hooks/loop-read-validator.sh:65-67`). If the filename does not parse as a round file, it exits successfully (`hooks/loop-read-validator.sh:77-80`). Historical allowlisted files bypass wrong-round blocking: `round-0-summary.md`, `round-1-summary.md`, `round-1-todos.md`, and `round-2-todos.md` (`hooks/lib/loop-common.sh:172-186`). Write/Edit paths do not use this template; they use `wrong-round-number.md` for wrong summary rounds (`hooks/loop-write-validator.sh:153-169`, `hooks/loop-edit-validator.sh:112-136`).
- validation_or_tests: Template rendering infrastructure is covered by `tests/test-template-loader.sh`, though its integration example uses the sibling `wrong-round-number.md` (`tests/test-template-loader.sh:147-164`). Comprehensive template tests also render `wrong-round-number.md` with all variables (`tests/test-templates-comprehensive.sh:493-514`). Round-read behavior is indirectly covered by allowlist validator tests that block non-allowlisted future summaries on `Read` (`tests/test-allowlist-validators.sh:245-281`).
