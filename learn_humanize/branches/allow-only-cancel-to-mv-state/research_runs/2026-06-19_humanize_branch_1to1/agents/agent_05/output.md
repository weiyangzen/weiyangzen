- core_role:
  The `scripts` directory is the command/setup and operator-monitoring layer for the RLCR loop. It contains three shell scripts:

- algorithmic_behavior:
  `setup-rlcr-loop.sh` is the algorithmic entry point. It parses a single plan path plus options such as `--max`, `--codex-model`, `--codex-timeout`, `--track-plan-file`, and `--push-every-round` (`scripts/setup-rlcr-loop.sh:104-180`). It writes `state.md` with YAML frontmatter containing round, limits, Codex config, plan path, tracking mode, start branch, and timestamp (`scripts/setup-rlcr-loop.sh:485-498`). It creates `goal-tracker.md` with immutable goal/acceptance sections plus mutable task/evolution/issues sections (`scripts/setup-rlcr-loop.sh:504-594`). Finally it creates `round-0-prompt.md`, embedding goal tracker initialization rules, the plan content, loop-state anti-bypass instructions, and summary/commit requirements (`scripts/setup-rlcr-loop.sh:600-653`).
  `portable-timeout.sh` detects timeout support in priority order `gtimeout`, GNU `timeout`, `python3`, `python`, then `none` (`scripts/portable-timeout.sh:10-27`). `run_with_timeout` dispatches to the chosen implementation and makes Python timeouts return `124`, matching GNU timeout semantics (`scripts/portable-timeout.sh:33-71`).

It discovers the latest timestamped session under `.humanize/rlcr` (`scripts/humanize.sh:23-41`), locates matching cached Codex logs under `$HOME/.cache/humanize/<sanitized-project-path>/<timestamp>/` (`scripts/humanize.sh:43-96`), resolves active versus terminal state files (`scripts/humanize.sh:98-136`), parses state frontmatter (`scripts/humanize.sh:138-154`), parses goal tracker sections into progress counts (`scripts/humanize.sh:156-238`), parses git status/diffstat (`scripts/humanize.sh:240-286`), and draws a fixed terminal status bar while tailing log updates (`scripts/humanize.sh:302-440`, `scripts/humanize.sh:519-675`).

- cursor: `[_]`
- core_role:
  This test is an executable specification for prompt-template integrity. It prevents validators and stop hooks from referencing missing templates and, for critical validator paths, enforces `load_and_render_safe` so a missing template does not collapse a blocking message into empty or unusable output (`tests/test-template-references.sh:3-9`, `tests/test-template-references.sh:172-200`).
- algorithmic_behavior:
  The script defines pass/fail/warn counters and colored reporting helpers (`tests/test-template-references.sh:18-49`). Section 1 scans a fixed list of hook scripts and shared library files for template loader calls using `$TEMPLATE_DIR` (`tests/test-template-references.sh:54-64`). For each non-comment line matching `load_template`, `load_and_render`, or `load_and_render_safe` with the first argument exactly `"$TEMPLATE_DIR"`, it extracts the second quoted argument using `sed`, increments `FOUND_REFERENCES`, and checks that `$PROJECT_ROOT/prompt-template/<template_path>` exists (`tests/test-template-references.sh:83-110`).
  Section 2 recursively discovers every `*.md` template under `prompt-template`, then warns if a template path is not directly quoted in any hook script (`tests/test-template-references.sh:119-142`). This is informational; unreferenced templates increment `WARNINGS` but do not fail the script.
  Section 3 hard-codes common block templates expected by `hooks/lib/loop-common.sh` message helpers and fails if any are absent (`tests/test-template-references.sh:145-167`). Section 4 checks critical read/write/edit validators for unsafe `load_and_render "$TEMPLATE_DIR"` calls that are not `load_and_render_safe` (`tests/test-template-references.sh:172-200`). The script exits `0` only when `TESTS_FAILED` is zero and exits `1` with a missing-template summary otherwise (`tests/test-template-references.sh:206-230`).
- inputs_outputs_state:
  Inputs are the repository-local filesystem, especially `hooks/loop-codex-stop-hook.sh`, `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, `hooks/loop-bash-validator.sh`, `hooks/lib/loop-common.sh`, and the full `prompt-template` tree (`tests/test-template-references.sh:14-17`, `tests/test-template-references.sh:56-64`, `tests/test-template-references.sh:121-126`). Outputs are human-readable PASS/FAIL/WARN lines, counters, and process exit status. The script mutates only local shell arrays/counters: `MISSING_TEMPLATES`, `FOUND_REFERENCES`, `TEMPLATE_FILES`, `UNREFERENCED`, `TESTS_PASSED`, `TESTS_FAILED`, and `WARNINGS` (`tests/test-template-references.sh:25-28`, `tests/test-template-references.sh:71-72`, `tests/test-template-references.sh:121-132`).
- gates_or_invariants:

  - The six common block templates used by shared loop messages must exist: todos access, prompt write, state modification, summary bash write, goal-tracker bash write, and goal-tracker modification (`tests/test-template-references.sh:152-167`).
  - Critical read/write/edit validators must not use plain `load_and_render` for `$TEMPLATE_DIR` calls; all such calls must be safe variants (`tests/test-template-references.sh:176-200`).
  Non-fatal invariant:
  - Templates not directly referenced are warnings because some may be loaded dynamically or appended conditionally (`tests/test-template-references.sh:130-140`).
- dependencies_and_callers:
  The test depends on the template-loader call style used by the hooks. It assumes calls look like `load_template "$TEMPLATE_DIR" "path/to/template.md"` or the corresponding render functions with a quoted literal template path (`tests/test-template-references.sh:66-69`, `tests/test-template-references.sh:83-96`). It is included in the aggregate test runner as one of the standard suites (`tests/run-all-tests.sh:32-46`). Its protected behavior matches `hooks/lib/template-loader.sh`, where `load_template` returns empty plus a warning on missing files, while `load_and_render_safe` uses a fallback message when loading or rendering produces empty content (`hooks/lib/template-loader.sh:33-48`, `hooks/lib/template-loader.sh:167-203`).
- edge_cases_or_failure_modes:
  The scanner is intentionally simple and misses non-literal or differently formatted template references. It only matches calls where `"$TEMPLATE_DIR"` appears quoted on the same line and extracts the next quoted string (`tests/test-template-references.sh:83-96`). Multi-line calls where the template path is on a later line, single-quoted paths, variable-computed names, or different first-argument expressions may not be counted. Conversely, Section 2 searches only `"$template"` under `hooks/`, so templates referenced from non-hook files or through shell variables can produce warnings even if they are real runtime dependencies (`tests/test-template-references.sh:134-140`).
  The safety check in Section 4 counts direct `load_and_render "$TEMPLATE_DIR"` occurrences but only for three critical validator files, not the stop hook or bash validator (`tests/test-template-references.sh:176-200`). That scope is deliberate for critical validator block messaging, but it is not a complete global policy check.
