### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-039 `file` `hooks/lib/template-loader.sh`
- cursor: `[_]`
- core_role: This shell library is the template resolution and rendering engine shared by RLCR hook scripts. It is core support for validator behavior because every block/codex/claude prompt fragment depends on its path discovery, loading, single-pass variable substitution, and fallback semantics.
- algorithmic_behavior: `get_template_dir` resolves plugin root from the caller’s `hooks/lib` directory and returns `$plugin_root/prompt-template` (`hooks/lib/template-loader.sh:26-31`). `load_template` concatenates a relative template path to the template dir, emits file contents if present, or returns empty content plus a warning when absent (`:36-48`). `render_template` accepts content plus `VAR=value` assignments, prefixes them as `TMPL_VAR_` environment entries, and runs an awk scanner that replaces `{{VAR}}` placeholders exactly once (`:58-132`). `load_and_render` composes load plus render (`:136-147`), `append_template` prints base plus another template (`:151-161`), `load_and_render_safe` falls back if loading or rendering is empty (`:170-203`), and `validate_template_dir` requires `block`, `codex`, and `claude` subdirectories (`:208-222`).
- inputs_outputs_state: Inputs are a caller-provided template directory, a repository-relative template name such as `block/git-not-clean.md`, and zero or more variable assignments. Output is rendered Markdown to stdout, or an empty/fallback string depending on function. It has no persistent state, but `loop-common.sh` initializes global `TEMPLATE_DIR` from it at `hooks/lib/loop-common.sh:46-55`.
- gates_or_invariants: Rendering is intentionally single-pass: if an injected value contains `{{OTHER_VAR}}`, that nested placeholder is not expanded (`hooks/lib/template-loader.sh:54-57`, `:115-125`). Missing variables are preserved literally (`:119-122`). Missing template directories are warned but not fatal in `loop-common.sh:53-56`, so hooks degrade to fallback messages rather than losing validation logic.
- dependencies_and_callers: Sourced by `hooks/lib/loop-common.sh:46-48`, then used across `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, `hooks/loop-bash-validator.sh`, `hooks/loop-plan-file-validator.sh`, and `hooks/loop-codex-stop-hook.sh`. It depends on standard shell utilities `cd`, `pwd`, `cat`, `env`, and `awk`.
- edge_cases_or_failure_modes: The comment documents uppercase placeholder names (`hooks/lib/template-loader.sh:9-13`), but the awk scanner actually accepts any text between `{{` and `}}`; caller discipline provides the real constraint. An unmatched `{{` is treated as literal text by appending `{{` and continuing (`:102-108`). `load_template` warns and returns empty for missing files; `load_and_render_safe` suppresses that warning and applies fallback (`:176-186`). Command substitution in `render_template` can normalize final trailing newlines, so this is suitable for prompt blocks but not byte-exact file templating.
