### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-016 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: Bash `PreToolUse` validator for the RLCR loop. It is registered for Bash tool calls in `hooks/hooks.json:42-48` and protects loop control files from shell-based bypasses of Write/Edit/Read validators. Its most branch-relevant role is enforcing that `state.md` is immutable except for one authorized cancellation transition: `state.md` may be moved to `cancel-state.md` only when the active loop directory contains `.cancel-requested`.
- algorithmic_behavior: The script reads hook JSON from stdin, extracts `.tool_name` and `.tool_input.command` with `jq`, and exits immediately unless the tool is Bash (`hooks/loop-bash-validator.sh:22-30`). It locates the active loop from `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize/rlcr` via `find_active_loop`, exits open when no loop exists, then parses `state.md` frontmatter to determine `current_round` and `push_every_round` (`hooks/loop-bash-validator.sh:36-49`). It then runs ordered gates: block `git push` unless `push_every_round` is true (`hooks/loop-bash-validator.sh:59-68`), block `state.md` modifications except authorized cancel (`hooks/loop-bash-validator.sh:81-89`, `91-226`, `228-241`), block loop plan backup writes (`hooks/loop-bash-validator.sh:249-253`), block goal tracker writes with different Round 0 vs later-round messages (`hooks/loop-bash-validator.sh:262-270`), block prompt writes (`hooks/loop-bash-validator.sh:278-280`), block summary writes through Bash (`hooks/loop-bash-validator.sh:288-291`), and restrict todos writes to only active-loop round 1 or 2 todos paths (`hooks/loop-bash-validator.sh:298-305`).
- inputs_outputs_state: Input is Claude hook JSON on stdin with `tool_name` and Bash command. The validator depends on active loop state from `.humanize/rlcr/<latest>/state.md`, especially `current_round` and `push_every_round`, parsed by `parse_state_file` in `hooks/lib/loop-common.sh:110-137`. Output is either silent success with exit `0`, or a blocking message to stderr with exit `2`. It does not itself mutate state; it authorizes or denies the pending Bash command. The only permitted state transition it recognizes is external execution of `mv <active-loop>/state.md <active-loop>/cancel-state.md`, and only when `<active-loop>/.cancel-requested` exists and `is_cancel_authorized` accepts the exact command shape (`hooks/lib/loop-common.sh:251-389`).
- gates_or_invariants: The first invariant is fail-open outside an active loop: if `find_active_loop` returns empty, all Bash commands pass (`hooks/loop-bash-validator.sh:40-43`). Inside a loop, `state.md` is system-owned and blocked for redirection, in-place editing, removal, destination writes, source moves/copies, and shell-wrapper bypasses (`hooks/loop-bash-validator.sh:71-241`). The cancel exception requires a signal file, rejects `$()` and backticks, rejects newlines and shell chaining, normalizes literal `$loop_dir` / `${loop_dir}` only, rejects any remaining `$`, requires command start `mv`, requires exactly two args, and requires source/destination equal to active loop `state.md` and `cancel-state.md` after path normalization (`hooks/lib/loop-common.sh:265-388`). The source-side `mv`/`cp` detector splits command segments on shell operators while preserving redirection operators through placeholders, strips leading redirections, and checks each segment for prefixed `mv`/`cp` variants (`hooks/loop-bash-validator.sh:102-226`). Goal tracker writes are always blocked through Bash, but Round 0 tells the actor to use Write/Edit while later rounds require a summary-file update request (`hooks/loop-bash-validator.sh:257-270`). Todos writes are basename-detected first, then allowed only when the command includes the full active loop path for `round-1-todos.md` or `round-2-todos.md`; same basename in another root is blocked (`hooks/loop-bash-validator.sh:298-305`).
- dependencies_and_callers: Called by the Claude hook dispatcher as a Bash `PreToolUse` command (`hooks/hooks.json:42-48`). It sources `hooks/lib/loop-common.sh` (`hooks/loop-bash-validator.sh:14-16`), which supplies active-loop discovery, frontmatter parsing, lowercasing, template-backed block messages, `command_modifies_file`, and `is_cancel_authorized`. The generic modifier detector recognizes redirection, `tee`, `sed -i`, `awk -i inplace`, `perl -i`, `mv`/`cp` destination writes, `rm`, `dd of=`, `truncate`, `printf >`, and `exec fd>` patterns (`hooks/lib/loop-common.sh:397-425`). Message rendering flows through `hooks/lib/template-loader.sh`, whose safe loader uses template files when present and fallback strings otherwise (`hooks/lib/template-loader.sh:167-203`). Related templates include `prompt-template/block/state-file-modification.md`, `prompt-template/block/goal-tracker-bash-write.md`, `prompt-template/block/goal-tracker-modification.md`, `prompt-template/block/summary-bash-write.md`, `prompt-template/block/prompt-file-write.md`, `prompt-template/block/todos-file-access.md`, and `prompt-template/block/git-push.md`.
- edge_cases_or_failure_modes: Because command analysis is regex-based rather than AST-based, the generic `command_modifies_file` helper documents a known limitation for multi-source `cp` forms like `cp file1.md file2.md goal-tracker.md` (`tests/test-bash-validator-patterns.sh:165-168`).
