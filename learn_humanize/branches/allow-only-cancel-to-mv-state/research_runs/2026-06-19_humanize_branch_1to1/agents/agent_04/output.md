### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-004 `directory` `prompt-template`
- cursor: `[_]`
- core_role: `prompt-template/` is the static prompt and block-message contract layer for the RLCR loop. It does not execute logic directly, but it defines the rendered Markdown that hook validators and stop-loop orchestration use when allowing, blocking, reviewing, or continuing a round. The directory is resolved from `hooks/lib/template-loader.sh` via `get_template_dir`, which maps `hooks/lib` to the repository-level `prompt-template` path (`hooks/lib/template-loader.sh:24-31`).
- algorithmic_behavior: The directory is consumed by `load_template`, `render_template`, `load_and_render`, and `load_and_render_safe` (`hooks/lib/template-loader.sh:33-203`). Templates use `{{VARIABLE_NAME}}` placeholders; the loader documents uppercase/numeric/underscore variable syntax, single-pass substitution, and preservation of missing placeholders (`hooks/lib/template-loader.sh:7-13`). `render_template` builds `TMPL_VAR_` environment variables, then uses an `awk` scanner to replace `{{...}}` once without recursively expanding injected placeholders (`hooks/lib/template-loader.sh:62-129`). `load_and_render_safe` falls back to inline text if a template is missing or renders empty (`hooks/lib/template-loader.sh:170-203`).
- inputs_outputs_state: Inputs are template names, template variables, hook-derived state, file paths, review content, Git status, and loop metadata. Outputs are rendered Markdown block reasons, Codex review prompts, and Claude next-round prompts. `block/` supplies guard messages such as Git cleanliness, blocked push, state-file modification, todos access, wrong round, wrong directory, wrong summary location, plan backup protection, schema outdated, incomplete todos, and missing work summary. `codex/` supplies review contracts: `regular-review.md`, `full-alignment-review.md`, and `goal-tracker-update-section.md`. `claude/` supplies continuation contracts: `next-round-prompt.md`, `next-round-footer.md`, post-alignment action items, push note, and goal-tracker update request instructions. The templates themselves hold no mutable state; state transitions are performed by callers such as `loop-codex-stop-hook.sh`, which writes review prompts and next-round prompts (`hooks/loop-codex-stop-hook.sh:681-709`, `hooks/loop-codex-stop-hook.sh:985-1018`).
- gates_or_invariants: `validate_template_dir` requires the three subdirectories `block`, `codex`, and `claude` (`hooks/lib/template-loader.sh:205-222`), and `loop-common.sh` warns but continues with fallbacks if validation fails (`hooks/lib/loop-common.sh:46-56`). The directory encodes key invariants in user-facing form: prompt files are read-only instructions (`prompt-template/block/prompt-file-write.md:1-10`), `state.md` is managed by the loop system (`prompt-template/block/state-file-modification.md:1-10`), summaries must be in the active loop directory (`prompt-template/block/wrong-summary-location.md:1-5`), wrong-round files must redirect to the current round (`prompt-template/block/wrong-round-number.md:1-7`), and goal tracker writes after round 0 must become summary-file update requests (`prompt-template/block/goal-tracker-modification.md:1-25`). For the branch theme, the templates communicate that normal state modification is blocked, while the actual “only cancel may move state” exception is implemented outside this directory in `is_cancel_authorized` and `loop-bash-validator.sh`.
- dependencies_and_callers: `hooks/lib/loop-common.sh` sources the template loader and exposes common message helpers (`hooks/lib/loop-common.sh:46-56`, `hooks/lib/loop-common.sh:188-237`). `hooks/loop-codex-stop-hook.sh` loads block notes for dirty Git state (`hooks/loop-codex-stop-hook.sh:431-445`), builds Codex review prompts from `codex/full-alignment-review.md` or `codex/regular-review.md` (`hooks/loop-codex-stop-hook.sh:681-709`), and builds Claude next-round prompts from `claude/next-round-prompt.md` plus footer/push/update request fragments (`hooks/loop-codex-stop-hook.sh:985-1018`). `hooks/loop-read-validator.sh`, `hooks/loop-write-validator.sh`, `hooks/loop-edit-validator.sh`, and `hooks/loop-bash-validator.sh` use block templates through common helpers or direct `load_and_render_safe` calls. `tests/test-template-references.sh` scans these shell call sites and fails missing template references (`tests/test-template-references.sh:51-111`), while `tests/test-templates-comprehensive.sh` validates directory structure, all-template loading, syntax, fallback behavior, and rendering (`tests/test-templates-comprehensive.sh:61-194`, `tests/test-templates-comprehensive.sh:441-606`).
- edge_cases_or_failure_modes: Missing templates produce stderr warnings from `load_template` and empty content (`hooks/lib/template-loader.sh:41-47`), then `load_and_render_safe` uses fallback text (`hooks/lib/template-loader.sh:179-186`). Missing variables remain as placeholders by design (`hooks/lib/template-loader.sh:119-122`). Values containing placeholder-like text are intentionally not recursively expanded (`hooks/lib/template-loader.sh:54-57`). A template directory missing `block/`, `codex/`, or `claude/` fails validation (`hooks/lib/template-loader.sh:216-219`). In this read-only sandbox, any render path using the here-string at `hooks/lib/template-loader.sh:129` failed because Bash could not create its temp file, so runtime validation could not get a clean signal here.
- validation_or_tests: `tests/test-template-loader.sh` directly specifies loader behavior. `tests/test-template-references.sh` verifies all referenced templates exist and critical validators use safe rendering (`tests/test-template-references.sh:172-200`). `tests/test-templates-comprehensive.sh` loads every template, checks placeholder syntax, exercises fallback paths, renders all templates with dummy values, and warns on unreplaced placeholders (`tests/test-templates-comprehensive.sh:90-107`, `tests/test-templates-comprehensive.sh:118-194`, `tests/test-templates-comprehensive.sh:563-606`). `tests/run-all-tests.sh` includes `test-template-loader.sh`, `test-template-references.sh`, and `test-templates-comprehensive.sh` in the full suite (`tests/run-all-tests.sh:32-46`).

### ALLOW_ONLY_CANCEL_TO_MV_STATE-HZ-064 `file` `prompt-template/block/wrong-summary-location.md`
- cursor: `[_]`
- core_role: `prompt-template/block/wrong-summary-location.md` is the specific block message for attempts to write a round summary file outside the active `.humanize/rlcr/<timestamp>/` loop directory. It is a guard-contract template: it tells the agent why the write was rejected and where the current summary must be written.
- algorithmic_behavior: The template contains a title, a location invariant, and one placeholder: `{{CORRECT_PATH}}` (`prompt-template/block/wrong-summary-location.md:1-5`). It is rendered by `hooks/loop-write-validator.sh` when the hook input is a `Write` tool call, the target path looks like a `round-N-summary.md` file, and the path is not inside the humanize loop directory (`hooks/loop-write-validator.sh:23-31`, `hooks/loop-write-validator.sh:56-61`, `hooks/loop-write-validator.sh:127-134`). The validator computes `CORRECT_PATH="$ACTIVE_LOOP_DIR/round-${CURRENT_ROUND}-summary.md"` and passes it into `load_and_render_safe` (`hooks/loop-write-validator.sh:127-133`), then exits 2 to block the write (`hooks/loop-write-validator.sh:134`).
- inputs_outputs_state: Input is only `CORRECT_PATH`, derived from the active loop directory and parsed current round. Output is Markdown sent to stderr by the write validator. The file itself mutates no state. The intended state effect is preventative: the wrong-location write does not occur, preserving the invariant that the active round summary is stored under the current loop directory.
- gates_or_invariants: The gate depends on several prior classifications. `is_round_file_type "$FILE_PATH_LOWER" "summary"` identifies `round-[0-9]+-summary.md` (`hooks/lib/loop-common.sh:144-151`). `is_in_humanize_loop_dir` is used by the write validator to determine whether the path is outside `.humanize/rlcr` (`hooks/loop-write-validator.sh:56-61`, `hooks/loop-write-validator.sh:127-135`). If there is no active loop, the write validator exits early (`hooks/loop-write-validator.sh:78-85`). The invariant encoded by this template is explicit: “Summary files MUST be in the loop directory” (`prompt-template/block/wrong-summary-location.md:3`).
- dependencies_and_callers: Direct caller is `hooks/loop-write-validator.sh` (`hooks/loop-write-validator.sh:127-134`). Rendering depends on `TEMPLATE_DIR` initialized by `hooks/lib/loop-common.sh` (`hooks/lib/loop-common.sh:46-56`) and `load_and_render_safe` from `hooks/lib/template-loader.sh` (`hooks/lib/template-loader.sh:170-203`). `tests/test-template-references.sh` scans shell scripts for `load_template`, `load_and_render`, and `load_and_render_safe` references and verifies referenced template files exist (`tests/test-template-references.sh:83-111`). `tests/test-templates-comprehensive.sh` loads every template and validates placeholder syntax across the directory (`tests/test-templates-comprehensive.sh:90-194`).
- edge_cases_or_failure_modes: If the template is missing or empty, the write validator supplies an inline fallback with the same `{{CORRECT_PATH}}` variable (`hooks/loop-write-validator.sh:129-133`). If `CORRECT_PATH` rendering fails, the fallback path still depends on `load_and_render_safe`; in the current read-only sandbox, render execution is blocked by the shell temp-file issue at `hooks/lib/template-loader.sh:129`. If the filename does not match the round-summary pattern, this template is not used. If the summary file is inside `.humanize/rlcr` but has the wrong round number or wrong active loop path, later gates use `wrong-round-number.md` or `wrong-directory-path.md` instead (`hooks/loop-write-validator.sh:153-188`).
- validation_or_tests: Static validation is covered by template reference and comprehensive template tests. The assigned `tests/test-template-loader.sh` does not directly render `wrong-summary-location.md`, but it verifies the same loader and safe-render mechanisms used to render this block. The broader `tests/test-template-references.sh` would flag this file if missing from the direct `loop-write-validator.sh` reference, and `tests/test-templates-comprehensive.sh` would validate its loadability and placeholder syntax.
