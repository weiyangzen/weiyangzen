### ADD_CAREFUL_MODE_V1_6_5-HZ-035 `file` `scripts/check-pr-reviewer-status.sh`
- cursor: `[_]`
- core_role: PR-loop startup classifier. It queries GitHub PR commits/comments/reviews and emits JSON describing which monitored review bots have responded, whether any responses are stale relative to the latest commit, and which startup case applies.
- algorithmic_behavior: Parses `<pr_number> --bots <bot1,bot2>` (`scripts/check-pr-reviewer-status.sh:49-85`), maps logical bot names to GitHub authors with special handling for Codex (`:91-100`), resolves the PR base repo for fork support (`:109-140`), fetches latest commit SHA/date (`:142-151`), fetches issue comments, review comments, and PR review submissions (`:153-168`), then classifies each configured bot as commented, missing, or stale (`:174-205`).
- inputs_outputs_state: Inputs are CLI args, `gh` auth/context, PR number, bot list, and `jq`; it sources `portable-timeout.sh` and uses `GH_TIMEOUT=60` (`scripts/check-pr-reviewer-status.sh:35-40`). Output is a JSON object with `case`, `reviewers_commented`, `reviewers_missing`, `latest_commit_sha`, `latest_commit_at`, `newest_review_at`, and `has_commits_after_reviews` (`:259-275`). It does not write persistent state.
- gates_or_invariants: Case logic is the primary gate: no reviewers commented => case 1; some missing and no stale commented bot => case 2; all commented and fresh => case 3; all commented and at least one stale => case 4; partial comments plus at least one stale commented bot => case 5 (`scripts/check-pr-reviewer-status.sh:220-245`). Staleness is per bot, based on comparing latest commit time against that bot’s newest review/comment time (`:190-196`), which avoids a later review from one bot masking stale feedback from another.
- dependencies_and_callers: Called by `scripts/setup-pr-loop.sh` to determine startup behavior and trigger requirements (`scripts/setup-pr-loop.sh:437-447`). Setup then handles cases 4/5 by checking for or posting trigger comments after the latest commit (`scripts/setup-pr-loop.sh:457-490`). `hooks/pr-loop-stop-hook.sh` also rechecks reviewer status during stop-hook flow (`hooks/pr-loop-stop-hook.sh:1547`).
- edge_cases_or_failure_modes: Commit info fetch failure is fatal (`scripts/check-pr-reviewer-status.sh:142-148`), while comment/review fetch failures degrade to empty arrays (`:156-165`). If repo resolution fails, it warns and uses `CURRENT_REPO`, which may be empty (`:137-140`). Timestamp comparison uses lexicographic string comparison (`:194`), correct for normalized ISO UTC but fragile for mixed formats/timezones. Bot list entries are not trimmed or validated after comma splitting (`:106-107`), so malformed bot lists can produce bad author names. Header comments describe case 5 as all-commented/future distinction (`:22-27`), but implementation emits case 5 for partial-comment stale-review state (`:223-229`).
- validation_or_tests: `tests/test-pr-loop-system.sh` covers no-comments case 1 (`:152-170`), partial case 2 (`:173-192`), PR review submission detection (`:236-249`), all-commented case 3 (`:493-506`), case 4 all-commented plus new commits (`:1315-1341`), and case 5 partial-comment plus new commits (`:1347-1378`). Tests were inspected but not run in this read-only export.
