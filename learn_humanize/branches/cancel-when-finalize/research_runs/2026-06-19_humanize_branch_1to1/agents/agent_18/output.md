### CANCEL_WHEN_FINALIZE-HZ-018 `file` `hooks/loop-edit-validator.sh`
- cursor: `[_]`
- core_role: Claude `PreToolUse` validator for the RLCR `Edit` tool. It prevents direct edits to loop-controlled artifacts, enforces current-round summary targeting, and protects the finalize/cancel state machine from agent-side mutation. It is registered in `hooks/hooks.json:24-30` for matcher `Edit`.
- algorithmic_behavior: The script reads hook JSON from stdin, extracts `.tool_name`, and exits successfully unless the tool is exactly `Edit` (`hooks/loop-edit-validator.sh:22-27`). For edit requests, it lowercases `tool_input.file_path` and applies layered guards. It first blocks `round-*-todos.md` edits unless the target is one of the explicit allowlisted active-loop files (`round-1-todos.md`, `round-2-todos.md`, `round-0-summary.md`, `round-1-summary.md`) via `find_active_loop` and `is_allowlisted_file` (`hooks/loop-edit-validator.sh:29-43`, shared allowlist in `hooks/lib/loop-common.sh:167-188`). It then blocks `round-*-prompt.md` files as generated Codex-to-Claude prompt artifacts (`hooks/loop-edit-validator.sh:46-49`). Non-loop paths are allowed early (`hooks/loop-edit-validator.sh:55-57`). For loop paths, it finds the newest active loop and chooses `finalize-state.md` over `state.md` when present, then parses `current_round` through `parse_state_file` (`hooks/loop-edit-validator.sh:63-79`; parser defaults missing `current_round` to `0` in `hooks/lib/loop-common.sh:112-138`). It blocks `finalize-state.md` before generic `state.md` because the generic `state.md` regex also matches `finalize-state.md` (`hooks/loop-edit-validator.sh:84-94`). It blocks `plan.md` backups under `.humanize/rlcr/` (`hooks/loop-edit-validator.sh:100-108`). It blocks direct `goal-tracker.md` edits after Round 0, redirecting work into the current round summary (`hooks/loop-edit-validator.sh:114-118`). Finally, for summary files, it extracts the round filename and blocks edits to a summary whose round number differs from parsed state unless it is allowlisted (`hooks/loop-edit-validator.sh:124-149`).
- inputs_outputs_state: Input is Claude hook JSON on stdin with at least `tool_name` and `tool_input.file_path`. Runtime context comes from `CLAUDE_PROJECT_DIR`, optional pre-set `PROJECT_ROOT`, `LOOP_BASE_DIR`, or `LOOP_DIR`, plus `.humanize/rlcr/<loop>/state.md` or `finalize-state.md`. Output is only stderr guidance on rejection; success is silent. Exit `0` means allow the edit, exit `2` means block the tool call. The state transition protected here is ownership of RLCR state files and round artifacts: the hook itself does not mutate state, but prevents Claude from editing files that drive state transitions, especially `state.md`/`finalize-state.md` and post-Round-0 `goal-tracker.md`.
- gates_or_invariants: Only `Edit` is validated; other tools pass through (`hooks/loop-edit-validator.sh:25-27`). Round todo files are generally blocked, with a narrow active-loop allowlist (`hooks/loop-edit-validator.sh:36-43`). Prompt files are immutable (`hooks/loop-edit-validator.sh:46-49`). If the path is outside `.humanize/rlcr/`, the validator does not inspect loop state and allows it (`hooks/loop-edit-validator.sh:55-57`). The newest active loop is authoritative; if none exists, loop-path edits are allowed because there is no active state to enforce (`hooks/loop-edit-validator.sh:65-69`; active-loop selection in `hooks/lib/loop-common.sh:58-81`). Finalize state protection is more specific than state protection and runs first (`hooks/loop-edit-validator.sh:86-94`). Goal tracker is editable through `Edit` only in Round 0; after that, requested changes must be placed in `round-${CURRENT_ROUND}-summary.md` (`hooks/loop-edit-validator.sh:114-117`). Summary filenames must match the active round unless allowlisted (`hooks/loop-edit-validator.sh:131-148`).
- dependencies_and_callers: Depends on `jq`, `sed`, `basename`, shell builtins, and shared `hooks/lib/loop-common.sh` loaded at `hooks/loop-edit-validator.sh:15-16`. Shared functions used include `to_lower`, `is_round_file_type`, `find_active_loop`, `is_allowlisted_file`, `is_in_humanize_loop_dir`, `parse_state_file`, `is_finalize_state_file_path`, `is_state_file_path`, `is_goal_tracker_path`, `extract_round_number`, and message renderers. Message rendering flows through `hooks/lib/template-loader.sh`; `load_and_render_safe` supplies a rendered template or fallback (`hooks/lib/template-loader.sh:167-180`). It coordinates with sibling validators: `loop-write-validator.sh` implements analogous Write gates, including finalize-summary allowance and directory validation (`hooks/loop-write-validator.sh:37-50`, `hooks/loop-write-validator.sh:79-149`, `hooks/loop-write-validator.sh:181-216`), while `loop-bash-validator.sh` blocks shell bypasses for the same protected files (`hooks/loop-bash-validator.sh:75-105`, `hooks/loop-bash-validator.sh:291-343`).
- edge_cases_or_failure_modes: The script uses `set -euo pipefail`; malformed JSON or missing `jq` can fail the hook before custom messaging. If no active loop directory is found, edits inside `.humanize/rlcr/` are allowed after the early active-loop check (`hooks/loop-edit-validator.sh:67-69`), which is intentional for inactive historical artifacts but means enforcement depends on `find_active_loop` seeing state files. Active-loop detection only considers the newest timestamp-like directory and ignores older directories even if they contain state (`hooks/lib/loop-common.sh:58-81`), avoiding zombie loops but making edits to older loop paths less constrained. The `is_state_file_path` pattern matches any path ending in `state.md`, including `finalize-state.md`; the script mitigates by checking finalize first (`hooks/loop-edit-validator.sh:84-94`). Path containment uses a substring regex for `.humanize/rlcr/` (`hooks/lib/loop-common.sh:415-419`), not canonical path resolution, so symlink or unusual path normalization behavior is not addressed here. Summary extraction supports both timestamped and flat RLCR layouts via two sed patterns (`hooks/loop-edit-validator.sh:126-129`). The allowlist is hard-coded and round-specific, which is narrow but can surprise future round files.
