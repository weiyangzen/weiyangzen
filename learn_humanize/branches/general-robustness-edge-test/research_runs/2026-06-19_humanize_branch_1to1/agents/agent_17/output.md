- algorithmic_behavior: The prompt prescribes a three-step classifier: quickly inspect repo documentation/structure/technologies, compare the draft against repo concepts/files/features, then emit exactly one of two verdict prefixes: `RELEVANT:` or `NOT_RELEVANT:` with a short reason (`agents/draft-relevance-checker.md:12-29`). - gates_or_invariants: The hard output contract is the verdict prefix format (`RELEVANT:` / `NOT_RELEVANT:`). The decision invariant is lenient semantic relevance rather than strict path/function matching. The agent is restricted to read/search tools, preventing it from writing while classifying.
- dependencies_and_callers: `commands/gen-plan.md:50-72` invokes `humanize:draft-relevance-checker` after IO validation and stops plan generation if the verdict is `NOT_RELEVANT`. `tests/test-gen-plan.sh` validates this agent file exists, has the expected name/model/tools, naming convention, required frontmatter, valid model, and English-only content.
- edge_cases_or_failure_modes: Possible false positives are expected by design because uncertainty is resolved as relevant. False negatives should be limited to drafts clearly unrelated to the repository. A malformed verdict prefix would break the caller’s phase gate because `gen-plan` expects the exact prefixes.
