- algorithmic_behavior: It requires `<pr_number> --bots <bot1,bot2>` and rejects missing/unknown/multiple PR args (`scripts/check-pr-reviewer-status.sh:46-85`). Bot names are mapped to GitHub author logins, with special handling for `codex -> chatgpt-codex-connector[bot]` and default `<bot>[bot]` for others (`scripts/check-pr-reviewer-status.sh:91-100`). It resolves the base repository for fork PRs by first reading the current repo, then trying `gh pr view` against the current repo, then the parent repo, finally warning and falling back to current repo (`scripts/check-pr-reviewer-status.sh:109-137`).
- algorithmic_behavior: It fetches latest PR commit SHA/date through `gh pr view --json headRefOid,commits` and fails hard if commit info cannot be fetched (`scripts/check-pr-reviewer-status.sh:139-145`). It then fetches three review surfaces: issue comments, inline review comments, and PR review submissions, using `--paginate`; each comments/reviews fetch falls back to `[]` on failure (`scripts/check-pr-reviewer-status.sh:150-163`). Those arrays are merged through `jq -s 'add // []'` (`scripts/check-pr-reviewer-status.sh:164-165`).
- algorithmic_behavior: For each configured bot, it filters combined comments by mapped author, records the bot as commented or missing, computes that bot’s newest review timestamp, and marks that bot stale when `LATEST_COMMIT_AT > BOT_NEWEST` (`scripts/check-pr-reviewer-status.sh:176-201`). Staleness is per-bot, not global; one fresh bot does not rescue another stale bot (`scripts/check-pr-reviewer-status.sh:189-197`).
- algorithmic_behavior: Case assignment is deterministic: case 1 means no configured bot commented; case 2 means some but not all commented and no commented bot is stale; case 5 means partial comments with at least one stale commented bot; case 3 means all commented and no stale bot; case 4 means all commented with at least one stale bot (`scripts/check-pr-reviewer-status.sh:208-242`). Output is JSON with `case`, `reviewers_commented`, `reviewers_missing`, `latest_commit_sha`, `latest_commit_at`, `newest_review_at`, and `has_commits_after_reviews` (`scripts/check-pr-reviewer-status.sh:248-272`).
- inputs_outputs_state: Inputs are CLI args, GitHub CLI auth/context, current repo or parent repo metadata, PR commit metadata, issue comments, review comments, PR review submissions, `jq`, and `scripts/portable-timeout.sh`. Output is stdout JSON; warnings/errors go to stderr. The script does not persist local state.
- gates_or_invariants: `set -euo pipefail` is active (`scripts/check-pr-reviewer-status.sh:29`). Missing PR number or bot list exits 1. Commit metadata fetch failure exits 1. Comment/review API failures are intentionally nonfatal and treated as empty arrays. Timestamp comparison assumes sortable ISO-8601 UTC strings. `run_with_timeout` gives all `gh` operations a default 60-second cap (`scripts/check-pr-reviewer-status.sh:35-40`), using GNU `timeout`, Homebrew `gtimeout`, Python subprocess timeout, or no timeout fallback (`scripts/portable-timeout.sh:9-71`).
- dependencies_and_callers: It depends on `gh`, `jq`, the portable timeout wrapper, GitHub API endpoint compatibility, and fixture-driven mocks in tests. It is used by PR-loop setup to decide startup case and trigger requirements. The assigned test suite exercises cases 1/2/3/4/5 and PR-review-submission inclusion (`tests/test-pr-loop-system.sh:152-193`, `tests/test-pr-loop-system.sh:236-249`, `tests/test-pr-loop-system.sh:493-507`, `tests/test-pr-loop-system.sh:661-675`, `tests/test-pr-loop-system.sh:1315-1378`).

If comment/review API calls fail, the script may classify missing comments instead of surfacing a degraded-data state. If PR base repo resolution falls back to an empty or wrong repo, commit fetch will fail. PR review submissions with null `submitted_at` become `created_at: null`, which can affect sorting/staleness. Timestamp comparison is string-based and assumes normalized comparable formats.
