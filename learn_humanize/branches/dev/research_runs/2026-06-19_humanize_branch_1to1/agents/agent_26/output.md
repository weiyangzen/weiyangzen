### DEV-HZ-056 `file` `scripts/cancel-rlcr-loop.sh`
- cursor: `[_]`
- core_role: User-facing RLCR cancellation transition script. It converts the newest active loop in the current project from an active state file into `cancel-state.md` and creates the `.cancel-requested` signal used by validators to authorize the otherwise-protected state move.
- algorithmic_behavior: Parses only `--force`, `-h`, and `--help`; unknown options exit `3` (`scripts/cancel-rlcr-loop.sh:24-63`). It sources `hooks/lib/loop-common.sh`, resolves the project root, computes `.humanize/rlcr`, and globally selects the newest active loop with `find_active_loop` without session filtering (`scripts/cancel-rlcr-loop.sh:69-92`). It detects the active phase by checking `state.md`, `methodology-analysis-state.md`, then `finalize-state.md`, extracts `current_round` and `max_iterations`, blocks finalize cancellation unless forced, touches `.cancel-requested`, removes pending setup/session markers, removes methodology marker, and moves the active state file to `cancel-state.md` (`scripts/cancel-rlcr-loop.sh:103-167`).
- inputs_outputs_state: Input is optional `--force`; implicit input is current project root from `CLAUDE_PROJECT_DIR` or git. Outputs are status tokens such as `NO_LOOP`, `NO_ACTIVE_LOOP`, `FINALIZE_NEEDS_CONFIRM`, `CANCELLED`, `CANCELLED_METHODOLOGY_ANALYSIS`, or `CANCELLED_FINALIZE` plus human-readable details (`scripts/cancel-rlcr-loop.sh:93-149`, `scripts/cancel-rlcr-loop.sh:172-184`). State transitions are active `state.md` / `methodology-analysis-state.md` / `finalize-state.md` to terminal `cancel-state.md`, with `.cancel-requested` created first (`scripts/cancel-rlcr-loop.sh:156-167`).
- gates_or_invariants: Exit codes are part of the contract: `0` cancelled, `1` no loop, `2` finalize confirmation required, `3` other error (`scripts/cancel-rlcr-loop.sh:11-15`). Finalize phase is protected unless `--force` is supplied (`scripts/cancel-rlcr-loop.sh:139-150`). The script intentionally cancels globally within the project because slash-command invocation has no session id (`scripts/cancel-rlcr-loop.sh:80-90`).
- dependencies_and_callers: Depends on `resolve_project_root` and `find_active_loop` from `hooks/lib/loop-common.sh` (`scripts/cancel-rlcr-loop.sh:69-77`). `find_active_loop` ignores older active loops when the newest project loop is terminal, preventing stale-loop revival (`hooks/lib/loop-common.sh:308-357`). The move is permitted through `is_cancel_authorized`, which requires `.cancel-requested` and validates a narrow `mv active-state cancel-state.md` command shape (`hooks/lib/loop-common.sh:1040-1255`).
- edge_cases_or_failure_modes: No project root exits `3`; loop directory with no active state exits `1`; finalize without `--force` exits `2`; missing round metadata prints `?` defaults (`scripts/cancel-rlcr-loop.sh:73-77`, `scripts/cancel-rlcr-loop.sh:118-133`, `scripts/cancel-rlcr-loop.sh:139-149`). Because `mv` is not atomic with a preexisting destination check, a stale `cancel-state.md` could be overwritten if present in the same loop directory; related validators are expected to mediate normal user commands.
- validation_or_tests: `tests/test-cancel-session.sh`, `tests/test-cancel-signal-file.sh`, `tests/test-state-exit-naming.sh`, and robustness tests cover cancel-state naming, signal creation, session behavior, and authorization. Bash validator allows `cancel-rlcr-loop.sh` during methodology analysis only as the leading command with shell metacharacters rejected (`hooks/loop-bash-validator.sh:81-115`).
