- core_role: `prompt-template/codex/regular-review.md` is the regular Codex review contract for RLCR loop rounds that are not full-alignment checkpoint rounds. It is not executable code, but it is algorithmically important because it defines the reviewer’s required inputs, review gates, goal-alignment checks, output location, and terminal completion token. The stop hook renders this template into `.humanize-loop.local/<timestamp>/round-N-review-prompt.md`, sends it to `codex exec`, and then interprets the resulting review file to decide whether the loop stops or continues.
- algorithmic_behavior: The template first requires Codex to read the original implementation plan via `@{{PLAN_FILE}}` before reviewing anything else, making the plan the primary requirement source (`regular-review.md:3-11`). It injects Claude’s round summary between explicit markers (`regular-review.md:14-18`) and then instructs Codex to compare the claimed completion against the original plan and `@{{PROMPT_FILE}}` (`regular-review.md:11`). The review behavior is deliberately skeptical: Codex must find gaps between “plan-design” and implementation, reject summary-led deferrals, and produce direct implementation plans for deferred or unfinished work (`regular-review.md:20-31`). It then requires a mandatory goal-tracker audit against `@{{GOAL_TRACKER_FILE}}`, checking acceptance criteria progress, forgotten plan items, deferred items, and plan evolution (`regular-review.md:33-45`). The final section defines the control output: write issues to `@{{REVIEW_RESULT_FILE}}` when gaps remain, and only put `COMPLETE` on the last line when every original plan task and acceptance criterion is fully satisfied with no pending or deferred work (`regular-review.md:49-57`).
- inputs_outputs_state: Inputs are all placeholder-driven: `CURRENT_ROUND`, `PLAN_FILE`, `PROMPT_FILE`, `SUMMARY_CONTENT`, `GOAL_TRACKER_FILE`, `DOCS_PATH`, `GOAL_TRACKER_UPDATE_SECTION`, and `REVIEW_RESULT_FILE`. The primary caller passes those values in `hooks/loop-codex-stop-hook.sh:595-607`. `SUMMARY_CONTENT` comes from `round-N-summary.md` (`hooks/loop-codex-stop-hook.sh:530-534`), `GOAL_TRACKER_FILE` is `.humanize-loop.local/<timestamp>/goal-tracker.md` (`hooks/loop-codex-stop-hook.sh:448`), and `REVIEW_RESULT_FILE` is `.humanize-loop.local/<timestamp>/round-N-review-result.md` (`hooks/loop-codex-stop-hook.sh:530-532`). The output is a rendered review prompt, not a direct state mutation. State transitions occur after Codex runs: if the last non-empty trimmed line of the review result is exactly `COMPLETE`, the hook calls `stop_loop "$STATE_FILE" "completed"` and exits (`hooks/loop-codex-stop-hook.sh:814-828`). If not, the hook increments `current_round`, writes the next round prompt from Claude-facing templates, and blocks exit so Claude must address the review (`hooks/loop-codex-stop-hook.sh:861-930`). The template therefore controls the semantic threshold for loop completion while the shell hook performs the mechanical state update.
- gates_or_invariants: The strongest invariant is that summary claims do not override the plan. The template says Claude’s deferrals must be treated as incomplete, and Codex should require all planned tasks to be completed rather than accepting future-phase deferrals (`regular-review.md:24-27`). It also treats honestly admitted pending work as review-blocking and requires a singular, directive implementation plan for that remaining work (`regular-review.md:28-31`). Goal tracking is mandatory, not advisory: Codex must read the goal tracker and report acceptance criteria coverage, forgotten items, deferrals, and plan changes (`regular-review.md:33-45`). Completion has a strict terminal-token gate: `COMPLETE` may be the last line only when all original plan tasks are done, all acceptance criteria are met, and there are no deferrals or pending items (`regular-review.md:52-57`). The hook enforces this with exact last-line matching, avoiding false positives such as longer phrases containing the token (`hooks/loop-codex-stop-hook.sh:814-821`).
