### VCS_CLI_ON_PLAN_FILE-HZ-039 `file` `prompt-template/block/goal-tracker-modification.md`
- cursor: `[_]`
- core_role: This is a blocking prompt template used by RLCR loop validators when an agent tries to modify `goal-tracker.md` after Round 0. It is not executable logic itself; its algorithmic role is to convert a denied file mutation into an actionable recovery contract: write a "Goal Tracker Update Request" into the current summary file instead. The template title and round interpolation are at [prompt-template/block/goal-tracker-modification.md](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/prompt-template/block/goal-tracker-modification.md:1), the hard prohibition is at lines 3-5, and the requested summary section format is at lines 7-23.
- algorithmic_behavior: When rendered, the block tells the worker that after Round 0 only Codex may modify the Goal Tracker, blocks direct Write/Edit/Bash mutation of `goal-tracker.md`, and prescribes a structured update request section in `{{SUMMARY_FILE}}`. The behavioral transition is: attempted direct mutation -> validator denial -> agent records requested tracker changes and justification in summary -> Codex later reviews and, if justified, updates the tracker. This coordinates with Codex review instructions in [prompt-template/codex/goal-tracker-update-section.md](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/prompt-template/codex/goal-tracker-update-section.md:1), which makes Codex responsible for evaluating and applying Claude's requests without modifying immutable goal/acceptance-criteria content.
- inputs_outputs_state: Inputs are two template variables supplied by `goal_tracker_blocked_message`: `CURRENT_ROUND` and `SUMMARY_FILE` ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/lib/loop-common.sh:351)). Output is a rendered Markdown error message emitted to stderr by validators before they exit with a blocking status. It does not mutate state. The state it depends on is the active loop directory and current round read from `.humanize-loop.local/.../state.md`; `get_current_round` defaults to `0` if no value is found ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/lib/loop-common.sh:88)).
- gates_or_invariants: The key invariant is that `goal-tracker.md` becomes read-only to non-Codex actors after Round 0. Write attempts are blocked when the target path is a goal tracker and `CURRENT_ROUND > 0` ([hooks/loop-write-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/loop-write-validator.sh:94)); Edit attempts use the same gate ([hooks/loop-edit-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/loop-edit-validator.sh:81)); Bash file-modification attempts route Round 0 to a different "use Write/Edit" block and later rounds to this template ([hooks/loop-bash-validator.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/loop-bash-validator.sh:166)). Path recognition is suffix-based through `is_goal_tracker_path`, matching paths ending in `goal-tracker.md` ([hooks/lib/loop-common.sh](/Users/wangweiyang/GitHub/humanize_branch_worktrees/vcs-cli-on-plan-file/hooks/lib/loop-common.sh:306)).
