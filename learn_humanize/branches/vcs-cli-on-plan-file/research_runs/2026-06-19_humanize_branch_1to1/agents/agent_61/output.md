### VCS_CLI_ON_PLAN_FILE-HZ-061 `file` `prompt-template/block/wrong-round-number.md`
- cursor: `[_]`
- core_role: This is a blocking prompt template for the RLCR loop hook system. It provides the human-facing rejection message when an agent tries to write or edit a `round-N-summary.md` file whose embedded round number does not match the active loop round. The file itself is only seven lines, but it sits on a validation path that prevents stale or future summary files from being mutated. The template title is declared at `prompt-template/block/wrong-round-number.md:1`, the rejected target is rendered at `prompt-template/block/wrong-round-number.md:3`, the corrected target path is rendered at `prompt-template/block/wrong-round-number.md:5`, and the invariant instruction "Do NOT increment the round number yourself" is at `prompt-template/block/wrong-round-number.md:7`.
- algorithmic_behavior: The template is rendered after a validator has already detected a mismatch between the requested summary filename round and the active `CURRENT_ROUND`. For write operations, `hooks/loop-write-validator.sh:130-147` checks summary files, extracts `CLAUDE_ROUND`, compares it to `CURRENT_ROUND`, renders this template through `load_and_render_safe`, writes the message to stderr, and exits with status `2` to block the tool action. For edit operations, `hooks/loop-edit-validator.sh:91-115` performs the same round mismatch check for summary files under `.humanize-loop.local/`, renders this template, and exits with status `2`. The behavioral transition is therefore: proposed file operation -> filename classified as summary round file -> requested round extracted -> active round loaded from loop state -> mismatch detected -> block message rendered -> operation rejected.
- inputs_outputs_state: Inputs are template variables `ACTION`, `CLAUDE_ROUND`, `FILE_TYPE`, `CURRENT_ROUND`, and `CORRECT_PATH`, all rendered into `{{...}}` placeholders in `prompt-template/block/wrong-round-number.md:3-5`. `ACTION` is supplied as `write to` by `hooks/loop-write-validator.sh:141` and as `edit` by `hooks/loop-edit-validator.sh:109`. `FILE_TYPE` is currently `summary` in both callers (`hooks/loop-write-validator.sh:143`, `hooks/loop-edit-validator.sh:111`). `CURRENT_ROUND` comes from `get_current_round "$ACTIVE_LOOP_DIR/state.md"` in the validators (`hooks/loop-edit-validator.sh:66`; write validator uses the same active-loop/current-round setup before its round check). The current round parser reads frontmatter key `current_round:` and defaults to `0` when absent, as shown in `hooks/lib/loop-common.sh:86-98`. `CLAUDE_ROUND` comes from `extract_round_number`, which lowercases the filename and matches `round-[0-9]+-(summary|prompt|todos).md` in `hooks/lib/loop-common.sh:243-253`. Output is not a data artifact; it is a rendered markdown diagnostic emitted to stderr before the validator returns blocking status `2`. The template does not mutate state; it preserves state by redirecting the actor to `CORRECT_PATH`.
- gates_or_invariants: The main invariant is that summary files are round-scoped and only the active round's summary file may be written or edited. The gate only fires when the extracted round is non-empty and differs from the active round (`hooks/loop-write-validator.sh:133`, `hooks/loop-edit-validator.sh:101`). It intentionally tells the actor not to self-increment the round in `prompt-template/block/wrong-round-number.md:7`; round advancement is owned elsewhere by loop state, not by manual filename choice. The validators also gate by tool type first: the write validator only handles `Write` (`hooks/loop-write-validator.sh:23-28`), and the edit validator only handles `Edit` (`hooks/loop-edit-validator.sh:22-27`). Template rendering uses a single-pass substitution mechanism, documented in `hooks/lib/template-loader.sh:35-43` and implemented in `hooks/lib/template-loader.sh:56-116`, so inserted values containing additional `{{VAR}}` text are not recursively expanded.
- dependencies_and_callers: Direct callers are `hooks/loop-write-validator.sh:140` and `hooks/loop-edit-validator.sh:108`, both using `load_and_render_safe`. The shared dependency chain is `hooks/lib/loop-common.sh`, sourced by both validators (`hooks/loop-write-validator.sh:15-17`, `hooks/loop-edit-validator.sh:14-16`), which provides template directory setup, active loop discovery, current round parsing, path classification, and round extraction. Template loading/rendering is implemented in `hooks/lib/template-loader.sh`: `get_template_dir` resolves `prompt-template` from the hook library location (`hooks/lib/template-loader.sh:9-16`), `load_template` reads the markdown file (`hooks/lib/template-loader.sh:18-33`), and `load_and_render_safe` falls back to an inline message if the template is missing or empty (`hooks/lib/template-loader.sh:152-188`). `tests/test-template-references.sh:173-201` explicitly verifies critical validators use safe template rendering rather than unsafe direct rendering.
- edge_cases_or_failure_modes: If no active loop directory exists, the edit validator exits without blocking before this template is used (`hooks/loop-edit-validator.sh:58-64`); the write validator has the same active-loop phase before summary round validation. If `current_round:` is missing in `state.md`, `get_current_round` returns `0`, which can cause any `round-N-summary.md` where `N` is not `0` to be treated as wrong (`hooks/lib/loop-common.sh:86-98`). If a filename does not match the expected `round-number-type.md` pattern, `extract_round_number` returns empty and the mismatch block is skipped (`hooks/lib/loop-common.sh:251-253`; guarded by `[[ -n "$CLAUDE_ROUND" ]]` in `hooks/loop-write-validator.sh:133` and `hooks/loop-edit-validator.sh:101`). If the template file is missing or renders empty, the callers still block because `load_and_render_safe` renders the caller-provided fallback message (`hooks/lib/template-loader.sh:164-188`). A broader edge is that the template hardcodes the displayed filename shape as `round-{{CLAUDE_ROUND}}-{{FILE_TYPE}}.md`; if future callers reuse it for a path pattern that is not round-file based, the message would be misleading.
