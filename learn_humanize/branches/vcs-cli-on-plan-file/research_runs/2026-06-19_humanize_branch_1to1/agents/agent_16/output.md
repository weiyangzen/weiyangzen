### VCS_CLI_ON_PLAN_FILE-HZ-016 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: `hooks/loop-bash-validator.sh` is the RLCR loop’s `PreToolUse` Bash gate. It is registered for Bash tool invocations in `hooks/hooks.json:42-48` and acts as the shell-command backstop for state-machine invariants that the structured Write/Edit/Read validators cannot enforce once a user tries to use shell redirection or command-line tools. Its stated purpose is to block shell bypasses such as redirection, `tee`, in-place edits, and loop-managed file mutations (`hooks/loop-bash-validator.sh:3-9`).
- algorithmic_behavior: The hook reads the hook JSON from stdin, extracts `.tool_name`, and exits allow-style for non-Bash tools (`hooks/loop-bash-validator.sh:22-27`). For Bash, it extracts `.tool_input.command`, lowercases it, finds the newest active loop under `${CLAUDE_PROJECT_DIR:-$(pwd)}/.humanize-loop.local`, and exits if there is no active `state.md` (`hooks/loop-bash-validator.sh:29-43`). Once active, it loads `current_round` and `state.md`, then evaluates gates in fixed order: git ancestry, git push policy, plan-file staged pre-commit policy, state-file mutation, goal-tracker mutation, prompt-file mutation, summary-file mutation, and todos-file mutation (`hooks/loop-bash-validator.sh:45-207`). The command mutation checks delegate to `command_modifies_file`, whose regex patterns cover output redirection, `tee`, `sed -i`, `awk -i inplace`, `perl -i`, `mv`/`cp` destinations, and `dd of=` (`hooks/lib/loop-common.sh:324-347`).
- inputs_outputs_state: Inputs are hook JSON on stdin, the Bash command string, `CLAUDE_PROJECT_DIR` or the process working directory, `.humanize-loop.local/<timestamp>/state.md`, git repository state, staged files, and prompt templates. State inputs read from `state.md` include `current_round`, `push_every_round`, `commit_plan_file`, `plan_file`, and `start_commit` (`hooks/loop-bash-validator.sh:45-53`, `hooks/loop-bash-validator.sh:85-107`); those fields are created by setup in `scripts/setup-rlcr-loop.sh:326-338`. Outputs are process exit status and stderr block messages: `exit 0` permits the command, while `exit 2` blocks the Bash tool call after rendering a template or fallback (`hooks/loop-bash-validator.sh:74-77`, `hooks/loop-bash-validator.sh:94-95`, `hooks/loop-bash-validator.sh:143-145`). The only direct state transition in this file is the branch-change path: failed ancestry calls `stop_loop "$STATE_FILE" "unexpected"`, which renames `state.md` to `unexpected-state.md` via `hooks/lib/loop-common.sh:31-38`, then emits the branch-changed block message and exits (`hooks/loop-bash-validator.sh:55-78`). All other block paths leave the loop active and only prevent the attempted shell command.
- gates_or_invariants: The hook enforces “no active loop means no restriction” (`hooks/loop-bash-validator.sh:40-43`) and “active loop HEAD must descend from `start_commit`” (`hooks/loop-bash-validator.sh:49-78`), using `git merge-base --is-ancestor` through `check_start_commit_ancestry` (`hooks/lib/loop-common.sh:40-61`). It prevents `git push` unless `push_every_round: true` is set (`hooks/loop-bash-validator.sh:80-97`). It prevents `git commit` when `commit_plan_file` is not true and the configured plan file is staged, by comparing `git diff --cached --name-only` against the repo-relative plan path (`hooks/loop-bash-validator.sh:100-148`). It treats `state.md` as system-owned and always blocks shell modifications (`hooks/loop-bash-validator.sh:150-158`). It blocks `goal-tracker.md` shell modifications in all rounds, but message routing differs: round 0 directs the actor to Write/Edit, while later rounds direct updates through the current round summary (`hooks/loop-bash-validator.sh:160-175`). It blocks shell writes to generated prompt, summary, and todos round files, pushing summary work back to Write/Edit and todos work back to native TodoWrite (`hooks/loop-bash-validator.sh:177-205`).
