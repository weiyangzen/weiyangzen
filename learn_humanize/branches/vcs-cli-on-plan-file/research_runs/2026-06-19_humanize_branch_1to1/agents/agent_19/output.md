### VCS_CLI_ON_PLAN_FILE-HZ-019 `file` `hooks/loop-plan-validator.sh`
- cursor: `[_]`
- core_role: Pre-prompt RLCR plan-file guard. The file is registered as a Claude `UserPromptSubmit` command hook in `hooks/hooks.json:4-10`, so it runs before the user prompt is processed. Its role is to prevent a running RLCR loop from continuing against a stale, moved, deleted, incompatible, or policy-invalid plan file. It belongs to the RLCR state machine boundary around `.humanize-loop.local/<timestamp>/state.md` and `.humanize-loop.local/<timestamp>/plan-backup.md`.
- algorithmic_behavior: The script starts with strict shell mode and sources shared loop helpers from `hooks/lib/loop-common.sh` (`hooks/loop-plan-validator.sh:25-29`). It locates the project root from `CLAUDE_PROJECT_DIR` or `pwd`, derives `.humanize-loop.local`, and selects only the newest active loop through `find_active_loop` (`hooks/loop-plan-validator.sh:35-41`; helper at `hooks/lib/loop-common.sh:67-84`). If no active loop, no state file, or no `plan_file:` value exists, it allows the prompt by exiting `0` (`hooks/loop-plan-validator.sh:39-50`, `hooks/loop-plan-validator.sh:80-87`). For active loops, it parses `start_commit:`, `plan_file:`, and `commit_plan_file:` directly from `state.md` using `grep`/`sed` (`hooks/loop-plan-validator.sh:59-82`). It rejects legacy pre-1.1.2 state files that have `plan_file:` but lack `start_commit:` by calling `stop_loop "$STATE_FILE" "unexpected"`, rendering a block message, writing it to stderr, and exiting `2` (`hooks/loop-plan-validator.sh:59-78`). It determines whether the plan is inside the Git repository by comparing the absolute plan path to `git rev-parse --show-toplevel` through `get_relative_path`; a path whose git-relative form does not start with `../` is treated as inside (`hooks/loop-plan-validator.sh:94-104`). It then runs an ancestry guard: if `HEAD` is not a descendant of `start_commit`, it blocks only when the plan file is in-repo, tracked, `--commit-plan-file` is false, and git history between `start_commit` and `HEAD` contains plan-file commits in either direction (`hooks/loop-plan-validator.sh:107-167`). After ancestry handling, it enforces the four documented cases: commit-plan plus outside repo is a configuration conflict (`hooks/loop-plan-validator.sh:169-198`); commit-plan plus inside repo requires the plan file to be tracked and clean (`hooks/loop-plan-validator.sh:200-251`); all non-conflict cases compare current plan contents to `plan-backup.md` with `diff -q` and block on modification or deletion (`hooks/loop-plan-validator.sh:253-317`).
- inputs_outputs_state: Inputs are environment `CLAUDE_PROJECT_DIR`, current working directory, Git repository metadata, `.humanize-loop.local/<newest>/state.md`, the configured plan file path, and `.humanize-loop.local/<newest>/plan-backup.md`. `state.md` is produced by `scripts/setup-rlcr-loop.sh`, which records `commit_plan_file:`, `plan_file:`, `plan_file_tracked:`, and `start_commit:` after copying the starting plan into `plan-backup.md` (`scripts/setup-rlcr-loop.sh:317-339`). Outputs are process exit status and optional stderr block text. Exit `0` means the prompt may continue; exit `2` means Claude should block prompt processing with the rendered reason (`hooks/loop-plan-validator.sh:76-77`, `hooks/loop-plan-validator.sh:196-197`, `hooks/loop-plan-validator.sh:222-223`, `hooks/loop-plan-validator.sh:248-249`, `hooks/loop-plan-validator.sh:316-317`). Persistent state transition occurs only for unexpected/incompatible loop state: `stop_loop` renames `state.md` to `unexpected-state.md`, preserving the old file while terminating active-loop detection (`hooks/loop-plan-validator.sh:61-64`, `hooks/loop-plan-validator.sh:143-145`; helper at `hooks/lib/loop-common.sh:31-38`). For normal plan-content mismatch, missing plan file, untracked plan file, uncommitted plan file, or outside-repo conflict, the script blocks but does not rename state.
- gates_or_invariants: Main invariant: an active RLCR loop may process a new prompt only when the current plan file still matches the backup captured at loop start, except when validation is impossible because no backup exists (`hooks/loop-plan-validator.sh:258-263`). In commit-plan mode, the plan file must be inside the repository, tracked, and have no `git status --porcelain` changes (`hooks/loop-plan-validator.sh:173-198`, `hooks/loop-plan-validator.sh:204-250`). In non-commit-plan mode, Git dirtiness is intentionally allowed, but the content still must match `plan-backup.md` before a prompt proceeds (`hooks/loop-plan-validator.sh:12-15`, `hooks/loop-plan-validator.sh:253-317`). A legacy state file with `plan_file:` but no `start_commit:` is considered incompatible and terminates the loop (`hooks/loop-plan-validator.sh:53-78`). Diverged branch ancestry is not a universal hard block; it blocks only when tracked in-repo plan-file history differs between `start_commit` and `HEAD` and `--commit-plan-file` is false (`hooks/loop-plan-validator.sh:115-167`). Hook messages are loaded through `load_and_render_safe`, which falls back to inline text if a template is missing (`hooks/lib/template-loader.sh:155-188`).
- dependencies_and_callers: The direct caller is Claude’s hook framework through `hooks/hooks.json:4-10`. Shared dependencies are `hooks/lib/loop-common.sh` for `find_active_loop`, `get_relative_path`, `check_start_commit_ancestry`, and `stop_loop`, plus `hooks/lib/template-loader.sh` for safe block-message rendering. Shell/runtime dependencies are `bash`, `git`, `grep`, `sed`, `diff`, `realpath` or the fallback relative-path implementation, and optionally `grealpath` on macOS (`hooks/lib/loop-common.sh:143-232`). It coordinates with `scripts/setup-rlcr-loop.sh`, which creates the loop directory, copies `plan-backup.md`, and writes the state fields consumed by this hook (`scripts/setup-rlcr-loop.sh:309-339`). It also coordinates with `hooks/loop-codex-stop-hook.sh`: that stop hook explicitly delegates plan-file validation to this pre-prompt hook (`hooks/loop-codex-stop-hook.sh:215-220`) while retaining post-round checks for accidental plan-file commits when `--commit-plan-file` is false (`hooks/loop-codex-stop-hook.sh:301-335`).
- edge_cases_or_failure_modes: If `.humanize-loop.local` is missing, the newest loop has no `state.md`, or `state.md` omits `plan_file:`, the validator silently allows the prompt (`hooks/loop-plan-validator.sh:35-50`, `hooks/loop-plan-validator.sh:80-87`). If `plan-backup.md` is absent, content validation is skipped and the prompt is allowed, which is intentionally fail-open for missing backup (`hooks/loop-plan-validator.sh:258-263`). If the current plan file is missing, the script blocks with status text `missing/deleted` (`hooks/loop-plan-validator.sh:265-289`). If not in a Git repo, `GIT_TOPLEVEL` is empty, `PLAN_FILE_INSIDE_REPO` remains false, and commit-plan mode becomes an outside-repo configuration conflict (`hooks/loop-plan-validator.sh:97-104`, `hooks/loop-plan-validator.sh:173-198`). A notable path edge case is that inside-repo detection only checks `[[ "$PLAN_FILE_REL_GIT" != ../* ]]`; a relative result of exactly `..` is outside but does not match `../*`, so it would be classified as inside. In practice the plan is expected to be a file path and `setup-rlcr-loop.sh` rejects outside paths for commit-plan mode earlier (`scripts/setup-rlcr-loop.sh:251-260`), but the guard’s predicate itself is narrower than a full outside-path test. Another edge case: plan-file paths are parsed from YAML-like state using simple `grep`/`sed`, so paths containing leading/trailing intentional spaces, newlines, or YAML quoting are not robustly decoded (`hooks/loop-plan-validator.sh:81-82`). The script uses `set -euo pipefail`; most optional commands are protected with `|| echo ""` or `|| true`, but failures in sourced helper initialization could still abort the hook before graceful fallback (`hooks/loop-plan-validator.sh:25-29`; `hooks/lib/loop-common.sh:11-20`).
