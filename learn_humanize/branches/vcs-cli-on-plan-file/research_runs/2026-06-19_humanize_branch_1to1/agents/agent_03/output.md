### VCS_CLI_ON_PLAN_FILE-HZ-003 `directory` `hooks`
- cursor: `[_]`
- core_role: `hooks/` is the runtime enforcement and review layer for the RLCR loop. It installs Claude Code hook entrypoints via `hooks/hooks.json:4-63`, then uses shell/Python validators to protect loop-owned files, validate the plan-file contract, block premature exits, and drive the Claude-to-Codex review iteration. This is core algorithm code because it governs whether a loop round may proceed, whether tool use is allowed, and whether the loop advances, completes, stops, or is terminated as unexpected.

- algorithmic_behavior: `hooks/hooks.json:4-13` wires `UserPromptSubmit` to `loop-plan-validator.sh`; `hooks/hooks.json:14-51` wires `PreToolUse` gates for `Write`, `Edit`, `Read`, and `Bash`; `hooks/hooks.json:52-60` wires `Stop` to `loop-codex-stop-hook.sh` with a 7200-second timeout. Shared behavior lives in `hooks/lib/loop-common.sh`, including active-loop discovery by newest `.humanize-loop.local/<timestamp>/` directory with `state.md` (`hooks/lib/loop-common.sh:67-84`), round extraction from YAML frontmatter (`hooks/lib/loop-common.sh:86-98`), round-file pattern matching (`hooks/lib/loop-common.sh:234-252`), state termination by renaming `state.md` to `<prefix>-state.md` (`hooks/lib/loop-common.sh:31-38`), and git ancestry validation with `git merge-base --is-ancestor` (`hooks/lib/loop-common.sh:43-61`). Template rendering is centralized in `hooks/lib/template-loader.sh`: it locates `prompt-template/` (`hooks/lib/template-loader.sh:11-16`), performs single-pass `{{VAR}}` substitution to avoid placeholder injection (`hooks/lib/template-loader.sh:35-117`), and provides safe fallback rendering when templates are absent (`hooks/lib/template-loader.sh:152-188`).

- algorithmic_behavior: The read/write/edit/bash validators enforce loop file protocol before tools run. `loop-read-validator.sh` blocks `round-*-todos.md` reads (`hooks/loop-read-validator.sh:36-39`), then requires prompt/summary reads to target the active loop directory and current round (`hooks/loop-read-validator.sh:56-133`). `loop-write-validator.sh` blocks todos and prompt writes (`hooks/loop-write-validator.sh:37-45`), blocks `state.md` writes (`hooks/loop-write-validator.sh:85-88`), blocks `goal-tracker.md` writes after round 0 (`hooks/loop-write-validator.sh:94-98`), blocks summary writes outside `.humanize-loop.local/` (`hooks/loop-write-validator.sh:104-112`), validates summary round number (`hooks/loop-write-validator.sh:130-148`), and enforces the exact active-loop path (`hooks/loop-write-validator.sh:154-165`). `loop-edit-validator.sh` mirrors edit-time restrictions for todos, prompts, state, post-round-0 goal tracker, and stale summary rounds (`hooks/loop-edit-validator.sh:36-117`). `loop-bash-validator.sh` closes shell bypasses: it blocks branch/ancestry violations and stops the loop as `unexpected` (`hooks/loop-bash-validator.sh:53-78`), blocks `git push` unless `push_every_round: true` (`hooks/loop-bash-validator.sh:85-97`), blocks committing a staged plan file when `commit_plan_file` is false (`hooks/loop-bash-validator.sh:106-148`), and detects redirection/tee/sed/perl/mv/cp/dd edits to protected files through `command_modifies_file` (`hooks/lib/loop-common.sh:324-347`, `hooks/loop-bash-validator.sh:155-205`).

- algorithmic_behavior: `loop-plan-validator.sh` is the pre-prompt plan-file gate. It finds the active loop (`hooks/loop-plan-validator.sh:35-42`), terminates pre-1.1.2 state files lacking `start_commit` as `unexpected` (`hooks/loop-plan-validator.sh:59-78`), reads `plan_file` and `commit_plan_file` from `state.md` (`hooks/loop-plan-validator.sh:80-87`), determines whether the plan file is inside the git repository using git-top-level-relative paths (`hooks/loop-plan-validator.sh:94-104`), and conditionally blocks branch divergence only when the plan file changed between `start_commit` and `HEAD` (`hooks/loop-plan-validator.sh:115-167`). It then enforces the four plan-file modes documented in-file: `--commit-plan-file` outside the repo is a configuration conflict (`hooks/loop-plan-validator.sh:169-198`); `--commit-plan-file` inside the repo requires a tracked and clean plan file (`hooks/loop-plan-validator.sh:200-251`); all non-conflict modes require current plan content to match `.humanize-loop.local/<timestamp>/plan-backup.md` (`hooks/loop-plan-validator.sh:253-317`).

- algorithmic_behavior: `loop-codex-stop-hook.sh` is the round completion state machine. It first exits if no active loop exists (`hooks/loop-codex-stop-hook.sh:52-57`). It blocks exit if the latest native `TodoWrite` state contains incomplete todos by delegating to `check-todos-from-transcript.py` (`hooks/loop-codex-stop-hook.sh:67-95`, `hooks/check-todos-from-transcript.py:17-84`). It blocks changed code/docs files over 2000 lines (`hooks/loop-codex-stop-hook.sh:99-177`). It blocks uncommitted changes, filtering the plan file out when `commit_plan_file` is false (`hooks/loop-codex-stop-hook.sh:222-294`), blocks accidental plan-file commits since `start_commit` (`hooks/loop-codex-stop-hook.sh:296-341`), and blocks unpushed commits only when `push_every_round` is true (`hooks/loop-codex-stop-hook.sh:344-376`). It parses state frontmatter and validates numeric `current_round` (`hooks/loop-codex-stop-hook.sh:379-416`), requires `round-N-summary.md` (`hooks/loop-codex-stop-hook.sh:417-442`), requires round-0 goal tracker initialization by checking placeholder markers (`hooks/loop-codex-stop-hook.sh:444-504`), stops at max iterations with `stopped-state.md` (`hooks/loop-codex-stop-hook.sh:506-516`), builds either regular or every-fifth-round full-alignment Codex review prompts (`hooks/loop-codex-stop-hook.sh:527-608`), runs `codex exec` with configured model/effort/timeout (`hooks/loop-codex-stop-hook.sh:614-704`), handles Codex execution/result-file failures (`hooks/loop-codex-stop-hook.sh:709-809`), and interprets the last non-empty review line as the transition signal: `COMPLETE` renames to `completed-state.md` (`hooks/loop-codex-stop-hook.sh:820-829`), `STOP` renames to `stopped-state.md` (`hooks/loop-codex-stop-hook.sh:831-858`), otherwise it increments `current_round`, emits the next Claude prompt, and blocks exit (`hooks/loop-codex-stop-hook.sh:861-931`).

- inputs_outputs_state: Primary inputs are Claude Code hook JSON on stdin (`tool_name`, `tool_input.file_path`, `tool_input.command`, and `transcript_path`), environment variables such as `CLAUDE_PROJECT_DIR`, `CLAUDE_PLUGIN_ROOT`, `CODEX_TIMEOUT`, git repository state, `.humanize-loop.local/<timestamp>/state.md`, `plan-backup.md`, `goal-tracker.md`, `round-N-summary.md`, and prompt templates under `prompt-template/`. Outputs are either exit status `0` allow, exit status `2` with stderr denial for PreToolUse/UserPromptSubmit hooks, or Stop-hook JSON objects with `"decision": "block"` plus `reason` and `systemMessage` (`hooks/loop-codex-stop-hook.sh:86-94`, `hooks/loop-codex-stop-hook.sh:285-293`, `hooks/loop-codex-stop-hook.sh:921-929`). State transitions are file-based: active loop state is `state.md`; terminal states are renamed to `completed-state.md`, `stopped-state.md`, `unexpected-state.md`, or `cancelled-state.md` via `stop_loop` (`hooks/lib/loop-common.sh:23-38`). Continuing review rounds mutate `current_round` in `state.md`, create `round-N-prompt.md`, and expect Claude to create `round-N-summary.md` (`hooks/loop-codex-stop-hook.sh:865-900`).

- gates_or_invariants: Active loop selection intentionally only considers the newest loop directory, preventing older “zombie” state files from being revived (`hooks/lib/loop-common.sh:63-84`). Todos files are forbidden as round files because the native `TodoWrite` tool is the source of task state (`hooks/lib/loop-common.sh:255-264`; enforced in read/write/edit/bash validators). Prompt files are read-only Claude inputs (`hooks/loop-write-validator.sh:42-45`, `hooks/loop-edit-validator.sh:41-44`, `hooks/loop-bash-validator.sh:177-185`). `state.md` is managed only by loop hooks (`hooks/loop-write-validator.sh:81-88`, `hooks/loop-edit-validator.sh:68-75`, `hooks/loop-bash-validator.sh:150-158`). `goal-tracker.md` mutable initialization is limited to round 0; later changes must be requested through summaries (`hooks/loop-write-validator.sh:90-98`, `hooks/loop-edit-validator.sh:77-85`, `hooks/loop-bash-validator.sh:160-175`). Plan-file immutability is checked against `plan-backup.md` before a prompt is processed (`hooks/loop-plan-validator.sh:253-317`). A plan file may be excluded from commits only when `commit_plan_file` is false; both pre-commit staged checks and post-commit log checks enforce this (`hooks/loop-bash-validator.sh:100-148`, `hooks/loop-codex-stop-hook.sh:296-341`). Codex completion is accepted only when the last non-empty review line is exactly `COMPLETE`; this avoids false positives like “CANNOT COMPLETE” (`hooks/loop-codex-stop-hook.sh:814-821`).

- dependencies_and_callers: `hooks/hooks.json` is the caller map used by Claude Code plugin hook registration. `scripts/setup-rlcr-loop.sh` creates the state consumed by this directory: it backs up the plan file, records `start_commit`, writes `push_every_round`, `commit_plan_file`, `plan_file`, and `plan_file_tracked` into frontmatter (`scripts/setup-rlcr-loop.sh:317-340`), creates `goal-tracker.md` (`scripts/setup-rlcr-loop.sh:343-434`), and seeds `round-0-prompt.md`/`round-0-summary.md` paths (`scripts/setup-rlcr-loop.sh:440-493`). The directory depends on `jq`, `git`, `sed`, `grep`, `awk`, `python3`, and optionally GNU `timeout`/`gtimeout` and `codex`. Template dependencies are under `prompt-template/block`, `prompt-template/claude`, and `prompt-template/codex`; template directory shape is validated by `validate_template_dir` (`hooks/lib/template-loader.sh:190-207`).

- edge_cases_or_failure_modes: Missing template directories do not hard-fail; the common library warns and safe renderers use inline fallback messages (`hooks/lib/loop-common.sh:18-21`, `hooks/lib/template-loader.sh:155-188`). Missing active loop or missing state generally allows normal behavior (`hooks/loop-read-validator.sh:60-62`, `hooks/loop-plan-validator.sh:39-51`, `hooks/loop-codex-stop-hook.sh:52-57`). Pre-1.1.2 state files lacking `start_commit` are treated as incompatible and renamed `unexpected-state.md` (`hooks/loop-plan-validator.sh:53-78`, `hooks/loop-codex-stop-hook.sh:190-213`). `get_relative_path` has GNU/macOS/fallback branches and a basename last resort, so path checks degrade if absolute normalization fails (`hooks/lib/loop-common.sh:143-232`). Bash modification detection is regex based and covers common write forms but may miss complex shell constructs, quoted paths with spaces, generated paths, or indirect scripts (`hooks/lib/loop-common.sh:327-347`). `find_active_loop` uses reverse lexical sort of timestamp directories, so nonstandard directory names could affect active-loop choice (`hooks/lib/loop-common.sh:75-80`).
