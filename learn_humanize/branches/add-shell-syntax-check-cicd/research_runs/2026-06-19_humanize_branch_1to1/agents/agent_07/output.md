- core_role: Defines the hidden slash-command workflow for cancelling an active RLCR loop by deleting loop state files. The command is not the loop algorithm itself, but it is a lifecycle control gate for the plan/RLCR automation: deleting `.humanize-loop.local/*/state.md` makes the stop hook and validators treat the loop as inactive. See `commands/cancel-rlcr-loop.md:1-5` for command metadata and `commands/cancel-rlcr-loop.md:7-24` for the workflow.
- algorithmic_behavior: The behavior is a small state-machine transition:
  - Discover candidate active loops with `ls .humanize-loop.local/*/state.md 2>/dev/null || echo "NO_LOOP"` at `commands/cancel-rlcr-loop.md:11-15`.
  - If no state file is found, emit the fixed response `No active RLCR loop found.` from `commands/cancel-rlcr-loop.md:17`.
  - If one or more state files are found, read the state file to extract round progress, remove matching state files with `rm .humanize-loop.local/*/state.md`, then report `Cancelled RLCR loop (was at round N of M)` from `commands/cancel-rlcr-loop.md:19-22`.
  - It intentionally preserves the containing loop directories and historical artifacts, as stated at `commands/cancel-rlcr-loop.md:24`.
- inputs_outputs_state: Inputs are only filesystem state under `.humanize-loop.local/*/state.md`; there are no command arguments. The expected state format is created by `scripts/setup-rlcr-loop.sh:223-232`, which writes YAML-like frontmatter containing `current_round`, `max_iterations`, Codex configuration, `push_every_round`, `plan_file`, and `started_at`. The command’s user-facing report derives `N of M` from `current_round` and `max_iterations`. Outputs are a human-readable cancellation/no-loop message and deletion of the state file(s). The durable state transition is `active loop present` -> `no active state.md`, while summaries, review results, and goal tracker files remain on disk. Related stop-hook behavior confirms the same state deletion convention: missing `state.md` exits early at `hooks/loop-codex-stop-hook.sh:322-324`; max-iteration, COMPLETE, and STOP terminal states also remove `STATE_FILE` at `hooks/loop-codex-stop-hook.sh:473-476`, `hooks/loop-codex-stop-hook.sh:820-827`, and `hooks/loop-codex-stop-hook.sh:830-857`.

The command is hidden from the slash-command tool listing via `hide-from-slash-command-tool: "true"` at `commands/cancel-rlcr-loop.md:4`. The intended invariant is that `state.md` is the loop liveness token: common hook messaging says `state.md` contains current round, max iterations, and Codex configuration and must not be modified because it is system-managed (`hooks/lib/loop-common.sh:110-123`). The broader loop helper treats only the newest timestamped loop directory as active and ignores older directories even if they contain `state.md` (`hooks/lib/loop-common.sh:11-13`, `hooks/lib/loop-common.sh:23-30`), while this cancel command removes all matching `state.md` files under the loop base.
- dependencies_and_callers: This command is paired with `commands/start-rlcr-loop.md`, whose stop conditions include user cancellation through `/humanize:cancel-rlcr-loop` (`commands/start-rlcr-loop.md:51-55`). The start script creates the state directory and state file under `.humanize-loop.local/<timestamp>/state.md` (`scripts/setup-rlcr-loop.sh:207-214`, `scripts/setup-rlcr-loop.sh:223-232`). The stop hook consumes that file as `STATE_FILE="$LOOP_DIR/state.md"` and exits if it is absent (`hooks/loop-codex-stop-hook.sh:44-57`, `hooks/loop-codex-stop-hook.sh:322-324`). Validators also derive the current active loop and round from the same state file, including bash, edit, write, and read validators referenced at `hooks/loop-bash-validator.sh:37-53`, `hooks/loop-edit-validator.sh:59-66`, `hooks/loop-write-validator.sh:79-86`, and `hooks/loop-read-validator.sh:68-75`. - edge_cases_or_failure_modes: If no loop directory or no matching `state.md` exists, the command returns the no-loop message and performs no deletion (`commands/cancel-rlcr-loop.md:11-17`). There is also a race between discovery and removal: a state file could disappear after `ls` and before `rm`, producing a shell-level failure not handled by the written workflow. Because the glob is relative, running the command outside the intended project root would miss the loop state.
