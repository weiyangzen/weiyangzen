### ADD_SHELL_SYNTAX_CHECK_CICD-HZ-019 `file` `hooks/lib/loop-common.sh`
- cursor: `[_]`
- core_role: Shared Bash library for the RLCR loop hook layer. It centralizes active-loop discovery, current-round extraction, round-file classification, protected-file path predicates, shell-command modification detection, and standard block messages used by the Read/Write/Edit/Bash validators and the Codex Stop hook. The library is not an executable state machine by itself, but it is a core support module for enforcing the loop’s current-round and protected-file invariants.
- algorithmic_behavior: `find_active_loop` selects the lexicographically newest child directory under `.humanize-loop.local` and returns it only if it contains `state.md`, intentionally ignoring older directories to avoid reviving stale loops after abnormal exits (`hooks/lib/loop-common.sh:11-33`). `get_current_round` reads YAML-style frontmatter from `state.md`, extracts `current_round:`, strips spaces, and defaults to `0` in normal empty-value cases (`hooks/lib/loop-common.sh:35-47`). `to_lower`, `is_round_file_type`, and `extract_round_number` normalize paths and recognize `round-N-summary.md`, `round-N-prompt.md`, and `round-N-todos.md` by suffix regex (`hooks/lib/loop-common.sh:49-73`). Message functions emit consistent blocking guidance for todos, prompt writes, state file edits, summary writes via Bash, and goal-tracker restrictions (`hooks/lib/loop-common.sh:75-158`, `hooks/lib/loop-common.sh:203-236`). Path predicates identify `goal-tracker.md`, `state.md`, and any path containing `.humanize-loop.local/` (`hooks/lib/loop-common.sh:160-176`). `command_modifies_file` scans a lowercased shell command for common write patterns such as redirection, `tee`, `sed -i`, `awk -i inplace`, `perl -i`, `mv`/`cp` destination writes, and `dd of=` (`hooks/lib/loop-common.sh:178-201`).
- inputs_outputs_state: Inputs are shell function arguments: loop base directory, state file path, file path strings, filename strings, command strings, and regex file patterns. Outputs are stdout values for lookup/parsing helpers, predicate exit codes, and stderr-ready block text from message helpers. The file itself performs no persistent writes and does not mutate loop state. It reads loop state indirectly via `state.md`, whose structure is created by `scripts/setup-rlcr-loop.sh` with `current_round`, `max_iterations`, Codex settings, `push_every_round`, and `plan_file` frontmatter (`scripts/setup-rlcr-loop.sh:223-234`). State transitions are implemented by callers: the Stop hook removes `state.md` on COMPLETE/STOP or max-iteration termination (`hooks/loop-codex-stop-hook.sh:473-476`, `hooks/loop-codex-stop-hook.sh:819-827`, `hooks/loop-codex-stop-hook.sh:830-857`) and increments `current_round` after review finds issues (`hooks/loop-codex-stop-hook.sh:864-867`).
- gates_or_invariants: The newest session directory is the only eligible active loop; older sessions are ignored even if they still contain `state.md` (`hooks/lib/loop-common.sh:11-14`, `hooks/lib/loop-common.sh:23-32`). Round files are recognized only by canonical `round-<number>-<summary|prompt|todos>.md` suffixes (`hooks/lib/loop-common.sh:54-72`). Todos files are always blocked in validators so users use native TodoWrite instead (`hooks/loop-read-validator.sh:32-39`, `hooks/loop-write-validator.sh:33-45`, `hooks/loop-edit-validator.sh:32-44`, `hooks/loop-bash-validator.sh:117-124`). Prompt files are read-only for Write/Edit/Bash paths (`hooks/loop-write-validator.sh:42-45`, `hooks/loop-edit-validator.sh:41-44`, `hooks/loop-bash-validator.sh:96-104`). `state.md` writes/edits/modification attempts are blocked because loop state is managed by the loop system (`hooks/loop-write-validator.sh:88-95`, `hooks/loop-edit-validator.sh:68-75`, `hooks/loop-bash-validator.sh:69-77`). Goal tracker writes are allowed in Round 0 through Write/Edit but Bash modification is blocked even in Round 0, and after Round 0 direct Write/Edit access is blocked with instructions to request updates through the summary file (`hooks/loop-write-validator.sh:97-105`, `hooks/loop-edit-validator.sh:77-85`, `hooks/loop-bash-validator.sh:79-94`). Summary writes must target the current round and active loop directory; wrong round or wrong directory is blocked (`hooks/loop-write-validator.sh:139-171`, `hooks/loop-edit-validator.sh:87-115`). Reads of prompt/summary files must target the current round in the active directory; wrong location, wrong round, or old session directory is blocked (`hooks/loop-read-validator.sh:94-149`).

- edge_cases_or_failure_modes: `find_active_loop` depends on lexicographic timestamp-style directory names; if the newest directory lacks `state.md`, the function returns no active loop and intentionally does not fall back to older directories (`hooks/lib/loop-common.sh:23-32`). That prevents stale-loop revival but can also hide an older valid loop after a partial newer setup. `get_current_round` is tolerant of a missing/empty value in intent, but because callers run with `set -euo pipefail`, a malformed `state.md` without a matching `current_round:` line can risk early hook failure before the final default echo, depending on shell error behavior around the `grep | sed | tr` assignment (`hooks/lib/loop-common.sh:40-46`; callers set strict mode at `hooks/loop-write-validator.sh:13`, `hooks/loop-bash-validator.sh:12`, `hooks/loop-edit-validator.sh:12`, `hooks/loop-read-validator.sh:12`). The Stop hook has a separate later numeric validation path that removes corrupted state when `current_round` is nonnumeric, but that is after its own parsing path (`hooks/loop-codex-stop-hook.sh:327-348`). `is_in_humanize_loop_dir` is a substring check, not path canonicalization, so symlinks, unusual relative paths, or path text containing `.humanize-loop.local/` can affect classification (`hooks/lib/loop-common.sh:172-176`). `command_modifies_file` is heuristic: it catches common shell write bypasses but not arbitrary writers such as `python -c`, `truncate`, `install`, `rsync`, editor commands, shell functions, or paths with complex quoting/spaces (`hooks/lib/loop-common.sh:185-193`). Conversely, simple regex matching can produce conservative false positives when a protected filename appears in a write-like command string.
