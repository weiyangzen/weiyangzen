### ADD_SHELL_SYNTAX_CHECK_CICD-HZ-011 `file` `hooks/loop-bash-validator.sh`
- cursor: `[_]`
- core_role: `hooks/loop-bash-validator.sh` is the Claude `PreToolUse` validator for Bash commands inside an active RLCR loop. It is registered as the Bash hook in `hooks/hooks.json:32-38`, alongside sibling validators for Write/Edit/Read. Its core role is to prevent Bash from bypassing the higher-level file validators when the loop’s protected state, prompt, summary, todo, or goal-tracker files are involved.
- algorithmic_behavior: The script reads a JSON hook payload from stdin, extracts `.tool_name`, and exits successfully for non-Bash tool calls (`hooks/loop-bash-validator.sh:22-27`). For Bash calls, it extracts `.tool_input.command`, lowercases it via `to_lower`, resolves the project root from `CLAUDE_PROJECT_DIR` or `pwd`, and searches for an active loop under `.humanize-loop.local` (`hooks/loop-bash-validator.sh:29-42`). If no active loop exists, every command is allowed. When a loop is active, it gets the current round from `state.md` and applies ordered gates: optional `git push` blocking, state file modification blocking, goal tracker modification blocking with round-specific messages, prompt file write blocking, summary file Bash modification blocking, and todo file Bash modification blocking (`hooks/loop-bash-validator.sh:45-124`). Successful validation exits `0`; blocked validation prints guidance to stderr and exits `2`.
- inputs_outputs_state: Input is a Claude hook JSON payload on stdin containing `tool_name` and, for Bash, `tool_input.command` (`hooks/loop-bash-validator.sh:22-30`). It reads state from `.humanize-loop.local/<newest-active-loop>/state.md`, specifically `current_round` through `get_current_round` and `push_every_round` through a frontmatter-style grep/sed path (`hooks/loop-bash-validator.sh:45-53`). It does not write files or mutate loop state. Output is either no stdout/stderr with exit `0`, or a blocking diagnostic on stderr with exit `2`. The relevant persistent state is external: the newest loop directory selected by `find_active_loop`, which intentionally only considers the newest timestamped directory with `state.md` to avoid reviving older loops (`hooks/lib/loop-common.sh:11-33`).
- gates_or_invariants: The first invariant is scope: only Bash tool calls are considered, and only while an active loop exists (`hooks/loop-bash-validator.sh:25-42`). The second invariant is local-loop push policy: when `push_every_round` is anything other than literal `true`, a command matching leading `git push` is blocked (`hooks/loop-bash-validator.sh:53-67`). The third invariant is protected file ownership: Bash must not modify `state.md`, `goal-tracker.md`, `round-*-prompt.md`, `round-*-summary.md`, or `round-*-todos.md` (`hooks/loop-bash-validator.sh:74-124`). Goal-tracker behavior is round-sensitive: in round `0`, Bash is redirected to Write/Edit for the active tracker path; after round `0`, the user is told to request changes in the current summary (`hooks/loop-bash-validator.sh:85-93`). The actual shell-command detection is centralized in `command_modifies_file`, which recognizes redirection, `tee`, `sed -i`, `awk -i inplace`, `perl -i`, `mv`/`cp` to a target path, and `dd of=...` patterns (`hooks/lib/loop-common.sh:181-200`).
- dependencies_and_callers: Runtime dependencies are `/bin/bash`, `jq`, `grep`, `sed`, `tr`, `find_active_loop`, `get_current_round`, `to_lower`, `command_modifies_file`, and shared message helpers from `hooks/lib/loop-common.sh` (`hooks/loop-bash-validator.sh:12-16`, `hooks/lib/loop-common.sh:15-235`). The direct caller is the Claude plugin hook system via `hooks/hooks.json:32-38`. It coordinates with sibling validators: Write/Edit validators enforce direct file-tool path rules, while this Bash validator closes shell-based bypasses for the same loop files. README documents this file as part of the lifecycle hook surface under `hooks/` and describes the active state directory files that the validator protects, including `state.md`, `goal-tracker.md`, prompt files, summary files, and review artifacts (`README.md:229`, `README.md:245-252`).
- edge_cases_or_failure_modes: If `jq` is missing or receives invalid JSON, `set -euo pipefail` makes the hook fail rather than silently allowing the command (`hooks/loop-bash-validator.sh:12`, `hooks/loop-bash-validator.sh:22-30`). If `.humanize-loop.local` does not exist, or the newest child lacks `state.md`, no active loop is found and Bash commands are allowed (`hooks/lib/loop-common.sh:18-32`, `hooks/loop-bash-validator.sh:40-42`). `push_every_round` is accepted only as exact `true`; whitespace, capitalization, or YAML quoting may cause push blocking because the comparison is literal (`hooks/loop-bash-validator.sh:53-55`). The modification detector is regex-based over a lowercased command string, so it catches common direct writes but may miss shell-obfuscated writes, quoted/escaped paths with spaces, variable-expanded target paths, multi-step temp-file flows that do not name the protected file in the final modifying primitive, or commands that modify through utilities not listed in `command_modifies_file` (`hooks/lib/loop-common.sh:185-193`). Conversely, because it looks for target filename patterns anywhere in broad command fragments, it may conservatively block some commands that mention a protected path in a write-like context even if a human would consider the invocation harmless.
