- cursor: `[_]`
- core_role: Runtime PR-loop data ingestion script. It fetches all relevant GitHub PR discussion surfaces, normalizes them into one comment stream, optionally filters by timestamp, sorts comments for review priority, groups active bot comments, and writes the Markdown comment snapshot consumed by PR-loop prompt generation.
- algorithmic_behavior: The script parses positional args `<pr_number> <output_file>` plus `--after <timestamp>` and `--bots <bot1,bot2>` (`scripts/fetch-pr-comments.sh:27`). It rejects missing/invalid PR numbers and missing output files (`scripts/fetch-pr-comments.sh:92`, `scripts/fetch-pr-comments.sh:103`). It requires `gh` and `jq` (`scripts/fetch-pr-comments.sh:112`, `scripts/fetch-pr-comments.sh:117`). Repository resolution handles fork PRs by first using the current repo, then falling back to parent/upstream repo to resolve the PR base repository (`scripts/fetch-pr-comments.sh:126`, `scripts/fetch-pr-comments.sh:138`, `scripts/fetch-pr-comments.sh:142`). It fetches issue comments, inline review comments, and PR review summaries through `gh api --paginate`, with three retries and non-fatal per-endpoint failure accounting (`scripts/fetch-pr-comments.sh:182`, `scripts/fetch-pr-comments.sh:211`, `scripts/fetch-pr-comments.sh:215`, `scripts/fetch-pr-comments.sh:219`). Each comment class is transformed into a common JSON object with `type`, `id`, `author`, `author_type`, `created_at`, `updated_at`, `body`, `path`, `line`, and `state` (`scripts/fetch-pr-comments.sh:260`, `scripts/fetch-pr-comments.sh:280`, `scripts/fetch-pr-comments.sh:302`). Empty PR review bodies are preserved with a review-state placeholder (`scripts/fetch-pr-comments.sh:300`, `scripts/fetch-pr-comments.sh:311`).
- inputs_outputs_state: Inputs are GitHub PR number, output Markdown file path, optional ISO-8601 lower bound, optional active bot names, GitHub CLI authentication/context, and repository remotes. Outputs are a Markdown file beginning with PR number, fetch timestamp, and resolved repository (`scripts/fetch-pr-comments.sh:244`), then either `*No comments found.*` or grouped human/bot sections (`scripts/fetch-pr-comments.sh:347`, `scripts/fetch-pr-comments.sh:357`, `scripts/fetch-pr-comments.sh:394`, `scripts/fetch-pr-comments.sh:424`), plus a warning if any endpoint failed (`scripts/fetch-pr-comments.sh:444`). State is transient in `mktemp -d`, deleted by trap (`scripts/fetch-pr-comments.sh:168`). It does not mutate repo state.
- gates_or_invariants: Hard gates: numeric PR number, output file present, `gh` installed, `jq` installed, and resolvable base repository (`scripts/fetch-pr-comments.sh:103`, `scripts/fetch-pr-comments.sh:112`, `scripts/fetch-pr-comments.sh:117`, `scripts/fetch-pr-comments.sh:158`). Soft gate: endpoint fetch failures are logged and represented by empty arrays, then summarized as incomplete output rather than failing the whole script (`scripts/fetch-pr-comments.sh:198`, `scripts/fetch-pr-comments.sh:199`, `scripts/fetch-pr-comments.sh:445`). Deduplication invariant is `unique_by(.id)` over the combined streams (`scripts/fetch-pr-comments.sh:321`), and ordering invariant is human comments before bots, then newest first via `fromdateiso8601` (`scripts/fetch-pr-comments.sh:333`, `scripts/fetch-pr-comments.sh:336`).
- dependencies_and_callers: Depends on `gh`, `jq`, `mktemp`, `sed`, `date`, and GitHub API endpoints under `repos/<owner>/<repo>/issues/<pr>/comments`, `pulls/<pr>/comments`, and `pulls/<pr>/reviews`. `scripts/setup-pr-loop.sh` uses the output sentinel `*No comments found.*` to decide whether Round 0 should wait for automatic reviews or instruct the user to address existing comments (`scripts/setup-pr-loop.sh:643`, `scripts/setup-pr-loop.sh:645`, `scripts/setup-pr-loop.sh:687`). It shares bot-author mapping logic with `hooks/lib/loop-common.sh` where `codex` maps to `chatgpt-codex-connector[bot]` and others to `<bot>[bot]` (`hooks/lib/loop-common.sh:626`, `hooks/lib/loop-common.sh:631`).
- edge_cases_or_failure_modes: Fork PR lookup is explicitly handled, but failure to resolve current or parent repo exits with error (`scripts/fetch-pr-comments.sh:131`, `scripts/fetch-pr-comments.sh:146`, `scripts/fetch-pr-comments.sh:150`). API failures after retries do not fail the process, so downstream prompt generation may operate on incomplete comments while a warning is appended (`scripts/fetch-pr-comments.sh:198`, `scripts/fetch-pr-comments.sh:448`). Deduplication solely by `id` may collide across GitHub object namespaces if issue comments, review comments, and reviews share numeric IDs; the script intentionally uses `unique_by(.id)` without type prefix (`scripts/fetch-pr-comments.sh:323`). Timestamp filtering uses lexical string comparison on `created_at > $after` rather than parsed dates, assuming normalized ISO 8601 timestamps (`scripts/fetch-pr-comments.sh:326`). Null `created_at` entries are dropped before sorting (`scripts/fetch-pr-comments.sh:337`). The `format_timestamp` helper is defined but not used (`scripts/fetch-pr-comments.sh:237`).
- validation_or_tests: PR-loop hook tests validate related comment sorting, deduplication, PR review placeholder behavior, and bot filtering logic. Examples include `fromdateiso8601` sorting (`tests/test-pr-loop-hooks.sh:224`), human-before-bot sort logic (`tests/test-pr-loop-hooks.sh:245`), `unique_by` deduplication (`tests/test-pr-loop-hooks.sh:273`), placeholder review bodies (`tests/test-pr-loop-hooks.sh:1169`), and bot response filtering (`tests/test-pr-loop-hooks.sh:1206`). `tests/mocks/gh` provides mocked `gh` JSON behavior for PR-loop tests (`tests/mocks/gh:42`, `tests/mocks/gh:135`).
