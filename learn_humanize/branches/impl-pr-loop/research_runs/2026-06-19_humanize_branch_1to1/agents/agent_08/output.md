### IMPL_PR_LOOP-HZ-038 `file` `scripts/portable-timeout.sh`
- cursor: `[_]`
- core_role: Cross-platform runtime abstraction for bounded command execution. It normalizes timeout behavior for macOS/Linux and underpins git, GitHub CLI, and Codex invocations throughout RLCR and PR-loop workflows.
- algorithmic_behavior: `detect_timeout_impl` selects the first available backend in this order: Homebrew GNU `gtimeout`, GNU `timeout`, `python3`, `python`, then `none` (`scripts/portable-timeout.sh:9-27`). `TIMEOUT_IMPL` is computed once at source time (`scripts/portable-timeout.sh:29`). `run_with_timeout <seconds> <command> [args...]` dispatches based on that implementation (`scripts/portable-timeout.sh:33-71`).
- inputs_outputs_state: Inputs are timeout seconds and argv array (`scripts/portable-timeout.sh:31-36`). Output is the wrapped command’s stdout/stderr and exit status. For Python fallback, `subprocess.run(..., timeout=timeout_secs)` returns the child exit code; timeout exits `124`, matching GNU timeout (`scripts/portable-timeout.sh:47-62`). The script exports `TIMEOUT_IMPL` for sourcing scripts (`scripts/portable-timeout.sh:73-76`).
- gates_or_invariants: GNU `timeout` is accepted only when `timeout --version` succeeds, avoiding non-GNU or incompatible implementations (`scripts/portable-timeout.sh:13-19`). Python fallback preserves argv boundaries by passing the command as `sys.argv[1:]` rather than shell-evaluating (`scripts/portable-timeout.sh:49-61`). If no timeout backend exists, the invariant weakens deliberately: command runs without a timeout and emits a warning to stderr (`scripts/portable-timeout.sh:64-68`).
- dependencies_and_callers: Direct callers include `scripts/setup-rlcr-loop.sh`, which sources the wrapper (`scripts/setup-rlcr-loop.sh:26-28`) and uses it for git repository checks and branch capture (`scripts/setup-rlcr-loop.sh:242-250`, `scripts/setup-rlcr-loop.sh:483-487`); `scripts/setup-pr-loop.sh`, which uses it for git/gh PR discovery and trigger comments (`scripts/setup-pr-loop.sh:33-35`, `scripts/setup-pr-loop.sh:243-250`, `scripts/setup-pr-loop.sh:295-354`, `scripts/setup-pr-loop.sh:446-455`); `hooks/loop-codex-stop-hook.sh`, which wraps `codex exec` with the configured Codex timeout (`hooks/loop-codex-stop-hook.sh:798-845`); and `hooks/pr-loop-stop-hook.sh`, which uses it in push-state and GitHub checks (`hooks/pr-loop-stop-hook.sh:318-334`).
- edge_cases_or_failure_modes: If the selected command does not exist, the underlying backend determines the exit code. Python fallback catches generic exceptions and exits `1` (`scripts/portable-timeout.sh:58-60`). A zero-second timeout is backend-specific. The no-backend path can hang indefinitely because it intentionally runs without timeout (`scripts/portable-timeout.sh:64-68`).
- validation_or_tests: Related executable specs are `tests/robustness/test-timeout-robustness.sh` call sites found for success, sleep timeout, argument preservation, command exit-code propagation, large output, nonexistent command, empty command, special characters, signal handling, and subshell execution. Assigned file `tests/test-templates-comprehensive.sh` does not test this script directly.

- cursor: `[_]`
- core_role: Executable robustness specification for active RLCR session discovery. It pins the behavior of `find_active_loop`, especially the “newest directory only” invariant that prevents stale or finished sessions from being revived.

The implementation under test checks whether the base dir exists, chooses the lexicographically newest immediate child directory using `ls -1d ... | sort -r | head -1`, and returns it only if it contains `state.md` or `finalize-state.md` (`hooks/lib/loop-common.sh:147-170`).
- inputs_outputs_state: Inputs are a loop base directory and its immediate child directories. Output is either the active session path or empty string. - gates_or_invariants: Positive gates require newest session with `state.md` to be returned (`tests/robustness/test-session-robustness.sh:33-46`), `finalize-state.md` to count as active (`tests/robustness/test-session-robustness.sh:48-61`), many sessions to resolve quickly under 1000 ms (`tests/robustness/test-session-robustness.sh:63-87`), and sessions with both active markers to be accepted (`tests/robustness/test-session-robustness.sh:105-118`).
